# 索引与分类系统深度整理方案（2026-06-06）

## 结论先行

> **后续执行状态更新（2026-06-06）**：本轮整理已补齐 `CITATION_TRACKING.md`、`cross-reference-map.md`、`graph.json`、`file-colors.css`，并完成 023／057 的重命名区分与总索引导航增强。

当前这套笔记库**不是内容少，而是总控层失配**。

核心问题不在“有没有索引”，而在于：

1. **总索引、映射表、分类方案三套系统彼此没有真正闭环**。
2. **设计规格明显超前于实际落地**。
3. **存在重复条目、缺失依赖文件、元数据不一致**，会让后续维护成本越来越高。

所以整理优先级应该是：

1. **先修总控层一致性**
2. **再补缺失依赖**
3. **最后再做大规模美化和批量标签化**

---

## 一、现状诊断

### 1．总索引体量大，但定位混杂

`000-总索引-Index.md` 既承担：

- 全书目目录
- 场景速查
- 个人化应用说明
- 概念导航

这会导致它从“索引”膨胀成“总控手册”。

问题不是不能大，而是**索引职责不纯**：

- 目录型信息应该稳定
- 场景型信息应该可迭代
- 个性化解释应该尽量独立

否则后面一改场景结构，整个总索引都会被牵连。

### 2．映射表质量高，但引用了不存在的依赖文件

`问题拆解-书籍映射-Question-Decomposition-Map.md` 的结构其实很强，已经接近“问答路由中枢”。

但它明确引用了：

- `CITATION_TRACKING.md`
- `000-总索引-Index.md`

其中 `CITATION_TRACKING.md` 当前并不存在。

这意味着这张映射表的“维护规则”是**设计成立、实现缺位**。

### 3．分类系统方案写得完整，但仍停留在规划稿阶段

`Obsidian分类系统-优化方案-v2.md` 提供了：

- graph.json 配置方案
- CSS snippet 方案
- frontmatter 标准化方案
- 三阶段实施路线

但当前仓库里没有找到：

- `CITATION_TRACKING.md`
- `cross-reference-map.md`
- `graph.json`
- `file-colors.css`

也就是说，这份文件目前是**架构说明书**，不是已执行系统。

### 4．重复书目已经出现，说明主数据源没有统一

在本轮整理前，至少存在两组高风险重名冲突：

- `011-认知觉醒-Cognitive-Awakening.md`
- `023-认知觉醒-速查版-Cognitive-Awakening-Quick-Reference.md`

以及：

- `028-我们如何学习-How-We-Learn.md`
- `057-我们如何学习-Stanislas-Dehaene.md`

这不一定是错误，因为可能是同书异版，或者同名异书。

但**命名层面高度相似，极易污染索引、双链、搜索和图谱识别**。

尤其是：

- 011 与 023 原本共用同名框架
- 028 与 057 都叫《我们如何学习》，但作者不同
- 011 与 023 的中文名、英文别名都一样
- 028 与 057 的中文名一样，英文也一样，但作者不同

这说明你现在缺一个“主数据规则”：

- 书名冲突如何区分
- alias 如何写
- 索引如何标注“同名异书／同主题异版本”

### 5．Frontmatter 存在风格不统一问题

从已检索内容看，不同文件里出现了：

- `domain: "D2"`
- `tags: [...]`
- `category:`
- `status: "已分析"`
- `citation_count:`（在方案里要求，但未见普遍落地）

说明当前元数据体系至少有两套：

- 早期编号域模型
- 后期语义标签模型

如果不统一，后面做 Dataview、图谱分组、引用频率统计都会越来越乱。

## 二、整理目标

这次整理建议只做“总控层修复”，不碰 62 本正文内容。

目标分三层：

### A．建立一个真正可维护的总控层

包括：

- 总索引负责“总览与导航”
- 映射表负责“问题→书籍路由”
- 分类方案负责“实施规范”
- 引用追踪表负责“热度控制与均衡引用”

### B．消除结构性歧义

包括：

- 给重复命名书籍加区分说明
- 明确缺失依赖文件
- 统一总控文件之间的互相引用

### C．让后续批量整理可自动化

包括：

- 固化 frontmatter 规范
- 补齐图谱配置与 CSS 文件
- 为后续 Dataview 与脚本化维护打基础

## 三、建议的执行顺序

### Phase 1．先修“总控层缺口”

优先级最高。

建议立即补 3 个文件：

1. `CITATION_TRACKING.md`
2. `cross-reference-map.md`
3. `file-colors.css`

原因：

- 前两个是现有方案中被多次引用的缺失依赖
- 第三个是低成本高感知收益

### Phase 2．给总索引增加“系统导航区”

在 `000-总索引-Index.md` 头部加入：

- 系统总览
- 相关总控文件入口
- 同名书区分说明
- 使用顺序说明

这样总索引才真正成为入口，而不只是长列表。

### Phase 3．统一重复命名规则

建议新增一套最小规则：

- 同名不同作者：标题中保留作者识别
- alias 不再只写英文名，增加“作者限定别名”
- 在总索引里增加“同名书辨析”小节

例如：

- `认知觉醒（周岭）`
- `认知觉醒（精简版／速查版）` 或者重新命名 023 的定位
- `我们如何学习（Benedict Carey）`
- `我们如何学习（Stanislas Dehaene）`

### Phase 4．最后才做 frontmatter 批量标准化

因为只要主数据规则没定，批量标签就是把混乱规模化。

## 五、这次整理里最值得警惕的点

### 1．不要现在就急着给 62 本书批量打标签

因为你现在的分类系统定义还没完全收敛。

如果现在硬打：

- 后面领域边界一变，要全部返工
- 重复命名问题会污染统计
- citation/high、mid、low 没有真实数据支撑

### 2．不要把所有东西继续塞进总索引

总索引已经很重了。

再继续加，会从“导航”变成“知识垃圾桶”。

正确做法是：

- 总索引只做导航
- 细分功能拆到专用总控文件

### 3．不要忽略“缺失依赖”这种小问题

它看起来只是少几个文件，实际会造成两个后果：

- 维护规则失真
- 你以后看到方案时会误以为系统已经存在

这类问题最容易积累成知识库的“假组织化”。
