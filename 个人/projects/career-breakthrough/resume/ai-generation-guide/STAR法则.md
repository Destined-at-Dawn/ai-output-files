# STAR 法则实操指南

> 用STAR写项目经历，不是背公式，而是讲故事。

---

## STAR是什么

- **S**ituation（背景）— 什么情况下做的？
- **T**ask（任务）— 要解决什么问题？
- **A**ction（行动）— 你具体做了什么？
- **R**esult（结果）— 产出了什么？量化！

---

## 在简历中怎么用

简历空间有限，不需要把S-T-A-R四个标签都写出来。**关键是按这个逻辑组织语言：**

### ❌ 反面案例

```
参与了国家级竞赛的FPGA项目，负责模块设计和验证，使用Vivado和ModelSim工具。
```

问题：没有具体动作、没有量化、没有结果。

### ✅ 正面案例

```
作为竞赛项目负责人，独立设计通信接口模块RTL代码，
自写Testbench覆盖正常/异常/边界三类场景，仿真100%通过。
```

拆解：
- S: 竞赛项目（背景隐含）
- T: 设计通信接口模块（任务）
- A: 手写RTL + 自写Testbench（具体行动）
- R: 标准速率 + 100%通过率（量化结果）

## 更多对比

### 例1：ROS项目

**❌ 差：**
"负责ROS系统的集成和调试工作。"

**✅ 好：**
"从0配置move_base导航框架（全局/局部代价地图+TF坐标变换），调参base_local_planner实现动态避障，通过逐文件比对发现3处代码-规则不一致。"

### 例2：软件项目

**❌ 差：**
"开发了一个Web应用，使用了前后端技术。"

**✅ 好：**
"独立设计并开发校园二手交易平台（Flask+Vue3），数据库索引优化将搜索响应从800ms降至120ms，Docker部署后99.5%在线率，内测50+用户。"

### 例3：硬件调试

**❌ 差：**
"参与硬件测试和调试。"

**✅ 好：**
"编写Python自动化测试脚本覆盖5种外设接口，将单轮测试时间从2小时降至20分钟，输出测试报告推动3个硬件问题修复。"

## 万能句式模板

```
[角色] + [具体动作] + [技术/工具] + [量化结果]

示例：
- 独立设计[模块名]的[实现方式]，使用[工具]验证，达到[量化指标]
- 优化[指标]从[旧值]到[新值]，通过[具体方法]
- 从0搭建[系统/框架]，支持[规模/范围]，[稳定性/性能指标]
- 编写[文档/脚本]覆盖[范围]，[效率提升/问题减少]
```

---

## 常见错误

1. **只写"参与"不写"做了什么"** → 用"独立完成/主导/设计/实现"替代
2. **只写技术名词不写具体动作** → "使用Vivado"不如"用Vivado综合后资源占用≤500 LUT"
3. **没有量化结果** → 尽量用数字。没数字时用"从0到1""100%""首次"等程度词
4. **一整段话不分行** → 每条一个要点，方便快速扫读

## 自查清单

写完每段经历后，逐项检查：

- [ ] 有没有明确的动词开头？（设计/实现/优化/搭建/独立完成）
- [ ] 有没有量化结果？（数字/百分比/性能指标）
- [ ] 技术栈是否在描述中自然出现？（不单独堆砌）
- [ ] 面试时这个点你接得住追问吗？
- [ ] 去掉形容词后，这段话还有信息量吗？
