# R16 根目录优先约束架构

> R15 的升级版。R15 解决了"proj-ID vs 主目录"的问题，R16 进一步解决"约束、记忆、产出不能重复，必须分层读取"的问题。
> 来源：2026-05-25 小黎对五大工作区统一治理的要求。

---

## 核心原则

**根目录（中文主目录）是唯一权威。项目文件夹只做补充，不重复。**

---

## 一、目录职责划分

### 根目录（中文主目录）放什么

| 类型 | 路径 | 说明 |
|------|------|------|
| 全局约束 | `CLAUDE.md` | 每次对话必读，不可跳过 |
| 工程法则与约束 | `.claude/rules/` | Claude Code 自动加载，不需手动读取 |
| SOP 入口 | `SOPs/` | 标准流程文档 |
| 技能调用规则 | `skill-routing-rules.md` | 哪个场景调用哪个 skill |
| 统一索引 | `index.md` | 本工作区所有重要文件的索引 |
| 长期记忆 | `memory/long-term.md` | 跨对话持久记忆 |
| 每日记忆 | `memory/{YYYY-MM-DD}.md` | 按天记录 |
| 最终产出 | `outputs/` | 已完成的交付物 |

### 项目文件夹（proj-ID）放什么

| 类型 | 路径 | 说明 |
|------|------|------|
| 项目增量约束 | `.claude/rules/` | 仅当项目有特殊规则时，写入 `.claude/rules/` |
| 中间产物 | `outputs/` | 未完成的草稿、临时计算结果 |
| 会话草稿 | `drafts/` | 本轮对话的临时文件 |
| `.session-meta.json` | 根目录 | 指向主目录的元数据 |

### 项目文件夹禁止放什么

- ❌ 不放全局约束（已在根目录）
- ❌ 不放 SOP（已在根目录）
- ❌ 不放记忆文件（已在根目录）
- ❌ 不放技能调用规则（已在根目录）
- ❌ 不放统一索引（已在根目录）

---

## 二、读取顺序（每次对话必须执行）

```
Step 1: 主目录/CLAUDE.md              ← 全局约束（必读）
Step 2: 主目录/memory/long-term.md     ← 长期记忆（必读）
Step 3: 主目录/memory/{今天日期}.md    ← 今日记忆（必读）
Step 4: .claude/rules/                 ← 工程法则（Claude Code 自动加载）
Step 5: 主目录/SOPs/                   ← SOP 入口（按需）
Step 6: 主目录/skill-routing-rules.md  ← 技能路由（按需）
Step 7: 项目目录/.claude/rules/       ← 项目增量约束（如有）
Step 8: 项目目录/outputs/             ← 中间产物（按需）
```

**禁止跳过 Step 1-4。** Step 5-8 按场景读取。

## 三、写入规则

### 写入根目录的场景

| 场景 | 写入路径 | 说明 |
|------|---------|------|
| 用户纠正偏好 | `memory/long-term.md` | 追加，不覆写 |
| 当日进展记录 | `memory/{日期}.md` | 追加或创建 |
| 完成的交付物 | `outputs/` | 最终版本 |
| 新发现的规则 | `.claude/rules/` | 验证后写入 |
| 新 SOP | `SOPs/` | 验证后写入 |

### 写入项目目录的场景

| 场景 | 写入路径 | 说明 |
|------|---------|------|
| 项目特有的约束 | `.claude/rules/` | 仅项目目录下的 `.claude/rules/`，标注"仅本项目适用" |
| 中间计算结果 | `outputs/` | 会话结束后决定是否提升到根目录 |
| 临时调试文件 | `drafts/` | 会话结束后清理 |

### 写入前检查清单

1. 根目录是否已有同类文件？→ 有则不重复创建
2. 这个内容是否所有对话都需要？→ 是则写根目录
3. 这个内容是否只是本项目临时需要？→ 是则写项目目录
4. 这个内容会话结束后是否还有价值？→ 是则写根目录

## 四、会话生命周期

### 会话启动

1. 读取 `proj-ID/.session-meta.json` → 获取主目录路径
2. 按"读取顺序"Step 1-6 加载主目录内容
3. 检查 `proj-ID/.claude/rules/` 是否有增量约束
4. 检查 `proj-ID/outputs/` 是否有上次遗留的中间产物

### 会话中

- 临时文件 → `proj-ID/drafts/`
- 中间产物 → `proj-ID/outputs/`
- 记忆更新 → 主目录 `memory/`
- 规则更新 → 主目录 `.claude/rules/`

### 会话结束

1. 扫描 `proj-ID/outputs/` → 有价值的提升到主目录 `outputs/`
2. 扫描 `proj-ID/drafts/` → 有价值的提升，其余删除
3. 确认主目录 `memory/` 已更新
4. proj-ID 目录 → 备份后移到 `归档/`

## 六、索引机制

### 主目录索引（index.md）

每个主目录必须维护一个 `index.md`，列出：
- 所有 `.claude/rules/` 下的文件及其适用范围
- 所有 `SOPs/` 下的文件及其触发条件
- 所有 `outputs/` 下的文件及其状态
- 所有 proj-ID 关联及其当前状态

### proj-ID manifest（.session-meta.json）

```json
{
  "mainDirectory": "E:\\ai产出文件\\牛马\\mutual\\mutual\\20260518-生态优化",
  "created": "2026-05-25T10:00:00",
  "purpose": "本次会话的具体目的",
  "incrementalRules": [],
  "status": "active"
}
```

## 七、反模式

| 反模式 | 为什么错 | 正确做法 |
|--------|---------|---------|
| 在 proj-ID 创建自己的 CLAUDE.md | 与根目录重复，且不会被读取 | 只在根目录维护 |
| 在 proj-ID 复制一份 SOP | 修改根目录时 proj-ID 的不会同步 | 只读根目录的 SOP |
| 在 proj-ID 存长期记忆 | 下次对话读不到 | 存根目录 memory/ |
| 根目录和项目目录放相同规则 | 维护两份 = 两份都可能过时 | 根目录唯一权威 |
| 会话结束不清理 proj-ID | 碎片化积累 | 提升有价值的，清理其余 |
