# R09 — 教训→skill/SOP 反馈闭环（真源 v1.0）

> 核心原则：**教训只记不改 = 白记。** 每条教训必须流向它能影响的 skill 或 SOP，否则这条教训会在下一次对话中被遗忘。

---

## 一、反馈闭环总图

```
用户批评/踩坑/表扬
    ↓
自我进化/做得差的避免/ 或 做得好的鼓励/
    ↓ 同时写入
lessons-learned.md（结构化记录）
    ↓ AI 自动分析
┌─────────────────────────────────────────┐
│ 这条教训影响哪个 SOP？哪个 skill？       │
│ → 查「教训→SOP/skill 映射表」            │
└─────────────────────────────────────────┘
    ↓ 影响明确
SOP 迭代日志（追加一行：日期/触发/原流程/问题/更新/版本号）
    ↓ 如果影响 skill 的行为逻辑
CLAUDE.md 对应规则段落（Edit 更新）
    ↓ 3 次同类教训
SOP 核心规则升级（从「避坑指南」升级为「铁律」）
```

---

## 二、反馈闭环铁律（5 条）

### 铁律 F1：教训产生即流向

**每当自我进化文件新增一条教训，AI 必须在同一次对话内完成以下操作：**

1. **定位**：查映射表，确定这条教训影响哪个 SOP + 哪个 skill
2. **回写**：在对应 SOP 的「迭代日志」追加一行
3. **升级判断**：如果这是第 3 次同类教训 → 将教训升级为 SOP 的「铁律/核心规则」
4. **通知**：如果影响 skill 的行为逻辑 → 更新 CLAUDE.md 中对应的规则段落

### 铁律 F2：SOP 迭代日志格式（统一）

每个 SOP 文件末尾必须包含：

```markdown
## 迭代日志

| 日期 | 触发事件 | 原流程 | 发现的问题 | 更新内容 | 版本 |
|------|---------|--------|-----------|---------|------|
| YYYY-MM-DD | 用户做了X / 发现了Y | 步骤A→B→C | 步骤B多余/缺失 | 删除/新增步骤X | v1.1 |
```

### 铁律 F3：3 次同类 → 升级为铁律

| 同类教训次数 | 动作 |
|-------------|------|
| 第 1 次 | 记录到自我进化 + lessons-learned + SOP 迭代日志 |
| 第 2 次 | 在 SOP 对应步骤旁加 **⚠️ 高频坑** 标记 |
| 第 3 次 | 升级为 SOP **铁律**（不可跳过），同时更新 CLAUDE.md 对应规则段落 |
| 第 5 次 | 评估是否需要新建独立 SOP 或将规则提升到共享规则 |

### 铁律 F4：成功经验也回流

**不只是失败教训。成功的经验也要回流到 SOP 的「最佳实践」段落。**

| 经验类型 | 回流目标 |
|---------|---------|
| 发现更高效的步骤 | SOP 对应步骤旁加 **💡 最佳实践** |
| 发现更好的 skill 选型 | SOP 决策表更新 skill 列 |
| 发现更安全的操作方式 | SOP 检查清单新增条目 |

### 铁律 F5：映射表维护

「教训→SOP/skill 映射表」是活文件。当出现以下情况时必须更新映射表：

| 情况 | 动作 |
|------|------|
| 新建了 SOP | 映射表新增对应行 |
| 教训找不到对应 SOP | 标记为「待建SOP」，评估是否需要新建 |
| 一个教训影响多个 SOP | 映射表记录所有受影响的 SOP |

## 四、自动触发条件（AI 行为规则）

### 4.1 何时触发反馈闭环

| 触发信号 | AI 必须做的事 |
|---------|-------------|
| 自我进化/做得差的避免/ 新增文件 | 查映射表 → 回写 SOP 迭代日志 |
| 自我进化/做得好的鼓励/ 新增文件 | 查映射表 → 回写 SOP 最佳实践 |
| lessons-learned.md 新增 ❌ 条目 | 同上 |
| lessons-learned.md 新增 ✅ 条目 | 同上 |
| 用户直接说"优化到XX skill里" | 立即回写对应 SOP + skill |
| 用户说"教训记下来了" | 检查是否已回流到 SOP，未回流则补上 |

### 4.2 回流操作流程

```
Step 1: 确定教训内容（从自我进化文件或 lessons-learned 中提取）
Step 2: 查映射表，找到受影响的 SOP
Step 3: Read 受影响的 SOP 文件
Step 4: 在 SOP 的「迭代日志」追加一行
Step 5: 判断是否需要更新 SOP 正文：
  - 第 1 次 → 只记日志
  - 第 2 次 → 在对应步骤旁加 ⚠️ 标记
  - 第 3 次 → 升级为铁律，Edit 更新 SOP 正文
Step 6: 如果教训影响 skill 行为 → Edit 更新 CLAUDE.md 对应段落
Step 7: Read 验证所有修改的文件
Step 8: 更新 MEMORY.md 的 PK 速查表（如适用）
```

### 4.3 禁止事项

- **禁止只记教训不回流**：教训记录完成 ≠ 闭环完成，必须回写 SOP
- **禁止跳过映射表**：每条教训必须查映射表，不能凭直觉判断影响范围
- **禁止覆盖式更新 SOP**：SOP 迭代日志只能追加，正文修改必须通过 Edit 精确替换
- **禁止忽略成功经验**：成功经验也要回流到 SOP 最佳实践，不只回流失败

## 七、认知科学支撑（百大认知书籍）

### 反馈闭环的认知原理

| 认知机制 | 来源 | 在反馈闭环中的应用 |
|---------|------|-------------------|
| **操作性条件反射** | 016-习惯的力量 §概念1 | 反馈闭环=操作性条件反射的工程化——"教训产生→回写SOP→下次避免"=行为(教训)→后果(系统改进)→行为频率变化(教训减少)。没有闭环→没有后果→行为不变 |
| **测试效应** | 009-认知天性 §概念2 | "教训只记不改=白记"的科学依据：仅记录教训=被动编码→遗忘率80%。回写SOP=主动提取+重新组织→测试效应→记忆巩固率提升至80% |
| **间隔重复** | 009-认知天性 §概念3 | "3次同类→升级为铁律"=间隔重复的变体——同一教训在不同时间点被激活→每次激活都在巩固记忆→3次后从"知道"升级为"内化" |
| **图式更新** | 025-认知负荷理论 §概念4 | SOP是"工作流程图式"。教训=图式中的"错误节点"。回写SOP=更新图式→下次执行时自动绕过错误→减少认知负荷 |
| **元认知监控** | 023-认知觉醒 §概念3 | 反馈闭环的本质是"系统级元认知"——不仅执行任务，还监控"执行质量"并持续改进。没有元认知→只能重复；有元认知→可以进化 |
| **损失厌恶利用** | 010-思考快与慢 §概念4 | "禁止只记教训不回流"利用了损失厌恶——如果教训已经记录但没有回流=已投入的认知资源"浪费"→驱动回流行为 |
