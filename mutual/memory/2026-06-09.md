## Agent 故障诊断与修复（mimo 用户不切模型方案）

### 故障根因（5 层）
1. **模型继承**：Agent 继承主会话 mimo-v2.5-pro，不支持 multimodal（PDF/图片）
2. **Prompt 泛化**："搜索 FPGA 文件" → Agent 不知目标/格式/停止条件
3. **Agent 类型用错**：Explore 读不全文件、general-purpose 太慢、Plan 不能搜
4. **无故障恢复**：Agent 挂了就挂了
5. **上下文断裂**：Agent 看不到主对话历史

### 修复方案（不切主模型，保留 mimo 免费额度）
1. **Agent Prompt 铁律**：新增 `.claude/rules/agent-prompt-ironclad.md`
   - 每次 Agent 调用必须含三要素：具体目标 + 输出格式 + 停止条件
   - 禁止泛化 prompt（"搜索一下"→禁止）
   - Agent 类型匹配表（Explore/Plan/general-purpose/code-reviewer 各司其职）
   - 模基混合策略：纯文本继承 mimo，multimodal 任务显式 `model: "opus"`
   - Agent 故障三问排查流程
2. **subagent-strategy.md v2.0**：增加模型混合策略 + 故障自动恢复 + 决策树

### 关键决策
- 用户指定不使用 Anthropic 渠道商（保留 mimo 免费额度）
- Agent 的 `model` 参数可单独覆盖，不改变主会话模型
- multimodal Agent 调用时加 `model: "opus"` 即可

## skill-routing-table 关键词冲突治理

**问题**：30 个泛化短触发词（如"分析""文档""学习""视频"）作为独立触发词注册在某 skill 上，同时是其他 skill 的复合触发词的子串，导致 184 个子串冲突，路由歧义。

**根因**：触发词粒度不一致——有些用泛词（"分析"），有些用精确词（"分析图片内容"），泛词短路了精确匹配。

**执行的修复**：
1. 删除 30 个毒触发词，替换为更精确的复合词
2. 清除 8 个同路由内重复词
3. 修复 1 个直接冲突（"记忆检索"同时路由 li-manage 和 li-memory）
4. 删除 r079（thinking-coach 不应路由简历，r100 resume-modification 已覆盖）
5. 删除 r010（URL 触发词应由上下文判断，不应用关键词匹配）

**量化结果**：
- 路由数：111 → 105（-6）
- 子串冲突：184 → 31（-83%）
- 直接冲突：1 → 0
- 剩余 31 个全部是可接受的最长匹配重叠（长触发词始终优先胜出）

**备份**：`E:\ai产出文件\牛马\归档\skill-routing-table-backup-2026-06-09.json`

**教训**：
- 触发词设计铁律：复合词优先，禁止注册短泛词（"分析""文档""学习"等）作为独立触发词
- 短泛词导致 substring 匹配短路，长的精确触发词永远无法胜出
- 路由表需要定期做冲突扫描，类似代码的 lint 检查

### FileStorage 索引发现

- `msg/file/`：1,397个文档（PDF 481, DOCX 228, HTML 153），10.6GB，**未加密，保留原始文件名**

- `msg/attach/`：34,959张图片，3.09GB，**全部AES加密**（magic bytes `07 08 56 32`），无法在不解密DB的情况下映射
- `db_storage/hardlink/hardlink.db`（1.3MB）是hash→路径映射源，但解密需要微信运行+管理员权限
