## 2026-06-05 15:55 Antigravity 登录 OAuth 连接超时排查

- 现象：Antigravity 登录时报 `Post "https://oauth2.googleapis.com/token": dial tcp ...:443 connectex`，说明失败点在 Google OAuth token 端点的 TCP 连接。
- 排查：`nslookup oauth2.googleapis.com` 可解析；`Test-NetConnection oauth2.googleapis.com -Port 443` 超时；Windows 用户代理为 `127.0.0.1:10808`；WinHTTP 为 Direct access；PowerShell 通过用户代理访问 OAuth 端点可达。
- 结论：Antigravity 登录进程大概率没有继承/使用 Windows 用户代理，导致 OAuth token 交换直连 Google 超时。
- 经验：桌面 AI IDE 登录失败时，先区分“账号问题”和“OAuth 网络链路问题”；若错误包含 `oauth2.googleapis.com/token` + `connectex timeout`，优先检查代理继承、WinHTTP 代理、环境变量 `HTTP_PROXY` / `HTTPS_PROXY` / TUN 模式。

## 2026-06-05 16:11 Antigravity 双进程与单应用代理方案

- 新发现：机器上同时存在 `D:\Antigravity IDE\Antigravity IDE.exe` 和 `%LOCALAPPDATA%\Programs\antigravity\Antigravity.exe` 两套 Antigravity 进程；`Get-Process Antigravity` 只能停后者，不能停 `Antigravity IDE.exe`。
- 排查：`cloudcode-pa.googleapis.com:443` 直连超时，但 `curl -x http://127.0.0.1:10808 https://cloudcode-pa.googleapis.com/v1internal:loadCodeAssist` 可返回 Google `404`，说明显式代理链路可用。
- 处理：新增 `scripts/launch-antigravity-proxy.ps1`，只给 Antigravity IDE 注入代理环境变量和 `--proxy-server` 参数；新增 Antigravity IDE 用户级 `settings.json` 的 `http.proxy` 配置，避免写全局 `setx` 影响其他 CLI/网站访问。

## 2026-06-05 16:20 Antigravity ineligible 资格错误判断

- 新现象：Antigravity 显示 `Sorry, this account is ineligible to use Antigravity IDE`，并说明 `not currently available in your location`。
- 判断：这已经不是 TCP/代理链路错误，而是 Google 服务端基于账号资格做的拒绝；优先检查个人 Gmail、年龄验证、Google 账号关联国家/地区、Workspace 管理限制。
