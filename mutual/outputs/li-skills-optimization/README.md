# li-skills 体系级优化项目

> **项目目标**：检查和优化所有 li- skills 的体系级设计（模式路由 + Progressive Disclosure + 自进化 + 生态联动）
> **当前状态**：评估完成，路线图 v2.0 已完成（基于失败教训重构），待执行
> **创建时间**：2026-06-25
> **最后更新**：2026-06-25（v2.0 修订：新增 Phase 0 / 修正 li-zhongshu 方向 / 批量处理代码类 / 双轨制验收）

---

## 一、项目概述

### 1.1 优化需求

对所有 li- skills 进行体系级设计检查和优化，四维度标准：

| 维度 | 定义 | 满分 |
|------|------|------|
| 模式路由 | SKILL.md 中是否有 Mode/Phase/Flow 分支路由机制 | 2 |
| Progressive Disclosure | 是否有 references/ 分层加载机制 | 2 |
| 自进化 | 是否有 golden_rules/ + eval/ + evolution-log.md | 2 |
| 生态联动 | 是否有联动其他 skill 的机制 | 2 |

**总分 8 分**，目标是让每个 skill 达到体系级设计标准。

### 1.2 例外原则

**如果发现某个 skill 有更好的设计方式，记录和学习其设计思路，不强行套用四件套。**

已识别的特殊设计模式：
- **li-hardware**：知识密集型设计（27 个 references），场景靠内容覆盖而非路由分支
- **li-manage**：超复杂路由（32 个 Flow A-J），是"skill 里的 skill"，复杂度上限

## 七、关键决策记录

### 7.1 评估标准

- **模式路由**：SKILL.md 中有 Mode/Phase/Flow 分支 = 2 分
- **Progressive Disclosure**：有 references/ + 分层加载机制 = 2 分
- **自进化**：有 golden_rules/ + eval/ + evolution-log.md = 2 分
- **生态联动**：有明确的联动声明和协议 = 2 分

### 7.2 例外处理

- **li-hardware**：模式路由=0 是合理设计（知识密集型），不强制补路由
- **li-manage**：复杂度上限参考，不建议其他 skill 达到同等路由密度
- **li-persona-qa**：已 DEPRECATED，优先合入 li-autoreply

### 7.3 质量标准

- **T1（满分）**：体系级设计完成，可作为标杆
- **T2（7分）**：高质量，缺 1 维度，修复难度低-中
- **T3（5-6分）**：中等质量，缺 2-3 维度，需要中等工作量
- **P0（≤3分）**：关键缺失，需要重做/合入/弃用
