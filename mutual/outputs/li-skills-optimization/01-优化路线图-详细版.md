# li-skills 优化路线图 — 详细版

> **创建时间**：2026-06-25
> **最后更新**：2026-06-25（v2.0 基于失败教训重构）
> **基于**：四维度评估结果（49 个 li- skills）+ 跨区教训库
> **目标**：制定逐个 skill 的优化计划

## ⚠️ v2.0 核心修订说明

v1.0 计划存在以下系统性偏差，v2.0 全面修正：

| 偏差 | v1.0 错误做法 | v2.0 修正 |
|------|-------------|---------|
| **触发层缺失** | 只修结构（golden_rules/目录），不修触发（路由词） | 新增 Phase 0，触发层优先于结构层 |
| **优先级按分数不按影响力** | li-script(5分)列为 T3 低优先 | 改为按实际失误频率排序 |
| **li-zhongshu 方向错误** | 建议合入 li-manage | 实为知识中枢协议，应独立强化 |
| **批量机会未识别** | 6 个代码类 skill 各写独立计划 | 归为代码类通用模板批量处理 |
| **验收只看结构** | 检查文件是否存在 | 新增行为维度（实际调用率） |
| **联动声明 ≠ 联动执行** | 有声明=2分 | 区分声明完整度 vs 执行率 |

## 零、Phase 0：触发层修复（最高优先，0.5 天）

> **根因**：技能库结构再好，触发层有漏洞 → skill 存在但从未被调用 → 等同于没有。
> **教训来源**：`20260613_脚本开发跳过li-script导致6轮迭代浪费.md`

### 0.0 现有触发架构（先理解，再修）

我们已经有完整的三层路由机制，**Phase 0 是修漏洞，不是从零建设**：

```
Layer 0：skill-routing-table.json（关键词精确匹配，95个路由条目）
   ↓ 未命中
Layer 1：li/SKILL.md 路由表（人类可读的意图→skill映射，已含li-script等）
   ↓ 仍不确定
Layer 2：li-local-search（语义搜索，扫描196+已安装skill的回退层）
```

**已知问题**：`li-script` 在 `li/SKILL.md` 里有路由，但**不在 skill-routing-table.json（Layer 0）**。
用户不说"li 系列"、直接说"写个脚本" → Layer 0 未命中 → li-local-search 语义搜索兜底（成功率不确定）。

## 二、阶段 1：P0 处理（关键缺失）+ 高影响 T3

### 2.1 li-embedded（总分 1/8，最严重退化）

**现状**：
- 331 行全堆在 SKILL.md，无分层
- 无模式路由、无 references/、无 golden_rules/、无联动
- 评分：路由(0) + PD(0) + 进化(1) + 联动(0)

**问题诊断**：
- 知识密度高但组织混乱
- 缺乏分层机制，用户认知负荷过重
- 无进化机制，无法自我优化

**修复方案**：

**方案 A：知识密集型重构（参考 li-hardware）**
- 保留知识全量，但拆分到 references/
- SKILL.md 只做索引 + 路由
- 不强制加模式路由（硬件开发场景相对固定）

**方案 B：功能型重构（参考 li-study）**
- 按功能拆分模式（FPGA/STM32/PCB 三大场景）
- 加模式路由，每个模式对应不同 references
- 加联动（li-hardware/li-debug/li-code）

**推荐方案**：方案 A（知识密集型）
- **理由**：li-embedded 是硬件开发 skill，领域知识复杂但流程相对固定
- **参考**：li-hardware 的设计哲学（27 个 references，模式路由=0）
- **执行步骤**：
  1. 拆分 SKILL.md → references/（按主题拆分）
  2. SKILL.md 精简为索引 + 核心流程
  3. 加 golden_rules/ + evolution-log.md
  4. 加联动声明（li-hardware/li-debug/li-code/li-test）
  5. 不加模式路由（场景靠 references 覆盖）

**预计工作量**：中-高（需要拆分 331 行内容）

## 四、阶段 3：P2 优化（T3 梯队）

### 4.1 批量处理策略（v2.0 新增）

> **v2.0 识别的批量机会**：T3 梯队中有 6 个代码类 skill，路由设计和联动设计高度相似，可用同一套模板批量处理，节省约 60% 工作量。

**代码类通用模板（li-code / li-debug / li-test / li-review / li-refactor / li-deploy）**：

```markdown
## 🎯 模式路由（代码类通用）

### 模式 A：快速模式
- 触发条件：用户说"快速/简单/小改"
- 流程：直接执行核心步骤，跳过深度分析

### 模式 B：标准模式（默认）
- 触发条件：默认
- 流程：分析 → 执行 → 验证

### 模式 C：深度模式
- 触发条件：用户说"全面/严格/安全性"
- 流程：完整检查 + 边界验证 + 文档更新

## 🔗 联动机制（代码类通用）

### 上游 skill
- li-code：实现层（li-debug/li-test/li-review 的上游）

### 下游 skill（自动联动）
- li-test：实现后自动触发测试
- li-review：测试通过后代码审查
- li-deploy：审查通过后部署

### 联动协议
代码链：li-code → li-test → li-review → li-deploy（标准发布链）
调试链：报错 → li-debug → li-test → 验证修复
```

### 4.2 分组修复清单

**组 A：代码类（用通用模板批量处理，预计共 3 小时）**

| skill | 已提前处理？ | 补充内容 |
|-------|------------|---------|

## 五、质量门禁（v2.0：结构 + 行为双轨制）

### 5.1 结构门禁（原有）

**新建 skill 必过四维度评估**：
- [ ] 模式路由：有 Mode/Phase/Flow 分支
- [ ] PD：有 references/ + 分层加载
- [ ] 自进化：有 golden_rules/ + eval/ + evolution-log.md
- [ ] 联动：有明确的联动声明和协议

**最低标准**：总分 ≥ 6/8

### 5.2 行为门禁（v2.0 新增）

> **根因**：结构存在不等于被执行。`20260607_朋友圈任务伪造执行5次_技能链零调用.md` 说明有 SKILL.md 但路由从未触发。

**验收需同时满足**：

| 指标 | 测量方法 | 通过标准 |
|------|---------|---------|
| **触发词命中率** | 用 3 个典型用户消息走路由，grep 检查是否匹配 | ≥2/3 命中 |
| **联动执行率** | 观察 skill-usage-log.md 中联动列是否有记录 | 有执行记录（非空） |
| **漏调率** | 对比"应该调用但没调用"的情况 | ≤1次/周 |

### 5.3 路由表注册联动

**强制流程**：
1. 创建 skill → 自动触发路由注册（li-skillcreate Phase 3.5）
2. 路由表更新 → 自动同步到所有工作区
3. 联动声明 → 自动更新相关 skill 的上游/下游

### 5.4 定期审计

**频率**：每月一次
**内容（v2.0 扩展）**：
- 四维度评分（结构层）
- skill-usage-log.md 调用统计（行为层）
- 触发词命中率测试
- 联动执行率统计
- 沉睡 skill 识别（30天未调用）→ 考虑弃用或路由优化

## 六、风险和应对（v2.0 基于历史教训）

### 6.1 主要风险

| 风险 | 历史教训依据 | 应对 |
|------|------------|------|
| **Phase 0 只改文件不测试** | 路由词加了但从未验证触发效果 | Phase 0 必须走"典型消息→路由→触发"验证闭环 |
| **阶段1工作量低估** | li-embedded 有 331 行，1-2天根本不够 | v2.0 调整为 2-3 天，li-embedded 单独分配 1 天 |
| **联动声明 vs 联动执行断层** | 声明写了但 AI 执行时跳过 | 必须在 skill-execution-discipline.md 中加强制联动检查 |
| **批量处理质量风险** | 套模板可能不适配具体 skill 场景 | 批量处理后每个 skill 独立走行为验收 |
| **路由表膨胀** | 触发词过多导致歧义匹配 | 新增触发词前先 grep 检查是否已被其他 skill 占用 |

### 6.2 已知死路（v2.0 新增）

> **法则 7：负结果是一等公民**——历史上已验证失败的方向必须归档，不能重复踩。

| 死路 | 为什么看起来行但不行 | 替代方案 |
|------|----------------|---------|
| li-zhongshu 合入 li-manage | li-zhongshu 是知识中枢协议，li-manage 是工作区管理，职责完全不同 | 功能明确化 + 独立补全 |
| 只补结构不验证行为 | 有文件 ≠ 被执行，历史多次证明 | 双轨制验收（结构+行为） |
| 给所有 T3 单独设计路由 | 代码类 6 个 skill 路由几乎一样，分别设计浪费时间 | 批量模板 |

### 6.3 应急预案

- **Phase 0 触发词验证失败**：不进阶段1，先彻底修好触发层
- **阶段1 P0 延期**：优先 li-embedded（最严重），li-zhongshu/li-autoreply 可延至阶段2
- **批量模板套用后行为验收不过**：降级为独立设计，接受工作量增加
