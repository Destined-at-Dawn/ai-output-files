## 一、T1 满分梯队设计模式

### 1.1 li-study — 模式路由 + PD + 自进化 + 联动的教科书

**总分**：8/8（满分）

**设计亮点**：
- **24 种学习模式**：按学习场景（概念学习/技能训练/知识迁移等）细分
- **模式路由**：用户意图 → 自动匹配最优学习模式
- **PD 完整**：references/ 按主题分层（基础/进阶/专家）
- **自进化**：golden_rules/ 学习效果规则 + eval/ 学习质量评估
- **联动**：li-analyze（深化）+ li-memory（沉淀）+ li-devil（质疑）

**可复用模式**：

**模式路由设计**：
```markdown
## 🎯 模式路由

### 模式 A：快速学习
- 触发条件：用户说"快速/简要/大概了解"
- 流程：Phase 1（概览）→ Phase 3（总结）
- 跳过：深度分析和实践环节

### 模式 B：深度学习
- 触发条件：默认/用户说"深入/详细/全面"
- 流程：Phase 1 → Phase 2 → Phase 3 → Phase 4
- 包含：完整学习闭环

### 模式 C：实践学习
- 触发条件：用户说"练习/实操/动手"
- 流程：Phase 1 → Phase 2 → 实践环节 → 反馈优化
- 重点：动手实践 + 即时反馈
```

**学习点**：
- 模式路由应基于用户意图，而非功能划分
- 每个模式应有明确的触发条件和流程差异
- 模式数量应与场景多样性匹配（24 种可能过多，但覆盖全面）

### 1.2 li-manage — 超复杂路由的极限设计

**总分**：8/8（满分）

**设计亮点**：
- **32 个 Flow（A-J 大类 + 子类）**：路由密度最高
- **Flow 分层**：大类（A-J）→ 子类（A1/A2/A3...）
- **路由逻辑**：用户意图 → 匹配 Flow → 执行对应流程
- **复杂度管理**：每个 Flow 有明确的职责边界

**可复用模式**：

**Flow 路由设计**：
```markdown
## 🎯 Flow 路由

### Flow A：工作区管理
- A1：创建新工作区
- A2：删除工作区
- A3：迁移工作区

### Flow B：skill 管理
- B1：创建新 skill
- B2：修改 skill
- B3：弃用 skill

### Flow C：记忆管理
- C1：记忆沉淀
- C2：记忆清理
- C3：记忆迁移
```

**学习点**：
- Flow 路由适合复杂度高的 skill（多个功能模块）
- Flow 分层可以降低单个路由的复杂度
- **风险**：过度拆分可能导致 Flow 之间职责不清（li-manage 已逼近上限）

**警告**：不是所有 skill 都需要 32 个 Flow。路由密度应与场景多样性匹配。

### 1.3 li-analyze — 用认知框架做模式路由

**总分**：8/8（满分）

**设计亮点**：
- **道法术器四框架路由**：用认知框架（道/法/术/器）划分模式
- **非功能分**：不是按"做什么"分，而是按"怎么想"分
- **框架驱动**：每个框架对应一套分析方法论

**可复用模式**：

**认知框架路由**：
```markdown
## 🎯 道法术器路由

### 道（Why）— 战略层
- 触发条件：用户问"为什么/目的/意义"
- 流程：目标分析 → 价值判断 → 战略建议

### 法（How）— 方法层
- 触发条件：用户问"怎么做/方法/策略"
- 流程：方法论匹配 → 方案设计 → 可行性评估

### 术（What）— 执行层
- 触发条件：用户问"具体做什么/步骤"
- 流程：任务拆解 → 执行计划 → 进度跟踪

### 器（Tool）— 工具层
- 触发条件：用户问"用什么工具/技术"
- 流程：工具选型 → 技术评估 → 实施建议
```

**学习点**：
- 认知框架路由比功能分更有深度（引导用户思考，而非只执行）
- 适合分析/咨询类 skill
- **风险**：用户可能不理解框架含义（需要在 SKILL.md 中解释清楚）

### 1.4 li-xhs — 内容创作 + 文风注入 + 自检的三件套

**总分**：8/8（满分）

**设计亮点**：
- **创作流程**：选题 → 标题 → 内容 → 优化
- **文风注入**：自动调用 voice-dna-auto-inject
- **自检机制**：自动调用 li-devil 进行质量审视
- **联动完整**：上游（li-research/li-analyze）+ 下游（voice-dna/li-devil）

**可复用模式**：

**创作三件套**：
```markdown
## 🔗 联动机制

### 上游 skill
- li-research：信息补全
- li-analyze：深度分析

### 下游 skill（自动联动）
- voice-dna-auto-inject：文风注入
- li-devil：质量自检

### 联动协议
1. 主 skill 执行完成（内容创作）
2. 自动调用 voice-dna（注入个人风格）
3. 自动调用 li-devil（质量审视）
4. 整合最终输出
```

**学习点**：
- 内容创作类 skill 应标配"创作 + 文风 + 自检"三件套
- 联动应该是自动的，不需要用户手动触发
- 上游（信息补全）+ 下游（质量保证）的分工模式

### 1.5 li-intent — 意图路由器 + 自学习闭环

**总分**：8/8（满分）

**设计亮点**：
- **意图识别**：Phase 0 提取用户意图
- **SOP 匹配**：Phase 1 匹配已有 SOP
- **自学习**：Phase 4 记录路由结果，优化匹配算法
- **路由表驱动**：基于 skill-routing-table.json

**可复用模式**：

**意图路由 + 自学习**：
```markdown
## 🎯 意图路由

### Phase 0：意图提取
- 从用户消息中提取核心意图
- 识别关键词和上下文

### Phase 1：SOP 匹配
- 查询 skill-routing-table.json
- 匹配最佳 skill

### Phase 2：执行
- 调用匹配的 skill
- 记录执行结果

### Phase 3：自学习
- 记录路由结果（成功/失败）
- 优化匹配算法（提升 confidence）
```

**学习点**：
- 路由器 skill 应有自学习机制（记录路由结果，优化匹配）
- skill-routing-table.json 是路由的核心数据结构
- 意图识别应基于关键词 + 上下文，而非单一关键词

### 1.6 li-memory — 记忆沉淀 + 置信度 + 过期机制

**总分**：8/8（满分）

**设计亮点**：
- **记忆沉淀**：从对话中提取重要信息
- **置信度**：每条记忆有置信度标签（高/中/低）
- **过期机制**：记忆有过期时间，定期清理
- **分类存储**：按主题分类存储记忆

**可复用模式**：

**记忆管理三要素**：
```markdown
## 记忆管理

### 1. 记忆提取
- 从对话中识别重要信息
- 分类：决策/教训/偏好/知识

### 2. 置信度评估
- 高置信：用户明确陈述 ≥ 2 次
- 中置信：AI 观察到的规律
- 低置信：单次观察

### 3. 过期机制
- 30 天未更新 → 标记为"需刷新"
- 90 天低置信 → 归档
- 180 天所有记忆 → 重新评估
```

**学习点**：
- 记忆管理应有置信度和过期机制（防止记忆过时）
- 分类存储可以提高检索效率
- 定期清理可以防止记忆膨胀

### 1.7 li-plan — 任务规划 + 预验尸 + 归档联动

**总分**：8/8（满分）

**设计亮点**：

- **归档联动**：自动调用 li-manage 进行归档
- **联动完整**：上游（li-analyze）+ 下游（li-devil/li-manage）

**可复用模式**：

**规划 + 预验尸**：
```markdown
## 🔗 联动机制

### 上游 skill
- li-analyze：需求分析

### 下游 skill（自动联动）
- li-devil：预验尸（风险评估）
- li-manage：归档（计划存档）

### 联动协议
1. 主 skill 执行完成（任务规划）
2. 自动调用 li-devil（风险评估）
3. 自动调用 li-manage（计划归档）
```

**学习点**：
- 规划类 skill 应标配"预验尸"机制（风险评估）
- 归档联动可以确保计划不丢失
- 上游（需求分析）+ 下游（风险评估 + 归档）的分工模式

### 1.9 li-image — 图片生成 + 一致性参数 + 多平台适配

**总分**：8/8（满分）

**设计亮点**：
- **图片生成**：调用图片生成工具
- **一致性参数**：锁定风格/视角/色调等参数
- **多平台适配**：自动适配不同平台的尺寸要求
- **联动**：li-design（设计）+ li-xhs（小红书）

**可复用模式**：

**生成 + 一致性 + 适配**：
```markdown
## 🎯 模式路由

### 模式 A：单图生成
- 触发条件：用户说"生成一张图"
- 流程：参数锁定 → 生成 → 输出

### 模式 B：多图一致
- 触发条件：用户说"生成一组图/保持风格一致"
- 流程：母版生成 → 参数锁定 → 批量生成 → 一致性检查

### 模式 C：多平台适配
- 触发条件：用户说"适配小红书/公众号/朋友圈"
- 流程：生成 → 裁剪/缩放 → 多平台输出
```

**学习点**：
- 图片类 skill 应有"一致性参数"机制（防止风格漂移）
- 多平台适配是常见需求，应内置支持
- 模式路由应基于使用场景（单图/多图/多平台）

## 二、特殊设计模式

### 2.1 li-hardware — 知识密集型设计

**总分**：4/8（特殊设计，不强制四件套）

**设计哲学**：把领域知识全量存进 references，SKILL.md 只做索引。

**数据**：
- **27 个 references**：所有 skill 中最多
- **模式路由=0**：场景变化靠 references 内容覆盖
- **适用场景**：硬件开发（FPGA/STM32），领域知识复杂但流程相对固定

**设计模式**：

```markdown
## 知识密集型结构

### SKILL.md（索引层）
- 核心流程（简要）
- references/ 索引（按主题分类）
- 使用说明（如何选择 references）

### references/（知识层）
- 按主题拆分（FPGA/STM32/PCB/调试/工具等）
- 每个 references 文件独立完整
- 支持按需加载（不一次性全读）
```

**学习点**：
- 不是所有 skill 都需要模式路由
- 知识密集型领域，references 质量比路由分支更重要
- SKILL.md 作为索引 + references 作为知识库的分工模式
- **适用场景**：领域知识复杂、流程相对固定的 skill

**不适用场景**：
- 场景多变、需要灵活路由的 skill
- 用户意图多样、需要模式切换的 skill

### 2.2 li-manage — 超复杂路由的极限设计

**总分**：8/8（满分，但复杂度已逼近上限）

**设计哲学**：用 Flow（A-J 大类 + 子类）实现"skill 里的 skill"。

**数据**：
- **32 个 Flow**：路由密度最高
- **Flow 分层**：大类（A-J）→ 子类（A1/A2/A3...）
- **复杂度上限**：已逼近单个 skill 的复杂度极限

**设计模式**：

```markdown
## 超复杂路由结构

### 大类（A-J）
- A：工作区管理
- B：skill 管理
- C：记忆管理
- ...

### 子类（A1/A2/A3...）
- A1：创建新工作区
- A2：删除工作区
- A3：迁移工作区
- ...

### 路由逻辑
用户意图 → 匹配大类 → 匹配子类 → 执行对应 Flow
```

**学习点**：
- 复杂任务需要复杂路由，但要警惕复杂度膨胀
- Flow 分层可以降低单个路由的复杂度
- **风险**：过度拆分可能导致 Flow 之间职责不清
- **建议**：其他 skill 不建议达到 32 个 Flow 的复杂度

**适用场景**：
- 功能模块多、场景多样的 skill
- 需要精细路由的 skill

**不适用场景**：
- 功能单一、场景固定的 skill
- 用户意图简单的 skill

## 三、设计模式总结

### 3.1 四种设计模式

| 模式 | 适用场景 | 代表 skill | 复杂度 |
|------|---------|-----------|--------|
| **模式路由** | 场景多变、意图多样 | li-study, li-analyze | 中 |
| **知识密集型** | 知识复杂、流程固定 | li-hardware | 低 |
| **超复杂路由** | 功能模块多、场景多样 | li-manage | 高 |
| **创作三件套** | 内容创作、需要质量保证 | li-xhs | 中 |

### 3.2 选择指南

**选择模式路由**：
- 用户意图多样（快速/深度/特殊场景）
- 场景差异明显（不同场景需要不同流程）
- 需要灵活切换

**选择知识密集型**：
- 领域知识复杂
- 流程相对固定
- 用户主要需要知识，而非流程

**选择超复杂路由**：
- 功能模块多（>10 个独立功能）
- 场景多样（>20 种使用场景）
- 需要精细路由

**选择创作三件套**：
- 内容创作类 skill
- 需要文风注入
- 需要质量自检

### 3.3 避免的陷阱

**过度路由**：
- 不是所有 skill 都需要 32 个 Flow
- 路由密度应与场景多样性匹配
- 过度路由会增加维护成本

**过度简化**：
- 不是所有 skill 都只需要一个模式
- 场景多变时需要模式路由
- 过度简化会降低用户体验

**忽略联动**：
- 单打独斗的 skill 效率低下
- 联动可以实现自动化和质量保证
- 每个 skill 都应有联动声明
