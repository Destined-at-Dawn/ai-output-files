# V15 最终最小可执行协议

## 用户反馈

“我觉得你还可以优化得更好。最后给我一个真正能执行的版本。”

## 最终判断

你的目标不是“文件同步”，而是：

> 任何想法、项目、产出、复盘，都能被正确路由、持续推进、自动登记、受控沉淀，最后变成能力资产。

## 最小可执行协议

### 1. 一个中枢

`mutual` 是协议中枢，不是所有内容的堆放地。

负责：

- 项目卡
- 产出注册表
- 工作流改进收件箱
- 任务路由器
- 仪表盘

### 2. 五个工作区

```text
个人：长期学习、认知、复盘
创作：调研、文档、表达、PPT
竞赛：代码、工程、实验、技术证据
求职：简历、面试、公司研究、能力转化
mutual：项目上下文、注册、路由、协议
```

### 3. 三个 Agent

```text
Claude/Newmax：主控、写作、规划、语义判断
Codex：代码、文件审计、验证、工程执行
Hermes：定时巡检、日报、长期监控
```

### 4. 三个核心文件

```text
project-context/PROJECT-xxx.md
artifact-registry.md
workflow-inbox.md
```

如果只能先做三个，就做这三个。

### 5. 三条铁律

```text
跨区项目先建项目卡
重要产出必须登记
长期记忆必须候选确认
```

### 6. 三种启动

```text
L1 轻启动：日常小任务
L2 项目启动：读项目卡 + 注册表
L3 高风险启动：读底层铁律 + 归档/记忆协议
```

### 7. 一周节奏

```text
每天：推进 Active 项目
每周：清理 workflow-inbox
每周：复盘项目与资产
每周：确认长期记忆候选
```

## 最终工作流

```text
想法出现
  → workflow-inbox
  → 判断是否立项
  → 建 project-context
  → 路由到对应工作区
  → 产出写回 artifact-registry
  → 求职/创作/竞赛复用
  → 项目复盘
  → 候选记忆/SOP
  → 用户确认后沉淀
```

## 最重要的取舍

放弃“五区规则全文同步”，保留“五区项目上下文同步”。

这能同时满足两个目标：

- 不丢上下文
- 不让规则系统膨胀失控
