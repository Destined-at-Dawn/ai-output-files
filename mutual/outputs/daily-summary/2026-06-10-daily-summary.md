## 💡 关键收获与洞察

### 技术洞察

1. **Windows PATH 优先级问题**
   - 多个 git.exe 可能导致命令冲突
   - npm 全局目录可能意外覆盖系统工具
   - **解决方案**：使用完整路径或调整 PATH 顺序

2. **MCP server 认证机制**
   - MCP server 需要显式传递环境变量
   - `-e` 参数是标准做法
   - **最佳实践**：在 MCP 配置中明确列出所需环境变量

3. **markitdown 包的启动陷阱**
   - 原生启动可能进入无限循环
   - 需要使用 wrapper 脚本控制启动行为
   - **解决方案**：`runpy.run_module()` 是可靠启动方式

## 🎯 经验教训

### 教训 1：Windows PATH 冲突是隐蔽杀手

**场景**：执行 git 命令时报"不是内部或外部命令"
**根因**：npm 全局目录中的 git.cmd 覆盖了系统 git.exe
**解决方案**：使用完整路径或调整 PATH 优先级
**预防措施**：定期检查 PATH 顺序，避免工具冲突

### 教训 2：MCP server 启动需要特殊处理

**场景**：markitdown 启动后无限循环
**根因**：markitdown_mcp 包的启动逻辑存在缺陷
**解决方案**：使用 wrapper 脚本 + runpy 控制启动
**预防措施**：新安装 MCP 后必须测试启动行为

### 教训 3：环境变量传递不能想当然

**场景**：MCP-Tools 调用返回 401
**根因**：子进程未继承父进程的环境变量
**解决方案**：使用 `-e` 参数显式传递
**预防措施**：MCP 配置中明确列出所需环境变量
