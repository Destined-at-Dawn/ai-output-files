# 微信聊天分析踩坑铁律（22条）

> 每一条都是真实事故，不是理论推演。

---

## 密钥提取（5条）

1. **密钥只在进程内存中**：AES-256 加密，密钥不在配置文件、不在磁盘，只在微信运行时的内存中。没有"离线破解"方案。
2. **微信必须正在运行**：提取密钥前不能关闭微信。wx_key 通过 DLL Hook 注入微信进程读取内存。
3. **需要管理员权限**：读取其他进程的内存需要 SeDebugPrivilege，普通权限不行。
4. **密钥会变**：微信重启后密钥可能变化，每次都要重新提取。
5. **密钥缓存要备份**：提取的密钥 JSON 保存到安全位置，Temp 目录随时可能被清理。

## 数据库结构（5条）

6. **表名 = Msg_ + md5(chatroom_wxid)**：每个聊天对象有独立的表，表名是 wxid 的 MD5 哈希。
7. **zstd 压缩是最大隐藏坑**：新版微信消息内容用 zstd 压缩，magic bytes `28 B5 2F FD`。不解压会得到二进制垃圾。
8. **sender_id 因聊天而异**：同一个数字 sender_id 在不同聊天中代表不同的人。不能假设 "sender=2 一定是小黎"。
9. **消息内容前缀格式**：部分消息内容开头是 `{wxid}:{实际内容}`，可以用来反推发送者身份。
10. **real_sender_id vs sender**：`real_sender_id` 是表内局部 ID，`sender` 是全局 ID。群聊用前者，私聊用后者。

## 编码与脚本（6条）

11. **Bash heredoc 含中文必炸**：MSYS2/Git Bash 下，即使单引号 heredoc 也会破坏 UTF-8 中文。一律用 Python 写文件。
12. **PowerShell 的 `$_` 在 inline 模式下必失败**：Bash 先展开 `$_`，PowerShell 看到的是空值。必须写 .ps1 文件再执行。
13. **GBK 终端显示问题**：Windows 终端默认 GBK，UTF-8 文件显示乱码但内容正确。用 Python 验证，不靠终端。
14. **Python 脚本优先**：涉及中文路径的任何操作，一律用 Python（`os`/`shutil` 原生支持 Unicode），不用 Bash/PowerShell。
15. **zstd 解压需要 `import zstandard`**：不是标准库，需要 `pip install zstandard`。
16. **Read 工具对中文路径可能返回假阴性**：文件存在但 Read 报 "File does not exist"。用 Python 做 ground truth。

## 导出与分析（6条）

17. **拍一拍占群聊 40%+ 噪音**：Type=10000 系统消息中拍一拍是最大噪音源，分析前必须过滤。
18. **信息密度 < 2%**：77,000 条群聊消息中只有 288 个真正有用的资源（0.37%）。先去噪再分析。
19. **关键词匹配对聊天记录无效**："工作"匹配到"工作群"、"分享"匹配到"分享日常"。必须用上下文判断。
20. **导出后必须验证**：消息数对比、发送者分布、二进制垃圾检测、UTF-8 编码确认。
21. **不要硬编码路径**：解密 DB 可能在创作区、Temp、或用户指定目录。每次都要探测。
22. **write_project_file 是覆写不是追加**：记忆文件必须用 Read → Edit 追加，不能用 write_project_file。曾导致 339 行记忆丢失。
