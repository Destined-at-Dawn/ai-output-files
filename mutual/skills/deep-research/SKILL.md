# 深度调研智能体 v3（Ecosystem-Aware）

> 调研不产出文件 = 没做。搜索不点开网页 = 没搜。迭代没有递增 = 复制粘贴。
> 产出不喂回生态 = 孤岛。越用越不懂用户 = 失败。

---

## 设计哲学：一鱼多吃

**每个调研动作必须产出 5+ 个生态副作用**：

```
用户说"帮我调研 X"
  → 读取 research-profile.md（不用重新问偏好）
  → 执行调研（多平台搜索+分析）
  → 产出 ① 调研报告 outputs/
  → 自动更新 ② memory/{today}.md
  → 自动更新 ③ memory/research-profile.md（新发现+偏好变化）
  → 自动更新 ④ memory/skill-usage-log.md（路由+效果）
  → 候选更新 ⑤ memory/long-term.md（高价值发现）
  → 自动登记 ⑥ artifact-registry.md（产出路径）
  → 自动记录 ⑦ evolution-calendar.md（进化信号）
```

**下次调研时，AI 读这些文件就直接知道**：用户喜欢什么平台、关注什么领域、哪些路走过是死路、上次迭代到第几轮。不用重新问。

## Phase 1：调研澄清（行动锁）

> 来源：jc-clarifier 行动锁 + 镜像复述 + 行家视角校准

**铁律：在用户确认镜像复述之前，禁止执行任何搜索。**

### Step 1.1：读取用户项目上下文

如果用户消息中提到了文件路径 → 必须读取。用户提到的代码/文档/方案/问题是调研的关键上下文。

检查 `outputs/` 是否有相关历史调研：
- 有 → 读取报告的"核心结论"和"边界声明"，避免重复搜索
- 无 → 进入澄清

### Step 1.2：澄清追问（一次性 3-4 个问题）

每个问题附带猜测（不要问空题），从 research-profile.md 中的偏好自动填充默认值：

```
基于你说的，我理解你要调研的是 [X]。

几个关键问题确认：

1. **调研目标**：你想找到的是 [A.竞品分析 B.需求验证 C.技术方案 D.市场机会]？
   我猜是 [X]，因为 [证据]。

2. **平台范围**：默认搜 GitHub + X + Reddit + 学术 + 产品网站。
   需要加减吗？你上次调研偏好的平台是 [从 research-profile.md 读取]。

3. **深度标准**：你要的是 [A.快速概览 B.标准调研 C.深度穿透]？
   你之前的习惯是 [从 research-profile.md 读取]。

4. **已有上下文**：你提到的 [文件/方向]，我读取后融入调研。
   还有其他相关材料吗？
```

### Step 1.3：镜像复述

用户回答后，输出镜像复述：

```
确认一下我的理解：

**调研主题**：[精确描述]
**目标**：[具体目标]
**平台**：[列出所有要搜的平台]
**深度**：[概览/标准/深度]
**融入上下文**：[已有材料的关键发现 + 已有调研的历史结论]
**产出位置**：outputs/[文件名]-[日期].md

对吗？确认后我开始搜。
```

**用户确认后才进入 Phase 2。**

## Phase 2：多平台搜索（每平台 5+5）

> 来源：SenseNova 搜索矩阵 + academic-research-skills 多维度独立取证 + subagent-strategy 并行搜索

### 搜索矩阵

每个平台执行两轮：

| 轮次 | 策略 | 目标 |
|------|------|------|
| Round 1：热门 | 按 stars/upvotes/views 排序 | 最主流的 5 个 |
| Round 2：新鲜 | 按发布时间倒序 + 质量筛选 | 最新高质量的 5 个 |

### 平台清单（根据主题动态选择 3-5 个）

| 平台 | Round 1 | Round 2 |
|------|---------|---------|
| GitHub | `{topic} site:github.com stars` | `{topic} site:github.com created:>2026-01-01` |
| X/Twitter | `{topic} site:x.com` | `{topic} site:x.com 2026` |
| Reddit | `{topic} site:reddit.com` | `{topic} site:reddit.com 2026` |
| 学术 | `{topic} site:arxiv.org OR site:semanticscholar.org` | `{topic} paper 2025 2026` |
| 产品 | `{topic} tool OR platform OR app` | `{topic} launch 2026` |
| 中文社区 | `{topic} site:zhihu.com OR site:juejin.cn` | `{topic} 2026 最新` |

### 搜索执行规则

1. **至少 3 个平台 × 2 轮 = 6 次搜索**
2. **至少 WebFetch 点开 5 个最相关网页**读全文
3. **搜索结果立即落盘**到报告的"附录 A"
4. **与历史调研去重**：读取 `memory/research-profile.md` 历次调研索引，避免重搜已覆盖的内容
5. **失败处理**：某平台失败 → 记录原因 → 继续其他平台 → 报告中标注

### 搜索结果分层标记

- 🔴 高相关：直接匹配目标，有实质内容
- 🟡 中相关：部分匹配，有参考价值
- ⚪ 低相关：间接相关，仅作背景

## Phase 3：分析综合（道法术器 + 认知框架穿透）

> 来源：dao-fa-shu-qi 四层穿透 + 百大认知书籍 + Karpathy 简洁审查

### Step 3.1：竞品卡片化

每个工具/产品/方案：

```markdown
### [名称]
- **一句话**：[它是什么]
- **链接**：[URL]
- **核心能力**：[3-5 个要点]
- **用户画像**：[谁在用、为什么用]
- **数据指标**：stars/downloads/users/活跃度
- **不足**：[用户抱怨什么、缺什么]
- **与用户方案的关系**：[互补/竞争/可借鉴]
```

### Step 3.2：差距分析

| 维度 | 用户现有方案 | 竞品 A | 竞品 B | 差距 | 机会 |
|------|------------|--------|--------|------|------|

### Step 3.3：道法术器四层穿透

**道（Why）**：用户的底层心理需求是什么？（不是功能需求）
- 引用认知科学：[具体理论 + 《书名》编号]

**法（How）**：行业最佳实践 / 成功产品的共同模式

**术（What）**：哪些技术方案被验证过？哪些是伪需求？

**器（With What）**：具体工具/框架/平台对比

### Step 3.4：刚需验证（三问过滤）

> 来源：精益创业 + 创新者的窘境 + 反脆弱

对每个发现的需求：

1. **频率**：多久遇到一次？（每天 > 每周 > 每月 > 偶尔）
2. **痛度**：现有方案有多差？（无方案 > 很差 > 凑合 > 还行）
3. **付费意愿**：愿意付多少？（已有付费竞品 > 愿意试用 > 免费才用）

**三个都指向"高频+高痛+有付费"才是刚需。**

## Phase 5：迭代模式（1.25 倍递增）

> 来源：auto-evolution eval.json + 用户"3 小时 100 次迭代"要求

### 迭代协议

**铁律：每次迭代必须比上一轮多产出 25% 的文件/发现。**

| 轮次 | 最少新增 | 说明 |
|------|---------|------|
| 第 1 轮 | 20 | 首轮至少 20 |
| 第 2 轮 | 25 | 20 × 1.25 |
| 第 3 轮 | 32 | 25 × 1.25 |
| 第 N 轮 | ceil(prev × 1.25) | 递增无上限 |

### 每轮迭代流程

1. 读取上一轮报告 + `research-profile.md` 中的历史调研索引
2. 确定本轮焦点（深挖/扩展/验证/用户指定）
3. 执行搜索（至少 WebFetch 点开 3 个新网页）
4. 更新报告，标注"第 N 轮迭代更新"
5. 差距检查：新增 >= ceil(上轮 × 1.25)？
6. 覆盖保存原报告 + 追加今日记忆
7. 汇报 + 继续问要不要迭代

### 迭代汇报格式

```
---
🔄 第 N 轮迭代完成

新增发现：X 个（目标 Y 个，✅/❌）
新增搜索：Z 次，点开 W 个网页
累计发现：总计 N 个

关键新发现：
1. [发现 1]
2. [发现 2]
3. [发现 3]

**继续迭代？** 指定方向或说"继续"。
说"停"→ 产出最终版报告。
---
```

### 终止条件

1. 用户说"停"/"结束"/"够了"
2. 发现高频+高痛+有付费的刚需 → 向用户汇报
3. 时间上限 3 小时
4. 连续 3 轮未达标（新增 < 目标 80%）→ 搜索饱和

### 迭代终止后

产出最终版报告 + 再次执行一鱼多吃（更新 research-profile + 产出登记等）。

## Phase 6：知识沉淀 + 生态进化

> 来源：十条工程法则 #10（规则结晶）+ auto-evolution 评估闭环

### 自动沉淀

每次调研/迭代完成后，检查以下信号：

**进化信号**（如果发现以下模式，记录到 `evolution-calendar.md`）：
- 同类调研被请求 ≥3 次 → 该领域应建立独立 skill
- 某个平台反复搜不到有价值内容 → 路由表降低该平台优先级
- 用户反复迭代同一方向 → 该方向应升级为长期项目

**路由优化**（如果发现以下模式，更新 `skill-routing-table.json`）：
- 用户手动指定"用 deep-research" → 说明路由漏了触发词
- 搜索到的内容更适合用其他 skill → 建议用户切换

**使用画像更新**（自动，不需要用户确认）：
- 迭代轮次 → 更新到 research-profile.md 的"迭代习惯"
- 用户偏好的分析深度 → 更新到 research-profile.md
- 用户关注的新领域 → 更新到 research-profile.md
