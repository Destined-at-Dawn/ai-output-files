# Session Summary — 会话总结审计系统

> **每次对话结束前的必触发技能。审计本次对话的一切：用了什么、漏了什么、下次怎么改。**
> **路径：** `C:\Users\13975\.newmax\skills\session-summary\SKILL.md`
> **版本：v1.0 | 2026-05-09**

---

## 触发判定规则（AI 必读，最高优先级）

### 必触发场景

以下场景**必须执行会话总结**：

| 触发信号 | 示例表达 | 执行方式 |
|----------|---------|----------|
| **用户主动要求** | "总结一下"、"复盘"、"做个总结"、"收工" | 立即执行 |
| **对话自然结束** | 问题已解决、无新任务、用户说"好的/OK/明白了" | 自动触发 |
| **用户告别** | "下次见"、"拜拜"、"先这样"、"结束吧" | 立即执行 |
| **任务完成** | 所有 TODO 已标记 completed | 自动触发 |
| **会话过长** | 对话超过 30 轮交互且未做过任何总结 | 提醒用户是否需要总结 |

### 不触发场景

| 场景 | 处理方式 |
|------|---------|
| 用户明确说"不用总结" | 跳过，不输出审计 |
| 纯闲聊（无实质任务） | 简化为一句话："本次对话为闲聊，无技能调用" |
| 用户中途切换话题 | 不触发（等最终结束时再触发） |

### 默认倾向：宁可多总结不要漏总结

- 判断不清时，**默认触发**
- 用户可以随时用"不用总结"跳过

## 核心输出格式

### 标准格式（有实质任务时）

```
📋 技能审计

本次调用：{技能A}（{次数/详情}）、{技能B}（{次数/详情}）、{工具C}（{次数/详情}）

被忽视的：{应该调用但没调用的技能} —— {为什么没调用，或下次应该调用}

下次改进：{遇到的问题、踩的坑、可以优化的方向}

📅 改进周期检查

当前工作区：{工作区名}
上次审计：{日期} → {今天已审计/今天未审计}
→ {已执行增量审计/明天首次对话时自动审计}
```

### 示例

```
📋 技能审计

本次调用：browser-use（6篇抓取）、project-tools（write_project_file ×9 / read_project_file ×3）、Edit ×4

被忽视的：wechat-article-collector（可替代浏览器手动抓取，但该skill之前测试只返回模板不执行实际采集）

下次改进：CSS视觉数据提取失败——browser_eval多行JS返回undefined，需改为单行JSON.stringify表达式
```

### 简化格式（轻量对话时）

```
📋 技能审计：本次对话为{对话类型}，{一句话总结}。
```

## 工作流程

### Step 1：收集调用记录

回顾本次对话，收集以下信息：

```
1. 调用的技能（Skill）
   - 技能名称 + 调用次数 + 关键细节
   - 包括：mcp__skill-handler__Skill、mcp__browser-use__*、mcp__convert-to-project__*、mcp__scheduled-tasks__*、mcp__web-search__*

2. 调用的工具（Tool）
   - 工具名称 + 调用次数
   - 包括：Read、Write、Edit、Bash、Grep、Glob、Agent、WebFetch、AskUserQuestion

3. 被触发的技能组（如有）
   - Core Loop 是否触发了 R08、L2 记录？
   - Workspace Builder 是否完整执行了 4 步？
   - Sync & Save 是否完整执行了 3 步？
```

### Step 2：分析被忽视的机会

检查是否有**应该调用但没调用**的技能：

```
检查清单：
- 做了大量手动文件操作 → 是否可以用 scaffold-workspace 或 project-init？
- 用浏览器抓取了网页 → 是否可以用 wechat-article-collector 或 step-search？
- 修改了多个工作区的同类文件 → 是否应该触发 li-sync？
- 对话中有个人信息透露 → 是否触发了 R08 自动捕获？
- 对话中有重要决策/事件 → 是否记录到了 L2？
- 做了复杂诊断 → 是否应该 dbs-save 存档？
- 写了大量代码 → 是否应该 li-improve 记录经验？
- 生成了内容 → 是否应该 style-dna-writer 做风格分析？
- 对话中有重要决策/教训写入了 daily memory → 是否需要触发 longmemory 提前归档？
```

### Step 3：记录踩坑和改进

回顾本次对话中遇到的问题：

```
踩坑类型：
- 工具调用失败（如 browser_eval 返回 undefined）
- 方法选择错误（如用 WebFetch 而不是 web_search）
- 流程遗漏（如忘记归档就删除了文件）
- 效率低下（如手动重复操作，应该写脚本）
- context浪费（如一次性读了太多文件）
```

### Step 4：输出审计报告

按标准格式输出。如果有被忽视的技能或踩坑记录，必须输出完整格式。

### Step 5：写入 L2 记忆

如果当前工作区有 `memory/daily/` 目录，将审计摘要写入今日日记：

```markdown
## {HH:MM} 会话总结

- 调用技能：{列表}
- 被忽视的：{列表}
- 改进方向：{列表}
- 本次对话主题：{一句话概括}
```

如果当前是诊断对话（有 dbs-save 相关内容），同时在审计报告末尾提示：

```
💡 如需存档本次诊断状态，输入 /dbs-save
```

本技能属于以下技能组：

| 技能组 | 角色 | 触发位置 |
|--------|------|---------|
| **Core Loop** | 收尾（最后执行） | 对话结束前 |

### 联动行为

```
session-summary 触发时，检查：
1. 本次对话是否触发过 li-sync？→ 在审计中记录同步操作
2. 本次对话是否触发过 li-memory？→ 确认事件已写入 L2
3. 本次对话是否触发过 dbs-save？→ 在审计中记录存档路径
4. 本次对话是否触发过 scaffold-workspace？→ 输出完整构建报告而非普通审计
5. 本次对话有实质性教训？→ 提示调用 conversation-to-knowledge 固化教训
```

## 与 dbs-save 的区别

| 维度 | session-summary | dbs-save |
|------|----------------|----------|
| **触发时机** | 每次对话结束 | 用户说"存档"时 |
| **输出内容** | 技能审计 + 改进建议 | 诊断状态快照 |
| **存储位置** | memory/daily/{日期}.md | ~/.dbs/sessions/{slug}/ |
| **用途** | 追踪技能使用效率 | 跨会话恢复诊断上下文 |
| **必触发？** | ✅ 每次对话必触发 | ❌ 用户主动触发 |

两者可以同时触发：session-summary 输出审计报告后，如果用户说"存档"，dbs-save 会额外保存诊断快照。

## 改进周期自动检查（R09 联动）

> 每次对话**开始时**读取 tracker 判断是否需要审计，**结束时**输出审计结果。

### 检查逻辑（每日首次对话自动审计）

```
Step 1: 读取 improvement-cycle-tracker.md（路径：E:\ai产出文件\牛马\mutual\mutual\improvement-cycle-tracker.md）
  ↓
Step 2: 确定当前工作区（从对话上下文判断：创作/竞赛/求职/个人/mutual）
  ↓
Step 3: 查该工作区的「上次审计日期」
  IF 上次审计日期 < 今天 → 触发该工作区的增量审计
  IF 上次审计日期 = 今天 → 跳过
  ↓
Step 4: 如触发 → 调用 audit-optimizer skill 执行增量审计
  ↓
Step 5: 审计完成 → 更新该工作区的「上次审计日期」为今天
  ↓
Step 6: 输出「📅 改进周期检查」段（已在上方标准格式中定义）
```

### 增量审计自动执行

1. 调用 `audit-optimizer` skill（Phase 0-5，仅限当前工作区范围）
2. 扫描本次对话产生的自我进化文件
3. 查 R09 映射表 → 找到受影响的 SOP
4. 如有需要修复的问题 → 备份后修复 → Read 验证
5. 更新 improvement-cycle-tracker.md 该工作区的「上次审计日期」
6. 审计结果追加到当天的 memory 文件

## 铁律

1. **不可跳过**：每次对话结束前必须输出 📋 技能审计。没有例外。这不是"建议"，是铁律。
2. **不打断**：在用户对话过程中不触发，只在收尾时触发
3. **不鼓励**：审计报告是事实记录，不加感叹号或鼓励语
4. **写入记忆**：审计摘要必须写入 L2（daily note），确保跨会话可追溯
5. **技能组联动**：审计时必须检查技能组的执行完整性
6. **违反即失败**：如果一次对话结束时没有输出 📋 技能审计 = 本次对话质量不合格
7. **R10 技能审计铁律**：审计时必须检查本次对话是否遵守了以下规则：
   - 写作前是否读了约束文件？
   - 输出后是否检查了标点符号？
   - Write 后是否 Read 验证？
   - 如果违反了任何一条，必须在「下次改进」中明确指出

---

## 为什么这个规则不可跳过

> 创作区真实案例（2026-05-10）：AI做完新闻稿写作任务，21个约束文件一个都没读，英文引号反复出现，声称修复但没验证。如果有 📋 技能审计，"被忽视的：21个写作约束文件"就会立即暴露问题。
>
> 技能审计不是走形式。它是发现"该用没用"的唯一机制。没有审计 = 盲飞。
