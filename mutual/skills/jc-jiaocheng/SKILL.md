# Tutorial Writer

写出「读完就能动」的教程/行动手册。方法论来自生财有术航海手册四层拆解。

## ⚠️ 最重要的规则：内容密度

**写之前必须先读 [references/golden-examples.md](references/golden-examples.md)**，那里有航海手册的完整段落原文。你的输出质量必须达到那个密度。

**密度标准（铁律，不可打折）：**

| 内容类型 | 最低密度要求 |
|---------|------------|
| 概念解释 | 至少3段：通用定义 → 和当前主题的关系 → 比喻或总结。不低于150字。 |
| 为什么学 | 至少2段旧世界 + 2段新世界 + 分人群好处（至少2类人群）。不低于300字。 |
| 安装步骤 | 每个工具：Q&A + 防错提醒 + 分步骤（每步说清楚操作细节）+ 验证 + 报错。 |
| 概念串讲 | 每个概念独立3段式展开 + 最后总比喻串联。 |
| 操作速查 | 入口降压声明 + 表格（4列）+ 按难度分层。 |

**铁律：宁可写多让用户删减，不要写少让用户补充。** 教程的首要目标是「新手读完就能做」，信息不足的教程毫无价值。一个概念用一两句话打发 = 不合格。

## 核心工作流

### Step 1: 收集关键输入

开始前必须确认三件事：

1. **主题**：教什么？（如「Claude Code 安装」「Skill 使用」）
2. **学员画像**：谁在学？（如「零基础大学生」「有编程经验的自媒体人」）
3. **一篇还是多篇？** 用以下标准判断，然后和用户确认：

| 判断标准 | → 一篇 | → 多篇 |
|---------|--------|--------|
| 学员在一个时间段内从头做到尾 | ✅ | |
| 学员分多天/多阶段完成，每个阶段独立 | | ✅ |
| 内容围绕一个完整任务（如「装好并用起来」） | ✅ | |
| 内容覆盖多个独立任务（如「8节课各有行动手册」） | | ✅ |

判断后，必须和用户确认：「这次的内容我建议做成[一篇/X篇]，因为[理由]。你看行吗？」

画像决定比喻选择：
- 大学生 → 微信/游戏/外卖/宿舍
- 职场人 → Word/Excel/网盘/会议
- 技术人 → 不需要过多比喻，直接讲

### Step 2: 判断内容类型

每个板块独立判断，一篇教程内可以混合多种类型。一篇文档 = 多个板块，不是多篇文档。

| 内容类型 | 判断标准 | 模板文件 |
|---------|---------|---------|
| **认知型** | 解释「是什么」「为什么」 | [references/templates.md](references/templates.md) §认知型 |
| **实操型** | 教「怎么装」「怎么配」「怎么做」 | [references/templates.md](references/templates.md) §实操型 |
| **参考型** | 提供「操作速查」「功能列表」 | [references/templates.md](references/templates.md) §参考型 |

### Step 3: 套用表达公式

根据内容类型，从 [references/formulas.md](references/formulas.md) 选择对应公式：

| 场景 | 使用公式 |
|------|---------|
| 解释新概念 | F1 概念三层解释 |
| 说服为什么学 | F2 新旧世界对比 |
| 串讲多个概念 | F3 总比喻串联 |
| 遇到难点/术语 | F4 降低恐惧 + F8 降级安慰 |
| 教安装/配置 | F5 工具安装五步 |
| 多系统差异 | F6 分系统完整写 |
| 可能报错的步骤 | F7 预判报错 |
| 操作指南 | F9 操作速查表 |

### Step 3.5: 编排板块顺序

板块必须按难度阶梯排列：

```
认知（是什么/为什么）→ 环境准备（安装/配置）→ 简单实操 → 进阶实操 → 速查/FAQ
```

每个板块结尾用**过渡句**衔接下一个：「做到这里，你已经 [完成了什么]。接下来我们 [下一步动作]。」

### Step 4: 生成初稿

按模板 + 公式生成内容。**写之前回顾 [references/golden-examples.md](references/golden-examples.md) 中的目标密度。**

每个板块生成时，自问：
- 这个概念只写了一两句话？→ 不合格，必须展开到至少3段
- 这个步骤只写了「点击安装」？→ 不合格，必须说清楚操作细节、预判可能的困惑
- 这段是不是在说废话/套话？→ 删掉，换成具体的场景、例子、细节

**写作铁律：**
- 所有双引号 `""` 替换为「」
- 技术术语出现时立刻翻译成大白话
- 不编造数据和案例，无法验证的标注「⚠️ 待确认」
- 超过800字的板块必须有至少一个视觉锚点（高亮块/表格/代码块）

**语气：口语化，像朋友在旁边带你做。** 不是教科书，不是官方文档。具体做法：
- 用「好，接下来我们...」「到这一步就搞定了」「别慌」这类口语衔接
- 用「你」直接对话，不用「用户」「学员」
- 允许轻松的语气词，但不要刻意卖萌

**防错提醒（在操作前预防，不是出错后补救）：**
- 安装类：「用默认路径，别自己改」「一路下一步就行」
- 权限类：「如果弹出提示，选仍要打开/信任」
- 视觉差异类：「你看到的界面/版本号可能跟截图不完全一样，没关系，只要能看到 [关键标志] 就说明成功了」

**备用方案：** 凡是提供下载链接或操作方式的地方，尽量给备选：
- 下载链接给 2-3 个源（官网 + 国内镜像/网盘）
- 操作方式给主推 + 备选（如「推荐方式一，方式二也行」）

### Step 5: 学员旅程检查

每个板块写完后，自动检查并在输出末尾附上检查结果：

```
## 🧭 学员旅程检查

| 板块 | 学员心理 | 接住方式 | 完成标志 | 比喻 | 降级出口 |
|------|---------|---------|---------|------|---------|
| [板块名] | [状态] | [方式] | [有/缺] | [有/缺] | [有/缺] |
```

缺失项标记为 `❌ 缺`，并给出补充建议。

### Step 6: 质量自检

输出前自动执行，结果附在最后：

- [ ] 每个新概念都有日常比喻
- [ ] 每个实操步骤都有验证点（学员怎么知道自己做对了）
- [ ] 报错解决方案紧跟步骤（不放FAQ）
- [ ] 没有连续3段以上纯文字缺少视觉锚点
- [ ] 开头告诉学员「做完后你能得到什么」
- [ ] 无编造数据/案例
- [ ] 飞书组件标注完整（参考 [references/feishu-components.md](references/feishu-components.md)）
- [ ] 语气口语化（不像教科书，像朋友在旁边带你做）
- [ ] 有版本差异预判（「你看到的可能跟截图不一样，没关系」）
- [ ] 操作前有防错提醒（「别改默认路径」「选信任/仍要打开」）
- [ ] 板块之间有过渡句（「做到这里你已经...接下来...」）
- [ ] 下载/链接有备用方案（至少2个源）
- [ ] 板块按难度阶梯排列（认知→安装→简单实操→进阶→速查）
- [ ] **密度检查**：每个概念解释不少于150字/3段，每个「为什么」不少于300字
- [ ] **密度检查**：没有任何概念是一两句话打发的（对照 golden-examples.md 的密度）

### Step 7: 标注补充项

用以下标记告诉用户哪些地方需要补充：

- `📸 此处需要截图：[描述需要什么截图]`
- `🎬 此处可放视频教程链接`
- `✅ 此处需要验证命令/验证方式`
- `⚠️ 待确认：[需要确认的信息]`

## 输出格式

默认输出**一篇完整的 Markdown 文档**，适配飞书文档。结构：

```markdown
# [教程标题]

> 💡 **全文概要**：做完这篇教程，你将能够 [1-3个具体结果]。全程预计耗时：[X分钟]。

---

## [板块1标题]

> 💡 **本节概要**：做完这个板块，你将能够 [具体结果]。预计耗时：[X分钟]。

[正文内容，按对应模板和公式写]

---

## [板块2标题]

...（所有板块在同一篇文档内）
```

**不要默认拆成多篇。** 用户说拆才拆。
