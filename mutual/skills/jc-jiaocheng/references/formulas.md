# 9个表达公式

从生财有术航海手册拆解提炼，每个公式配用法说明和示例。

---

## F1: 概念三层解释

**结构**：一句话定义 → 标准定义 → 日常比喻

**用于**：第一次出现的新概念/术语

**示例**（解释 Claude Code）：

> **一句话**：Claude Code 是一个能用自然语言和你协作写代码的AI编程助手。
>
> **标准定义**：它基于 Anthropic 的大语言模型，在命令行中运行，能读懂整个项目、生成代码、修复bug、重构逻辑。
>
> **比喻**：如果说写代码像盖房子，Claude Code 就是一个你用中文指挥的包工头——你说「客厅要大一点」，它就帮你改图纸、搬砖、刷墙。

**要点**：
- 比喻必须用学员画像熟悉的事物
- 大学生：微信/外卖/游戏/宿舍/选课系统
- 职场人：Word/网盘/会议/报销
- 一句话定义不超过30字

## F2: 新旧世界对比

**结构**：旧世界的痛 → 新世界的爽 → 分人群说好处

**用于**：解释「为什么要学/用这个」

**示例**（为什么用 Claude Code）：

> **旧世界**：过去想做个小工具，要先学编程语法、搞环境配置，光入门就要几个月。很多人不是学不会，是这条路太陡了。
>
> **新世界**：现在你只需要用中文描述你想要什么，Claude Code 帮你写代码。核心能力从「记语法」变成「说清楚需求」。
>
> **分人群**：
> - 零基础的你：不用学语法也能做出能用的东西
> - 有点基础的你：效率翻倍，一天做完过去一周的活
> - 想创业的你：一个人就能搭出完整产品

**要点**：
- 旧世界要具体到「痛在哪」，不要抽象
- 新世界要具体到「爽在哪」，不要画饼
- 分人群不超过3类

## F3: 总比喻串联

**结构**：逐个解释概念 → 最后用一个场景把所有概念串起来

**用于**：一个板块里出现3个以上新概念

**示例**（串联终端/命令行/Node.js/Claude Code/VS Code/Git）：

> 把整个流程想象成一场演出：
> - **终端**是舞台
> - **命令行**是你在舞台上说的台词
> - **Node.js**是后台的发电机，提供动力
> - **Claude Code**是主演，负责理解你的需求、写代码
> - **VS Code**是化妆间，你在这里查看和修改代码
> - **Git**是录像师，记录每一次修改

**要点**：
- 总比喻必须是一个**完整场景**（演出/点外卖/打游戏），不能是零散的比喻
- 每个概念在场景里的角色要合理，不能强行对应

## F4: 降低恐惧

**结构**：先说「不重要/不用深学」→ 再讲内容

**用于**：出现可能吓到学员的内容（术语密集、技术底层知识）

**示例**：

> ⚠️ 以下是一些技术名词的解释，帮你在后面操作时有个印象就行，**不需要记住，更不需要深学**。遇到不懂的直接跳过，后面用到时再回来看。

**要点**：
- 放在内容**前面**，不是后面
- 用 ⚠️ 警告块让视觉上就能看到

---

## F5: 工具安装五步

**结构**：为什么要装（Q&A）→ 视频教程（可选）→ 分步骤图文 → 验证 → 常见报错

**用于**：任何需要学员安装/配置的步骤

**示例骨架**：

```markdown
### 安装 [工具名]

> 💡 **Q：为什么需要安装 [工具名]？**
> A：[一句话说清楚这个工具在整个流程里的角色]

🎬 视频教程：[链接]

**第一步：下载**
[具体操作]

**第二步：安装**
[具体操作]
📸 此处需要截图：[安装过程关键界面]

**第三步：验证是否安装成功**
在终端输入：
`[验证命令]`
看到 [什么结果] 就说明安装成功了。
📸 此处需要截图：[验证成功的终端界面]

> ⚠️ **常见问题**
> 如果出现 [报错信息]，执行 [解决命令] 即可。
> 📸 此处需要截图：[报错界面]
```

**要点**：
- Q&A 开头让学员知道「为什么」，不是无脑执行
- 验证步骤是**必须的**，不能省略
- 报错紧跟步骤，不放到最后的FAQ

## F6: 分系统完整写

**结构**：每个系统（Windows/Mac/Linux）完整写一遍所有步骤

**用于**：安装/配置涉及不同操作系统

**示例骨架**：

```markdown
#### Windows 版本

第一步：...
第二步：...
验证：...

#### Mac 版本

第一步：...
第二步：...
验证：...
```

**要点**：
- **不要写「Mac 类似」**——学员只看自己系统的部分，「类似」会让他们不确定
- 如果用户告知学员环境已统一（如都用Mac），则不需要分系统写

## F7: 预判报错

**结构**：报错信息 + 原因 + 解决方案，紧跟在可能出错的步骤后面

**用于**：任何可能出错的实操步骤

**示例**：

```markdown
**第二步：安装**

执行命令：`npm install -g @anthropic-ai/claude-code`

> ⚠️ 如果出现红色报错 `permission denied`：
> 这是因为没有管理员权限。在命令前加 `sudo` 再试一次：
> `sudo npm install -g @anthropic-ai/claude-code`
> 然后输入电脑开机密码（输入时屏幕不会显示，正常的，直接输完回车就行）。
```

**要点**：
- 报错信息要**具体**（写出报错文字），不要只说「如果报错」
- 解决方案要**直接给命令**，不要让学员自己去搜
- 「输入时不显示」这类细节必须预判——新手会以为键盘坏了

## F8: 降级安慰

**结构**：在难点前加一句「看不懂也没关系」+ 告诉学员什么时候会再遇到

**用于**：出现学员可能卡住/放弃的内容

**示例**：

> 如果这里没看懂，完全不需要担心。这些概念在后面的实操中会自然用到，到时候你一边做一边理解，比现在硬记有效得多。需要时随时回来翻这一节就行。

**要点**：
- 不是敷衍的「没关系」，要告诉学员**什么时候会再遇到**，让他知道跳过是安全的
- 放在内容**前面或旁边**，不是学员已经看完困惑了才说

## F9: 操作速查表

**结构**：表格（操作名/功能/场景/操作方法）+ 按难度分层

**用于**：功能介绍、操作指南、快捷键列表

**示例**：

```markdown
### 基础操作

| 操作 | 功能 | 什么时候用 | 怎么操作 |
|------|------|-----------|---------|
| 自动编辑模式 | 让 Claude 自动改代码，不用每次确认 | 写代码、修bug时 | Mac: Shift+Tab 一次 |
| @ 指定文件 | 告诉 Claude 只处理某个文件 | 修改单个文件时 | 输入 @文件名 |

### 进阶操作

| 操作 | 功能 | 什么时候用 | 怎么操作 |
|------|------|-----------|---------|
| Plan 模式 | 先想方案再写代码 | 复杂功能/项目起步时 | Shift+Tab 连按两次 |
```

**要点**：
- 按难度分层：入门 → 高频 → 进阶
- 学员不需要从头读到尾，按需查找
- 「什么时候用」比「功能」更重要——学员知道功能但不知道什么时候该用
