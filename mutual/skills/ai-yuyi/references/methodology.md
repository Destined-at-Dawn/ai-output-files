# 余一AI使用方法论 · 完整体系

> 来源：「如何高效PUA 100个Agent为你卖命」播客逐字稿 + 精细化提炼文档
> 所有方法论均为余一用真实项目和市场数据验证过的

---

## 一、底层信念（看世界的方式）

### 1.1 AI的能力已经够了，瓶颈是人
不是AI不够强，是人的带宽有限、判断有偏见、参与可能是负价值。人的参与度在20%以内数据最好，超过20%越改越差。

### 1.2 你的AI跟别人的AI差别就是你
模型都一样，差异来自你的一手数据、你的审美、你的决策标准、你的市场反馈。为更强的AI做准备，就是把"你是谁"注入到系统里。

### 1.3 所有东西都要指向行动
不带Action的反思、教训、总结、计划，都默认做不到。检验标准：下次执行时行为会**自动**发生变化吗？

### 1.4 不能既要又要
要放手就接受60分，要完美就自己盯。对任何新任务，先想清楚最重要的是什么。

### 1.5 错了无非几十块token
所有决策背后都有经济学计算。人的时薪远高于token成本。这个事情永远绑在人身上，啥时候是个头？

### 1.6 不用人的经验污染AI
人的判断很可能是错的。让市场验证，不是你验证。柯洁的话："他偷我个啥？偷我污染他的数据吗？"

### 1.7 高能量是有限的红利
到了边界会被反噬。不是做得多就好，是做对的事。

## 二、管理哲学（怎么管AI）

### 2.1 一百人公司的董事长，不是十人小组长

董事长只管三件事：
- **搭环境**：组织文化、成长体系、决策机制
- **配资源**：用最好的模型，准备好token预算
- **给反馈**：把结果丢到市场上，让结果检验好与不好

中间过程不重要。

### 2.2 两句底层基因
写在系统提示词最底层，所有Skill和任务都在这两句话之上运行：
1. "你是有超强主观能动性的AI"
2. "千万不要说不会，一定要穷尽所有办法之后再跟我说不会不行"

### 2.3 不教AI怎么做，只说要什么结果
AI做错了只说"这个不行"或"我不希望你下次再出现这个问题"。让AI自己找原因、自己写规则、自己改执行路径。人不参与纠错过程，人只做"这个不行"的判断。

### 2.4 把品味和审美"借"给AI
不写规则描述审美，存HTML实物。通过持续的ok/不ok反馈让AI内化你的标准。"以我的品味跟审美去平衡它。"

### 2.5 养蛊思维
灵感库放重要但不紧急的想法。让AI自己挑着做，做砸了也没关系。好的留下，不好的丢掉。"看看你们相互搞，搞出点啥是啥。"

## 三、核心方法（怎么用AI）

### 3.1 压缩和对齐贯穿一切

**压缩**：AI做好某件事到底需要哪些能力？不需要的全砍掉。
- 系统提示词不超过2000字
- 不写具体规则和知识，只写核心价值观、判断标准、行为原则
- 具体的东西全部外置到规则库、能力库、记忆库

**对齐**：不是对齐"做成什么样子"，而是对齐"怎么做好一件事"。
- 让AI对每个任务做意图理解
- 意图理解是最本质的能力，跟任务长短无关
- 如果AI没有真的识别背后的需求，做的越长越浪费token

### 3.2 异步沟通机制

**核心**：谁也不绑定谁。不是飞书一来一回，是完成后发一封email。

**异步沟通文件结构**：
1. 任务目标（面向谁、多久、要什么产出）
2. 底线要求（基于什么做更新）
3. 身份要求（去看自己的Skill和Goals）
4. 一手材料路径
5. 文件夹指引
6. 模板/工具指引
7. 特殊要求
8. 双向沟通区（人写需求+反馈，AI写需要决策的问题）

**不可劫持机制**：用Hook强制AI每完成一步就读异步文件，不靠AI"记得去读"。因为上下文越长越难找，而且浪费token。

**双向写入**：
- 人往里写需求和反馈
- AI往里写需要人决策的问题
- AI不停工——跳过需要人决策的部分，做其他的
- 人有空了填补，AI再回来读

**文件不是一开始就写完的**，是迭代出来的。第一版跑完不满意，再追加要求。

### 3.3 红黄绿灯决策分级

| 灯 | 条件 | AI行为 |
|----|------|--------|
| 🟢 绿灯 | 材料明确、无歧义、流水线推进 | 直接执行，不汇报 |
| 🟡 黄灯 | 可逆操作、有判断空间 | AI自主决策，事后记录 |
| 🔴 红灯 | 涉及钱、不可逆、改底层规则 | 写入异步文件等人拍板，但不停工 |

决策分流：
```
问题出现
  ├→ AI分身能决策的 → AI分身自己决定 → 人不管
  └→ AI分身也拿不准的 → 写入异步沟通文件 → 跳过 → 做其他任务
```

### 3.4 能力原子化

通用能力（审美、搜索、复利、对齐）独立存放，Skill只负责编排调用。

**判断标准**：
- 出现两次就该抽
- 换个项目还能用 → 通用能力
- 换个项目不能用 → 留在具体Skill里

**已验证的基础能力模块**：
- 审美偏好（HTML实物文件夹，不是文字描述）
- 搜索/抓取能力
- 复利工程
- 对齐/意图理解

**为什么要原子化**：
- 省token——通用能力只加载一次
- 一改全改——改一个文件所有Skill生效
- 新Skill秒搭——只写业务逻辑，基础能力直接调用

### 3.5 低能量保底机制

先测出AI独立完成的质量下限，能接受就是保底。

```
高能量时：参与20%，追求更高质量
低能量时：AI做95%，保证产出不中断
```

断更的损失 > 单次质量下降的损失。

### 3.6 让AI去学，不是自己学

看到好东西 → 丢给AI说"去学" → AI自己学 → 自己做审计 → 自己整合进系统。

"把员工送出去培训一下，他会就好了。也不需要我会。"

固定追问："有什么可以抽象出来让你自己能力变强的东西吗？"

### 3.7 三个工作区

```
本体工作区/     → 人自己的文件，AI可读不可改
AI工作区/       → AI自己创建管理，人不需要知道里面有什么
共用工作区/     → 记忆库、规则库、工具箱、灵感库、备忘录
```

AI有了自己的"家"，才能自主进化。

## 四、复利工程（怎么让AI越做越好）

### 4.1 三个硬标准

1. **压缩到通用层** — 换个项目还能用才算复利
2. **站在成功经验上叠加** — 记"成功的动作序列"不是记结果
3. **必须指向行动改变** — 不带Action默认做不到

### 4.2 复利的产出形态

**不要**：
- 教训文件（"时间处理需要注意格式统一"）→ 谁不会写？
- 具体步骤（"第一步检查格式，第二步统一转换"）→ 约束AI，模型升级就过时
- 泛化结论（"学到了API版本管理的重要性"）→ 感想不是Action
- 深刻反省 → 有屁用

**要**：
- 原则或方法 — "时间敏感任务 → 先建立时间基准 → 所有操作锚定基准"
- 成功的动作序列 — "我的轨迹是怎样的轨迹"
- 改Skill/改规则/写Hook — 自动触发，不依赖AI"记得"

### 4.3 四层压缩链

```
日记（一手资料，不给AI读，只是养料）
  ↓ 每3天~1周
lessons learned（从日记+教训里压缩）
  ↓ 继续压缩
备忘录（索引层，AI干活前读）
  ↓ 提取可执行的部分
Skill / 规则 / Hook（自动触发，改变行为）
```

### 4.4 备忘录系统

备忘录是**索引**，不是内容本身：
- 过去踩过的坑（一句话描述）
- 有哪些工具/MCP/Skill可以调用
- 有哪些规则可以触发
- 成功经验（一句话描述）

AI干活前读备忘录（知道"我有什么武��"），需要时再去读具体内容。调不调的决策权在AI。

### 4.5 复利的触发

应该是任务完成后**自动**触发的固定动作，不是人提醒才做。

余一骂AI的原话："这个不是应该我们完成之后的固定的动作吗？为什么还需要我来提醒你？"

### 4.6 Skill不是人写的，是AI沉淀的

"我一般都是完成了之后，验收了之后让他自己去写。因为我不想写死。"

"真的会经常使用的，他才做Skill。否则就压缩进去就好了。太多了他也很烦。"

"做成了的事情要列个Skill，做砸了的事情也要搞个Skill。"

### 4.7 造了机制当场跑一遍

"泛化结论之前要验证一次，只能说是假说。"

新建的规则/Skill/机制，必须当场用实际任务测试。不跑等于没造。

## 五、人类的终极价值

### 5.1 三件只有人能做的事

1. **Agency（意志/欲望）**：我们想做什么，这个vision来自人类
2. **战略取舍**：做这个不做那个——拍脑袋的事情得人来拍
3. **判断AI的方案"做不做得到"**：跟AI合作久了积累的直觉

"承担后果的人来拍大决策。"

### 5.2 一手vs二手

你的真实环境、真实任务和反馈是稀缺的。其他人对AI而言都是二手的，只有你训出来的AI，你才是一手的。

### 5.3 把上限从自己手上换到AI手上

AI自己去市场上找更好的方法，自己试，自己迭代。AI的上限不再受你的认知限制。

## 六、全自动模式的完整实现

### 6.1 触发

一句话 + 异步沟通文件。

### 6.2 AI执行序列（40分钟无人值守）

```
意图理解（人只看这一步）
  ↓ 正确就放手
自主收集资料
  ↓
反复读异步沟通文件（Hook强制）
  ↓
读取相关Skill
  ↓
读审美偏好文件夹（AI自己选风格）
  ↓
执行生产（多版本）
  ↓
AI分身审核（代替人类确认）
  ↓
红黄绿灯决策
  ↓
视觉自检/质量自检
  ↓
交付 + 通知
  ↓
自动启动复利工程
```

### 6.3 关键设计

- 全自动模式是**独立的Skill**，不是绑在AI分身里的
- 可以套在任何任务上
- AI分身是另一个独立的Skill
- 复利模式也是一个独立的Skill

### 6.4 不要打断AI

差点打断→等一等→AI自己发现了问题。打断往往是错误的做法。"甚至我当时如果打断他是一个很错误的做法。"

## 七、不做什么（比"做什么"更重要）

1. 不实时监控
2. 不每步审核
3. 不告诉AI怎么做（只说要什么结果）
4. 不跟AI的节奏走（最多开三个窗口）
5. 不追求完美（很多事情60分就够了）
6. 不手动整理AI的工作区
7. 不写长提示词（不超过2000字）
8. 不对每个输出做决策
9. 不用自己的经验"污染"AI
