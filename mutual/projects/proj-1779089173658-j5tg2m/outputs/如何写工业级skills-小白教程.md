# 如何写工业级 Skills：一个踩了 9 个坑的人手把手教你

> 我用 4 天造了 44 个 skill，删了 9 个，弃用了 8 个。
> 这篇教程是从那堆"废墟"里提炼出来的。
> 如果你是 AI 小白，读完这篇就能造出能用的 skill。

---

## 0. 先搞清楚一个问题：skill 到底是什么？

**一句话**：skill = 一段写给 AI 的"工作手册"，告诉它"遇到 X 场景该怎么做"。

你可以把它想象成——你雇了一个新员工，skill 就是你写的 SOP 文件。写得好，员工照着干就行；写得烂，员工要么瞎干，要么站着不动。

**好 skill 和烂 skill 的区别**：

| | 烂 skill | 工业级 skill |
|---|---|---|
| 比喻 | 一张便利贴："把事情做好" | 一本带检查清单的操作手册 |
| AI 表现 | 要么跳步骤，要么反复问你 | 拿到任务直接执行，每步有依据 |
| 你能得到 | "我觉得应该这样……" | "根据 references/case-studies.md 第 12 行的案例……" |

## 1. 写 skill 之前的三件事（别急着动手）

### 1.1 先搜再建——别闭门造车

**铁律：写任何 skill 之前，先搜有没有现成的。**

我犯过的错：花 2 小时写了一个"逐字稿清洗"skill，写完才发现社区里有个 `transcript-cleaner` 已经做了 80% 的工作。我只需要改个名字、加上自己的框架就行。

**搜索三件套**：
1. **本地已装的 skill**（`li-local-search`）——看看你手里有什么牌
2. **GitHub / skills.sh**（`li-bestskill`）——看看别人造了什么轮子
3. **MCP Registry**——看看有没有现成的工具能直接用

> 🧠 认知科学原理：**确认偏误**（《思考快与慢》§概念4）——你脑子里已经有了"我要写一个 XX skill"的念头，就会自动忽略已有的同类工具。先搜再建 = 强制打断这个偏误。

### 1.2 想清楚这个 skill 解决什么问题

问自己三个问题：
- **谁用？** 你自己用 / 团队用 / 公开发布？
- **什么时候触发？** 用户说什么话的时候该激活它？
- **产出是什么？** 一份报告 / 一段代码 / 一个决策？

如果你答不上来，说明你还没想清楚，别动手。

### 1.3 确认这个 skill 不是"伪需求"

**检验方法**：想象你已经有了这个 skill——你会每周用它 ≥2 次吗？

- 会 → 值得做
- 不会 → 大概率是伪需求，别浪费时间

## 2. Skill 文件长什么样（最小可用结构）

一个 skill 就是一个文件夹，里面至少有一个 `SKILL.md`：

```
my-skill/
├── SKILL.md              ← 主文件（必须有）
└── references/           ← 参考资料（按需）
    ├── case-studies.md   ← 案例库
    └── anti-patterns.md  ← 禁止清单
```

### 2.1 SKILL.md 的骨架（直接抄这个模板）

```markdown
# my-skill

## 这个 skill 干什么
一句话说清楚。不要写"本技能旨在为用户提供全方位的……"——废话。

## 触发条件
什么情况下该用这个 skill？列 3-5 个具体场景。

## 执行流程
Step 1: 做什么 → 产出什么
Step 2: 做什么 → 产出什么
Step 3: 做什么 → 产出什么
（每步必须有明确的产出，不能是"分析一下"这种模糊指令）

## 禁止清单
- ❌ 不准做什么
- ❌ 不准做什么
（这比"应该做什么"更重要——AI 天生爱自由发挥，你得给它画红线）

## 产出格式
最终输出长什么样？给一个具体的模板或示例。
```

### 2.2 一个真实案例：li-transcript 的演化

我来讲一个 skill 从"能用"到"好用"的真实过程：

**第 1 版**（原始形态）：`transcript-cleaner`
- 功能：ASR 转文字后去口头禅、修正术语
- 问题：只能"清洗"，不能"分析"

**第 2 版**（重命名 + 注入框架）：`li-transcript`
- 改了名字，纳入 li- 统一体系
- 注入了"道法术器"四层分析框架
- 从"洗衣服"升级成"洗衣服 + 分析面料成分"

**第 3 版**（加 reference）：
- 增加了 5 个参考文件：案例库、反模式、科学理论、风格指南、术语词典
- AI 执行时有据可查，不再凭感觉

**第 4 版**（瘦身）：
- 52 个文件 1240KB → 15 个文件 189KB
- 砍掉了预览图、开发文档、冗余词典
- **教训：文件多 ≠ 质量高，少而精 > 多而杂**

## 3. 工业级 skill 的 10 项质量门禁

这是从 225 个 skill 中提炼出来的"及格线"。你的 skill 至少要过 7 项才算能用：

| # | 门禁 | 什么意思 | 反面教材 |
|---|------|---------|---------|
| 1 | **核心公理** | 这个 skill 的 1-3 条不可违反的原则 | 没有原则 = AI 随便发挥 |
| 2 | **案例库** | ≥3 个真实案例，告诉 AI "长这样才是对的" | 没案例 = AI 编造（我删的 9 个 skill 就是这么来的） |
| 3 | **对话机制** | 什么时候该问用户，什么时候自己决定 | 不问 = 可能跑偏；全问 = 烦死人 |
| 4 | **禁止清单** | 明确列出"不准做什么" | 没红线 = AI 越线你都不知道 |
| 5 | **条件式下一步** | "如果 A 则 X，如果 B 则 Y"的分支逻辑 | 没分支 = 一刀切，遇到意外就卡住 |
| 6 | **信号追踪** | 怎么判断执行成功还是失败 | 没信号 = 做完了不知道对不对 |
| 7 | **认知科学引用** | 为什么这样设计？有什么理论支撑？ | 没依据 = "我觉得这样好" |
| 8 | **联动技能表** | 和哪些其他 skill 配合使用 | 孤岛 skill = 用一次就扔 |
| 9 | **反模式表** | 哪些做法看起来对但实际是坑 | 没反模式 = 新手必踩坑 |
| 10 | **触发词覆盖** | ≥15 个触发词（中文 + 英文） | 触发词少 = 用户说了需求但 skill 没被激活 |

> **我的惨痛教训**：li-transcript 有 231 行代码，但质量评分只有 2/10。为什么？因为它只过了门禁 1 和 7，其他 8 项全缺。**行数多 ≠ 质量高。**

## 4. 架构设计：三个关键原则

### 4.1 Progressive Disclosure（渐进式披露）

**原则：主文件 ≤300 行，细节放 references/ 按需加载。**

为什么？因为 AI 的上下文是有限的。你把 1000 行全塞进 SKILL.md，AI 读了一半就忘了前面的。

```
✅ 正确做法：
SKILL.md (200行) → 只放流程骨架
references/case-studies.md (500行) → AI 需要案例时才读
references/anti-patterns.md (300行) → AI 需要避坑时才读

❌ 错误做法：
SKILL.md (1500行) → 全塞进去，AI 上下文爆炸
```

### 4.2 Tier 三层架构

把你的 skill 生态分成三层：

```
Tier 1（核心引擎）  ← 5 个最重要的 skill，解决 80% 的问题
  ├── li-analyze    ← 深度分析
  ├── li-research   ← 深度研究
  ├── li-create     ← 内容创作
  ├── li-manage     ← 管理协调
  └── li-improve    ← 自我进化

Tier 2（能力模块）  ← 30+ 个专项 skill，各管一摊
  ├── li-transcript ← 逐字稿
  ├── li-xhs        ← 小红书
  └── ...

Tier 3（SOP 编排）  ← li-intent，自动路由用户需求到正确的 skill
```

**为什么分层？** 因为你不可能把所有功能塞进一个 skill。分层 = 每个 skill 职责清晰、可独立维护。

### 4.3 Progressive Disclosure 的反面：什么不该拆

不是所有东西都要拆成独立 skill。如果你的"skill"只有 7 行代码，它大概率不值得独立存在——要么合入父 skill，要么作为 reference。

> **教训**：我曾经把一个 7 行的 `zoom-out` 技能独立出来，结果它从来没被触发过。后来合入 `li-analyze` 作为一个子功能，反而用得多了。

## 5. 最容易踩的 3 个坑（血泪教训）

### 坑 1：做了不连接

**现象**：skill 写好了，SKILL.md 漂亮得很，但路由表里没注册 → 用户永远不会触发它。

**我犯了 6 次。** 不是 1 次，是 6 次。

**正确做法**：skill 创建后，必须立即做以下三件事：
1. 在 `skill-routing-table.json` 注册路由（触发词 ≥15 个）
2. 更新父级 skill 的索引
3. 同步到所有工作区

### 坑 2：扫表面不扫深层

**现象**：只看了 skill 的顶层目录就说"质量不错"，漏掉了深层文件里的问题。

**正确做法**：审计一个 skill 时，必须读完所有文件，不能只看 SKILL.md。

### 坑 3：批量创建 = 低质量

**现象**：一次性创建 10 个 skill，每个都是模板复制粘贴，质量远差于单点深度迭代。

**正确做法**：一次只做 1 个 skill，做到 7/10 质量门禁过了再做下一个。

> 🧠 认知科学原理：**沉没成本谬误**（《思考快与慢》§概念4）——"已经做了 5 个了，再做 5 个就能收工了"。不，那 5 个烂 skill 以后每个都会浪费你的时间。

## 6. 从"能用"到"好用"的迭代路线

```
v0.1  最小可用版（骨架 + 3 个步骤 + 1 个禁止项）
  ↓
v0.5  加案例库（≥3 个真实案例）+ 触发词（≥15 个）
  ↓
v1.0  过 7/10 质量门禁 + 注册路由 + 同步工作区
  ↓
v1.5  加 references/（反模式、术语表、科学理论）
  ↓
v2.0  被真实使用 ≥10 次后，根据反馈迭代
```

**关键**：每个版本之间必须有真实使用反馈。没用过的 skill = 没验证过的设计 = 纸上谈兵。

## 7. 速查清单：创建一个 skill 的完整流程

```
□ Step 0: 搜现有 skill（li-local-search → li-bestskill → GitHub）
□ Step 1: 确认不是伪需求（每周会用 ≥2 次？）
□ Step 2: 写 SKILL.md 骨架（≤300 行，用上面的模板）
□ Step 3: 加 ≥3 个真实案例到 references/
□ Step 4: 写禁止清单（≥3 条红线）
□ Step 5: 过质量门禁（至少 7/10）
□ Step 6: 注册路由（触发词 ≥15 个）
□ Step 7: 同步到所有工作区
□ Step 8: 真实使用 ≥3 次，记录反馈
□ Step 9: 根据反馈迭代
```

## 8. 给小白的最后一句话

写 skill 不是写代码，是写"工作手册"。

你不需要会编程。你需要的是：
1. **知道这个任务该怎么做**（你自己做过 ≥3 次）
2. **能把步骤写清楚**（让一个陌生人照着做能做对）
3. **知道哪里容易出错**（把坑提前标出来）

如果你自己都没做过这个任务，就别写 skill——先手动做几次，找到手感，再把它固化成 skill。

> **一句话总结**：工业级 skill = 搜到已有的 + 写清骨架 + 加真实案例 + 标红线 + 注册路由 + 用过再迭代。就这么简单，但每一步都不能跳。
