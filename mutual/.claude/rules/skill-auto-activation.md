# 技能自动激活规则（Skill Auto-Activation v2.0 — 联动优先）

> 技能和 MCP 应该像空气一样——用户不需要知道"用哪个"，系统根据上下文自动调用。
> **v2.0 核心升级**：单技能调用 → 多技能联动链；被动路由 → 自我学习闭环。

---

## 核心原则

**用户说出来的是需求，不是技能名。路由是 AI 的责任，不是用户的。**

- 用户说"帮我想个标题" → 自动激活 li-xhs（标题策略）+ voice-dna（文风注入）
- 用户说"研究一下XX行业" → li-research（深度研究）+ li-analyze（道法术器）+ li-devil（预验尸）
- 用户说"用 xxx 技能"（用户主动说技能名）→ 说明路由失败，需要记录并修复路由表

**v2.0 新增三条铁律**：

1. **联动优先 > 单打独斗**：一个任务能调2个skill就不只调1个，除非任务极简单（<3步完成）
2. **探索性任务强制用技能**：用户自己不确定方向的任务，必须调技能（li-analyze/li-research/li-devil），禁止原生裸跑
3. **不满 = 学习信号**：用户对结果不满时，必须当轮记录+调整联动链，下次同类任务自动应用新链

---

## 路由优先级链（v2.0 — 三层决策）

```
收到用户消息
  │
  ▼
Layer 0: 技能路由表命中？（skill-routing-table.json）
  ├── 命中 → 检查该 skill 的 SKILL.md 中是否定义了联动链
  │         ├── 有联动链 → 按链调度（主skill + 辅助skill）
  │         └── 无联动链 → 单技能执行，但仍检查 Layer 1 是否有增强
  │
  ▼（Layer 0 未命中）
Layer 1: SOP 索引命中？（li-intent Phase 0 扫描工作区 SOP）
  ├── 命中 → SOP 中定义了哪些技能 → 按 SOP 流程调度
  │
  ▼（Layer 1 也未命中）
Layer 2: 触发词语义匹配（li-local-search Layer 2 语义搜索）
  ├── 命中 → 推荐 skill + 检查是否需要联动
  │
  ▼（全部未命中）
Layer 3: 任务复杂度判断
  ├── 简单任务（<3步，明确答案）→ 原生能力处理（唯一允许不用skill的情况）
  └── 探索性/复杂任务 → 强制调用 li-analyze 或 li-research（禁止原生裸跑）
```

**优先级排序**：技能路由(Layer 0) > SOP(Layer 1) > 触发词(Layer 2) > 原生能力(Layer 3)

**关键**：不问用户确认。技能调用是内部决策，不是外部确认点。用户要的是结果，不是过程透明。

## 联动链机制（v2.0 核心新增）

### 什么是联动链

一个任务不只触发一个 skill，而是触发一条「主skill + 辅助skill」的协作链。

### 预定义联动链

| 任务类型 | 主 Skill | 辅助 Skill（自动联动） | 联动逻辑 |
|----------|---------|----------------------|----------|
| 内容创作 | li-xhs / blog-post-writer | voice-dna-auto-inject + li-devil(自检) | 写完 → 文风检查 → 泼冷水审视 |
| 深度研究 | li-research | li-analyze(道法术器) + li-devil(预验尸) | 研究 → 结构化 → 质疑 |
| 商业分析 | li-dbs | li-analyze + li-industry + li-devil | 战略 → 行业 → 挑战 |
| 学习新概念 | li-study | li-analyze(认知框架) + li-memory(沉淀) | 学 → 深化 → 存 |
| 任务规划 | li-plan | li-devil(预验尸) + li-manage(归档) | 规划 → 质疑 → 落地 |
| Skill 创建 | li-skillcreate | li-local-search(查重) + li-skills-mgmt(审核) | 创建 → 查重 → 审核 |
| 数字资产 | li-asset | li-xhs + blog-post-writer + li-storyboard | 采集 → 多平台分发 |
| 系统诊断 | li-diagnose | li-improve(教训) + li-infra(修复) | 诊断 → 记录 → 修复 |
| 探索性问题 | li-analyze | li-research(补信息) + li-devil(质疑) | 分析 → 补盲区 → 反驳 |

### 联动调用协议

```
Step 1: 确定主 skill（路由表命中的那个）
Step 2: 查联动链表 → 确定辅助 skill
Step 3: 执行主 skill Phase 流程
Step 4: 主 skill 产出后 → 自动调用辅助 skill 对产出进行增强/质疑/归档
Step 5: 整合最终输出交付用户
```

**联动的前提**：辅助 skill 不能拖慢主流程超过50%。如果辅助 skill 需要大量额外输入，降级为"建议联动"而非"强制联动"。

### 动态联动（非预定义）

当预定义联动链没有覆盖当前任务时：

```
AI 自主判断：
  1. 这个任务的输出，谁能增强它？（增强型联动）
  2. 这个任务的输出，谁能质疑它？（质疑型联动）
  3. 这个任务的产出，谁能帮它归档？（归档型联动）

至少选 1 类联动。3 类都选 = 最高质量。
```

## 探索性任务强制用技能（v2.0 铁律）

### 判断标准：什么是探索性任务

| 信号 | 判定 |
|------|------|
| 用户说"帮我想想""看看怎么搞""研究一下" | 探索性 |
| 用户没给具体方向，只给了主题 | 探索性 |
| 任务没有明确的"正确答案" | 探索性 |
| 用户说"不确定""试试""探索" | 探索性 |
| 涉及决策（A还是B？做不做？） | 探索性 |

### 强制调用规则

探索性任务**禁止原生裸跑**，必须至少调用以下之一：

| 技能 | 何时用 |
|------|--------|
| li-analyze | 需要深度分析框架时（道法术器） |
| li-research | 需要信息补充时（深度研究） |
| li-devil | 需要质疑/风险评估时（泼冷水） |
| li-dbs | 涉及商业/行业时（深度商业战略） |
| li-diagnose | 涉及系统问题时（熵增诊断） |

**禁止**：用户给了一个开放性问题，AI 直接用原生知识回答。必须走技能 + 联动。

## 路由表位置

**主路由表**：`E:\ai产出文件\牛马\mutual\mutual\skill-routing-table.json`

**路由表结构**（v2.1，新增 linkage 字段）：
```json
{
  "version": "2.1",
  "routes": [
    {
      "id": "r001",
      "triggers": ["..."],
      "context": ["..."],
      "skill": "skill-name",
      "mcp": null,
      "priority": 1,
      "auto": true,
      "confidence": 0.9,
      "linkage": ["skill-b", "skill-c"],
      "note": "..."
    }
  ]
}
```

字段说明：
- **triggers**：触发词列表（任一匹配即触发）
- **context**：上下文条件（必须同时满足）
- **skill**：要激活的主技能名
- **linkage**：联动 skill 列表（主 skill 执行后自动调用）
- **priority**：优先级（数字越小越优先，冲突时高优先级胜出）
- **auto**：是否自动激活（false = 需要用户确认）
- **confidence**：路由置信度（低于 0.7 时需要确认）

执行约束：
- `confidence >= 0.7` 且 `auto=true`：直接激活，不问用户
- 多个候选命中：选 confidence 最高 + priority 最小的
- 技能调用后必须把实际结果更新到 `memory/skill-usage-log.md`

## 自动激活的场景分类（含联动链）

### 必须自动激活 + 联动（不问用户）

| 信号 | 主技能 | 联动技能 | 理由 |
|------|--------|---------|------|
| 用户说"小红书/帖子/标题" | li-xhs | voice-dna + li-devil(自检) | 内容创作三件套 |
| 用户说"写文章/写公众号" | blog-post-writer | voice-dna + li-devil(自检) | 写作联动质检 |
| 用户说"研究/调研/分析行业" | li-research | li-analyze + li-devil | 研究三件套 |
| 用户说"PDF/解析pdf" | pdf | data-analysis(如有数据) | 解析+分析 |
| 用户说"ppt/幻灯片/演示" | pptx / baoyu-slide-deck | li-design(视觉) | 演示+设计 |
| 用户说"表格/excel/数据" | xlsx / data-analysis | li-data(可视化) | 数据全链路 |
| 用户说"学习/理解/搞懂" | li-study | li-analyze + li-memory | 学→深化→存 |
| 用户说"规划/计划/安排" | li-plan | li-devil + li-manage | 规划→质疑→归档 |
| 用户分享链接/URL | baoyu-url-to-markdown | li-analyze(提炼) | 提取+分析 |
| 探索性问题（无明确方向） | li-analyze | li-research + li-devil | 分析→补盲区→质疑 |
| 对话即将结束且有重要决策 | li-memory | li-improve(教训) | 沉淀+复盘 |

### 单技能即可（简单任务）

| 信号 | 技能 | 不需要联动的原因 |
|------|------|----------------|
| 查天气 | weather-forecast | 事实查询，无需深度 |
| 翻译一段话 | deepl | 工具型，无需质疑 |
| 压缩图片 | baoyu-compress-image | 机械操作 |
| 格式化markdown | baoyu-format-markdown | 格式操作 |

## 自我学习机制（v2.0 核心升级）

### 学习触发：用户不满

**信号检测**（出现以下任一 = 触发学习）：
- 用户说"不对""不好""太浅""没用""不是我要的""再深入点"
- 用户重新描述需求（说明第一次理解错了）
- 用户手动说"用XX技能"（说明路由漏了）
- 用户说"还有呢""不够""再想想"（说明联动不足）

### 学习闭环（5步，当轮强制执行）

```
Step 1: 识别不满类型
  ├── 路由错误（调了不该调的skill）
  ├── 联动不足（该联动的没联动）
  ├── 深度不够（该用li-analyze/li-devil没用）
  └── 风格不对（该注入voice-dna没注入）

Step 2: 确定修正方案
  - 路由错误 → 调整触发词 confidence / 添加新触发词
  - 联动不足 → 更新联动链表（添加辅助skill）
  - 深度不够 → 该类任务的探索性评分 +1（更积极调深度skill）
  - 风格不对 → 检查voice-dna注入规则是否覆盖该场景

Step 3: 当轮应用修正（立即用新方案重做）

Step 4: 写入学习记录
  → memory/skill-usage-log.md 追加「学习事件」
  → 格式见下方

Step 5: 更新路由表（如需要）
  → Edit skill-routing-table.json
  → 调整 confidence / linkage / triggers
```

### 学习记录格式

```markdown
## [日期] 学习事件
- 用户原始需求：[原话]
- 首次路由：[调了什么skill]
- 不满信号：[用户说了什么]
- 不满类型：[路由错误/联动不足/深度不够/风格不对]
- 修正方案：[改了什么]
- 修正后结果：[用户满意/仍需迭代]
- 规则变更：[路由表/联动链是否更新]
```

### 自动优化规则

| 事件 | 动作 |
|------|------|
| 同一 skill 连续 3 次不满 | confidence 降 0.15，检查是否应该换主 skill |
| 用户纠正"不要用这个" | 立即降低该路由 confidence 0.2 |
| 用户主动说"用 xxx 技能" | 立即添加新路由 + 记录为路由失败 |
| 用户说"再深入/再想想" | 下次同类任务强制加 li-analyze + li-devil |
| 同类任务联动成功 3 次 | 联动链固化为预定义联动 |
| 技能连续 5 次满意 | confidence 升 0.05（上限 0.95） |

## 路由失败处理（v2.0 修订）

当无法确定该用哪个技能时：

1. **有 >=2 个候选**：选 confidence 最高的，同时联动次高的作为辅助
2. **没有候选但任务复杂**：强制调用 li-analyze（禁止原生裸跑）
3. **没有候选且任务简单**：原生能力处理（唯一例外）

**v2.0 铁律**：因为"不确定"就不调用任何技能 = 违规。探索性任务宁可调错也不要不调。

## MCP 自动调用规则

MCP 工具不需要"激活"，每次对话中都可用：

| MCP | 自动调用条件 | 说明 |
|-----|-------------|------|
| project-tools | 任务创建/查询、文件读写、项目状态 | 项目管理的默认工具 |
| skill-handler | 路由表匹配到具体技能时 | 技能调用入口 |
| ask-user | 需要用户做选择（>=2 个互斥选项）时 | 替代纯文本提问 |
| web-search | 涉及时效信息或外部事实时 | 实时搜索 |
| markitdown | 用户上传 Office/PDF 文件时 | 文件转换 |
