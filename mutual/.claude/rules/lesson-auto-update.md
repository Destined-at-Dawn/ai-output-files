# 教训闭环自动更新规则（Lesson Auto-Update — 最高优先级）

> 优先级：最高（与 skill-route-enforcement.md 同级）
> 创建：2026-06-11
> 根因：教训只记不改 = 白记。li-improve 记录了教训，但相关工作流文件没有跟着更新 → 下次还会犯。

---

## 铁律 1：用户纠正必须触发工作流更新

**触发**：用户说了"不对""不要""改成""你让我很失望""太单纯""扫表面不扫深层"等纠正信号。

**强制流程**：

```
Step 1: 识别教训来源和根因
Step 2: 分析影响范围（哪些 skill/SOP/CLAUDE.md/rules/ 需要改）
Step 3: 生成修改方案（列出所有需要修改的文件和具体内容）
Step 4: 归档原文件到 E:\ai产出文件\牛马\归档\{日期}-lesson-auto-update\
Step 5: 用 patch 逐个修改（不用 Write 覆盖）
Step 6: 每个修改后 Read 验证
Step 7: 输出修改总结
```

**不可跳过。没有"先记下来后面再改"的例外。**

---

## 铁律 2：事故必须触发规则更新

**触发**：发生了数据丢失、文件覆盖、错误断言、路由缺失等事故。

**强制流程**：

1. 检查事故是否违反了已有规则 → 违反 → 强化规则
2. 检查事故是否有对应规则 → 没有 → 创建新规则
3. **硬编码检查（Insight 2 — 2026-06-14 新增）**：事故解决后必问——「这个错误能不能变成一条硬编码规则？」
   - ✅ 能 → 立即写入对应规则文件（具体场景 + 具体禁止行为 + 具体触发条件）
   - ❌ 不能 → 说明为什么（太场景化/太罕见/已有上层规则覆盖）
4. 检查事故是否影响其他工作区 → 影响 → li-sync 同步

---

## 铁律 3：矛盾检测必须触发统一

**触发**：发现多个文档的数据不一致（如 6 个来源的 skill 数量不同）。

**强制流程**：

1. 识别所有矛盾的文件
2. 确定单一真相源（哪个文件是权威）
3. 更新所有非权威文件（与权威对齐）
4. 在权威文件中标注"统计口径"

---

## 铁律 4：修改必须用 patch 不用 Write

**原因**：Write 全文覆盖会丢失未读取的内容，导致数据丢失。

**正确做法**：
```python
# 用 patch 精确替换
old_string = "需要修改的旧内容"
new_string = "修改后的新内容"
patch(file_path, old_string, new_string)
```

**禁止**：
```python
# 用 Write 全文覆盖
write_file(file_path, new_content)  # ❌ 禁止
```

---

## 铁律 5：修改前必须归档

**原因**：防止修改失败后无法恢复。

**强制流程**：
```
修改文件前 → 复制原文件到 E:\ai产出文件\牛马\归档\{日期}-lesson-auto-update\
→ 确认归档成功 → 执行修改
```

---

## 铁律 6：审计必须全量扫描

> **来源**：2026-06-11 教训——只读 3 个输出文件就给建议 = "扫表面不扫深层"

**触发**：AI 要审计文档、检查状态、验证一致性时。

**强制流程**：

```
审计目标文档 → 必须同时读取：
1. 目标文档本身
2. 对应工作区的 CLAUDE.md
3. 对应工作区的 SOP 总索引
4. 对应工作区的 memory/long-term.md
5. 对应工作区的 memory/{今天日期}.md
6. skill-routing-table.json
7. .claude/rules/ 下的相关规则
8. 文件系统实际状态（ls/find 验证）
```

**验证工具层级**：文件系统 > memory > 输出文档（文档可能过时）

**违反信号**：
- 只读了输出文档就给建议
- 用户说"太单纯""扫表面不扫深层"
- 发现多个来源数据不一致（说明没读够）

## 违规检测信号

如果出现以下任何情况，说明本规则被违反了：

- [ ] 用户纠正后只写 memory 没更新工作流文件
- [ ] 事故后只记录教训没更新规则
- [ ] 矛盾检测后只标注没统一
- [ ] 用 Write 覆盖了已有文件
- [ ] 修改前没有归档
