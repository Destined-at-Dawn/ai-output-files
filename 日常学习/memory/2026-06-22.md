## 三、关键设计

### 两层产出体系

```
projects/{课程}/outputs/   ← 工作台（中间产物、草稿、过程文件）
outputs/{日期}-{任务名}/   ← 成果库（终态产物 + 心得 + SOP 改进日志）
```

### 收尾五步

```
Step 1: 产出提炼 → 从项目 outputs/ 筛选最终产物
Step 2: 心得沉淀 → 写心得体会.md
Step 3: SOP改进  → 写SOP改进日志.md
Step 4: 索引更新 → 更新 outputs/_INDEX.md
Step 5: mark_task_done → 填写 output_files 参数
```

### 质量门禁

- mark_task_done 是收尾的**最后一步**，不是第一步
- 不填写 `output_files` 参数 = 违规
- output_files 必须包含：产物 + 心得体会 + SOP 改进日志

## 四、li-diagnose 诊断结论

### 熵源分析

| 熵源 | 本质 | 根治方式 |
|------|------|---------|
| **边界熵**（主导） | 系统边界按"预期功能"画，不按"实际内容流"画 | 从"设计驱动"改为"实际驱动" |
| **信息熵** | 每个目录的"为什么存在"没有被文档化 | 建立目录生命周期规则 |
| **结构熵** | 新目录不断追加，旧的不验证是否还有效 | 建立目录健康审计节律 |

### 核心洞察

> 目录结构是一次性"设计"出来的，不是从实际内容流中"生长"出来的。
>
> 设计时画了"理论上应该有内容的地方"，但没有验证"实际上有没有内容在那里流动"。

## 五、教训沉淀

### 教训：用户要根治，不要治标

- **场景**：用户问"文件夹结构优化"，我给出"删除空目录"的建议
- **用户反馈**："你不是解决问题，你是要根治问题"
- **正确做法**：用 li-diagnose 六熵源框架找根因，设计系统性解决方案

### 教训：没用 li 技能 = 失望

- **场景**：用户问"文件夹结构优化"，我直接分析没调用 li-diagnose
- **用户反馈**："你没有使用/li，我很失望"
- **正确做法**：路由表没命中是路由漏点，需要补上触发词
