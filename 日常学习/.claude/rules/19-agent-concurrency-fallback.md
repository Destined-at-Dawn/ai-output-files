# R19 — Agent 并发降级协议（真源 v1.0）

> **核心原则：完成任务 > 追求并发速度。** 并发是锦上添花，不是必需品。429 限流 = 并发能力被收回，此时必须立刻切换执行策略，而不是等待恢复。

---

## 一、问题根因

并发 Agent 调用（一次消息中发送多个 Agent tool_use）会瞬间消耗大量 API 配额。当配额不足时：

- 所有并发 Agent 同时返回 429
- 重试 = 再次消耗配额 = 再次 429（恶性循环）
- **一整轮对话被浪费**，零产出

这是**执行策略错误**，不是技术故障。等待重试 = 用错误策略反复撞墙。

---

## 二、降级协议（3 步，强制）

### Step 1：识别 429 信号

Agent tool 调用返回以下任一情况时，**立即触发降级**：

| 信号 | 表现 |
|------|------|
| 429 错误 | HTTP 429 / "rate limit" / "too many requests" |
| Agent 超时 | Agent 调用超过 30 秒无响应 |
| 部分失败 | 多个并发 Agent 中任意一个返回错误 |

### Step 2：立即降级（零等待）

**发现 429 后，必须在同一轮响应中立即执行：**

1. **放弃**所有未返回的 Agent 调用（不等待、不重试）
2. **切换**为顺序直接工具调用（Read / Grep / Bash / Write 等）
3. **保持**任务目标不变——只是执行方式变了

**降级后的执行模式：**

```
❌ 并发模式（已失效）：
  Agent(A) + Agent(B) + Agent(C) → 全部 429

✅ 降级模式（立即切换）：
  直接工具调用 A → 直接工具调用 B → 直接工具调用 C
  （顺序执行，稳定可靠）
```

### Step 3：记录 + 审计

降级后必须在本轮对话结束前：

1. 技能审计中明确标注 `⚠️ 并发降级：Agent 429 → 顺序直接调用`
2. 如本轮有 memory 写入，记录降级事件

---

## 三、铁律（不可违反）

| 铁律 | 内容 | 违反后果 |
|------|------|---------|
| **禁止重试** | Agent 返回 429 后，禁止在同一次响应中再次调用 Agent | 重试 = 再次浪费一轮 |
| **禁止等待** | 不得输出"等待重试"/"稍后再试"等无效响应 | 等待 = 用户看到 AI 在发呆 |
| **禁止降级遗忘** | 降级后仍必须完成原始任务，不能因为"没用 Agent"就降低产出质量 | 降级是手段变化，不是标准降低 |
| **禁止隐匿** | 必须在审计中如实报告降级事件 | 隐匿 = 违反 F1（做了才说） |

## 四、决策树（每次想用并发 Agent 时必过）

```
需要多个信息源？
  ├─ 信息量小（<50行/文件）→ 直接顺序 Read，不用 Agent
  ├─ 信息量中等 → 单个 Agent（Explore）+ 直接工具辅助
  └─ 信息量大且确实需要并发 → 尝试并发 Agent
       ├─ 成功 → 正常继续
       └─ 任一返回 429 → 立即降级为顺序直接调用
```

**关键判断**：大多数信息收集任务用顺序直接调用就能在 3-5 步内完成。只有真正的跨目录大规模扫描才需要 Agent 并发。
