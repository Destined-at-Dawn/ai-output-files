# li-skillfusion 黄金规则

## GR-001：融合前先归档
融合两个 skill 前必须先 git commit + 归档目录备份。
- 来源：git-recovery 检查点规则
- 反模式：直接改文件，发现改错了无法恢复

## GR-002：功能重叠>80%才融合
重叠不到 80% → 保持独立，只更新交叉引用。
- 来源：li-skillfusion 融合决策矩阵
- 反模式：把所有相似 skill 都合到一个大杂烩里

## GR-003：弃用旧名必须全量替换
弃用 skill 名后必须 grep 全生态替换引用。
- 来源：2369 处断裂引用修复事件
- 反模式：改了文件名但 SKILL.md 里还引用旧名

## GR-004：拆分比融合更难
融合前想清楚：如果将来要拆，代价多大？
- 来源：损失厌恶（#003《思考，快与慢》）
- 反模式：合了 5 个 skill 发现太大，拆不回去了

## GR-005：融合后必须更新路由
融合完成后立即更新路由表 + 同步所有工作区。
- 来源：路由同步教训
- 反模式：skill 融合了但路由表还指向旧名
