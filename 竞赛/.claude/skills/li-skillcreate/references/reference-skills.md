# 融合的参考 Skill 完整清单

> 本文件记录 li-skillcreate 融合的所有参考技能、提取的核心机制、以及每个机制的设计原理。
> 来源：225 个本地 skill 全量扫描 + GitHub/skills.sh 联网搜索。

---

## 核心参考（深度提取，每次创建必参考）

### 1. skill-creator（本地）
**核心机制**：Progressive Disclosure（渐进披露）
**为什么好**：SKILL.md 只放执行骨架，详细参考放 references/——符合认知负荷理论（T2），AI 上下文窗口是稀缺资源
**提取的设计决策**：主文件 ≤ 300 行；Phase 用"详见 references/xxx.md"链接
**反面案例**：li-transcript 2039 行单文件 = Progressive Disclosure 的反面教材

### 2. dbs-diagnosis（本地）
**核心机制**：问题消解五层漏斗
**为什么好**：90% 的"造技能"需求是假需求——要么已有技能能解决，要么描述模糊到无法执行
**提取的设计决策**：Phase 0 必须先消解（语言陷阱/假设错误/发动机空转/好问题五项检查/量化评分）
**反面案例**：收到"造个工具"就直接开始 Phase 1 = 制造一个解决假问题的技能

### 3. jc-clarifier（本地）
**核心机制**：行动锁（镜像复述 → 用户确认 → 才能执行）
**为什么好**：防止 AI 理解偏差导致整轮工作白费
**提取的设计决策**：Phase 2 必须输出镜像复述，用户确认后才进入 Phase 3
**反面案例**：AI 理解成"造天气技能"但用户要的是"造天气推文生成器"，30 分钟白干

### 4. li-devil（本地）
**核心机制**：触发词从对话中长出来
**为什么好**：触发词不是猜的——从用户真实对话中采集，1 周无命中就调整
**提取的设计决策**：初始触发词来自对话记录 → 使用 1 周 → 无命中率 > 50% 则替换
**反面案例**：触发词写"技能创建"但用户实际说"帮我造一个"

### 5. deep-research v4（本地）
**核心机制**：一鱼多吃（每次调研更新 7 个生态文件）
**为什么好**：技能不是孤岛——不更新生态 = 磁盘垃圾
**提取的设计决策**：Phase 5 强制 ≥ 5 个生态副作用
**反面案例**：skill 创建了但路由表/记忆/画像全没更新 → AI 不知道它存在

## 增强参考（提取部分机制）

### 6. conversation-to-knowledge
**提取的机制**：教训四分类法（偏好/教训/决策/知识）+ 去重机制
**在技能创建中**：迭代时发现的教训必须分类写入 golden_rules.md，不是口头流传

### 7. output-skill
**提取的机制**：完整输出禁止截断 + 禁止占位符
**在技能创建中**：SKILL.md 必须写完整的 Phase 指令，不能用"（此处省略）"

### 8. skill-vetter
**提取的机制**：安全审查协议 + 风险分级（🟢🟡🔴）
**在技能创建中**：引用外部 skill 作为参考时必须安全审查

### 9. deep-review
**提取的机制**：模块化分析维度 + 数据支撑要求
**在技能创建中**：质量门禁的 9 项检查就是模块化分析

### 10. auto-evolution
**提取的机制**：进化日历 + 小改/大改分流
**在技能创建中**：Phase 7 自动沉淀 = 技能的自进化机制

### 11. li-sync
**提取的机制**：真源 + 分发架构（一个文件是权威，其他是副本）
**在技能创建中**：~/.newmax/skills/ 是真源，路由表/工作区是分发

### 12. longmemory
**提取的机制**：语义去重 + 唯一写入者
**在技能创建中**：golden_rules.md 不写重复规则，_meta.json 是元数据唯一写入者

### 13. prompt-optimizer
**提取的机制**：BROK 四维度拆解（Background/Role/Objective/Key constraints）
**在技能创建中**：SKILL.md 的 YAML description 就是 BROK 的简化版

### 14. 十条工程法则
**提取的机制**：证据分层 + 规则结晶 + 负结果归档
**在技能创建中**：质量门禁必须带口径标签（粗筛/中级/全签核）

## 外部参考（GitHub/社区）

### 15. mattpocock/skills（GitHub, 19 个 skill）
**核心价值**：极简 skill 设计（diagnose 117行/triage 103行/caveman 49行）
**提取的机制**：反馈循环优先 + 状态机分流 + 75% token 压缩
**启示**：好 skill 不需要 500 行——49 行的 caveman 比 500 行的臃肿 skill 更实用

### 16. peterskoett/self-improving-agent（GitHub, 641★）
**核心价值**：Hook 驱动 + 结构化日志 + 自动 skill 提取
**提取的机制**：Recurrence-Count ≥ 3 + 跨 ≥ 2 任务 → 自动晋升为规则
**启示**：教训沉淀不能靠 AI 记得——必须有自动化机制
