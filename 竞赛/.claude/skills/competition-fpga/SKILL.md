# competition-fpga：FPGA 竞赛全流程引擎

> 竞赛FPGA ≠ 工程FPGA。评委看的是：创新度 + 功能完整度 + 文档质量 + 现场答辩——四个维度都要赢。

---

## 理论锚点

| # | 原则 | 在竞赛中的应用 |
|---|------|--------------|
| T1 | 对标先行 | 先读往届优秀作品，提取评分高点，再定自己的技术路线 |
| T2 | 最小可演示 | 每个阶段都有可展示的功能，不要最后 deadline 前才跑通 |
| T3 | 安全余量 | 时序约束留 20% 余量；功能留 buffer 时间 |
| T4 | 评委视角 | 文档和答辩的目标是让非专家也能理解你做了什么 |
| T5 | 差异化 | 竞赛评分看创新点——找到同类作品没做的那个维度 |

## Phase 0：赛题分析与对标（开始前必做）

### Step 0.1：读懂赛题
- 明确：功能要求 / 性能指标 / 限制平台（FPGA型号/工具链）/ 评分维度
- 识别：哪些是硬性要求（必须做），哪些是加分项（可做）
- 输出：赛题理解表（功能矩阵 + 评分权重估算）

### Step 0.2：往届优秀作品对标
```
1. 搜集近 3 年同赛道/同赛题方向的获奖作品
2. 提取：技术路线 / 创新点 / 评委反馈（如有）
3. 输出：对标矩阵（往届做了什么 / 我们做什么 / 差异化在哪）
```
> 竞赛区的往届作品分析见：`{竞赛项目目录}/01-调研/`

### Step 0.3：技术路线决策
基于对标结果，输出：
- 核心功能清单（必做 + 加分）
- 技术选型（加密算法 / 通信协议 / IP核复用）
- 风险识别（时序风险 / 资源风险 / 时间风险）

## Phase 1：架构设计

### 典型 FPGA 安全通信架构（三层）

```
Top Layer  [top.v]           — 管脚约束 + 模块例化，只连线不算逻辑
Ctrl Layer [aes_ctrl.v ...]  — FSM 控制流，驱动各功能模块
Func Layer [aes_core.v ...]  — 具体算法实现（AES/RSA/CRC/UART/SPI）
```

> 三层架构铁律来自 li-hardware T2，FPGA 交付3个项目验证。

### 模块划分原则
- 每个加密/通信模块独立封装，接口用 valid/ready 握手
- 时钟域隔离：不同速率的接口用 FIFO 或 CDC 同步器
- 复位策略：统一用同步复位（Vivado STA 友好）

## Phase 2：安全通信实现要点

> 本 Phase 仅给竞赛级提示，RTL 代码细节走 li-hardware。

### AES 加密（最常见竞赛需求）
- 模式选择：ECB（最简单）→ CBC（推荐，更有安全性展示价值）→ CTR（高吞吐）
- 资源权衡：全展开 vs 迭代式（竞赛通常选迭代节省 LUT）
- 关键时序：S-Box 查表是关键路径，用 BRAM 实现更快

### 通信接口
- UART：最容易演示（串口助手可直观展示）
- SPI：适合板间通信展示
- AXI-Lite：如果用了 Zynq/MicroBlaze，加分项

### 竞赛答辩演示建议
- 准备"一键演示"：上电即跑，不需要复杂操作
- 准备对比数据：加密前/后波形，ILA 抓取
- 准备降级预案：如果最优功能跑不通，有 plan B 可演示

## Phase 3：时序优化（竞赛级）

```
优先级顺序：
1. 先跑通功能（不管时序）
2. 检查时序报告：违例路径 + slack
3. 针对性优化：加流水线寄存器 / 分解复杂组合逻辑
4. 约束文件：先写时钟约束，再写 I/O 约束
5. 综合策略：先 Performance 跑一次，不满足再换策略
```

**竞赛时序目标**：满足约束 + 留 10-20% 余量（答辩时环境不确定）

## Phase 4：文档准备（竞赛专用）

### 必交材料清单
- [ ] 设计说明书（技术报告）
- [ ] 源代码（含 testbench）
- [ ] 综合/实现报告（资源利用率 + 时序报告截图）
- [ ] 演示视频（如需）
- [ ] PPT（答辩用）

### 技术报告结构（评委看的顺序）
```
1. 摘要（30秒扫完知道你做了什么）
2. 系统架构图（一张图说清楚）
3. 创新点（必须单独列，加粗）
4. 功能实现（详细，有波形截图）
5. 性能指标（频率、资源、加密吞吐率）
6. 总结与展望
```

> 文档输出走 li-docs；PPT 走 li-office。

## Phase 5：答辩准备

### 高频评委问题
| 问题 | 准备方向 |
|------|---------|
| 为什么选这个加密算法？ | 性能/安全性/资源权衡，有数据 |
| 时序怎么收敛的？ | 说出关键路径 + 优化手段 |
| 和其他组的区别？ | 你的创新点，对标矩阵里找 |
| 如果给你更多时间会做什么？ | 展望要具体，不要说"做更多功能" |
| 功耗怎么样？ | Vivado Power Report 数据 |

## 反模式

| 反模式 | 后果 | 正确做法 |
|--------|------|---------|
| 不对标就开始写RTL | 做了往届已有的平庸方案 | Phase 0 对标先行 |
| 只追功能不管时序 | 最后时序不收敛无法上板 | 功能跑通后立刻看时序 |
| 技术报告写"代码很长" | 评委不知道你的创新在哪 | 创新点必须单独列出强调 |
| 演示需要人工配合 | 现场演示不稳定 | 一键演示脚本/状态机 |
| 答辩PPT抄技术报告 | 枯燥，评委无法快速抓到重点 | PPT = 3-5个故事线 + 数据 |
