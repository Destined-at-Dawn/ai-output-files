# No Blind Overwrite（禁止盲目覆写）

## 铁律
**任何写操作（write_project_file / Write / Edit）施加到已存在的文件上之前，必须先读取该文件当前内容。**

没有例外。没有"我记得内容"这回事。没有"大概差不多"这回事。

## 为什么要这条规则
write_project_file 是**覆写**，不是追加。
Write 工具是**覆写**，不是追写。
如果你没有先 Read，你就是在用新内容**销毁旧内容**，而且你自己不知道你销毁了什么。

这不是风格问题，是数据安全问题。

## 强制流程（每次写已有文件前）

```
1. Read 文件当前内容
2. 确认：我要写入的内容 = 旧内容 + 新增部分（而非只有新增部分）
3. 写入
```

如果跳过第1步直接写 → 这是数据事故，不是"小失误"。

## 常见陷阱

| 错误模式 | 后果 |
|---------|------|
| "我知道文件大概内容，直接追加写" | 覆盖，丢失所有旧内容 |
| "用 write_project_file 更新一条记录" | 整个文件被替换为只有那一条记录 |
| "模板里有这个文件，内容应该和模板一样" | 实际文件早已超越模板，覆写后全部丢失 |
| "这个文件不重要，直接写吧" | 你不判断不了它重不重要，先读再决定 |

## 反模式（绝对禁止）

- ❌ "我之前读过，还记得内容" → 你的上下文可能已截断，先 Read 确认
- ❌ "只改一小部分，直接写就行" → 用 Edit（精确替换），不要用 Write/覆写
- ❌ "文件很长，读起来浪费 token" → 丢失数据的代价远大于读一次的 token
- ❌ "用户催得急，先写了再说" → 急不是覆写数据的理由

## 能用 Edit 就不要用 Write

Edit 工具是精确替换，只改你指定的部分，不碰其他内容。
Write 工具是全文替换，旧内容全部销毁。

**决策顺序**：
1. 只改几行 → 用 Edit
2. 需要大幅重写但仍要保留部分内容 → Read → 拼接新旧 → Write
3. 全新文件（确实不存在） → 可以直接 Write

## 事故后的补救
如果已经覆写了：
1. 立即告知用户发生了覆写
2. 搜索可能的备份：归档目录、git history、其他副本
3. 如果无法恢复，坦诚说明，不要假装没事
