## 二、执行步骤

### Step 1: 需求确认
- **操作**：确认以下信息
  - 图的类型（框图/流程图/架构图/对比图/原理图）
  - 图的用途（报告/PPT/文档/答辩）
  - 需要展示的核心信息
  - 风格要求（技术风格/简洁风格）

### Step 2: 选择生成方式
- **操作**：根据图的类型选择工具
  - 技术架构图/流程图/状态机/时间线/象限图 → **Kami图表基元**（`assets/diagrams/`，14种SVG模板，暖色设计语言统一）
  - 简单流程图/框图 → Mermaid/HTML
  - 复杂技术图 → Gemini 生成
  - 代码相关图 → 代码生成（matplotlib/d3.js）
- **Kami图表基元速查**（安装路径: `~/.newmax/skills/kami/assets/diagrams/`）:
  | 图表类型 | 触发词 | 模板文件 |
  |---------|--------|---------|
  | 架构图 | "架构图/系统图/components" | architecture.html |
  | 流程图 | "流程图/决策流/flowchart" | flowchart.html |
  | 状态机 | "状态图/lifecycle" | state-machine.html |
  | 时间线 | "时间线/里程碑/roadmap" | timeline.html |
  | 象限图 | "象限图/2×2矩阵" | quadrant.html |
  | 分层图 | "分层架构/OSI/层叠" | layer-stack.html |
  | 柱状图 | "柱状图/对比" | bar-chart.html |
  | 折线图 | "趋势/时间序列" | line-chart.html |
  | 环形图 | "占比/分布" | donut-chart.html |
  | 树状图 | "层级/组织架构" | tree.html |

### Step 3: 生成图片
- **操作**：
  - 写出详细的提示词/代码
  - 包含所有必要的技术细节
  - 标注具体参数和算法

### Step 4: 质量检查
- **操作**：检查以下维度
  - 技术准确性（参数/算法/流程是否正确）
  - 视觉清晰度（文字可读/颜色区分/布局合理）
  - 信息密度（是否达到标杆密度）
  - 配色方案（5色配色，含HEX）

## 五、认知科学支撑（百大认知书籍）

### 技术视觉设计的认知原理

| 认知机制 | 来源 | 在技术图设计中的应用 |
|---------|------|-------------------|
| **认知负荷管理** | 025-认知负荷理论 §概念1 | 技术框图同时呈现>7个模块会超载。架构图应分层——顶层只显示3-5个核心模块，细节在下层展开 |
| **注意力分裂效应** | 025-认知负荷理论 §概念5 | 图例和图主体分离→评审需在两者间切换→工作记忆覆写。技术图的标签应直接嵌入图内（整合注意力） |
| **非注意盲视** | 043-看不见的猩猩 §概念1 | 评审在快速扫描PPT时，大多数技术图被过滤。核心创新点必须通过视觉异常（颜色对比/放大/标注箭头）突破注意力过滤器 |
| **空间-数字同构** | 048-数字觉醒 §概念2 | 数据在心理数轴上有空间位置——左侧=小/旧，右侧=大/新。技术对比图应利用空间布局传递信息（进步从左到右，退化从右到左） |
| **杏仁核快速通路** | 042-情绪大脑 §概念1 | 评审对技术图的"第一印象"在12ms内形成（快于理性分析）。技术图必须在纯视觉层面传递"专业感"——整洁布局、一致配色、对称结构 |
| **预测误差吸引** | 047-体验机 §概念1 | 评审预期"又一张标准框图"→当你的图包含意外元素（动态数据流/创新布局）→预测误差→注意力锁定→创新感建立 |
