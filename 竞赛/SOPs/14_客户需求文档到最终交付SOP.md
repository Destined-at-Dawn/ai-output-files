# 客户需求文档到最终交付 SOP

> **版本**：v2.0 | **编号**：SOP-14
> **触发条件**：客户/甲方/老师/比赛方发来需求文档、考核要求、任务书
> **前身**：`14_RoboMaster电控考核最终交付SOP.md`（已归档）
> **核心理念**：客户文档是唯一标准；模板加速产出；SOP 交叉引用避免重复

---

## 结论

接到客户文档 → Phase 0 吃透要求 → 匹配模板 → 调用对应 SOP 执行 → 门禁全过 → 交付。
**不要从零开始。模板是前人的经验复利。**

---

## 零、适用场景

| 场景 | 模板类型 |
|------|---------|
| FPGA/RTL 竞赛交付 | `templates/FPGA-RTL类/` |
| 电控/嵌入式竞赛交付 | `templates/嵌入式-电控类/`（待建） |
| 课程设计/实验报告 | 用 SOP-11 实验教学文档编写 |
| 答辩/演示/汇报 | 用 SOP-05 答辩与演示 |

**不适用**：纯口头需求、日常 Bug 修复、单次代码修改。

## 一、Phase 0：需求分析（动手前必做，15 分钟）

> 铁律：没有需求分析就开始做 = 方向赌博。

```
收到客户文档
  ├─ Step 1: 通读，用🔴🟡🟢标优先级
  ├─ Step 2: 提取交付物矩阵（名称 / 格式 / 必须或可选 / 权重 / 截止时间）
  ├─ Step 3: 识别隐含需求（竞赛要答辩、企业要接口文档、课程要参考文献）
  ├─ Step 4: 输出确认清单 → 与客户确认歧义
  └─ Step 5: 匹配模板（见 §二）
```

**自检**：所有「必须提交」项都在矩阵里？格式要求逐项标注？歧义已确认？

## 三、执行流程（交叉引用，不重复造轮子）

### 3.1 代码开发

按项目技术栈调用对应 SOP/Skill：

| 技术栈 | 调用 |
|--------|------|
| FPGA/Verilog | **SOP-06** + `li-hardware` skill |
| 嵌入式 C | `li-hardware` skill |
| Python/算法 | 直接开发 |

**代码交付铁律**（所有技术栈通用）：
- 交出去的代码必须能编译/运行（「应该能跑」≠「跑了」）
- README 写清编译方法、依赖、目录结构
- 删除调试残留（`print("debug")`、`// TODO`）
- 关键函数/模块必须有注释

### 3.2 文档编写

**不重复写文档规范** → 直接调用：
- **SOP-04 技术文档编写**（通用技术文档）
- **SOP-11 实验教学文档编写**（实验文档，含去 AI 味）
- 模板中已有该类型的章节结构 → 用模板结构

**格式铁律**：同时提供 `.md` + `.docx` + `.pdf`（除非客户只要单一格式）。PDF 必须抽检中文显示。

### 3.3 答辩材料

**不重复写答辩规范** → **SOP-05 答辩与演示**。

通用原则：
- 只说自己能解释的
- 没实测就说「待验证」
- 用工程事实替代空泛努力

### 3.4 打包与交付

```
1. 对照交付物矩阵逐项检查文件存在
2. 删除 §五 禁止清单中的文件
3. 重命名 → {项目名}_最终交付包/
4. 打包 .zip → 验证可解压
5. 门禁全过（§六）→ 输出交付报告（§七）
```

### 3.5 阶段性落盘验证铁律

**每阶段完成后立即 Glob+Read 验证落盘，不等最后一起验。**

```
阶段 N 完成 → Glob 确认文件存在 → Read 抽检内容正确 → 有缺口立即补写 → 才进入阶段 N+1
```

- 对话可能截断/超时，最后一起验 = 可能永远验不了
- 来源：2026-06-09 SOP-14 通用化重构，对话截断导致 2 个缺口未被发现

## 四、最终目录标准

```text
{项目名}_最终交付包/
  交付说明.md
  核心代码/
    src/
    README.md
  实验文档/              ← 按需，参见 SOP-04/11
  设计说明/              ← 按需
  答辩演示材料/          ← 按需，参见 SOP-05
  自我学习/              ← 按需
```

**裁剪规则**：只交代码 → `核心代码/ + 交付说明.md`；交代码+文档 → `+ 实验文档/`；全套 → 全部。

---

## 五、禁止放入最终 ZIP 的内容

- 调研过程报告、AI 工具快照、对话记录
- 临时渲染目录（`_pandoc_tmp`）
- 未说明用途的杂文件
- 中间版本（`_v1`、`_draft`、`_backup`）
- 含「临时」「测试」「草稿」字样的文件
- 未验证却声称实测的材料

---

## 六、最终门禁（全部通过才能交付）

| # | 门禁项 |
|---|--------|
| 1 | 交付物矩阵所有「必须」项有对应文件 |
| 2 | 无禁止清单中的文件 |
| 3 | 代码能编译/运行 |
| 4 | 文档格式齐全（按客户要求） |
| 5 | PDF 中文显示正常 |
| 6 | 所有文件中文命名（代码文件除外） |
| 7 | ZIP 能正常解压 |
| 8 | 无编造数据或测试结果 |

**任何一项未通过 → 不交付。没有「差不多」。**

```
## 交付报告

```

## 八、模板反哺（经验复利的关键）

**每个项目完成后必须执行**：

1. 本次用了哪个模板？
2. 模板缺少什么？（客户要求了但模板没有的章节/检查项）
3. 本次踩了什么坑？（模板应该提醒但没有的）
4. → 更新模板或创建新案例文件写入 `templates/{类型}/`

**铁律**：不做反哺 = 模板永远不进化 = 下次还从零开始。
