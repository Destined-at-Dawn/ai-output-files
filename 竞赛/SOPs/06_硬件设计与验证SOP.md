### 论文1：ChiPBench — AI芯片布局基准测试

| 项目 | 内容 |

**核心发现（直接可用于竞赛）：**
- **HPWL ≠ PPA**：过去所有AI布局论文都用 HPWL（半周长线长）作为优化目标，但 ChiPBench 用 20 个真实电路（含 ariane RISC-V CPU）证明——**HPWL 和最终 PPA（功耗/性能/面积）不相关**
- **6种AI布局算法集体翻车**：在真实 EDA 工具链（OpenROAD）中跑完综合→布局→布线→时序后，AI 方法与传统方法差距不大，甚至更差
- **竞赛启示**：如果竞赛题目涉及芯片布局，不要盲目用 HPWL 做优化目标，必须跑完整 PPA 流程

**可融入竞赛 SOP 的操作：**
- Step 5 综合约束时，HPWL 仅作初步参考，**必须以 PPA 为最终评估指标**
- 选题调研时，此论文是 AI+EDA 方向的必读基线

## 四、硬件设计与调试避坑指南（基于历史血泪教训）

### 1. Verilog 编码铁律（⚠️ 已从避坑指南升级为铁律 —— 基于 3+ 次历史教训）
- **位宽匹配**：`localparam` 位宽必须与被比较的 `reg` 寄存器位宽精确一致，防止溢出或截断。
- **循环变量隔离**：`for` 循环变量的作用域是整个模块，内外层循环**必须**使用不同变量名（如 `i` 和 `j`）。
- **`disable` 命名块规则**：`disable <label>` 只能退出包裹它的命名块，必须写成 `begin : label for(...) ... end`。
- **跨时钟域与异步信号**：外部/异步输入信号必须经过至少两级同步器，之后再检测边沿（`sig_d1 && !sig_d2`）。
- **持续高电平检测**：`done`、`valid` 等持续高电平信号，必须用 `sig && !sig_dly` 检测上升沿，不可直接作为触发条件。
- **避免悬空 `if`**：移除 debug 打印（如 `$display`）时，务必检查是否会留下空 `if`，防止下行代码被意外嵌套。

### 2. 仿真调试优先级（⚠️ 已升级为铁律 —— 禁止上来就看波形）
遇到仿真失败，**禁止上来就直接看波形**，应遵循以下顺序：
1. **查看仿真 `$display` 输出**：快速定位超时、CRC错误、状态卡住。
2. **检查状态机 `state` 和 `led_status`**：明确当前卡在哪一个具体状态。
3. **在波形中看顶层标志信号**：如 `frame_valid`、`error_code`，确认错误类型。
4. **追踪模块握手信号**：如 `busy`、`tx_en`、`done`，检查信号边沿对齐情况。
5. **最后再看底层波形**：比如逐字节 trace 帧内容、检查过采样时序。

### 3. Vivado大容量ROM综合铁律（⚠️ 2026-05-13 新增 —— 基于8次综合崩溃教训）

- **ROM深度>16384时**：禁止用 `$readmemh` 代码推断方案，**必须**用 Block Memory Generator IP 核。
- **create_ip前**：必须 `file mkdir` 输出目录。
- **generate_target后**：必须 `add_files` IP wrapper 文件。
- **禁止写死IP版本号**：让Vivado自动选。
- **添加综合约束前**：必须检查是否与已有约束矛盾（如 `max_bram_cascade_height 1` + ROM深度>16384 = 物理不可能）。
- **Windows中文环境铁律**：TCL脚本开头必须覆写 `::env(COMPUTERNAME)` 和 `::env(USERNAME)` 为纯ASCII字符串；IP核必须 `set_property generate_synth_checkpoint false` 禁用OOC综合（OOC子进程继承系统原始环境变量，TCL覆写无效）。
- **排查顺序铁律**：工具报错时必须按顺序排查——①环境变量 → ②路径存在性 → ③日志实际ERROR → ④最小化复现 → ⑤单变量修改。**禁止跳过前几步直接猜代码问题。**
- **Vivado综合崩溃诊断**：Messages 0 Errors + 右上角红灯 = 进程被杀（OOM或路径问题），检查Log最后几行。

**关联教训**：`自我进化/做得差的避免/知识库_教训.md` §六（8次综合崩溃完整记录）

### 3.1 Vivado IP 核管理与修改铁律（⚠️ 2026-05-16 新增 —— 避免IP被锁死）
- **禁止手动修改 `.xci` 文件**：修改IP配置**永远使用TCL的 `set_property` 命令**。手动编辑 `.xci` 会导致与 Vivado 自动生成的 `.xml` 缓存不一致，判定为 `stale content` 从而彻底锁死IP，导致修改无效。
- **IP解锁与缓存清理**：如果IP被锁死，必须执行：`set_property is_locked false [get_files xxx.xci]`，并强制删除对应的 `.xml` 缓存文件。
- **重配后必须重新生成**：执行 `set_property` 后，必须紧跟 `reset_target all` 和 `generate_target all`。
- **综合顺序**：必须先单独完成 IP 的综合 (`launch_runs xxx_synth_1` 并 `wait_on_run`)，再进行顶层综合。不可同时启动。

### 3.2 Vivado XSIM 仿真避坑铁律（⚠️ 2026-05-16 新增）
- **Verilog-2001 兼容性**：Vivado XSIM 对 `.v` 文件的解析默认严格遵循 Verilog-2001。**绝对不支持 SystemVerilog 语法**（如 `fork/join_any`, `disable fork`, `break`, `return`）。
- **跳出机制平替**：写 Testbench 时如需提前跳出块，必须使用 Verilog-2001 标准的命名块禁用：`begin : block_name ... disable block_name; end`。
- **日志输出位置**：TB 中的 `$display` 打印可能不会出现在 GUI 的 TCL Console 中，查找报错必须去查阅 `sim_1/behav/xsim/simulate.log` 文件。
- **验证守卫**：LCD等外设仿真的数据捕获验证，必须在收到确切的数据写入命令（如 0x2C Memory Write）后才开始，严禁在初始化阶段误把命令参数当作像素数据校验。

### 4. 文件编码铁律（⚠️ 2026-05-18 新增 —— 基于3次编码诊断连续犯错教训）

- **非UTF-8文件必须全程用Python操作**：`.v`/`.sv`/`.xci`/`.xdc` 等硬件文件可能不是UTF-8编码（常见GBK/Latin-1）。**禁止用 Claude Code 的 Read/Write/Edit 文本工具直接操作**，必须用 Python 的 `open(file, encoding='latin-1', errors='replace')` 读取，处理后再用 Python 写回。
- **修改前先检测编码**：`python -c "import chardet; print(chardet.detect(open('file','rb').read()))"`
- **TCL脚本必须纯ASCII**：Vivado TCL控制台用GBK编码读取，中文会导致乱码。脚本中禁止出现非ASCII字符。
- **诊断顺序铁律**：遇到文件读写异常 → ①先查编码 → ②再查路径 → ③最后查内容。禁止跳过①直接猜内容问题。

### 6. 多模块常量/命令编码一致性铁律（⚠️ 2026-05-20 新增 —— 基于I2C通信全乱教训）

- **多模块工程中，所有共享的命令编码/常量定义必须逐一比对**：master 发 CMD_START(=1)，slave 收到 CMD_WRITE(=1)——差 1 就是通信全乱，比没有通信更危险。
- **创建工程时的强制步骤**：列出所有模块间的共享常量（CMD_IDLE/START/WRITE/READ_ACK/READ_NAK/STOP 等），逐个验证数值一致。
- **禁止各自定义常量后"假设一致"**：每个模块单独定义 `localparam CMD_xxx = N` 时，必须交叉检查。推荐做法：公共常量放在一个 `include` 文件里，或用参数传递。
- **验证方法**：用 Grep 搜索所有模块中的 `CMD_` 或命令常量定义，**逐行对比数值**。

**关联教训**：
- `自我进化/做得差的避免/20260520_CMD编码不匹配_失败教训.md`（I2C CMD编码差1导致SCL/SDA全高，调试2天）

### 7. I2C SDA 必须用 Open-Drain 铁律（⚠️ 2026-05-22 新增 —— 基于SCL/SDA瞬间变全高教训）

- **禁止 behavioral tri-state 描述 I2C SDA**：`assign sda = sda_oe ? sda_out : 1'bz;` 会被 Vivado 综合成 push-pull 输出，ACK 阶段 FPGA 不释放 SDA，与 slave 的 ACK（拉低）总线冲突。
- **必须用 open-drain 写法**：`assign sda = (sda_oe & ~sda_out) ? 1'b0 : 1'bz;` — FPGA 只驱动低电平，高电平靠外部上拉电阻。
- **真值表**：`sda_oe=0` → 高阻（释放）| `sda_oe=1, sda_out=0` → 驱动低 | `sda_oe=1, sda_out=1` → 高阻（释放，pull-up 拉高）
- **XDC 约束**：SDA 引脚必须配置为 open-drain + 上拉（`IOSTANDARD LVCMOS33` + 外部 4.7kΩ 上拉）。

**关联教训**：
- `自我进化/做得差的避免/20260520_综合症状说明会_失败教训.md`（I2C SDA push-pull 导致偶尔有波形瞬间变全高）

### 8. 禁止全局错误检测铁律（⚠️ 2026-05-22 新增 —— 基于假阳性导致状态机循环卡死教训）

- **禁止在状态机外用"每个周期都检查"的方式检测错误**：`if (done && !ack_err) state <= S_ERROR;` 这种写法会捕获**上一条命令**的残留 ack_err，制造假阳性。
- **正确做法**：在每个具体 I2C 命令完成的 case 分支内**就地检查** ack_err。CMD_START 完成时 ack_err 无意义（START 不产生 ACK），不能用来判定 NAK。
- **原因**：`done` 脉冲和 `ack_err` 的时序关系取决于具体命令类型。全局检测无法区分"当前命令的 ACK"和"上一条命令的残留"。

**关联教训**：
- `自我进化/做得差的避免/20260520_综合症状说明会_失败教训.md`（偶尔有 I2C 波形然后瞬间变全高的根因）

## 六-B、认知科学支撑（百大认知书籍）

> **为什么需要这一节？** FPGA硬件设计是人类认知负荷最高的工程活动之一——同时涉及状态机思维、时序推理、信号追踪、协议理解等多维认知。理解认知瓶颈的物理机制，才能设计出"大脑友好"的开发流程。

### 硬件设计的认知瓶颈

| 认知机制 | 来源 | 在硬件设计中的应用 |
|---------|------|-------------------|
| **工作记忆物理极限** | 025-认知负荷理论 §概念1 | 同时追踪>5个信号的时序关系会超载。Step 2架构设计时，模块划分的目标是让每个模块的接口信号≤5个。超过时必须引入中间缓冲/分层抽象 |
| **单核处理器限制** | 022-认知神经科学 §概念4 | 大脑是"单核处理器"，强制多线程（同时写代码+查文档+看波形）会导致系统崩溃。Step 3 RTL编码应采用"专注块"模式：写代码时不查文档，查文档时不写代码 |
| **深度工作** | 005-深度工作 §概念1 | RTL编码是典型的深度工作。每次中断→注意力残留→23分钟后才能恢复到之前的认知深度。Step 3应设置"不可打扰"时间块（至少45分钟连续） |
| **刻意练习** | 017-刻意练习 §概念2 | 硬件设计能力提升=舒适区边缘练习。每次设计应在当前能力"刚好够不着"的复杂度上挑战。过于简单的模块→天真练习→不增长能力 |
| **反脆弱** | 046-反脆弱 §概念1 | Debug是反脆弱练习——每次找到bug都增强了代码和工程师的反脆弱性。Step 4仿真验证中故意注入边缘case（混沌工程），让设计在日常微痛中变得强韧 |
| **预测编码** | 062-冲浪不确定性 §概念1 | 调试本质=最小化预测误差。先形成"预期波形"（生成模型），再与实际波形对比→差异=预测误差=bug定位点。§四.2仿真调试优先级的认知学解释 |
| **认知负荷与图式积累** | 025-认知负荷理论 §概念4 | 每个已解决的硬件bug应固化为"图式"存入记忆系统。下次遇到类似症状→自动匹配图式→认知负荷降低→解决速度指数级提升。这是Step 0加载教训库的认知学原理 |

### 深度工作保护协议

```
开始RTL编码前：
1. 手机静音放另一个房间（消除物理中断源）
2. 关闭邮件/消息通知（消除数字中断源）
3. 在桌面贴标签："当前错误 = 预设的正常噪声"（047-体验机，降低精度权重）
4. 设置45分钟倒计时（番茄钟）
5. 编码期间禁止查文档——有疑问记在便签上，编码结束后统一查
```

### Debug的认知心理学

| Debug阶段 | 认知陷阱 | 正确做法 | 来源 |
|-----------|---------|---------|------|
| 看到bug时 | "我怎么这么笨"→自我谴责→杏仁核劫持→前额叶算力泄漏 | 切换"冷血机械师"视角：这是系统故障，不是人格缺陷 | 050-情绪 + 042-情绪大脑 |
| 定位bug时 | 管窥效应→只看一个信号→忽略全局 | 强制切换"鸟瞰视角"→画全局信号拓扑图 | 060-稀缺 §概念2 |
| 修复bug时 | 确认偏误→只找支持自己假设的证据 | 魔鬼辩护：主动寻找推翻自己假设的信号 | 010-思考快与慢 §概念5 |
| 验证修复时 | 流畅度错觉→波形"看起来对了" | 强制边界测试：最小值/最大值/溢出值/零值 | 009-认知天性 §概念3 |

## 五、FPGA 构建自动化模式库（v1.9 新增）

> 来源：FPGABuilder 开源项目逆向学习（2026-05-25）
> 详见：`li-hardware` SKILL.md §20（完整模式库）

### 5.1 配置驱动构建（铁律）

**规则**：FPGA 项目必须有配置文件（YAML 或等价格式），定义：项目名、器件型号、源文件路径、约束文件、综合策略、Hook 脚本。

**禁止**：每次从零手写 TCL，零复用、不可版本控制。

### 5.2 TCL 模板化（铁律）

**规则**：常用 TCL 片段必须模板化（工程创建、构建流程、清理、编程），参数化替代硬编码。

**6 个标准模板**：BasicProject / BDRecovery / BuildFlow / Clean / GUI / ProgramDevice

### 5.3 Hook 系统（建议）

**规则**：在构建流程的 6 个关键节点（pre_build / pre_synth / post_synth / pre_impl / post_impl / post_bitstream）支持插入自定义操作。

**智能分离**：TCL 命令嵌入脚本，外部脚本（Python/Bat）用 subprocess 执行。

### 5.4 版本适配器（建议）

**规则**：不同 Vivado 版本应有独立的适配器，自动切换推荐策略和命令参数。

### 5.5 设备编程封装（建议）

**规则**：JTAG/Flash 编程流程模板化，自动查找 bitstream 文件，支持配置 cable/target/bitfile。

### 5.6 三级清理（建议）

| 级别 | 操作 | 适用场景 |
|------|------|---------|
| soft | reset_runs + 删日志 | 日常开发 |
| hard | 删工程目录 | 重建工程 |
| all | 删全部生成文件 | 完全重置（需确认） |
