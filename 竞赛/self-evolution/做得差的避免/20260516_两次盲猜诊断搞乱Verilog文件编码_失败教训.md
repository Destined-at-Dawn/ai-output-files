# 两次盲猜诊断搞乱Verilog文件编码

## 日期
2026-05-16

## 场景
用户报告 lcd_ctrl.v 中文注释显示乱码，要求修复。

## 我做了什么

### 第1次诊断：猜"文件不同步"
- 现象：用户贴出乱码 `50MHz鏃堕搐/浠跨湡鍔犻`
- 我的判断：Vivado imports 目录下的文件是旧版，根目录才是新版
- 实际情况：**部分正确**——imports确实有旧版，但用户看到的乱码不是因为不同步
- 操作：把根目录版本复制到 Vivado imports + outputs 三处同步
- 结果：用户说"乱码更多了"

### 第2次诊断：猜"UTF-8要转GBK"
- 现象：用户贴出更多乱码
- 我的判断：Vivado + Windows 中文环境用 GBK，UTF-8 中文会乱码
- 实际情况：**完全错误**——没有验证Vivado到底用什么编码就执行转码
- 操作：把所有 .v 文件从 UTF-8 转成 GBK
- 结果：Read 工具（UTF-8 读 GBK）显示乱码，用户看到的乱码更多

### 根因
两轮诊断都是**先猜原因再行动**，没有：
1. 用 Python 检查文件原始字节和编码
2. 确认 Read 工具和 Vivado 各自用什么编码
3. 小范围测试再全量执行

## 错误模式
```
看到乱码 → 猜原因 → 立即全量执行修复 → 修复本身制造新问题 → 再猜 → 再修 → 越修越乱
```

## 正确做法
```
看到乱码 → Python检查原始字节 → 确认真实编码 → 小范围测试 → 验证后再全量执行
```

## 具体规则
1. **禁止盲猜编码**：看到乱码时，第一件事是 `python3 -c "open(f,'rb').read()[:100].hex()"` 检查原始字节
2. **禁止全量转码**：先对一个文件小范围测试，确认效果后再推广
3. **区分显示问题和内容问题**：Read工具/Bash终端/Vivado 三个显示方可能用不同编码，要分开诊断
4. **已修正**：项目CLAUDE.md §4 第6条已从"GBK铁律"改为"保持UTF-8，禁止盲猜转码"

## 影响范围
- 5个 .v 文件被转码两次（UTF-8→GBK→UTF-8）
- 已恢复为UTF-8，内容未丢失
- 项目CLAUDE.md的错误规则已修正
