## 二、失败模式 → SOP 映射

| 模式 | 严重度 | 关联SOP | 避坑规则 |
|------|--------|---------|---------|
| **声称已完成但未执行** | 🔴致命 | **所有SOP** | 任何"已完成"声明必须伴随 tool_use + Read 验证，否则=虚假操作 |
| **厂商代码误判** | 🔴致命 | SOP-08 | 厂商代码对比只能发现"差异"，差异≠错误。硬件实测数据才能判定严重性 |
| **无计划debug** | 🟡中 | SOP-01、SOP-06 | debug必须有计划：①定位异常信号 ②时域优先（对比） ③仿真最后 |
| **未主动写入记忆/教训** | 🔴致命 | **所有SOP** | 发现认知误区或纠正错误后，必须立即在同一轮对话中写入 memory 和自我进化，绝不等待用户要求 |
| **被忽视的：应该调用但没调用的Skill/文件** | 🟡中 | **所有SOP** | 宁可误触发Skill读一下确认不适用，也不能漏触发 |
| **Vivado综合崩溃(8连败)** | 🔴致命 | SOP-06 | ROM>16K→必须BMG IP核；COMPUTERNAME中文→禁用OOC；排查顺序：环境→路径→日志→代码 |
| **D盘清空+移动硬盘物理损坏(数据丢失)** | 🔴致命 | SOP-06 | 3-2-1备份铁律：3份副本、2种介质、1份异地。禁止单点存储。里程碑后备份。 |
| **多模块CMD编码不匹配** | 🔴致命 | SOP-06 | 两个模块之间的常量/命令编码必须逐一比对，不能假设一致。编码差1=通信全乱。创建工程第一天就列出跨模块常量表。 |
| **读了memory但不执行** | 🔴致命 | SOP-11、SOP-06 | memory里记录了"lcd_rst/lcd_bl/init_done不引出端口"，但生成实验文档时仍写错。规则必须在生成时严格执行，不能只读不用。 |
| **编码诊断连续犯错** | 🟡中 | SOP-11 | 看到乱码→猜原因→全量执行→修复制造新问题→再猜→越修越乱。正确做法：先Python查字节确认编码→小范围测试→验证后再全量执行。 |
| **工程文件三处不同步** | 🟡中 | SOP-06、SOP-11 | 修改代码后必须同步到：根目录+Vivado imports+outputs。三处不一致=Vivado读旧版=用户看到乱码/错误。 |
| **PDF工具猜错名** | 🟡中 | SOP-11 | 猜 `mcp__pdf__pdf_read` 不存在。正确方案：Python+PyMuPDF（fitz），已固化到CLAUDE.md §4.5。 |
| **I2C SDA被综合成push-pull** | 🔴致命 | SOP-06 | I2C SDA必须用open-drain写法`(sda_oe & ~sda_out) ? 1'b0 : 1'bz`，禁止behavioral tri-state。Vivado会把`sda_oe ? sda_out : 1'bz`综合成push-pull输出 |
| **全局错误检测假阳性** | 🔴致命 | SOP-06 | 禁止在状态机外"每个周期检查"错误。done脉冲时ack_err可能是上一条命令残留，制造假阳性推到ERROR循环卡死。必须在每个具体命令完成的case分支内就地检查 |
| **双路由文件混乱+技能激活系统性失败** | 🔴致命 | **所有SOP** | 两个路由文件（skill-routing-table.json + skill-rules.json）格式不同、内容冲突→AI迷路→整轮对话0次skill调用。修复：统一为单一真相源 + 创建skill-auto-activation.md硬协议 + 废弃旧文件。检测：新对话中AI必须输出技能审计且调用率≥80% |
