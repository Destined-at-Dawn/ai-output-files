# 2026-05-25 proj-ID 碎片化治理

## 问题
每次新建对话生成无意义的 `proj-{timestamp}-{random}` 目录，导致：
- 记忆分散在多个 proj-ID 目录中（mutual 区就有 6 个 memory 文件分散在两处）
- 产出碎片化（mutual 区 proj-ID 有 28 个产出文件，主目录也有 28 个，大量重叠）
- 跨对话记忆连续性断裂（新对话读不到旧对话的记忆）

## 涉及范围
6 个 proj-ID 目录，跨 5 个工作区：
- mutual: proj-1779089173658-j5tg2m → 20260518-生态优化
- 创作: proj-1777084456942-plvse9 → 20260521-AI创作与视觉设计
- 个人: proj-1777092260144-quq9li → 20260525-心理机制深度分析
- 求职: proj-1777554288697-9e2i8z → 20260523-求职准备
- 竞赛: proj-1777087251931-pj4ul9 → 20260516-触摸屏驱动开发
- 竞赛: proj-1778067689138-1wcqv9 → 跨校协作

## 解决方案
主目录优先架构（R15 共享规则）：
- 中文主目录（`YYYYMMDD-任务名`）= 权威数据源
- proj-ID = 会话暂存区，对话结束后归档
- 记忆/规则/SOP 统一从主目录读写

## 教训
1. **PK-INFRA-010**：系统自动生成的目录名（proj-ID）不能作为永久标识。必须有人类可读的语义名称
2. **PK-INFRA-011**：记忆文件分散在多处 = 跨会话记忆连续性断裂。记忆必须有唯一的权威源
3. **PK-INFRA-012**：迁移已有目录时，当前正在使用的目录无法移动（Windows 文件锁）。需要用 `.session-meta.json` 标记迁移状态，在会话结束后再归档

- 法则 7：负结果归档——proj-ID 碎片化是一种"负结果"（架构设计缺陷）
