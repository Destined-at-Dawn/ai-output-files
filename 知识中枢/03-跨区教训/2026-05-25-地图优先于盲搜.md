# 教训：地图优先于盲搜——上下文质量 > 模型能力

## 发生了什么
阅读 CodeGraph 项目分析文章（GitHub 7天1.4万Star）发现一个核心洞察：AI Agent 处理任务时，token 浪费的主因不是模型能力不足，而是没有"地图"——每次都从盲目探索开始（grep/Read 循环），而不是先查结构化索引。

CodeGraph 在 Tokio（Rust 790文件）项目上，成本降低 82%（$2.33→$0.42），Excalidraw 工具调用从 79 次降到 3 次（-96%）。**差异不是模型版本，是有没有一张地图。**

## 根因
- AI Agent 默认行为是"拿到任务就探索"（ls → grep → Read → 换关键词 → 再搜）
- 没有在 CLAUDE.md 中显式指示"先查索引再动手"
- MCP 工具虽然存在，但指令没到位 = 装了白装

## 修复方案
1. **CLAUDE.md 强制指令**：所有工作区的 CLAUDE.md 必须明确指示"有索引/注册表/MCP工具时，禁止直接 grep/Read 循环，必须先用结构化查询"
2. **优先级链**：MCP工具/索引查询 → 注册表/总索引 → 精确路径 Read → 最后才是 grep/Explore
3. **工具装了但指令没到位 = 没装**：每个 MCP 工具必须在 CLAUDE.md 中有对应的使用指令

## 适用场景
- 任何有 MCP 工具的工作区（project-tools、skill-handler 等）
- 任何有注册表/索引/SOP总索引的工作区
- 任何需要跨文件查找信息的任务

## 跨工作区影响
此教训影响所有 5 个工作区——每个工作区都有 MCP 工具和索引系统，但 AI 默认行为仍然是"先探索再查索引"。必须通过 CLAUDE.md 指令反转这个优先级。

## 核心公式
> **给模型更好的输入，是比升级模型版本更快、更便宜的路。**
>
> 地图（索引） > 盲搜（grep/Read 循环）
> 上下文质量 > 模型能力
> 工具指令到位 > 工具本身安装
