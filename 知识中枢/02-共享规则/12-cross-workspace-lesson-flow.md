# R12：跨工作区教训流动规则

> 版本：v1.0 | 创建：2026-05-24
> 维护方：mutual 工作区
> 适用：所有工作区

---

## 一、为什么需要这条规则

共享规则（02-共享规则/）解决了"规则同步"，但"教训同步"没有机制。
结果：竞赛踩的坑，创作区不知道；mutual发现的模式，求职区用不上。

**教训 ≠ 规则**：
- 规则 = 对AI的强制指令（"你必须做X"）
- 教训 = 从失败中提取的经验（"做X会导致Y，因为Z"）

两者都需要跨工作区流动，但机制不同。

---

## 二、教训流动机制

### 中转站
`E:\ai产出文件\牛马\知识中枢\03-跨区教训\`

### 流程
```
工作区A踩坑
  → 写入本项目 memory/long-term.md（原始记录）
  → 判断：是否≥2个工作区可能遇到？
     → 是：写入 知识中枢/03-跨区教训/YYYY-MM-DD-关键词.md
     → 否：仅保留在本项目 memory 中
```

### 自动加载
各工作区 CLAUDE.md 的"跨工作区感知"段落中，引用 `03-跨区教训/` 目录。
AI 进入任何工作区时，应扫描该目录最近7天的教训文件。

---

## 三、教训文件格式

```yaml
---
来源工作区: {工作区名}
来源项目: {项目名}
日期: {YYYY-MM-DD}
关键词: [标签1, 标签2]
适用工作区: [全部] 或 [工作区1, 工作区2]
严重程度: 高/中/低
---
```

正文必须包含：
1. **发生了什么**（事实）
2. **根因**（为什么）
3. **修复方案**（怎么做）
4. **跨工作区影响**（为什么其他工作区也需要知道）

---

## 四、什么教训值得跨区

| 条件 | 示例 |
|------|------|
| 涉及 MCP 工具行为 | write_project_file 路径不可预测 |
| 涉及文件安全 | 覆写陷阱、隐私泄露 |
| 涉及通用工作流 | memory 写入流程、git 操作 |
| 涉及 AI 行为模式 | 幻觉引用、路径依赖 |
| ≥2个工作区踩过相同坑 | 记忆碎片化 |

**不值得跨区的**：纯项目技术细节（如某个零件的CAD参数）。

## 六、维护

- **新增**：任何工作区踩坑后，判断是否需要跨区
- **淘汰**：30天内无新引用的教训，降级为"历史参考"
- **归档**：longmemory skill 每月清理过期教训
