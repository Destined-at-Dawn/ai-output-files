# 四位一体合规检查协议 v1.0

> **定位**：防止四个工具偏离职责规范、跳过协同流程的强制执行层。
> **原则**：自检 + 互审 + Hermes 审计，三层防漂移。

---

## 一、三层合规架构

```
Layer 1: 任务级自检（每个工具每任务必执行）
  → 启动门禁：进场前 5 问
  → 收工门禁：交付出门前 5 问

Layer 2: 跨工具互审（接收方校验发送方）
  → Handoff 完整性检查
  → 产出注册检查

Layer 3: Hermes 周期审计（每周 + 异常触发）
  → 合规扫描
  → 违规报告
  → 升级规则
```

---

## 二、Layer 1：任务级自检

### 2.1 启动门禁（Start Gate — 每个工具每任务必过）

**在动手做任何事之前，逐项回答——全部为「是」才能开始：**

```
□ SG-1 我知道自己是什么角色吗？
      [Hermes/Newmax/Codex/WorkBuddy]
      → 如果答错 = 角色混乱，停止

□ SG-2 这个任务在我的主战场内吗？
      [对照本工具职责规范 § 二、核心职责]
      → 如果不在 = 越界，应 handoff 给正确工具

□ SG-3 我从 Hermes 获取上下文了吗？
      [Hermes 上下文注入 / memory 文件 / project-context 项目卡]
      → 如果没获取 = 盲开，停止

□ SG-4 我知道前序工具留下了什么吗？
      [读取 handoffs/ 中给自己的交接文件]
      → 如果该有但没读 = 断链，停止

□ SG-5 我知道产出写到哪里吗？
      [绝对路径，在自己工作区范围内]
      → 如果不知道 = 乱写，停止
```

**违规响应**：
- 1 个「否」→ 停止，补齐再继续
- ≥2 个「否」→ 停止，向用户报告为什么无法启动

### 2.2 收工门禁（Finish Gate — 标记完成前必过）

**在说「做完了」之前，逐项回答——全部为「是」才能宣告完成：**

```
□ FG-1 产出是否写入了正确位置？
      [对照本工具职责规范 § 四、输出规范]
      → 路径不对 = 没有完成

□ FG-2 可复用产出是否写入了 artifact-registry.md？
      [判断标准见 § 3.3]
      → 该注册但没注册 = 没有完成

□ FG-3 是否需要写 handoff 给下一个工具？
      [任务产出需要其他工具接力？→ 是则必须写]
      → 该写但没写 = 没有完成

□ FG-4 handoff 文件是否通过了完整性检查？
      [对照 § 3.2 handoff 完整性门禁]
      → 不完整 = 没有完成

□ FG-5 边界声明是否写了？
      [可写入材料 / 不可写入材料 / 尚不能声称]
      → 正式交付必须写
```

**违规响应**：
- 任何一项「否」→ 任务未完成，补齐后再宣告

## 三、Layer 2：跨工具互审

### 3.1 接收方校验规则

**当工具 B 读到工具 A 写来的 handoff 文件时，B 必须校验：**

| 检查项 | 不通过时 |
|--------|----------|
| 文件包含「已完成」栏 | 退回——没有写清前序产出 |
| 文件包含「需要做」栏 | 退回——没有写清任务要求 |
| 文件包含「约束」栏 | 退回——没有写清限制条件 |
| 文件包含「产出位置」绝对路径 | 退回——接手后不知道往哪写 |
| 产出路径可访问（文件存在/目录存在） | 退回——前序产出不存在 |

**退回流程**：
1. 在 handoff 文件末尾追加 `## 退回记录`
2. 写清：`[日期] [退回原因] [需要工具A补充什么]`
3. 告知用户：「{工具A} 的 handoff 缺少 {字段}，已退回等待补充」

### 3.2 Handoff 完整性门禁

**任何 handoff 文件必须包含以下四个结构块——缺一不可投递：**

```markdown
# Handoff: {项目名} → 从 {工具A} 到 {工具B}

## {工具A} 已完成
[至少 1 条，含绝对路径]

## {工具B} 需要做
[至少 1 条具体任务]

## 约束
[至少 1 条，没有真实约束则写「无特殊约束」]

## 产出位置
[至少 1 个绝对路径]
```

**自动检查**：Hermes 扫描 handoffs/ 目录时，自动校验每个文件的四栏完整性。不完整的 → 标注 `⚠️ 不合规`。

### 3.3 产出注册合规

**判断标准**：产出满足以下任一条 → 必须写入 artifact-registry.md：
- 未来面试/简历/答辩会用到
- 项目说明/README/PPT/复盘材料
- 调研结论/资料清单
- 代码工具/脚本（可复用）
- 跨工作区引用素材

**不登记**：临时草稿、单次中间推理、无法复用的碎片。

**自动检查**（Hermes 每周执行）：
1. 扫描各工作区 `outputs/` 的新文件
2. 比对 `artifact-registry.md`，找出「有产出但未注册」的文件
3. 生成遗漏清单 → 报告用户

## 四、Layer 3：Hermes 周期审计

### 4.1 每周合规扫描（周一自动执行）

**Hermes 扫描以下 5 项，输出合规报告：**

| 扫描项 | 检查内容 | 违规定义 |
|--------|----------|----------|
| **Handoff 断链** | 超过 7 天未被消费的 handoff 文件 | 任务流转中断 |
| **产出未注册** | outputs/ 有新文件但 registry 无对应条目 | 产出不可发现 |
| **边界越界** | 输出文件内容与工具职责不匹配（如 Newmax 目录下有 .html 文件，WorkBuddy 目录下有长篇分析 .md） | 工具越界 |
| **规范漂移** | 同一工具连续 3 次跳过启动门禁/收工门禁 | 自我合规松懈 |
| **Handoff 不完整** | handoff 文件缺少四栏中任一栏 | 交接不规范 |

### 4.2 违规升级规则

| 违规次数 | 级别 | 动作 |
|----------|------|------|
| 第 1 次 | 🟡 提醒 | Hermes 在下次该工具进场时注入提醒 |
| 第 2 次（同类） | 🟠 警告 | Hermes 在任务开始前阻断，要求确认合规后再放行 |
| 第 3 次（同类） | 🔴 升级 | Hermes 向用户报告 + 写入 long-term.md 教训 |
| ≥5 次（任意类） | ⛔ 冻结 | 该工具的启动门禁从「自检」升级为「强制阻断」——必须用户手动确认每一项后才能继续 |

### 4.3 异常触发审计

以下情况 Hermes 立即触发针对性审计，不等到周一：

| 触发信号 | 审计动作 |
|----------|----------|
| 用户说「为什么 XX 没做」「XX 在哪里」 | 全链路回溯：handoff → 产出注册 → 工具职责 |
| 同一任务被两个工具重复执行 | 越界审计：检查谁的 handoff 没被正确处理 |
| 产出路径写到别的工具的工作区 | 隔离违规审计 |
| artifact-registry 出现冲突条目（同文件被注册两次） | 重复注册审计 |

## 五、合规信号速查表

**每个工具在执行中检测到以下信号时，立即触发合规检查：**

| 信号 | 含义 | 动作 |
|------|------|------|
| 「我顺手也把…」 | 越界 | 停止，对照职责规范 |
| 「这个我也能做」 | 越界 | 能做 ≠ 该做，handoff 给正确工具 |
| 「先做着，回头再…」 | 跳门禁 | 回头就忘。现在就做 |
| 「太简单了不用写 handoff」 | 偷懒 | 「简单」不是跳过协议的理由 |
| 「Hermes 应该知道了吧」 | 假设 | Hermes 只从文件系统读事实。没写 = 不知道 |
| 「上次是这样做的」 | 路径依赖 | 先对照当前规范，上次的做法可能已过时 |

## 六、各工具违规模式速查

### Hermes 特有违规

| 违规 | 检测方式 |
|------|----------|
| 上下文注入 > 500 字 | 用户反馈「太啰嗦」 |
| 未报告已发现的矛盾 | 新事实写入时跳过了对比 |
| 生成内容（写了分析/代码） | 输出中包含非结构化文本 |

### Newmax 特有违规

| 违规 | 检测方式 |
|------|----------|
| 写了复杂代码（> 100 行逻辑） | outputs/ 中出现 .py/.js 且有复杂逻辑 |
| 做了 HTML 页面 | outputs/ 中出现 .html 文件 |
| 产出未注册 artifact-registry | outputs/ 新文件 vs registry 比对 |
| 不经 Hermes 开始跨工具任务 | 跳过了启动门禁 SG-3 |

### Codex 特有违规

| 违规 | 检测方式 |
|------|----------|
| 写了营销文案 | 输出含 XHS/公众号风格文本 |
| 顺手重构不相关代码 | diff 范围明显大于任务要求 |
| 未写边界声明 | 交付物只有代码没有三栏边界 |
| 未守 Git 规范 | commit message 不含 Conventional Commits 格式 |

### WorkBuddy 特有违规

| 违规 | 检测方式 |
|------|----------|
| 写了长篇分析内容 | 输出含大量非 UI 文本 |
| 未读 handoff 就开始做 UI | 产出内容与 handoff 要求不匹配 |
| 引入重型框架做简单页面 | HTML 文件含 React/Vue CDN 引用 |
| 飞书内嵌用了 WebGL/Canvas | HTML 文件含 canvas/WebGL 相关代码 |

## 七、合规报告模板

### 每周合规报告（Hermes 产出，存入 mutual/outputs/）

```markdown
# 四位一体合规周报 — {YYYY-MM-DD}

## 总览
| 工具 | 本周任务数 | 合规通过 | 违规 | 合规率 |
|------|-----------|----------|------|--------|
| Hermes | N | N | N | N% |
| Newmax | N | N | N | N% |
| Codex | N | N | N | N% |
| WorkBuddy | N | N | N | N% |

## Handoff 链路健康
- 正常交接：N 条
- 断链（>7天未消费）：N 条
  - [handoff 文件路径]（{天数}天）

## 产出注册遗漏
- [文件路径]：已产出但未注册

## 越界事件
- [工具A] 做了 [工具B] 的活：[具体产出]（{日期}）

## 待处理违规
- [违规描述] → 建议：[纠正措施]
```

## 八、启动时合规注入

每个工具在启动序列中必须追加以下自检提示：

```
🛡️ 合规提醒：你正在以 [{工具名}] 角色运行。
- 你的主战场：[从职责规范 § 二]
- 你不该做：[从职责规范 § 六、禁止事项]
- 本次任务如果不在主战场内 → handoff 给正确工具
- 开始前先过启动门禁（SG-1 到 SG-5）
- 完成前先过收工门禁（FG-1 到 FG-5）
```

**注入方式**：各工具的职责规范 § 启动检查清单已包含门禁引用。
