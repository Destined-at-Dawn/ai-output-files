# 文件管理与SOP识别综合治理方案

> **诊断日期：** 2026-05-23
> **诊断范围：** 5个工作区（创作/竞赛/个人/求职/mutual）+ 知识中枢 + 全局配置
> **核心发现：** 文件管理混乱只是表象，真正的病根是SOP识别断裂——新对话中的AI不认识已学SOP，每次重新发明轮子

---

## 一、SOP识别断裂（核心痛点）

### 1.1 症状描述

- 新对话中AI不主动识别已有SOP，直接用默认行为执行
- AI不会说"我识别到你这个任务应该走SOP-02"，而是直接开始干活
- 每个对话都从零开始，不记得过去做过类似任务、用过什么流程
- 用户体验：明明建了15个SOP，但AI像个新手一样每次从头摸索

### 1.2 四个断裂点

#### 断裂点1：SOP索引不在对话启动链中

当前CLAUDE.md要求对话开始时读取的东西：
1. AGENT.md（第1行引用）
2. MEMORY.md（第66行）
3. skill-rules.json（第212行）
4. 铁律F1-F7（第73行）
5. 路由表（第187行）

**但没有要求读取 SOPs/SOP总索引.md。** 唯一提到SOP索引的地方在F5铁律层第149行：

```
写作前**必须**读 SOPs/SOP总索引.md → 匹配指令 → 读对应 SOP
```

这条规则**仅限于"写作前"**，不是通用强制规则。处理其他类型任务（分析文章、做PPT、找论文等）时，AI不会主动读SOP索引。

**结果：** 15个SOP中有12个永远不会被AI主动触发。

#### 断裂点2：skill-rules.json跳过SOP层

当前路由链路：
```
用户意图 → skill-rules.json → 具体Skill
```

应该的链路：
```
用户意图 → SOP总索引匹配 → SOP-00清晰度检查 → 执行SOP → SOP内调用Skill
```

skill-rules.json有81条路由规则，但只有3条涉及"SOP"关键词。它是一个**技能路由表**，不是SOP路由器。两者的设计目标不同：

| 维度 | skill-rules.json | SOP总索引 |
|------|-------------------|-----------|
| 粒度 | 单个Skill | 多步流程 |
| 包含内容 | 触发词→Skill名 | 触发词→完整执行步骤→质量检查→收尾 |
| 缺什么 | 没有流程、没有质量检查 | 没有被路由机制覆盖 |

**结果：** AI直接调用Skill，跳过了SOP中定义的标准化流程、质量检查和收尾步骤。

#### 断裂点3：记忆中没有SOP使用记录

读取 MEMORY.md（28个记忆文件），没有一条记录是：
- "今天用SOP-02采集了某篇文章"
- "上次写公众号用了SOP-01的Step 3概念寓言化"
- "SOP-00拦截了一次模糊指令，生成了3个变体"

记忆中只有**项目级**的记录（公众号分身系统、求职决策系统等），没有**SOP使用级**的记录。新对话读了记忆也读不到"我上次用了什么流程"。

**结果：** 记忆系统能记住"做了什么项目"，但记不住"用什么流程做的"，更记不住"这个流程好用不好用"。

#### 断裂点4：F5铁律的写作限定太窄

F5第149行的规则：
```
写作前**必须**读 SOPs/SOP总索引.md → 匹配指令 → 读对应 SOP
```

问题：
- **"写作前"** 限定了只有写文章/公众号时才触发，但实际上做PPT、做科研日报、分析文章等也需要走SOP
- 这条规则被埋在"中文标点与约束文件必查铁律"里，标题暗示它只管标点和约束文件
- AI没有理由在处理非写作任务时去读"中文标点铁律"

### 1.3 根因总结

```
根因链：
SOP索引不在启动链 → AI不知道有哪些SOP可用 → 跳过SOP直接调Skill
                                              ↓
记忆不记录SOP使用 → 新对话无法参考过去的SOP经验 → 每次从零开始
```

本质：**SOP系统是"建好了但没接入电网"的状态**。文件存在、结构完整，但没有被AI的决策链路覆盖。

### 1.4 修复方案

#### 方案A：最小改动（推荐先试）

在CLAUDE.md的路由表段（第187行附近）新增一条强制规则：

```markdown
### 🔴 SOP识别层（每次收到用户指令时必执行，在路由表之前）

> **此层优先于skill-rules.json路由。先匹配SOP，再匹配Skill。**

**执行流程：**
1. 读取 `SOPs/SOP总索引.md` 的「二、消息路由规则」段
2. 按触发条件匹配SOP编号
3. 匹配成功 → 读取对应SOP文件 → 按SOP步骤执行
4. 匹配失败 → 回退到skill-rules.json路由
5. SOP执行过程中需要调用具体Skill时，在SOP步骤内调用

**AI必须在回复开头声明：**
- 识别到了哪个SOP（如："匹配到SOP-01内容创作SOP，按流程执行"）
- 如果没匹配到任何SOP，说明为什么（如："未匹配到标准SOP，回退到Skill路由"）
```

#### 方案B：记忆增强（配合方案A）

在记忆中增加SOP使用记录维度。修改每日记忆的写入模板，增加：

```markdown
## SOP使用记录
- 任务：{简述}
- 使用SOP：{SOP编号+名称}
- 执行路径：{是否跳过了某些步骤、是否有偏差}
- 效果评估：{好/一般/差，简述原因}
```

#### 方案C：skill-rules.json增加SOP前置层

在skill-rules.json的所有规则之前，增加一条优先级最高的SOP识别规则：

```json
{
  "id": "sop-0",
  "priority": 0,
  "trigger": "ANY",
  "action": "READ_SOP_INDEX_FIRST",
  "target": "SOPs/SOP总索引.md",
  "description": "任何用户指令先过SOP索引匹配，匹配成功则走SOP流程而非直接调Skill"
}
```

### 1.5 预期效果

| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| 用户说"写一篇公众号" | AI直接开始写，可能用默认流程 | AI声明"匹配SOP-01"→读SOP→按5步流程执行 |
| 用户丢一个公众号链接 | AI可能开始分析 | AI声明"匹配SOP-02"→先过SOP-00清晰度检查→按采集流程执行 |
| 用户说"做个PPT" | AI开始做PPT | AI声明"匹配SOP-04"→读PPT制作SOP→按流程执行 |
| 用户说"帮我想个更好的提示词" | AI直接改提示词 | AI声明"匹配SOP-00"→3维度评分→生成3个变体让用户选 |

## 三、目录嵌套问题

### 3.1 现状

5个工作区全部存在"目录套同名目录"：

```
E:\ai产出文件\牛马\创作\          ← 外层（只有log + 创作子目录）
  └── 创作\                       ← 内层（真正的工作区内容）
      ├── CLAUDE.md
      ├── 公众号\
      ├── SOPs\
      └── ...
```

同样结构：`竞赛\竞赛\`、`个人\个人\`、`求职\求职\`、`mutual\mutual\`

### 3.2 影响

- 所有路径凭空多一层（如 `E:\ai产出文件\牛马\创作\创作\SOPs\` 本可以是 `E:\ai产出文件\牛马\创作\SOPs\`）
- 全局CLAUDE.md中的路径规则全部带双层目录名
- 知识中枢注册表中的路径全部带双层目录名
- 路径越长越容易出错（特别是Bash命令中的反斜杠问题）

### 3.3 可能的解决方案

#### 方案A：不动目录，只在注册表中记录（风险最低）

- 在知识中枢注册表中记录"当前路径包含冗余层级"的事实
- 所有新写的规则/文档使用当前实际路径
- 不改变任何现有路径

#### 方案B：逐步扁平化（中等风险）

- 选一个工作区（如创作）做试点
- 将 `创作\创作\` 的内容提升到 `创作\`
- 更新该工作区的CLAUDE.md、注册表、所有引用路径
- 验证无误后再推广到其他工作区
- **需要更新的文件**：全局CLAUDE.md + 工作区CLAUDE.md + 注册表 + 所有SOP中的路径引用

#### 方案C：用符号链接过渡（低风险但依赖系统支持）

- 将 `创作\` 直接设为符号链接指向 `创作\创作\`
- 不改变任何内部路径
- 但Windows符号链接需要管理员权限，且git对符号链接支持不佳

### 3.4 建议

**短期（1-2周）：方案A** — 先记录问题，不动结构
**中期（1个月内）：方案B** — 选一个工作区试点扁平化
**长期：方案B全面推广** — 在所有工作区完成后再做

## 七、关于记忆文件可读性的追问

用户在诊断过程中提出了一个关键问题：

> "你确定这个工作区的所有memory文件都是AI可以直接读的吗？为什么连最简单的分析文章和读取公众号在其他工作区或者新开一个对话之后AI就不懂继续它的SOP？"

**回答：** 记忆文件本身是可以读的（28个文件，内容完整）。但问题在于：

1. **记忆文件没有记录SOP使用经验** — 28个记忆文件都是关于项目/偏好/反馈的，没有一条是"上次用SOP-02采集文章的效果如何"
2. **记忆文件没有被强制读取** — 新对话启动时读的是MEMORY.md索引，但索引中没有"SOP使用"维度的条目
3. **SOP和记忆是两个独立系统** — SOP定义了"怎么做"，记忆记录了"做了什么"，但两者之间没有桥接

**真正缺失的是一句话**：AI在新对话中应该说"我看到你之前用SOP-01写过公众号文章，这次也走这个流程吗？"
