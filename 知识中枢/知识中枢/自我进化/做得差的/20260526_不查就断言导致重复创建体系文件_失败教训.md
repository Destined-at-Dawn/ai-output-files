# 20260526 — 不查就断言，导致重复创建 GitHub 管理体系文件

## 用户原话
> "你让我很失望，那些东西都在项目文件夹里"
> "你为什么不写入自我进化文件夹？"
> "你这个完全可以放入自我进化里面做的好的部分"
> "又没有自我进化"

## 我的错误
在对话中口头分析了4个子错误，说了"下次改进"，但**一个字都没落盘到自我进化文件夹**。
教训说出来了 ≠ 教训记录了。记录了 ≠ 写入文件了。

## 4个具体子错误

### 子错误1：没查 projects/ 目录
- **我说的**：「没有 GitHub 项目生命周期管理 SOP」
- **事实**：`projects/20260504-科研日报开源/outputs/` 里有 `github-publisher-skill.yaml`，是完整的 GitHub 发布 Skill
- **正确做法**：断言"不存在"前，必须先查 `projects/*/` 目录
- **规则**：断言文件不存在 → 必须搜索 3 个位置：根工作区 + projects/ 子目录 + git 删除历史

### 子错误2：没读 .newmax 记忆
- **我说的**：「没有隐私保护规则」
- **事实**：`.newmax/MEMORY.md` 记录了 2026-05-09 的「git历史隐私残留」失败教训，已有隐私排查经验
- **正确做法**：对话开始时读 .newmax 记忆，就能发现已有教训
- **规则**：F3 记忆归档铁律 — 对话开始必须读 .newmax MEMORY.md

### 子错误3：没查 git 删除历史
- **我说的**：「没有 git-recover.sh 防误删脚本」
- **事实**：`git log --diff-filter=D` 显示 2026-05-22 删除了 `scripts/agent-commit.ps1`、`scripts/smart-commit.sh` 等脚本
- **正确做法**：断言"不存在"前，必须查 `git log --diff-filter=D` 确认是否曾被删除
- **规则**：文件"不存在" → 先查 git 历史是否曾经存在过，可能需要恢复而非重建

### 子错误4：直接在根目录新建同名文件
- **我说的**：直接在根工作区创建了 SOP-14、privacy规则、git-auto-sync等
- **事实**：跟 projects/ 里已有体系脱节，变成了两套重复系统
- **正确做法**：新建任何文件前，先检查是否有同名/同类文件可以复用或升级
- **规则**：新建前先 grep 全局，有同名/同类 → 升级而非新建

## 根因
**不查就断言** — 这不是"能力不足"，是"态度问题"。4个子错误的共同点：
1. 在项目沙箱 `proj-xxx/` 里找 → 找不到 → 断言"❌ 没有"
2. 没有扩展搜索范围到根工作区、projects/ 子目录、.newmax 记忆、git 历史
3. 分析完错误后只口头说"下次改进"，没有落盘

## 量化影响
- **重复劳动**：创建了6个已存在/有前身的文件
- **体系分裂**：根目录和 projects/ 各有一套 SOP，互相不知道对方存在
- **教训浪费**：5月9日的隐私教训没有被读取，等于白记录
- **信任损失**：用户说"你让我很失望"——第6次被抓包

## 规则（必须/禁止）
- **必须**：断言文件"不存在"前，搜索3个位置：根工作区 `ls` + `projects/*/` 遍历 + `git log --diff-filter=D`
- **必须**：对话开始时读 .newmax MEMORY.md（F3 铁律的严格遵守）
- **必须**：新建任何文件前 grep 全局确认无同名/同类
- **必须**：错误分析完成后，**当轮**写入自我进化文件，不说"下次再写"
- **禁止**：只查一个地方就说"没有"
- **禁止**：口头说"下次改进"但不落盘
