# 2026-05-20 CMD 编码不匹配导致 I2C 通信全乱（Bug #9）

> **严重度**：🔴 致命
> **影响**：SCL/SDA 一直高电平，I2C 通信完全不工作，调试 2 天
> **教训来源**：触摸屏工程 `D:\AMD\touchscreen\`

---

## 发生了什么

i2c_master.v 和 touch_ctrl.v 两个模块独立定义了 I2C 命令编码：

```verilog
// i2c_master.v（旧版）
localparam CMD_START    = 3'd0;
localparam CMD_WRITE    = 3'd1;
localparam CMD_READ_ACK = 3'd2;
localparam CMD_READ_NAK = 3'd3;
localparam CMD_STOP     = 3'd4;

// touch_ctrl.v
localparam CMD_IDLE     = 3'd0;
localparam CMD_START    = 3'd1;
localparam CMD_WRITE    = 3'd2;
localparam CMD_READ_ACK = 3'd3;
localparam CMD_READ_NAK = 3'd4;
localparam CMD_STOP     = 3'd5;
```

**全部差 1。** touch_ctrl 发 CMD_START(=1)，i2c_master 收到的是 CMD_WRITE(=1)。

## 后果

- touch_ctrl 发 START → master 当 WRITE 执行（释放 SDA 放数据，不是拉低 SDA 产生 START 条件）
- touch_ctrl 发 WRITE 0x70 → master 当 READ_ACK 执行（释放 SDA 等从机数据）
- touch_ctrl 发 STOP(=5) → master 走 `default` 分支直接 done（什么也没做）
- FT6336U **从未收到一个正确的 I2C 事务**
- 结果：SCL/SDA 一直高电平，touch_valid 永远为 0

## 为什么发现这么晚

1. 两个模块分别在不同文件、不同时间写成
2. 都叫 `CMD_START`、`CMD_WRITE`，名字一样，**数值不一样**
3. 没有编译/仿真阶段能暴露这个问题（综合不检查跨模块常量一致性）
4. 只有上板后看到 I2C 无波形才暴露

## 根因

**两个模块各自独立定义了同名常量，但数值不同，且没有任何交叉检查机制。**

这是多模块工程中最隐蔽的 Bug 类型之一——常量名相同，语义相同，但数值差 1，编译器不会报错，仿真不一定能发现（取决于 TB 是否覆盖了完整的 I2C 事务）。

```
### 6. 多模块常量/命令编码一致性铁律（⚠️ 2026-05-20 新增）

- 多模块工程中，所有共享的命令编码/常量定义必须逐一比对：master 发 CMD_START(=1)，slave 收到 CMD_WRITE(=1)——差 1 就是通信全乱。
- 创建工程时的强制步骤：列出所有模块间的共享常量，逐个验证数值一致。
- 禁止各自定义常量后"假设一致"。推荐做法：公共常量放在一个 include 文件里，或用参数传递。
- 验证方法：用 Grep 搜索所有模块中的 CMD_ 定义，逐行对比数值。
```

## 触发条件

- 新建多模块工程
- 涉及主从通信（I2C/SPI/UART 协议模块之间）
- 多个模块各自定义 `localparam CMD_xxx`

## 防范措施

1. **创建工程的第一天**，列出所有跨模块常量表，逐一比对数值
2. 推荐将公共常量放在 `include` 文件或用参数传递，避免各自定义
3. 用 Grep `CMD_` 搜索全工程，检查是否有同名不同值的定义
4. Testbench 覆盖完整的 I2C/SPI 事务（不只是单字节），能更快暴露此问题
