# SOP-F：个人信息跨工作区同步（telos）

> **版本**：v1.0 | **创建**：2026-05-07 | **来源工作区**：竞赛
> **核心原则**：`C:\Users\13975\.claude\telos\` 是小黎个人信息的「单一真相源」，所有工作区共同维护。

---

## 一、为什么需要这个 SOP

小黎有多个工作区（创作/竞赛/求职/个人），每个工作区都有独立的 CLAUDE.md 和 memory/。当小黎在一个工作区说"我最近在做 XXX"，只有那个工作区记住了，其他工作区一无所知。

**telos 解决这个问题**：8 个文件存储全局个人信息，所有工作区读写同一份数据。

---

## 二、决策表：什么时候触发同步？

| 用户说了什么 | 更新哪个 telos 文件 | 操作 |
|------------|-------------------|------|
| "我最近在做科研日报" / "我退出了某个比赛" | `identity.md` → §当前状态 | 追加条目+日期 |
| "我的目标是 XXX" / "我改主意了" | `goals.md` | 修改对应目标+变更记录 |
| "我最近在关注 XXX 领域" | `interests.md` | 追加到对应分类 |
| "我学会了 XXX 工具/技能" | `skills.md` | 追加到对应分类 |
| "我觉得 XXX 是对的" / "我信 XXX" | `beliefs.md` | 追加到对应分类 |
| AI 被批评 / 小黎说"以后要 XXX" | `learned.md` | 追加教训或偏好条目 |
| "我以后回复简洁点" / "我喜欢 XXX 方式" | `learned.md` 或 `work-style.md` | 追加偏好条目 |
| 小黎的工作节奏/时间安排变化 | `work-style.md` | 修改对应部分 |

## 三、同步流程（4 步，不可跳过）

```
Step 1：识别 — 本轮对话中是否读到小黎的新个人信息？
  → 是 → 继续
  → 否 → 跳过同步

Step 2：读取 — Read telos 对应文件当前内容
  → 确认文件存在且可读

Step 3：追加 — Edit 在对应区域追加新信息
  → 格式：`- {新信息}（{日期}，来源：{当前工作区名}）`
  → 在文件末尾「变更记录」表追加：`| 日期 | 变更摘要 | 来源：竞赛工作区 |`

Step 4：验证 — Read 确认追加内容已落盘
  → 验证通过 = 同步完成
  → 验证失败 = 重新执行 Step 3
```

---

## 四、禁止行为

- ❌ **禁止**只更新本工作区而不更新 telos
- ❌ **禁止**覆盖 telos 文件——只允许追加和修改
- ❌ **禁止**删除 telos 中的已有信息（除非用户明确要求）
- ❌ **禁止**在本工作区 CLAUDE.md 中硬编码小黎的个人信息而不同步到 telos
- ❌ **禁止**Write 后不 Read 验证

## 五、telos 与本工作区 memory 的关系

| 系统 | 存什么 | 谁维护 |
|------|--------|--------|
| **telos/** | 全局个人信息（身份/目标/偏好/教训） | 所有工作区共同维护 |
| **memory/long-term.md** | 本工作区的长期记忆（项目决策/工具选择） | 本工作区 |
| **memory/{日期}.md** | 本工作区的每日进展 | 本工作区 |

**关键区别**：telos 是「我是谁」，memory 是「我在这个项目里做了什么」。两者互补，不替代。

## 六、示例场景

### 场景 A：小黎提到新经历

> 小黎："我最近在帮老师做一个电力系统仿真项目"

AI 行动：
1. 识别 → 这是新经历，需要同步
2. Read `telos/identity.md`
3. Edit 在「当前状态」区域追加：`- 正在参与电力系统仿真项目（2026-05-07，来源：竞赛工作区）`
4. Read 验证

### 场景 B：小黎调整目标

> 小黎："我改主意了，不想去国网了，想去做芯片设计"

AI 行动：
1. 识别 → 目标调整，需要同步
2. Read `telos/goals.md`
3. Edit 在「长期目标」区域修改+在变更记录追加
4. Read 验证

### 场景 C：AI 被批评

> 小黎："你每次回复都太长了，以后简洁点"

AI 行动：
1. 识别 → 用户偏好变化，需要同步
2. Read `telos/learned.md`
3. Edit 在「用户偏好」表追加：`| 回复风格 | 默认简洁，除非用户要求详细 | 2026-05-07 | 竞赛工作区 |`
4. Read 验证

### 场景 D：没有新信息

> 小黎："帮我查一下 FPGA 的时序约束怎么写"

AI 行动：
1. 识别 → 这是技术问题，不涉及新个人信息
2. 跳过同步，直接回答问题

## 八、认知科学支撑（百大认知书籍）

### 跨工作区记忆的认知架构

| 认知机制 | 来源 | 在个人信息同步中的应用 |
|---------|------|----------------------|
| **分布式认知** | 008-协同智能 §概念1 | 多工作区系统=分布式认知系统——每个工作区是一个"认知节点"，telos是"共享记忆"。如果节点之间不同步→每个节点的"认知模型"都会偏离"真实状态"→决策质量下降 |
| **编码特异性** | 009-认知天性 §概念1 | 信息在哪个工作区被记录，就会在那个工作区最容易被检索。同步到telos=创建"跨情境编码"→任何工作区都能检索到→信息可得性提升 |
| **WYSIATI** | 010-思考快与慢 §概念3 | "我在这个工作区看到了这些信息→小黎就是这样的"=WYSIATI陷阱。每个工作区只看到小黎的一部分→拼凑出不完整画像。telos同步=强制打破WYSIATI→所有工作区看到完整真相 |
| **记忆衰减** | 009-认知天性 §概念3 | 没有记录的信息以指数速度衰减——"小黎上周说了XX"→3天后只剩模糊印象→7天后完全遗忘。Step 3"立即追加"的设计就是对抗记忆衰减 |
| **一致性偏见** | 010-思考快与慢 §概念5 | 每个工作区会倾向于维护自己的"一致性叙事"——"小黎在这个工作区一直是这样的"→忽略来自其他工作区的更新。telos同步=打破一致性偏见→强制接受"小黎已经变了" |

### 同步流程的认知工程设计

| 流程步骤 | 认知原理 | 来源 | 设计理由 |
|---------|---------|------|---------|
| Step 1 识别 | 注意力引导 | 020-专注力管理 §概念1 | 不是所有对话内容都需要同步→需要"注意力过滤器"→只捕获"新个人信息"→降低同步的认知成本 |
| Step 2 读取 | 先验确认 | 010-思考快与慢 §概念2 | 先读取当前文件→建立"当前状态"先验→追加操作基于已知状态而非"记忆中的状态"→避免WYSIATI |
| Step 3 追加 | 增量编码 | 009-认知天性 §概念1 | 追加（而非覆写）=增量编码→保留历史轨迹→任何时间点都能看到"小黎的变化过程"→支持趋势分析 |
| Step 4 验证 | 反馈回路 | 016-习惯的力量 §概念1 | "追加→验证"形成正反馈回路→验证成功→确认操作正确→信任建立→习惯固化→同步成为自动行为 |

### 禁止行为的认知科学解释

| 禁止行为 | 违反的认知原理 | 来源 | 后果 |
|---------|--------------|------|------|
| 只更新本区不同步telos | 信息孤岛 | 008-协同智能 §概念1 | 各工作区形成独立的"小黎画像"→决策基于不完整信息→跨工作区任务出错 |
| 覆写telos文件 | 记忆丢失 | 009-认知天性 §概念3 | 覆写=用新信息替换旧信息→旧信息永久丢失→无法追溯"小黎曾经的目标/经历" |
| 删除已有信息 | 损失厌恶 | 010-思考快与慢 §概念4 | 删除的信息可能在未来有参考价值→"曾经有过的经历/目标"本身也是重要数据→保留比删除更安全 |
| Write后不Read验证 | 反馈缺失 | 016-习惯的力量 §概念1 | 没有验证→不确定操作是否成功→无法建立"操作→结果"的反馈回路→同步行为无法固化为习惯 |
