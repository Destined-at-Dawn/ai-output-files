## 2026-05-17 工作记录

### 10:00 PDF/docx文档生成能力确立
- **问题**：老板和学生需要 docx+pdf，不是 md/txt
- **解决**：安装 python-docx（已有）+ fpdf2（绕代理安装成功）
- **fpdf2 CJK Bug**：v2.8.7 的 `multi_cell(w=0)` + `align='C'` 触发 "Not enough horizontal space"，累计失败10+次
- **Workaround**：强制显式宽度 + align='L' + safe_text过滤 + try-catch
- **产出**：4个文档文件（2个docx + 2个pdf），转换脚本 `outputs/convert_md_to_docx_pdf.py`

### 11:00 永久参考资料库建立
- **教训**：上个对话的PDF提取结果全在 /tmp，会话结束丢失
- **修复**：Python+fitz提取8个PDF/docx到 `outputs/references/`（1.9MB）
- **数据链**：原始PDF → Python+fitz → outputs/references/*.txt → memory引用 → 实验文档

### 12:00 工程文件三处同步修复
- **根因**：lcd_ctrl.v 在根目录(20KB教学注释版) vs Vivado imports(13KB旧版) vs outputs(13KB旧版) 三处不同步
- **修复**：5个.v文件全部从根目录同步到三处
- **铁律固化**：项目CLAUDE.md新增"工程文件三处同步铁律"

### 14:00 Verilog文件编码修复（重大教训）
- **根因**：.v文件是UTF-8，Vivado+Windows中文系统用GBK读，中文注释全部乱码
- **AI连续犯错3次**：猜UTF-8→转GBK→Read工具验证乱码→误判转回UTF-8→更乱
- **最终修复**：Python `text.encode('gbk')` 直接转码，不经过Read/Edit工具
- **核心教训**：非UTF-8文件必须全程Python操作，绝不经过Read/Edit/Write工具

### 16:00 实验文档屏幕参数修正
- **问题**：规格书标320(H)×480(V)，代码用480列×320行，方向冲突
- **用户确认**：长边480在左右（水平），短边320在上下（垂直），代码正确
- **修复**：文档从"竖屏320×480"修正为"横屏480×320"

### 17:00 交付目录创建
- `D:\AMD\project_output\` — TFT-LCD完整工程 + 实验文档（docx+pdf）
- lab1_led_seq 双归档后从交付目录移除（用户要求只保留TFT实验）
