# 十条贯穿性工程法则

> 横轴 = 跨所有阶段始终生效。每条：**原理 / 触发信号 / 实操 / 反模式**。
> 来源：副教授「可复用 AI 工程方法论」第三层，经小黎团队适配。

---

## 法则 1　证据分层——任何「通过/达标」必须带口径标签

**原理**：「通过」不是布尔值，是带层级的；不标层级的「通过」无意义。

**触发**：要写下或引用任何 PASS、达标、分数、准确率时。

**实操**：
- 结果一律写成 `<指标> = <值> @ <口径>`
- 口径分层：
  - **粗筛**：快速验证、in-sample、单次运行 → 只能用来排序
  - **中级**：交叉验证、多次运行、k-fold → 可以做方向性判断
  - **全签核**：独立复现、完整流程、盲测、真实环境 → 可以下结论
- 报告里不同口径的结果用**不同字段名**，永不混用

**反模式**：把快速校验 / in-sample / 预签核结果直接写成最终签核 PASS。

## 法则 2　防假象审计——高分先审「产生方式」

**原理**：异常的好数字（与异常的坏数字）都先假设是假象，直到证明不是。

**触发**：出现明显高于预期的指标；跨轮/跨变体挑出来的「最优」。

**实操——五连问**：
1. 是不是 N 选 1？（从 N 个变体中挑了最好的那个）
2. 是不是挑最高？（从多次运行中取了最大值）
3. 是不是取曲线峰值？（参数扫描中取了最优点）
4. 是不是 in-sample 设计的规则？（在训练集上评估的）
5. 流程跑完整了吗？（有没有跳过验证步骤）

**审计双向**——坏数字也可能是 bug 假象，不一定真坏。

**反模式**：跨多轮逐类挑最优 + 内层用 test 选超参，然后报均值。

---

## 法则 3　实测优先于直觉——优化方向必须扫描验证

**原理**：复杂系统的响应常常非单调、反直觉；「越紧越好/越多越好/越短越好」都可能错。

**触发**：要调一个连续参数（任何可调变量）。

**实操**：扫一组值实测，**画响应曲线再决策**；端点和中段都跑。

**反模式**：假定单调直接取端点；凭"应该这样"决策。

---

## 法则 4　给成熟工具自由度——克制人工 hint

**原理**：成熟的自动化工具在其设计域内通常优于人工干预；人工 hint 常被静默忽略或起反效果。

**触发**：想用"硬约束""强制实现指令"去「帮」工具。

**实操**：先让工具自由跑；约束施加后**实测是否真生效**（验证 ≠ 假设生效）；人工 hint 是最后手段。

**反模式**：用强制 hint 锁死工具，反而把关键路径做得更长。

---

## 法则 5　跨边界先校验——格式/编码/单位/命名/口径

**原理**：bug 高发在「两个系统的接缝处」。

**触发**：数据/文件/模块在两个工具、两个团队、两个数据源之间传递时。

**实操**：接缝处先显式校验——文件编码、数值单位（公制/英制）、字段语义（同名字段两边含义可能不同）、模块命名（同名冲突需先 grep）、API 版本。

**反模式**：假定「另一边的格式跟我一样」直接喂进去。

---

## 法则 6　隔离契约——多 Agent 写入区严格隔离

**原理**：并行的前提是隔离；没有隔离的并行会互相覆盖、互相污染。

**触发**：启动任何多 Agent / 多 subagent / 多团队工作。

**实操**：
- 每个 Agent 独占一个写入目录
- 他人目录**只读**
- 复用走 Read → Write 回自己树
- 写入前用 stat/glob 确认目标在自己目录内

**反模式**：两个 background agent 同时写同一文件互相覆盖。

---

## 法则 7　负结果是一等公民——死路必须归档

**原理**：一条没被记录的死路，会被每个后续 Agent 重新踩一遍。

**触发**：任何方案被验证为「无效/更差/不可行」时。

**实操**：写进负结果日志（项目级 + 全局级），含「方案 + 为什么死 + 为什么看起来行其实不行 + 退出判据」。

**反模式**：只留下成功路径，死路靠记忆；负结果只口头流传不入档。

详见 `negative-results.md`。

---

## 法则 8　门禁不可跳过——「完成」= 过全部门禁

**原理**：「完成」不是自我感觉，是通过一组预定义的自动门禁。

**触发**：想宣告某阶段/某交付「做完了」。

**实操**：
- 为项目建一套门禁套件（典型项：静态检查/类型检查/单元测试/数据校验/文档完整性/边界声明）
- **修复任一问题后重跑全部门禁**

**反模式**：改完只检查刚改的那条就宣告完成；门禁挂掉就把那条门禁注释掉绕过。

---

## 法则 9　识别结构墙——平台期换角度，天花板就停

**原理**：「努力没用」分两种——**平台期**（方法没对，换角度还能进）与**结构墙**（物理/数据/工艺上限，再投入是负期望）。

**触发**：同一目标多次不同方法都失败。

**实操**：
- 用 ≥3 个**正交方法维度**独立验证
- 都撞同一极限 → 判定为结构墙，**停手**
- 转而提交「天花板证据」（说明为什么这是墙、需要什么前提才能突破）

**反模式**：在结构墙上继续硬凑；靠脏方案/挑口径/跳门禁制造表面突破。

---

## 法则 10　规则结晶与自迭代——经验 → 规则 → Skill

**原理**：经验若不结晶成规则，就会随上下文丢失；价值在于**复利沉淀**。

**触发**：每轮对话结束、每次调试结束、用户每次纠正。

**实操**：
- 立即把经验凝练成规则写入知识库
- 用户纠正 ≥2 次或同类踩坑 ≥2 次 → 升级为规则甚至独立 Skill
- 新 bug 解决 → 立即写进 bug-kb
- **反过度沉淀阀门**：单次未重复的 feedback 暂不入库（硬指令例外）；同义规则优先 update 旧的，不新增
- 新规则路由到**对应专项 Skill**，不堆进主 CLAUDE.md 导致膨胀

**反模式**：同一个坑排查两次；规则文件无限膨胀人脑记不住因此实际无人遵守。
