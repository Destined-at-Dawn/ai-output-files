# 防假象审计规则

> 任何异常的好数字，先假设是假象，直到证明不是。
> 审计双向——异常的坏数字也可能是 bug 假象。

---

## 核心问题

**AI 最危险的错误不是「答错」，而是「自信地给出错误的好结果」。**

用户不会质疑一个看起来很好的数字。如果这个数字是假象，后续所有基于它的决策都会跑偏。

---

## 五连问（任何「最终数字」出现时必问）

### 问 1：是不是 N 选 1？

**场景**：跑了 N 个变体/配置/方案，只报了最好的那个。

**为什么危险**：如果 N 足够大，随机噪声也能产生一个好看的数字。N 选 1 的最优值不等于方案的真实水平。

**正确做法**：报告所有变体的分布（中位数、标准差、最差值），而不只是最优值。

### 问 2：是不是挑了最高？

**场景**：同一方案跑了多次，取了最大值报告。

**为什么危险**：多次运行的最高值是极端值，不是期望值。如果样本量小，极端值可能是噪声。

**正确做法**：报告均值 ± 标准差，或者中位数 + 四分位距。

### 问 3：是不是取了曲线峰值？

**场景**：参数扫描中取了响应曲线的最高点。

**为什么危险**：峰值点可能是过拟合点（in-sample 最优但 out-of-sample 翻车）。非单调响应中的峰值尤其可疑。

**正确做法**：用独立测试集验证峰值点；报告峰值附近的稳定性（±10% 参数变化结果是否剧烈波动）。

### 问 4：是不是 in-sample / 训练集上的结果？

**场景**：评估指标是在参与模型设计/调参的数据上计算的。

**为什么危险**：in-sample 指标天然偏高，不反映真实泛化能力。

**正确做法**：必须在 held-out / 独立 / 盲测数据上验证。

### 问 5：流程跑完整了吗？

**场景**：跳过了某些验证步骤，或者只跑了部分流程。

**为什么危险**：跳过的步骤可能恰好是发现问题的步骤。部分流程的「通过」不代表全流程的「通过」。

**正确做法**：列出完整流程清单，逐项打勾确认。

---

## 反向审计：异常坏数字

坏数字也可能是假象——可能是因为 bug、数据污染、环境问题导致的。

**检查清单**：
- 数据是否正确加载？（空数据 / 部分数据 / 格式错误）
- 环境是否正确？（版本不匹配、依赖缺失、路径错误）
- 随机种子是否合理？（极端种子导致极端结果）
- 是否有数据泄漏？（测试数据混入训练数据）

## 审计报告模板

```markdown
## 审计对象
[具体指标和数值]

## 五连问结果
1. N 选 1？ [是/否] — [说明]
2. 挑最高？ [是/否] — [说明]
3. 取峰值？ [是/否] — [说明]
4. In-sample？ [是/否] — [说明]
5. 流程完整？ [是/否] — [说明]

## 审计结论
- 可信度：[高/中/低/不可信]
- 建议：[可以使用/需要额外验证/需要重做]

## 证据路径
- 数据来源：[文件路径]
- 运行命令：[具体命令]
- 时间戳：[日期时间]
```

## 信号检测：AI 可能正在制造假象

如果出现以下任何一种情况，**立即停下来做审计**：

- [ ] 报告了一个明显高于预期的数字，但没有质疑
- [ ] 从多个变体中只报告了最优的那个
- [ ] 用「通过」描述一个没有带口径标签的结果
- [ ] 跨多轮挑最优，然后算均值
- [ ] 跳过了某个验证步骤但仍然宣告完成
