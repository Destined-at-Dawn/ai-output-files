# 负结果归档规则

> 一条没被记录的死路，会被每个后续对话重新踩一遍。
> 负结果和正结果一样重要——甚至更重要。

---

## 为什么要归档负结果

**没有归档的死路 = 下一个对话会重新走一遍。**

AI 的上下文是有限的。上一轮对话验证过的「此路不通」，下一轮对话完全不知道。如果不归档，每个新对话都会：
1. 重新提出同样的方案
2. 重新花时间验证
3. 重新发现行不通
4. 重新换方案

**归档一次 = 所有未来对话省去这个成本。**

---

## 什么需要归档

### 必须归档

- 验证为「无效」的方案（做了实验，结果不行）
- 验证为「更差」的方案（做了对比，结果更差）
- 验证为「不可行」的方案（技术上做不通）
- 被用户否决的方案（用户说"不要这个方向"）

### 不需要归档

- 还没验证的猜测（只是想法，没试过）
- 明显不合理的方案（常识判断即可排除）
- 一次性的临时尝试（没有复现价值）

---

## 归档格式

### 标准格式

```markdown
# 负结果：[方案名称]

## 日期
[YYYY-MM-DD]

## 方案
[做了什么，具体到操作级别]

## 已尝试
[具体尝试了哪些变体/参数/方法]

## 失败现象
[观察到什么，用数据说话]

## 为什么看起来行其实不行
[这是最关键的部分——解释为什么这个方案表面上合理但实际走不通]

## 退出判据
[什么时候该放弃这个方向，不要无限投入]

## 关联法则
[涉及哪些工程法则，如法则 3（实测优先）、法则 9（结构墙）]

## 影响范围
[这个负结果影响哪些后续方案/决策]
```

### 简化格式（快速记录）

```markdown
**死路**：[方案] → [为什么不行] → [退出判据]
```

## 归档位置

### 项目级负结果日志

每个项目目录下维护一个 `negative-results.md` 文件：

```markdown
# [项目名] 负结果日志

## 1. [方案名称] (YYYY-MM-DD)
方案：...
失败现象：...
为什么看起来行不行：...
退出判据：...

## 2. [方案名称] (YYYY-MM-DD)
...
```

### 全局负结果日志

跨项目的通用教训归档到全局位置：

```markdown
# 全局负结果日志

## 通用教训
### [教训名称]
- 场景：[什么情况下会遇到]
- 死路：[通常会尝试什么]
- 为什么不行：[为什么走不通]
- 正确做法：[应该怎么做]
```

## 使用流程

### 对话开始时

1. 检查项目是否有 `negative-results.md`
2. 如果有，读取并记住已知的死路
3. 在提出方案时，先排除已知的死路

### 对话过程中

1. 如果发现某个方案不行，**立即归档**
2. 不要等到对话结束才归档（可能会忘记细节）
3. 归档时尽量包含具体数据和观察

### 对话结束时

1. 检查本轮是否有新的负结果需要归档
2. 检查是否有已知负结果需要更新（新的实验可能推翻旧结论）
3. 确保归档格式完整

---

## 常见错误

### 错误 1：只记录「什么行」，不记录「什么不行」

**后果**：后续对话重新踩坑。

**正确做法**：行和不行都记录，不行的往往更有价值。

### 错误 2：归档太笼统

**错误示例**：「方案 A 不行。」

**为什么错**：没有说清楚为什么不行、什么条件下可能行、退出判据是什么。

**正确示例**：「方案 A 在 N=100 时表现好，但 N=1000 时 OOM。原因是内存复杂度 O(N²)。退出判据：如果 N > 500，此方案不可行。」

### 错误 3：归档后不查看

**后果**：归档了但下次对话不读，等于没归档。

**正确做法**：每个新项目/新对话开始时，先读负结果日志。

---

## 与门禁的关系

负结果归档是门禁的一部分：

- **交付门禁**：本轮所有死路是否已归档？
- **启动门禁**：是否已读取已知负结果？
- **方案选择门禁**：候选方案是否排除了已知死路？
