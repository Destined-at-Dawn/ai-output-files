# 20260525 I2C协议致命Bug精准定位 成功案例

## 日期
2026-05-25

## 场景
用户的FT6336U触摸屏工程I2C通信失败，SCL/SDA一直为1。

## 做对了什么

### 1. 精准定位根因：I2C序列缺少设备地址字节
- 对比C代码和Verilog状态机，逐字节对齐
- 发现 CMD_WRITE 只发了寄存器地址(0x02)，没发设备地址(0x70)
- i2c_master是纯字节引擎，不会自动发设备地址
- 这个分析被用户和老板采纳，最终版本修复了这个Bug

### 2. 正确识别open-drain问题
- SDA的push-pull写法违反I2C协议
- 给出了正确的open-drain写法：
- 最终版本采纳了这个修复

### 3. 全局ACK检测假阳性的发现
- case外的全局检测会覆盖case内的正常状态转移
- CMD_READ_NAK的协议性NAK被误判为通信失败
- 最终版本删除了这个全局检测

### 4. 从ILA波形+示波器现象反推根因
- 用户说"按复位后短暂有波形然后全1"
- 正确推断：I2C启动了但FT6336U不回ACK → 设备地址错误
- 这个推理链完全正确

## 为什么成功
- 严格按照C代码逐字节对齐Verilog状态机
- 理解i2c_master的抽象层级（纯字节引擎，不理解协议语义）
- 从物理现象（波形）反推逻辑错误

## 可复用模式
- I2C移植铁律：每个Send_Byte()对应一个独立CMD_WRITE状态
- 调试优先级：bitstream下载 → 状态机启动 → 协议正确性 → 物理层
- 全局检测铁律：禁止在case外放覆盖性赋值
