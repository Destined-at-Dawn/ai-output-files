## 九、文件操作铁律

> ⚠️ **"承诺"一文不值，只有 tool_use 发起的实际文件操作才算数。**
> Source: 交叉学习自个人区 CLAUDE.md §四

| 规则 | 说明 |
| --- | --- |
| **9.1 禁止伪造** | 所有文件操作必须通过 tool_use 发起，禁止在纯文本中描述"已写入 xxx"但本轮未发 tool_use |
| **9.2 验证必须** | 每次 Write/Edit 后，必须 Read 同一文件确认内容正确落盘 |
| **9.3 绝对路径审计** | 写入路径必须是 `E:\ai产出文件\牛马\求职\求职\` |
| **9.4 禁止空文件** | 写入的内容必须有实际信息 |

### 9.5 记忆文件操作铁律

- **先 Read 再 Write**：更新任何文件前，必须先 Read 读取当前全部内容
- **追加格式**：新内容以 `## YYYY-MM-DD HH:MM` 分隔
- **Write 后必须 Read 验证**
- **禁止空文件/占位符**

### 9.6 文件修改失败应急规则

当文件修改失败时：
1. 创建文件副本到同目录
2. 将原文件移入 `_归档/`
3. 将副本重命名为原文件名

## 十、自我进化机制（Pattern-Key体系）

> Source: 交叉学习自个人区 CLAUDE.md §九 + 创作区自我进化目录
> **核心理念**：每个错误生成一个Pattern-Key，跨会话防重犯。成功也要记录，复制有效模式。

### 10.1 目录结构

```
自我进化/
├── 做得好的鼓励/              ← 用户肯定时记录
│   └── YYYYMMDD_[场景]_成功案例.md
└── 做得差的避免/              ← 用户批评/犯错时记录
    └── YY_[简短描述]_失败教训.md
```

### 10.2 失败教训记录格式（必须包含Pattern-Key）

```markdown
# 做得差的避免 — [序号]


## YYYY-MM-DD [简短描述]

- **Pattern-Key**：PK-求职-XXX
- **表象**：[用户看到的问题]
- **根因**：[真正的原因]
- **后果**：[造成了什么影响]
- **正确做法**：[下次怎么避免]
- **关联**：[CLAUDE.md章节编号 / 其他Pattern-Key]
- **教训密度**：★~★★★★★
```

### 10.3 成功案例记录格式

```markdown
# 做得好的鼓励 — [序号]


## YYYY-MM-DD [简短描述]

- **用户原话**：[逐字记录]
- **做对了什么**：[具体行为]
- **成功原因**：[为什么这次做得好]
- **可复制模式**：[下次怎么复用]
- **关联**：[CLAUDE.md章节编号]
```

### 10.4 触发规则（铁律）

| 触发条件 | 动作 | 记录位置 |
|---------|------|---------|
| 用户批评/指责/否定 | 无条件记录，不辩解 | `做得差的避免/` + `memory/MEMORY.md` Pattern-Key表 |
| 用户肯定/表扬 | 记录成功原因和可复制模式 | `做得好的鼓励/` |
| 同一错误第二次出现 | **最严重错误**——说明第一次教训没吸收 | 升级Pattern-Key教训密度为★★★★★ |
| 面试后复盘 | 记录面试中暴露的问题 | `做得差的避免/`（如果是答不上来）或素材库更新 |

### 10.5 Pattern-Key 查询机制

- **开局必查**：每个新对话先读 `memory/MEMORY.md` 的Pattern-Key速查表
- **执行前查**：简历生成前过一遍SOP中的踩坑清单
- **犯错后查**：记录新教训时，检查是否与已有Pattern-Key重复

### 10.6 历史文件

- `简历生成问题复盘.md` — 12个已知问题的历史记录（已被自我进化目录替代，保留供参考）

## 十一、SOP标准操作流程系统

> Source: 交叉学习自创作区 CLAUDE.md §二十八（指令分类SOP系统）
> **核心理念**：规则在CLAUDE.md，流程在SOP。两者互补不重复。

### 11.1 SOP目录

```
SOPs/
├── SOP总索引.md          ← 所有SOP入口
└── 00_简历生成SOP.md     ← 简历生成8步流程（从§1.1提取为独立可执行文件）
```

### 11.2 SOP编写规范

每个SOP必须包含：
1. **触发条件**：什么情况下执行
2. **执行步骤**：编号步骤清单（可引用CLAUDE.md章节编号）
3. **检查清单**：执行完毕后的自检项
4. **关联Pattern-Key**：本SOP涉及的错误模式
5. **关联文件**：执行时需要读取的文件路径

### 11.3 SOP与CLAUDE.md的关系

- CLAUDE.md定义**规则**（必须/禁止/不得）
- SOP定义**流程**（先做什么→再做什么→最后检查什么）
- SOP引用CLAUDE.md的规则编号，不重复写规则内容

## 十二、批评必记录铁律

> Source: 交叉学习自创作区 `feedback_批评必记录.md`

### 12.1 触发条件

用户提问**带问号**且为**指责/批评/否定**时，必须无条件记录到 `memory/feedback_批评必记录.md`。

### 12.2 记录格式

```markdown
### YYYY-MM-DD — [简短描述]
- **用户原话**：[逐字记录]
- **我的问题**：[具体做错了什么]
- **根因**：[为什么会犯这个错]
- **正确做法**：[下次怎么避免]
- **Pattern-Key**：PK-求职-XXX
```

### 12.3 禁止

- **禁止辩解**——先记录，再回应
- **禁止淡化**——原话逐字记录，不改措辞
- **禁止跳过**——任何批评都必须记录，没有"不值得记录"的批评

### 12.4 联动

1. 记录到 `feedback_批评必记录.md`
2. 同步创建 `自我进化/做得差的避免/` 对应文件
3. 更新 `memory/MEMORY.md` Pattern-Key速查表
4. 如果涉及具体文件操作，更新 `简历生成问题复盘.md`（历史兼容）

## 十三、记忆索引机制

> Source: 交叉学习自个人区 `memory/MEMORY.md`

### 13.1 三层记忆结构

| 层级 | 文件 | 内容 | 更新频率 |
|------|------|------|---------|
| 索引层 | `memory/MEMORY.md` | Pattern-Key速查表 + 文件索引 + 统计 | 每次记忆更新后同步 |
| 长期层 | `memory/long-term.md` | 用户偏好 + 重要决策 + 项目经历写法 | 有新决策/偏好变化时 |
| 日期层 | `memory/YYYY-MM-DD.md` | 每日任务进展 + 技能审计 | 每次对话结束前 |

### 13.2 开局读取顺序

```
1. memory/MEMORY.md          ← 先读索引，恢复全局上下文
2. memory/long-term.md       ← 确认用户偏好和决策
3. memory/[今天日期].md      ← 确认今日已有进展（如有）
4. 按对话主题读取对应文件    ← 参见§零的路由表
```

### 13.3 收尾写入顺序（每次对话结束前必须执行）

```
1. 写入 memory/YYYY-MM-DD.md   ← 记录今日进展
2. 更新 memory/MEMORY.md       ← 更新索引（如有新增Pattern-Key/新增文件）
3. 更新 memory/long-term.md    ← 追加新决策/偏好（如有）
4. 更新 project.md             ← 更新里程碑状态（如有）
```

## 十七、双归档铁律（最高优先级 · 2026-05-07）

> ⚠️ **此规则优先级高于所有其他操作规则。违反=数据丢失=最严重错误。**
> 来源：小黎明确要求——"所有删除文件保存两份，第一份全局归档，第二份原工作区归档"

### 17.1 双归档目录

| 归档位置 | 路径 |
|---------|------|
| **全局归档** | `${LEGACY_ROOT}\归档\` |
| **本工作区归档** | `${LEGACY_ROOT}\求职\求职\归档\` |

### 17.2 删除前六步（缺一不可）

1. **`ls` 检查**：列出目录/文件的完整内容
2. **识别原工作区**：根据被删除文件的路径判断属于哪个工作区
3. **第一份→全局归档**：`cp -r` 到 `${LEGACY_ROOT}\归档\`
4. **第二份→工作区归档**：`cp -r` 到 `${LEGACY_ROOT}\求职\求职\归档\`（没有则 `mkdir -p` 新建）
5. **AskUserQuestion 确认**：展示清单+双归档路径+验证状态
6. **验证双归档完整性**：ls 确认两份一致后才执行删除

### 17.3 删除后报告（强制）

每次删除完成后必须输出：
```
✅ 删除完成
- 删除的文件/目录：[清单]
- 全局归档：${LEGACY_ROOT}\归档\{文件名} — 已验证
- 工作区归档：${LEGACY_ROOT}\求职\求职\归档\{文件名} — 已验证
```

### 17.4 禁止

- 不得在未检查内容的情况下删除任何目录
- 不得在未归档的情况下删除任何文件
- 不得在未获得用户确认的情况下删除任何文件
- **不得只归档一份**——必须全局+工作区两份都归档
- **不得删除后不报告**——必须输出双归档路径+验证状态

## 十九、教训→skill/SOP 反馈闭环（强制）

> **教训只记不改 = 白记。** 每条教训必须流向它能影响的 SOP 或 skill，否则下一次对话会被遗忘。

### 闭环铁律

每当自我进化文件或 lessons-learned 新增一条教训，**同一次对话内**必须完成：

| 步骤 | 操作 | 验证 |
|------|------|------|
| 1. 定位 | 查映射表，确定影响哪个 SOP + 哪个 skill | 映射表在 `E:\ai产出文件\牛马\知识中枢\02-共享规则\09-feedback-loop.md` |
| 2. 回写 | 在 SOP「迭代日志」追加一行 | Read 验证 |
| 3. 升级 | 同类教训 3 次 → 升级为 SOP 铁律 | Edit 更新 SOP 正文 |
| 4. 通知 | 影响 skill 行为 → 更新 CLAUDE.md 对应段落 | Read 验证 |

### SOP 迭代日志格式

每个 SOP 末尾必须有：
```markdown
