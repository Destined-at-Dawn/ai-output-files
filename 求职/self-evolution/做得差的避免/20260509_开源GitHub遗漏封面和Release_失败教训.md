# 失败教训：开源 GitHub 时遗漏封面图和 Release

**日期**：2026-05-09
**场景**：将科研日报项目开源到 GitHub
**用户批评**：「我之前给你的流程不是包括了，在你生成了并上传了github之后，你要给我生成对应的封面图片的提示词吗？以及这明明是一个skill，你可以在release那里直接上传对应的skill文件」

## 问题

将科研日报项目开源到 GitHub 后，只做了基础的 push + 验证，遗漏了两个明确流程要求的步骤：

1. **封面图提示词** — 开源项目应该配有 GitHub Social Preview 图片，方便社交分享和视觉识别
2. **Release 上传 Skill 文件** — 科研日报本身是一个 Skill（`.yaml` 文件），应在 GitHub Release 中单独上传，方便用户直接下载安装

## 根因

- 没有在执行前回顾完整的「开源发布流程清单」，只凭记忆执行了最基本的 git push
- 对"开源"的理解太窄——认为 push 代码就完了，忽略了开源项目的完整包装（README + 封面 + Release + 安装引导）

## 正确做法

**开源发布完整流程（必须全部执行，缺一不可）：**

1. ✅ 创建 Private 仓库
2. ✅ 复制核心文件到暂存目录（排除个人数据）
3. ✅ 写 README + LICENSE + .gitignore
4. ✅ git init → commit → push
5. ✅ 验证远程仓库内容（gh repo view + gh api contents）
6. ❌ **生成封面图提示词**（GitHub OG Image / Social Preview）
7. ❌ **创建 Release + 上传 Skill 文件**（让用户可直接下载 .yaml）
8. ✅ 通知用户检查

**核心教训**：开源 ≠ push 代码。开源 = 让别人能发现 + 能看懂 + 能用起来。封面图是"发现"，Release 是"能用"。

## 对小黎的影响

- 浪费了一轮对话时间，用户需要再次提醒
- 如果用户不提醒，这个仓库就是个"残缺的开源项目"——没有社交预览图，没有一键下载入口
