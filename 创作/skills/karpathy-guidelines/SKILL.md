## 四条原则

### 原则 1：先思考再编码 Think Before Coding

**不要假设。不要隐藏困惑。暴露权衡。**

实现前必须：
- 明确说出你的假设。不确定就问。
- 如果存在多种解读，全部列出——不要悄悄选一个。
- 如果有更简单的方案，说出来。该推回就推回。
- 如果某处不清楚，停下来。说清楚哪里困惑你。然后提问。

> 牛马适配：与 `think-before-act.md` 的"强制决策门"互补。think-before-act 管"做什么"，本原则管"怎么做之前先对齐认知"。

---

### 原则 2：简洁优先 Simplicity First

**最少代码解决问题。不做投机性设计。**

- 不做超出需求的功能。
- 不为单次使用创建抽象。
- 不加没要求的"灵活性""可配置性"。
- 不为不可能的场景写错误处理。
- 如果写了 200 行但 50 行能搞定——重写。

自问：**"一个资深工程师会觉得这过度设计了吗？"** 如果会，简化。

> 牛马适配：直接注入 `.claude/rules/think-before-act.md` 的新 Step。这是原 repo 最核心的增量价值——牛马现有规则覆盖了"想清楚再做"但没有覆盖"做完后自检是否过度设计"。

---

### 原则 3：外科手术式改动 Surgical Changes

**只碰必须碰的。只清理自己制造的垃圾。**

编辑已有代码时：
- 不要"顺手改进"相邻代码、注释或格式。
- 不要重构没坏的东西。
- 匹配已有风格，即使你会写得不一样。
- 发现无关的死代码——提一嘴，但别删。

你的改动产生孤儿代码时：
- 删除**你自己的改动**导致未使用的 import/变量/函数。
- 不删改动前就存在的死代码（除非被要求）。

**检验标准：每一行改动都能直接追溯到用户需求。**

> 牛马适配：直接注入 think-before-act.md 的新 Step。这解决了 AI 最常见的代码审查问题——"我来帮你顺便重构一下"然后改了 20 个文件。

---

### 原则 4：目标驱动执行 Goal-Driven Execution

**定义成功标准。循环直到验证通过。**

将任务转化为可验证目标：
- "加上验证" → "为无效输入写测试，然后让它们通过"
- "修 bug" → "写一个复现 bug 的测试，然后让它通过"
- "重构 X" → "确保重构前后测试都通过"

多步任务必须列出简明计划：
```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

强成功标准 → 可以独立循环。弱成功标准（"让它跑起来"）→ 需要反复确认。

> 牛马适配：与 `lifecycle-sop.md` 阶段 4（分层验证）和法则 8（门禁不可跳过）一致。本原则补充了"微观任务级"的目标定义方法。

## 激活条件

本 skill 在以下场景自动激活（通过 skill-routing-table.json 路由）：

| 触发词/场景 | 动作 |
|-------------|------|
| 写代码 / 实现功能 / 开发 | 原则 1+2 全部执行 |
| 重构 / 优化 / 清理代码 | 原则 3 全部执行 |
| debug / 修 bug / 排查 | 原则 4 全部执行 |
| code review / 代码审查 | 原则 3+4 全部执行 |
| 简单配置修改 / 明确指令 | 可跳过（用判断力） |

## 这些指南有效的标志

- diff 中不必要的变更更少了
- 因过度设计导致的重写更少了
- 澄清问题在实现前提出，而非犯错后才问
