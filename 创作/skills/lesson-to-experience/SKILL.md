# 教训自动沉淀系统（Lesson-to-Experience）

> 核心理念：**每一次踩坑都应该产生复利**——不是记个笔记就完，而是自动沉淀为可检索、可复用的标准化案例。

---

## 一、问题域

### 1.1 本 Skill 解决什么问题

每次对话中出现「用户纠正」「方案验证为死路」「踩坑」时，当前流程是：

1. 写 memory（当日记忆）
2. 写 negative-results（负结果日志）
3. 写 self-evolution（自我进化系统） ← **最容易遗漏**
4. 回写 SOP（如果相关）

问题：**Step 3 经常被跳过**（教训027已触发3次），而且经验库（`社群运营/经验库/`）完全是手动写，没有标准化。

### 1.2 行家怎么做

真正做知识管理的人（如 Obsidian + Zettelkasten 玩家）的核心原则：

- **原子化**：一条教训 = 一个独立卡片，不混在大文件里
- **双向链接**：教训卡片 ↔ SOP ↔ CLAUDE规则 ↔ 案例模板，互相引用
- **自动索引**：新增卡片后，索引文件自动更新
- **定期回顾**：同类教训 ≥2 次自动提醒升级

### 1.3 新手/AI 常见问题

- 把教训写进 memory 就以为完了（memory 是日记，不是知识库）
- 教训只记"发生了什么"，不记"为什么发生"和"怎么预防"
- 经验库和 self-evolution 是两套独立系统，不同步
- 案例文件命名不规范，找不到、检索不了

### 1.4 如果做错了会是什么样

- 教训文件散落在 memory/、negative-results.md、self-evolution/ 三处，没有统一入口
- 新对话 AI 读不到历史教训，重复踩坑
- 经验库空壳化——文件夹建了但里面没东西
- CLAUDE 规则更新了但经验库没同步

### 1.5 典型用户

- 小黎本人（社群运营者/内容创作者）
- 任何使用牛马AI工作区并希望"越用越聪明"的用户

## 二、子技能清单

| 编号 | 名称 | 功能 | 触发条件 |
|------|------|------|---------|
| A | lesson-sink | 教训提取与归档 | 对话中出现「用户纠正/方案失败/踩坑」 |
| B | case-generate | 案例卡片生成 | 教训归档后，自动生成标准化案例文件 |
| C | case-encrypt | 隐私脱敏 | 案例文件进入公开经验库前 |
| D | auto-assign | 自动分类与索引更新 | 新案例文件写入后 |

### 工作流总览

```
对话中出现踩坑/纠正
    │
    ▼
┌─────────────┐
│ A: lesson-sink │  提取教训 → 写 memory + negative-results + self-evolution
└──────┬──────┘
       │ 教训归档完成
       ▼
┌─────────────┐
│ B: case-generate │  生成标准化案例卡片 → 写入经验库
└──────┬──────┘
       │ 案例文件生成
       ▼
┌─────────────┐
│ C: case-encrypt │  脱敏检查（如果是公开经验库）
└──────┬──────┘
       │ 脱敏通过
       ▼
┌─────────────┐
│ D: auto-assign │  自动分类 → 更新 README 索引 → 检查是否需要升级规则
└──────┬──────┘
       │
       ▼
    完成 ✅
```

## 三、双写验证

### 3.1 验证脚本

路径：`skills/lesson-to-experience/scripts/dual-write-verify.py`

功能：检查经验库文件是否与 self-evolution 同步。

```bash
# 全量检查
python skills/lesson-to-experience/scripts/dual-write-verify.py

# 只检查一个模块
python skills/lesson-to-experience/scripts/dual-write-verify.py --module 社群运营

# 自动修复缺失的副本
python skills/lesson-to-experience/scripts/dual-write-verify.py --fix
```

### 3.2 双写规则

每条教训必须同时存在于：

| 位置 | 文件格式 | 用途 |
|------|---------|------|
| `self-evolution/做得差的避免/` | `{编号}-{主题}.md` | 全局教训库（所有工作区共享） |
| `{模块}/经验库/做得差的/` | `{CASE-编号}-{主题}.md` | 模块级经验库（子模块专属） |
| `self-evolution/做得好的/` | `{编号}-{主题}.md`（如有成功经验） | 全局成功经验 |

### 3.3 不可做清单

- ❌ 只写经验库不写 self-evolution
- ❌ 只写 self-evolution 不写经验库
- ❌ 不运行双写验证脚本就声称"已同步"
- ❌ 案例文件写完不更新 README 索引
- ❌ 用 PowerShell 操作中文路径（必须用 Python）

## 四、使用方式

### 4.1 手动触发

在对话中说：

```
"记录教训" / "这个坑要记下来" / "经验库更新"
```

AI 将自动执行 lesson-sink → case-generate → case-encrypt → auto-assign 四步。

### 4.2 自动触发（CLAUDE 规则集成）

以下信号自动触发本 Skill：

1. **用户纠正**（"不对"、"不是这个意思"、"你误解了"）
2. **方案失败**（验证结果比预期差、被用户否决）
3. **踩坑**（出现错误、遗漏、违规）
4. **F6 铁律触发**（教训沉淀 checklist 启动时）

### 4.3 输入格式

AI 从对话上下文中自动提取，不需要用户手动输入：

- **触发事件**：用户原话 or 错误现象
- **根因分析**：AI 自行判断
- **正确做法**：从用户纠正中提取
- **影响范围**：判断涉及哪些模块/SOP/CLAUDE规则

### 4.4 输出格式

#### 案例卡片（经验库）

```markdown
# CASE-{NNN}-{主题}

## 元信息
- **案例编号**：CASE-{NNN}
- **日期**：{YYYY-MM-DD}
- **类别**：{做得差的/做得好的}
- **模块**：{社群运营/公众号/...}
- **严重度**：{🔴高/🟡中/🟢低}
- **触发信号**：{用户原话}

## 踩坑经过
{客观描述发生了什么，不超过200字}

## 根因分析
{为什么会发生？直接原因 + 深层原因}

## 正确做法
{下次遇到同类问题，应该怎么做？}

## 关联
- 教训编号：E{NNN}（self-evolution）
- 影响 SOP：{SOP文件名}
- 影响 CLAUDE 规则：{规则文件名}
- 同类教训计数：{N}次

## 可复用模式
{这次踩坑能提炼出什么通用规则？}
```

#### 索引条目（README.md）

```markdown
| CASE-{NNN} | {日期} | {主题} | {做得差的/做得好的} | {严重度} | E{NNN} |
```

## 五、配置

### 5.1 可配置参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `经验库根目录` | `projects/20260425-内容创作系统/社群运营/经验库/` | 经验库文件写入位置 |
| `self-evolution目录` | `self-evolution/` | 全局教训库位置 |
| `案例编号前缀` | `CASE-` | 案例文件命名前缀 |
| `教训编号前缀` | `E` | self-evolution 教训编号前缀 |
| `自动脱敏` | `true` | 案例进入公开经验库前是否自动脱敏 |
| `双写验证` | `true` | 写入后是否自动运行双写验证 |
| `索引自动更新` | `true` | 新案例后是否自动更新 README |

### 5.2 模块配置

当新的子模块创建时，在本文件的 `模块注册表` 中添加：

```markdown
| {模块名} | `{模块路径}/经验库/` | `{模块路径}/CLAUDE-{模块名}.md` |
```

### 5.3 扩展点

- **新模块接入**：在模块注册表添加一行 + 初始化经验库目录
- **新分类维度**：在案例卡片模板中添加字段
- **新触发信号**：在 CLAUDE 规则中添加自动触发条件
- **集成 MCP**：可接入 notification MCP，在飞书/微信推送"今日教训"
