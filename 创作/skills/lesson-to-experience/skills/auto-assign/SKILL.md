# D: auto-assign（自动分类与索引更新）

---

## 一、问题域

### 1.1 解决什么问题

案例文件写入经验库后，还需要：

1. 更新经验库 README.md 的案例索引表
2. 检查同类教训是否达到升级阈值（≥3次 → 升级为铁律）
3. 检查是否需要更新 CLAUDE 规则或 SOP
4. 运行双写验证脚本确认同步率 100%

这些步骤经常被遗忘，导致索引过时、规则不更新。

### 1.2 行家怎么做

- **自动索引**：文件系统监控 + 新文件自动索引
- **阈值触发**：同类教训达到阈值时自动提醒升级
- **闭环验证**：每一步都有验证，不靠"我记得做了"

### 1.3 做错了什么样

- README 索引表只有前 3 个案例，后面 7 个没更新
- 同一个教训出现 4 次了，但还没升级为铁律
- 双写验证脚本从来没人跑过

## 二、使用方式

### 2.1 触发条件

由 case-generate（B）自动触发。新案例文件写入后自动执行。

### 2.2 执行流程

```
案例文件写入完成
    │
    ▼
Step 1: 更新经验库 README.md
  - 在案例索引表中追加新条目
  - 格式：| CASE-{NNN} | {日期} | {主题} | {类别} | {严重度} | E{NNN} |
    │
    ▼
Step 2: 检查同类教训计数
  - 扫描 self-evolution/lessons.md 中的教训列表
  - 如果当前教训的主题与已有教训匹配 → 计数 +1
  - 计数 ≥ 3 → 标记为"需升级为铁律"
    │
    ▼
Step 3: 规则升级检查（如果计数 ≥ 3）
  - 提醒 AI：此教训已出现 {N} 次，建议升级为 CLAUDE 规则
  - 生成升级建议（添加到哪个铁律、什么内容）
  - 但不自动执行——需要用户确认
    │
    ▼
Step 4: SOP 回写检查
  - 查 F6 映射表，确认影响的 SOP
  - 在 SOP 的迭代日志中追加条目
  - 如果 SOP 有"高频坑"标记区 → 添加标记
    │
    ▼
Step 5: 运行双写验证
  - python scripts/dual-write-verify.py
  - 同步率 < 100% → 运行 --fix
    │
    ▼
Step 6: 输出完成报告
  - 案例编号
  - 索引更新状态
  - 同类教训计数
  - 双写验证结果
```

### 2.3 输出格式

```
✅ 教训沉淀完成

案例：CASE-003（做得差的）
教训：E33
索引：README.md 已更新（共 3 个案例）
同类计数：F8技能链跳过 × 3 → 🔴 建议升级为铁律
双写验证：100% 同步
SOP回写：SOP-01 迭代日志已追加
```

## 三、配置

### 3.1 可配置参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `升级阈值` | `3` | 同类教训出现 N 次后建议升级 |
| `自动SOP回写` | `true` | 是否自动在SOP迭代日志中追加 |
| `双写验证` | `true` | 是否自动运行双写验证脚本 |
| `验证脚本路径` | `skills/lesson-to-experience/scripts/dual-write-verify.py` | 双写验证脚本位置 |

### 3.2 升级路径

| 同类教训次数 | 动作 |
|------------|------|
| 1次 | 记录到 self-evolution + 经验库 |
| 2次 | SOP 对应步骤旁加 **高频坑** 标记 |
| 3次 | 建议升级为 CLAUDE 铁律（需用户确认） |
| 5次 | 评估是否需要新建独立 SOP |

---

## 四、不可做清单

- ❌ 不更新 README 索引就声称"案例已归档"
- ❌ 同类教训 ≥3 次但不提醒升级
- ❌ 不运行双写验证就声称"已同步"
- ❌ 自动升级铁律（必须用户确认）
