# B: case-generate（案例卡片生成）

---

## 一、问题域

### 1.1 解决什么问题

教训归档（lesson-sink）完成后，需要把教训"翻译"成经验库的标准案例格式。

- lesson-sink 写的是 `self-evolution/` 格式（教训+经验分开）
- 经验库需要的是 `CASE-{NNN}-{主题}.md` 格式（一个文件包含全部信息）
- 两者的信息相同，但格式不同、存放位置不同

### 1.2 行家怎么做

案例库管理的最佳实践：

- **标准化模板**：每个案例卡片有固定结构，方便检索和批量处理
- **自动生成**：从结构化数据生成，不靠人工填写
- **可追溯**：每个案例都能追溯到原始对话和教训编号

### 1.3 新手/AI 常见问题

- 案例文件格式不统一（有的有元信息，有的没有）
- 案例和教训是同一份信息的两个副本，但内容不同步
- 编号不连续，新增案例时不知道该用什么编号

### 1.4 做错了什么样

- 经验库里有 10 个案例文件，但格式各不相同
- 想查某个教训的案例，但找不到对应关系
- 案例编号冲突（两个 CASE-003）

## 二、使用方式

### 2.1 触发条件

由 lesson-sink（A）自动触发，不需要手动调用。

### 2.2 执行流程

```
lesson-sink 输出教训编号（如 E33）
    │
    ▼
Step 1: 确定目标模块
  - 从教训的"影响范围"字段判断属于哪个模块
  - 查模块注册表确定经验库路径
    │
    ▼
Step 2: 确定案例编号
  - 扫描目标模块的 做得差的/ 和 做得好的/ 目录
  - 取最大 CASE 编号 +1
  - 如果目录为空，从 CASE-001 开始
    │
    ▼
Step 3: 填充案例模板
  - 从 lesson-sink 的输出中提取：
    - 触发事件 → 「踩坑经过」
    - 根因分析 → 「根因分析」
    - 正确做法 → 「正确做法」
    - 影响范围 → 「关联」
  - 生成「可复用模式」（从教训中提炼通用规则）
    │
    ▼
Step 4: 写入经验库
  - 做得差的：`{模块}/经验库/做得差的/CASE-{NNN}-{主题}.md`
  - 做得好：`{模块}/经验库/做得好的/CASE-{NNN}-{主题}.md`
    │
    ▼
Step 5: Read 验证
  - 确认文件已落盘
  - 确认内容完整
    │
    ▼
返回案例编号（如 CASE-003）供 auto-assign 使用
```

### 2.3 输出格式

```markdown
# CASE-{NNN}-{主题}

## 元信息
- **案例编号**：CASE-{NNN}
- **日期**：{YYYY-MM-DD}
- **类别**：做得差的
- **模块**：{模块名}
- **严重度**：{🔴高/🟡中/🟢低}
- **触发信号**："{用户原话}"
- **教训编号**：E{NNN}

## 踩坑经过
{客观描述，≤200字}

## 根因分析
- **直接原因**：{...}
- **深层原因**：{...}

## 正确做法
{下次遇到同类问题的标准操作}

## 关联
- **影响 SOP**：{文件名}
- **影响 CLAUDE 规则**：{规则名}
- **同类教训计数**：{N}次

## 可复用模式
{提炼出的通用规则，可直接写入 CLAUDE 规则或 SOP}
```

## 三、配置

### 3.1 可配置参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `案例编号前缀` | `CASE-` | 案例文件命名前缀 |
| `编号位数` | `3` | 如 CASE-001（3位补零） |
| `严重度自动判断` | `true` | 根据影响范围自动判断严重度 |
| `可复用模式自动生成` | `true` | 从教训中自动提炼通用规则 |

---

## 四、不可做清单

- ❌ 手动指定编号而不检查现有最大编号（可能冲突）
- ❌ 案例文件不写元信息（无法检索）
- ❌ 不关联教训编号（无法追溯）
- ❌ 写完不 Read 验证

## 五、与其他 Skill 的关系

| Skill | 关系 |
|-------|------|
| `lesson-sink`（上游） | 提供教训数据作为输入 |
| `case-encrypt`（下游） | 案例文件生成后，如需公开则调用脱敏 |
| `auto-assign`（下游） | 案例文件生成后，调用更新索引 |
