## 2026-06-13 负结果：脚本开发——6轮手动迭代 v1→v6，li-script 全程闲置

### 方案
收到"PDF→MD→深度拆解→书籍"任务后，直接手写6个Python脚本(v1→v6)，每次迭代修改模板/格式/数据结构。

### 失败现象
- v1(.format冲突)、v2(模板太小0.37MB)、v3(3.3MB远不达标)、v4(格式冲突再次)、v5(变量未定义)
- 5轮失败后才在v6(程序化批量生成)成功
- 每轮约5-10分钟，总计~50分钟浪费

### 为什么看起来行其实不行
"手写6个脚本"是可行路径（最终v6成功了），但方式错了——没有先用li-script检索库存。li-script库里有现成脚本模板和管理框架，本应：
1. 先搜库确认无同类→直接新写（跳过"写脚本前不看库"这一步就是在重复造轮子+管理混乱）
2. 写完归位（入库/归档）→本次v2-v5本来应该判为一次性归档，而不是散落在outputs/

### 退出判据
→ li-script Phase 1检索结论=无命中时，才是新写的时机。**先检后写，不是先写后检。**

### 系统加固
1. 脚本任务必须走 li-script → Phase 1（检索引擎先行）
2. 将"写脚本"加入 li-script 路由触发词表

### 与已有教训的关联
- 模式同：skill闲置（li-content闲置→F8升级 教训027）| 本次：li-script闲置→路由升级
- 根因同：约束文件/技能存在但没被触发
> 来源：工程法则 7——负结果是一等公民。

### 退出判据
HTML 生成任务 ≠ 快速写一个 HTML 文件 = 先做参考品质量化分析再动笔

### 关联法则

## 2026-06-07 负结果：HTML转MD任务跳过F8路由 + 技能链零调用（教训E38）

### 方案
用户给3个HTML直播笔记文件（5.23/5.30/6.6），指令"总结是md文件"/"帮我继续整理成md笔记"。AI将任务分类为"格式转换"而非"内容创作"，直接写MD笔记，未走F8路由、未调任何技能。

### 已尝试
- 第1个HTML（6.6-如何AI面试）：直接生成MD，未走F8
- 第2-3个HTML（5.30/5.23）：同样直接生成MD，未走F8
- 写完3份笔记后经验库未更新（用户提醒才做——同类错误第2次）

### 失败现象
- F8 路由表：❌ 跳过
- li-content 五维诊断：❌ 未调
- niuma-voice-dna：❌ 未调
- 嘉宾笔记整理SOP：❌ 未读
- 笔记整理铁律（6条+11项checklist）：❌ 未读
- 已有笔记格式参考：❌ 未对照
- 经验库/私域认知更新：❌ 未主动执行

### 为什么看起来行其实不行
表面上"HTML→MD"已完成，笔记质量本身不差（格式对齐、道法术器蒸馏到位、Prompt完整保留）。但核心问题不在产出质量，而在**流程合规**：

1. **格式伪装**：用户说"整理成md"，AI将"格式转换"当理由绕过F8——但最终产出是人类可读内容，本质是内容创作
2. **规则沉睡**：存在专门的笔记整理铁律和嘉宾笔记整理SOP，但AI没有读取——"有约束文件不用"
3. **经验库被动更新**：写完笔记不主动更新经验库，等用户提醒——同类错误第2次

三者叠加 = P3格式伪装 + P1系统软约束 + P2对话输出当交付，三个根因模式同时触发。

### 退出判据
- 任何"X→Y格式转换"任务 → 自问"最终产出是什么？" → 人类可读内容 → F8
- 任务涉及嘉宾/直播/分享/笔记 → 强制读笔记整理铁律
- 写完产出后 → 经验库/私域认知更新不再依赖用户提醒

### 影响范围
所有HTML/TXT/DOCX→MD的"格式转换"类任务。已修复：
- CLAUDE.md F8 新增"任务实质判断优先于表面形式"（v3.0）
- CLAUDE.md F8技能链速查表+1行（嘉宾笔记/直播笔记/HTML转MD整理）
- CLAUDE.md 主路由表+1行 + 子模块路由表+1行
- 笔记整理铁律加固：识别信号+格式对齐步骤+checklist 9→11项
- 私域认知新增：AI协作-任务表面伪装与规则失效模式.md
- 私域认知加固：防错机制-再也不会犯错.md 新增规则8
- 变更记录 CLAUDE.md v2.6

## 负结果 042：GitHub仓库学习任务跳过li-research直接动手

- **日期**：2026-06-14
- **方案**：收到3个GitHub仓库URL后直接Clone+逐文件探索+产出笔记
- **已尝试**：Clone了GSAP（55K★）、gsap-cc-starter（1.3K★）、gsap-skills（200★），读了多个源文件和README
- **失败现象**：
  - 没有用li-research做结构化调研就开始动手
  - 边探索边产出，方向多次调整（先做学习地图，再加动画演示，再加学习成果展示）
  - token消耗高，效率低
  - 用户明确指出：「GitHub仓库学习任务应先用li-research做结构化调研再动手」
- **为什么看起来行其实不行**：
  - 「直接动手」看起来效率高，但缺少前期调研导致后期频繁调整方向
  - 「边探索边产出」看起来灵活，但缺少全局视野导致产出质量不稳定
  - 没有利用li-research的结构化能力（目标→范围→方法→预期产出），浪费了已有工具
- **退出判据**：
  - 收到GitHub仓库URL → 必须先用li-research做结构化调研
  - 调研阶段必须产出：学习目标、核心文件清单、三个仓库的分工关系
  - 调研完成后才开始Clone+深度阅读
- **关联法则**：法则3（实测优先）、法则10（规则结晶）
- **影响范围**：所有涉及GitHub仓库学习/开源项目研究的任务
