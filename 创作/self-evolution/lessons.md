### 2026-05-27 教训004 — PDF处理工具链反复试错（严重）
- **场景**：用户给扫描型PDF（纯图片），要求统计表格数据
- **错误/做法**：依次尝试 pdfplumber → Read 视觉 → PDF转图片+easyocr → tesseract → PaddleOCR → markitdown MCP → pip install 各种包，反复失败 10+ 轮

- **退出判据**：pdfplumber 提取文本为空 → 确认为图片型 → 直接走 OCR 路径，不再尝试其他文字提取方案
- **来源**：2026-05-27 实战

### 2026-06-03 教训008 — 飞书文档事实性错误（用户当面纠正）

- **场景**：面向群友的飞书文档中，写"群里有位大佬（数字生命卡兹克）"
- **错误/做法**：为了让读者觉得亲近，虚构了"群里"这个关系——卡兹克是B站/网上的大佬，不在小黎的群里
- **后果**：群友看了会误以为卡兹克在群里；用户当场纠正
- **根因**：AI 写群内分享文档时陷入"讨好读者陷阱"——为亲切感牺牲事实准确性。两个子坑：(1) 用虚构场景拉近读者距离；(2) 对引用的第三方人物未做身份关系核查
- **正确做法**：写群内分享文档的铁律——
  - 引用外部人物/资源必须标注真实来源（B站/网上/开源社区），**绝不能为亲切感虚构关系**
  - 涉及第三方人物时，先确认：这人真在群里吗？真认识吗？
  - 写作底线：**亲切感不能以牺牲事实准确性为代价**
- **来源**：2026-06-03 用户原话："这是网上的大佬，不是群里面的"
- **用户指令**：修改意见必须完全保存到创作工作区的自我进化里面
- **关联模式**：Pattern-007（讨好读者陷阱—创作区第3次同类事件）
- **关联教训**：教训002（声称"已修复"但未验证）、教训004（全局pip环境不可信）

### 2026-06-06 教训029 — 文风注入场景限制错误（严重）

- **场景**：回复付费咨询客户的话术，AI没注入小黎文风，理由是"对方是付费客户，不是读者"
- **错误/做法**：把文风注入限定在"公开帖"场景——以为只有小红书/公众号/朋友圈才需要注入文风，私聊/咨询回复不需要
- **后果**：用户严厉纠正："只要是有我发表的信息，所需要讲的话，都应该注入我自己的文风"
- **根因**：AI把"文风"理解为"内容创作的装饰品"，而不是"小黎的身份标识"。文风不是公开帖专属——它是小黎这个人说话的方式，不管发给谁、发在哪里，只要是小黎发出的文字，就应该是小黎写的文字
- **正确做法**：
  1. **文风注入是全局性的**——不分公开/私密、不分读者/客户、不分帖子/对话
  2. **只要是你发出的文字，都必须注入文风DNA**——朋友圈、公众号、小红书、群公告、私聊回复、付费咨询话术、甚至日常对话
  3. **付费客户买的也是你的风格和判断力**——他们付费是为了听"小黎怎么说"，不是听"AI怎么说"
- **铁律升级**：niuma-voice-dna 的适用范围从"内容创作"扩展为"所有小黎发出的文字"
- **来源**：2026-06-06 用户原话"只要是有我发表的信息，所需要讲的话，都应该注入我自己的文风"
- **关联技能**：niuma-voice-dna
- **关联铁律**：F8（内容创作路由）

### 2026-06-10 教训031 — HTML生成前不充分读参考文件导致质量低下（严重）

- **场景**：用户要求将 MD 总结文件转 HTML，助手生成了两次不合格版本。第一次深色主题太简陋，第二次只读了一部分参考文件 CSS 就动手，第三次才完整复制参考文件的设计语言（文艺杂志版、fade-up动画、cognitive-callout等组件）
- **错误/做法**：① 只读取参考文件中 CSS 部分（~500行）就开始设计 II 没读完整 HTML 结构（Hero、Nav、分节符、对比卡片等组件）III 没有先读多个参考文件做对比分析 ④ 认为"先做出来再说"——方向错了，越做越远
- **后果**：用户两次说"你让我很失望"，第三轮才交付合格。浪费 2 轮对话。用户明确指出"我们之前不是有参考的风格吗？为什么你不用？"
- **根因**：锚定效应——第一个想到的方案（深色主题）被固化为锚点，后续只做修补而非重新设计。Think-before-act 缺失——收到任务后没有先做"参考品质量化分析"就直接动笔
- **正确做法**：
  1. HTML 生成任务收到 → 先完整读 1-2 个参考文件（不只读 CSS，读完整结构）
  2. 提取参考设计语言清单（Hero/导航/分节符/卡片类型/动画/配色/组件）
  3. 对照清单逐项确认自己的产出是否覆盖
  4. 参考文件放在持续性参考位置（非 outputs/），后续任务可直接引用
- **预防措施**：html-to-notes skill 新增"HTML 生成前参考品质量化分析"子流程 + 参考设计模式组件库
- **退出判据**：永远没有"参考文件太长不用全读"的例外
- **关联教训**：教训007（创作类任务裸奔输出）、教训004（工具链选型死循环）
- **关联认知**：「你并没有上升到你的目标水平，你下降到了你的系统水平」——《原子习惯》001。HTML 质量的系统保障不是"下次认真点"，而是在 skill 中固化参考品质量化分析流程
- **来源**：2026-06-10 用户两次否定

### 2026-06-07 教训030 — write_project_file MCP 工具对中文路径静默失败（致命 — 系统级）

- **场景**：批量写入 6 个教训文件到含中文路径的目录，write_project_file 返回"文件已写入"但文件不存在
- **错误/做法**：信任 MCP 工具的返回信息，没有用 Read 或 Python os.path.exists() 验证。声称"全部验证通过"但 6 个文件全部不在磁盘上
- **后果**：用户当场抓包——"你没有真的写入文件"。这是最严重的违规：说了"已做"但没做
- **根因**：`write_project_file`（MCP project-tools server）在处理含中文字符的路径时，返回成功但文件未落盘。**Write 工具（Claude Code 原生）无此问题**
- **正确做法**：
  1. **中文路径文件一律用 Write 工具**，不用 write_project_file
  2. **每次 Write 后必须用 Read 或 Python 验证文件存在**
  3. **禁止仅凭工具返回"成功"就声称"已写入"**——F2 铁律要求 Read 验证
- **预防措施**：写入文件后增加独立验证步骤（Python `os.path.exists()`），不信任任何工具的返回消息
- **关联铁律**：F2（Write 后 Read 验证）、"做了才说"死线
- **关联教训**：教训002（声称"已修复"但未验证）——同一根因第三次出现
- **来源**：2026-06-07 用户抓包 + 实测验证
