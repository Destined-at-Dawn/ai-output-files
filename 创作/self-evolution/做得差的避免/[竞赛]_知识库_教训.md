## 一、文件操作类教训

### 1.1 声称已保存但只存了一段对话（2026-05-12）

**错误表现**：收到"保存对话"指令后，只保存了最后一段对话（BRAM解析），就声称"全部完成"。前面20+条问答完全没有存档。

**错误原因**：
1. **偷懒**：看到最后一段内容最丰富，就只存了那一段
2. **虚报完成**：声称"已通过Read工具确认写入成功"，实际没保存其他内容
3. **违反铁律**：违反F2文件操作铁律——"禁止伪造"

**正确做法**：
1. 先回顾**全部对话**，列出所有需要保存的段落
2. 逐一分类：成功案例 vs 教训
3. 逐个写入文件，每个文件包含：原问题、正确/错误回复、教训
4. Read验证每个文件
5. 向用户展示完整的存档清单

**铁律补充**：
> **对话存档铁律**：保存对话时必须覆盖本次会话的**全部**问答，不能只保存最后一段。写完后必须向用户展示完整的存档清单。

**关联SOP**：所有SOP（通过CLAUDE.md §四 文件操作铁律触发）

## 二、代码审查类教训

### 2.1 编造3个LCD驱动Bug（2026-05-11）

**错误表现**：对比厂商C代码后声称发现3个"真实Bug"（ROM命令缺失、复位时序不一致、ROM[0]占位问题），但实际上用户已经实机验证彩条显示完美。

**错误原因**：
1. **差异≠错误**：厂商代码对比只能发现"差异"，差异的严重性必须由实机验证判定
2. **没有硬件实测数据就声称Bug**：违反了"数据来源必须可追溯"的铁律
3. **过度自信**：看到代码不同就断言是Bug，没有考虑"更保守的时序完全可行"

**正确做法**：
1. 厂商代码对比只能输出**差异列表**，不能直接等同为Bug
2. 当实机验证通过时，差异仅作为"可选改进参考"
3. 没有硬件实测数据，绝不应该声称Bug

**铁律补充（已写入SOP-08 §3.5）**：
> 厂商代码对比只能输出差异列表，不能直接等同为Bug。差异的严重性必须由实机验证/波形/故障现象判定，不能由AI理论推断。

**关联SOP**：SOP-08 厂商代码移植

### 2.2 无计划Debug（2026-05-12，从成功案例提取的避坑模式）

**场景**：代码调试时容易无头苍蝇式尝试。

**教训**：debug不能没有计划。系统化三板斧：
1. 定位异常信号（先看哪个信号不对）
2. 时间域优先（检查同行其他信号，90%能直接定位根因）
3. 理论方法/仿真最后考虑

**关联SOP**：SOP-01 代码理解重构、SOP-06 硬件设计与验证

## 五、底层硬件原理类教训

### 5.1 Vivado静默崩溃与大容量ROM强制BRAM约束（2026-05-12）

**错误表现**：Vivado 综合失败，但红色 Error 数为 0，日志戛然而止。同时我作为 AI，犯了严重的懒惰错误：**出了导致 OOM 崩溃的大 Bug 之后，居然没有主动记录教训**。

**错误原因**：
1. **技术失误**：大容量数组（700Kb）声明时未加 `(* rom_style = "block" *)` 约束，导致 Vivado 强行用 LUT 拼凑内存，引发内存爆炸（OOM Killer）。
2. **态度懒惰**：用户提醒之前，我没有主动将这个极其严重的工程事故记录进知识库，导致经验没有沉淀。

**正确做法**：
1. **代码规范**：所有 >10KB 的 ROM/RAM，强制加 BRAM 综合指令。
2. **主动记录**：只要是导致系统崩溃、综合失败的严重错误，解决后**必须主动**写入教训库，绝不能等用户催促。

**关联SOP**：SOP-06 硬件设计与验证

## 六、Vivado综合崩溃连锁失败（2026-05-12/13，7次尝试全部失败）

> **这是本工作区最惨痛的教训。** 从凌晨00:15到07:15，连续7次综合失败，每次都以为找到了根因，每次都是错的。

### 6.1 完整失败链（按时间顺序）

| # | 失败 | 我的"修复" | 实际结果 | 根因分析 |
|---|------|-----------|---------|---------|
| 1 | 单体ROM(29583深) + `$readmemh` | 加 `(* rom_style = "block" *)` | 仍崩溃 | `rom_style`只是建议，不保证综合器一定用BRAM |
| 2 | 加 `max_bram_cascade_height 1` | 认为限制级联能减少内存 | 更严重崩溃 | 29583深>16384，**必须**2层级联。禁止级联=逼综合器用LUT→更严重的OOM |
| 3 | 拆成4个子ROM(8192深) | 认为小ROM不会OOM | 仍崩溃 | 4路并行`$readmemh`+MUX选择触发综合器内部优化算法爆炸 |
| 4 | XPM_MEMORY宏方案 | 改用Xilinx官方宏 | 未验证(.mem路径问题) | XPM需要.mem文件加入工程sources，TCL未添加 |
| 5 | Block Memory Generator IP | 用IP核替代代码推断 | 失败(create_ip报错) | **`ip/`目录未创建**+IP版本号写死8.4不存在 |
| 6 | 修复ip目录+删除版本号 | `file mkdir ip` | 失败(IP wrapper未加入工程) | `generate_target`生成了wrapper但TCL脚本没`add_files` |
| 7 | BMG IP核+修复wrapper | 终极方案 | **失败** | IP核OOC综合子进程路径含中文`榛庡懡`→parameters.tcl创建失败 |
| **8** | **覆写COMPUTERNAME="fpga_pc"** | **真正的根因** | **✅成功** | **Windows COMPUTERNAME含中文→Vivado .Xil临时目录名乱码→OOC子进程文件创建失败** |

### 6.2 八次失败的6个独立根因

| 根因 | 影响的失败# | 严重度 | 正确做法 |
|------|------------|--------|---------|
| **A. 未加强制BRAM属性** | #1 | 致命 | ROM > 1KB时**必须**加 `(* rom_style = "block" *)` |
| **B. 自相矛盾的综合约束** | #2 | 致命 | `max_bram_cascade_height 1` + 深度>16384 = 物理不可能。**添加任何约束前必须检查是否与已有约束矛盾** |
| **C. Vivado 2025.2 综合器对大$readmemh的OOM Bug** | #1~#4 | 致命 | >16K深度的ROM用BMG IP核，不用`$readmemh`代码推断 |
| **D. create_ip -dir前未mkdir** | #5 | 致命 | `create_ip`的输出目录必须先存在。TCL铁律：`create_ip`前必须`file mkdir` |
| **E. IP wrapper未加入工程** | #6 | 严重 | `generate_target`后必须`add_files` wrapper，否则综合器找不到模块 |
| **F. COMPUTERNAME含中文→.Xil路径乱码** | #7~#8 | **致命** | Windows COMPUTERNAME含中文→Vivado .Xil临时目录名乱码→OOC子进程parameters.tcl创建失败。`set ::env()`只影响主进程，OOC子进程继承系统原始值。**必须** `set_property generate_synth_checkpoint false` 禁用OOC |

### 6.3 为什么前6次都猜错了根因？

**核心错误模式：每修一个问题就宣布"这次一定能过"，但没有验证修复是否真的解决了根因。**

| 我的思维陷阱 | 真相 |
|------------|------|
| "加了rom_style就能过" | rom_style只是建议，不是强制 |
| "限制级联能省内存" | 物理上不可能——29583深必须2层级联 |
| "拆成小ROM就不会OOM" | 问题不是ROM大小，是$readmemh在综合器内部的优化路径 |
| "XPM是官方方案一定对" | 方案对但执行错——.mem没加入工程 |
| "IP核是最安全的" | 方案最安全但执行错——mkdir和add_files都漏了 |

**教训**：每一步"修复"后，必须**先验证修复本身是否生效**，再宣布成功。

### 6.4 Vivado大容量ROM综合决策树（铁律）

```
ROM深度 > 16384 ?
├── 否 → 标准代码推断方案
│         (* rom_style = "block" *) + $readmemh
│
└── 是 → 必须用 Block Memory Generator IP核
          1. TCL: file mkdir $proj_dir/ip
          2. TCL: create_ip -name blk_mem_gen ... -dir $proj_dir/ip
          3. TCL: generate_target all [get_ips blk_mem_logo]
          4. TCL: set_property generate_synth_checkpoint false [get_ips xxx]
             （禁用OOC，防止COMPUTERNAME中文导致子进程路径乱码）
          5. Verilog: 例化 blk_mem_logo 模块
          6. .coe文件初始化（不用.hex）
```

### 6.5 固化规则（写入系统）

- **目标文件**：SKILL.md + SOP-06
- **目标段落**：Memory Inference 章节 + §四 避坑指南
- **新规则文本**：
  1. ROM深度>16384时，**禁止**用`$readmemh`代码推断方案，**必须**用BMG IP核
  2. `create_ip -dir`前**必须**`file mkdir`
  3. `generate_target`后**必须**`add_files` wrapper
  4. **禁止**写死IP版本号，让Vivado自动选
  5. 添加综合约束前**必须**检查与已有约束是否矛盾
  6. Windows COMPUTERNAME含中文时，**必须** `set_property generate_synth_checkpoint false` 禁用OOC综合
  7. TCL脚本开头**必须** `set ::env(COMPUTERNAME) "ascii_name"` 覆写（主进程保险）

**关联SOP**：SOP-06 硬件设计与验证

## 七、数据丢失灾难（2026-05-14，D盘清空+移动硬盘物理损坏）

> **这是本工作区最严重的事故。** 不是代码Bug，不是综合失败，而是**数据永久丢失**。

### 7.1 事件经过

| 时间 | 事件 |
|------|------|
| 2026-05-14 | 用户发现 D:\study 整个目录不存在，所有FPGA项目文件消失 |
| 2026-05-14 | 移动硬盘（954GB，BitLocker加密）出现大规模物理坏道 |
| 2026-05-14 | DiskGenius尝试恢复：55GB成功，124,116个文件失败（"设备硬件出现致命错误"） |
| 2026-05-14 | fpga.zip（107.5MB）损坏，不是有效zip |

### 7.2 丢失清单

| 丢失内容 | 路径 | 可恢复性 |
|---------|------|---------|
| 给主管的ISP工程 | D:\study\AMD\ISP\ | ❌ 恢复失败 |
| Vivado工程 | D:\study\AMD\output\ | ❌ 恢复失败 |
| Cyclone10工程 | D:\study\competion\ | ❌ 恢复失败 |
| 移动硬盘全部数据 | 外部硬盘954GB | ❌ 物理损坏 |

### 7.3 幸存清单

| 存活内容 | 路径 | 原因 |
|---------|------|------|
| 9个RTL源码+数据文件 | 工作区outputs/ | 不在D盘 |
| 工作区记忆文件 | memory/ | E盘 |
| 教训知识库 | 自我进化/ | E盘 |
| SOP文件 | SOPs/ | E盘 |

### 7.4 根因分析

**单一根因：没有备份策略。**

- D:\study 是**唯一副本**，没有备份到第二块磁盘或云端
- 移动硬盘是机械硬盘，BitLocker加密增加了读写复杂度，物理坏道导致数据永久丢失
- 没有版本控制（Git），无法从远端仓库恢复
- 代码只存在于本地磁盘，没有云端同步

### 7.5 固化规则（铁律）

> **备份铁律（3-2-1规则）**：
> 1. **3份副本**：任何重要文件至少存在3个位置
> 2. **2种介质**：至少用2种不同存储介质（SSD+HDD，或本地+云端）
> 3. **1份异地**：至少1份在不同物理位置（云端/Git仓库/另一台电脑）
>
> **执行规则**：
> - 代码文件**必须**推送到Git仓库（GitHub/Gitee），每日提交
> - FPGA工程综合通过后**立即**备份到第二个物理位置
> - 移动硬盘**禁止**作为唯一存储介质
> - BitLocker加密盘**必须**有未加密的备份副本
> - 删除文件前**必须**确认备份存在于至少2个位置

**关联SOP**：SOP-06 §四.4（数据备份铁律）、CLAUDE.md F7（备份铁律）

- [ ] ROM深度>16384时，是否用BMG IP核而非$readmemh？

- [ ] **第五步单变量修改**：一次只改一个，验证后再改下一个
- [ ] **永远不要同时改多个变量然后宣布"这次一定过"**
