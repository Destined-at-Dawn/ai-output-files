# 教训文件覆盖公众号文章——Git 救回

> **教训 E35** | 2026-06-06 | 来源：本轮对话——飞书 wiki 权限突破教训写入时

---

## 踩坑过程

飞书 wiki 权限突破后，要写一个教训文件记录经验。使用 `mcp__project-tools__write_project_file` 时，路径参数复用了 `outputs/社群分享-小黎经验分享-公众号版.md`——这个路径上**已有用户的公众号文章**。

教训内容覆盖了公众号文章，原文被替换。

## 根因

1. **没先 Read 目标路径**——违反 F2（先 Read 再 Write）
2. **write_project_file 的路径参数没精确到新文件名**——直接复用了旧路径
3. **认为"教训文件是新文件"就跳过了 Read 验证**——但路径上已有文件

## 恢复过程

```bash
git log --oneline -5  # 找到最近一次 commit 包含原文
git checkout 9ec25e9 -- "outputs/社群分享-小黎经验分享-公众号版.md"  # 恢复
```

10 秒内恢复，内容完整。Git 检查点机制（git-recovery.md）救命。

## 正确做法

1. **Write 任何文件前必须 Read 目标路径**——即使"看起来是新文件名"
2. **教训文件用独立命名**——不要复用已有文件的路径
3. **教训文件命名规范**：`outputs/feishu-wiki-权限突破-经验教训.md`（带前缀区分）

## 退出判据

- Write 操作前 Read 目标路径 → 发现已有内容 → 停止，换新文件名
- Write 操作后没有 Read 验证 → 立即补上验证
