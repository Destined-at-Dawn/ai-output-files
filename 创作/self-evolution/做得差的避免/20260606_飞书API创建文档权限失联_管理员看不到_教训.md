# 失败教训：飞书 API 创建文档后设 closed 权限——管理员反而看不到

## 日期
2026-06-06

## 场景
小黎要求将本地 .mm 思维导图和 .md 文件上传为私密飞书文档。助手用飞书 Docx API（tenant_access_token）创建文档后，设置了 `link_share_entity: closed`，意图是"仅自己可见"。结果：**管理员（小黎）完全无法访问自己组织里的文档**。

## 踩坑清单

### 坑1：`closed` 权限让管理员失联
**表现**：API 创建文档 → 设置 `link_share_entity: closed` → 浏览器访问显示"没有权限访问"
**用户批评**："我作为这个组织的管理员，我都没有查看我组织里面的文档的权限，这完全是让我很失望的。"
**根因**：飞书的权限模型是「创建者 ≠ owner」。用 tenant_access_token 创建的文档，创建者身份是**飞书应用**（workbuddy），不是用户本人。设 `closed` 后只有应用自己能访问，**人反而看不到**。
**正确做法**：飞书 API 创建文档绝不能设 `closed`。正确流程——创建时默认 `tenant_readable` → 显式添加管理员为 `full_access` 成员 → 浏览器验证可访问。

### 坑2：应用创建的"孤岛文档"
**表现**：即使权限改为 `tenant_readable`，早期创建的 4 个文档在飞书 UI 里找不到入口
**根因**：API 直接创建的独立文档不挂在任何人的云盘下，是"孤岛"——权限改了但没有入口。
**正确做法**：在用户的 Wiki 知识库中创建文档（通过 Wiki API），而不是创建独立文档。

### 坑3：不同租户 open_id 不一致
**表现**：添加管理员为文档成员时，用了 API 里查询到的 open_id，但和浏览器登录用户的 open_id 不一致
**根因**：飞书的 open_id 是按 app 维度分配的——不同应用看到的同一个用户的 open_id 不同。浏览器登录和 API 用的是不同的 app 上下文。
**正确做法**：添加成员时用 email 或 user_id（跨 app 通用），不用 open_id。

### 坑4：权限修复后没有浏览器验证
**表现**：API 返回权限修改成功（HTTP 200），但浏览器打开仍显示无权限
**根因**：API 成功 ≠ 用户能访问。必须在浏览器中用用户身份验证。
**正确做法**：每次权限修改后，用 browser_open 打开文档 URL 验证可访问性。

### 坑5：飞书 API 默认权限是 tenant_readable 不是 closed
**表现**：助手误以为需要"显式设置私密"，但实际上 API 创建的文档默认就是租户内可见
**根因**：对飞书 API 默认行为不了解。显式设 `closed` 反而把原本可用的权限锁死了。
**正确做法**：不设置任何权限参数 = 默认 tenant_readable = 租户内所有人可见。这是最安全的默认值。

## 用户纠正原话

2. "这完全是让我很失望的"

## 根因分析

**最深层根因：不了解飞书「应用身份 vs 用户身份」的权限模型差异。**

- tenant_access_token 创建的文档 → 创建者是应用 → closed = 只有应用能看
- user_access_token 创建的文档 → 创建者是用户 → closed = 用户能看
- 当前只有 tenant_access_token → 绝不能设 closed

## 正确流程（飞书 API 文档创建权限检查清单）

```
1. 创建文档（默认 tenant_readable，不碰权限参数）
2. 用 browser_open 打开文档 URL → 验证管理员可访问
3. 如需更高权限 → 添加管理员为 full_access 成员（用 email，不用 open_id）
4. 再次浏览器验证
5. 绝不调用 /permissions/{token}/public 设 closed
```
