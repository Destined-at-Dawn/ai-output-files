# 教训：分段整理逐字稿——道法术器深度拆解（根因级）

> **目标**：不是记录"出了什么错"，而是建立"永远不会再犯"的防错机制。
> **框架**：道法术器 × 百大认知 × 决策树

## 日期
2026-06-06

## 场景
用户上传 6.3 JC 分享逐字稿（何锦成 + Michelle，1608行），要求整理为知识库文档并存入经验库和私域认知。

---

## 一、道——根因原理

> 道 = 为什么一定会犯这个错？底层认知机制是什么？

### 1.1 即时响应陷阱（系统1劫持）

「系统1是自动的、快速的、直觉的、无意识的；系统2是需要努力的、缓慢的、逻辑的、有意识的。我们大部分时间里生活在系统1的错觉中。」——《思考，快和慢》

**本次表现**：用户发第一段，AI立即开始整理——这是系统1的"收到任务就执行"本能。没有经过系统2的判断："这是完整版吗？应该等齐再动吗？"

**根因**：AI没有"等待"的默认行为。人类专家处理分段输入时，会自然地说"等你发完我再看"，但AI被训练为"即时响应"，这个倾向在长文整理场景下变成劣势。

### 1.2 知识搬运 ≠ 知识建构

「你并没有上升到你的目标水平，你下降到了你的系统水平。」——《原子习惯》

**本次表现**：
- 经验教训文件只记录了"坑→正确做法"，是**信息搬运**
- 私域认知文件只列了Michelle的观点，是**信息搬运**
- 没有用道法术器框架重新组织，没有用认知科学支撑

**根因**：记录教训的习惯停留在"记流水账"层面。真正有用的教训是**知识建构**——把经验压缩成可执行的决策树和防错机制，下次遇到类似场景时系统自动触发。

### 1.3 认知负荷过载

「工作记忆是学习过程中最致命的瓶颈——它的容量和持续时间极其有限，任何超越其极限的认知活动都会导致学习完全失效。」——《认知负荷理论》

**本次表现**：分段整理意味着每段都要：
1. 重新读取上文已整理的内容（增加认知负荷）
2. 判断新内容与已有内容的边界（增加认知负荷）
3. 去重合并（增加认知负荷）

一次性整理时，这些负荷只需要承担一次。

### 1.4 习惯回路：没有"等齐再动"的触发器

「习惯是不能被消除的，它们只能被替代。一旦你理解了大脑中习惯回路的运作方式，你就可以把坏习惯替换成好习惯。」——《习惯的力量》

**本次表现**：现有的习惯回路是"收到内容→立即整理"。没有建立新的回路"收到分段内容→先确认完整性→再整理"。

**根因**：CLAUDE.md 的路由表和 F8 技能链中，没有"长文整理"这个场景的专用协议。

## 二、法——机制与决策树

> 法 = 遇到这类场景时，应该走什么机制？

### 2.1 「等齐再动」协议（新建）

```
收到长文输入？
  ├── 是分段粘贴（用户说"这是后面的内容" / 多次粘贴 / 路径是 Temp 临时文件）
  │   └── ① 存临时文件 ② 回复"已收到第X段，等完整版到了再一次性整理" ③ 不开始正式整理
  │
  ├── 是完整版（文件路径稳定 / 用户说"这是完整版" / 文件 > 800行）
  │   └── 进入 2.2 输出矩阵确认
  │
  └── 不确定（内容很短 / 没有上下文）
      └── 问"这是完整版吗？还有后续内容吗？"
```

### 2.2 输出矩阵确认（新建）

```
确认完整后，正式整理前，必须确认输出矩阵：
  ├── 本地文档（outputs/）→ 必选
  ├── 经验库（经验库/做得好的/ 或 做得差的/）→ 按需
  ├── 私域认知（私域认知/）→ 按需
  ├── 飞书知识库 → 按需
  └── 其他（SOP更新/技能链路由等）→ 按需

确认方式：一句问用户"整理完放哪里"，列出候选目标。
```

### 2.3 降级读取机制

```
Read 工具失败？
  ├── 第1次失败 → 换 Python 读取（open(encoding='utf-8')）
  ├── Python 也失败 → 检查文件是否存在（os.path.exists）
  ├── 文件不存在 → 检查路径编码（中文路径常见问题）
  └── 都失败 → 报告用户，不盲目重试

核心原则：同一方法失败2次 = 必须换方法。永远不要重试3次以上。
```

### 2.4 「教训深度化」协议

```
记录教训时，必须走道法术器四层：
  ├── 道：根因原理（引用百大认知原句，解释"为什么一定会犯"）
  ├── 法：防错机制（决策树/协议/checklist）
  ├── 术：具体操作步骤（可直接执行的SOP）
  └── 器：工具和模板（检查清单/脚本/速查卡）

禁止：只记"现象→正确做法"（信息搬运）
```

## 三、术——具体操作步骤

> 术 = 每一步具体怎么做？

### 3.1 分段输入处理SOP

**Step 1**：收到第一段内容后，立即判断：
- 内容是否完整？（有开头有结尾？还是戛然而止？）
- 文件大小：> 500行 → 大概率不是完整版
- 路径是否是临时目录？（Temp/ → 大概率是分段粘贴）

**Step 2**：如果判断为分段：
```python
# 1. 存临时文件
import os
temp_dir = r"E:\ai产出文件\牛马\创作\创作\outputs\temp"
os.makedirs(temp_dir, exist_ok=True)
# 存为 temp_part_1.txt, temp_part_2.txt ...

# 2. 回复用户
"已收到第1段（约XXX行），已缓存。等完整版到了再一次性整理，避免重复合并。"
```

**Step 3**：收到完整版后：
```python
# 1. 读取所有临时段，合并验证
# 2. 与完整版对比，确认内容一致
# 3. 一次性整理
# 4. 清理临时文件
```

### 3.2 输出矩阵确认SOP

**一句话确认**：
> "整理完放哪里？我列了几个候选：① outputs/本地文档 ② 经验库 ③ 私域认知文件夹 ④ 飞书知识库。你选哪些？"

**如果没有回复**：默认只写 outputs/，不自行决定其他目标。

### 3.3 教训深度化SOP

记录任何教训时，必须包含：

```markdown
# 教训标题

## 道——根因原理
「引用百大认知原句」——《书名》
为什么一定会犯这个错？底层认知机制是什么？

## 法——防错机制
决策树 / 协议 / checklist（可直接执行的判断逻辑）

## 术——具体操作步骤
Step 1 → Step 2 → Step 3（每步都有具体命令/模板）

## 器——工具和模板
检查清单 / 速查卡 / 脚本

## 为什么看起来合理实际不行
表面逻辑 vs 实际结果

## 影响范围
哪些场景会遇到同类问题
```

## 四、器——工具和检查清单

### 4.1 分段输入检查清单

每次收到长文输入时，逐项检查：

- [ ] 这是完整版吗？（有开头有结尾？）
- [ ] 文件 > 500行 → 追问"还有后续吗"
- [ ] 路径是 Temp 目录 → 大概率分段，追问
- [ ] 用户说了"后面的内容" → 确认分段，缓存不整理
- [ ] 用户说了"完整版" → 进入输出矩阵确认

### 4.2 输出矩阵检查清单

开始任何整理任务前：

- [ ] 确认输出目标（至少问一次）
- [ ] 本地 outputs/ → 必选
- [ ] 经验库 → 有踩坑就写
- [ ] 私域认知 → 有新认知框架就写
- [ ] 飞书 → 用户确认后才上传
- [ ] 一次性写入所有目标，不分批

### 4.3 教训深度化检查清单

记录教训时：

- [ ] 有"道"层（引用了百大认知原句）
- [ ] 有"法"层（有决策树/checklist）
- [ ] 有"术"层（有具体操作步骤）
- [ ] 有"器"层（有工具/模板）
- [ ] 有"为什么看起来合理实际不行"
- [ ] 不是纯"现象→正确做法"的信息搬运

### 4.4 降级读取检查清单

Read 工具失败时：

- [ ] 第1次失败 → 换 Python `open(encoding='utf-8')`
- [ ] Python 失败 → `os.path.exists()` 确认文件存在
- [ ] 文件不存在 → 检查路径编码/中文路径问题
- [ ] 同一方法最多试2次，必须换方法
- [ ] 不要重试3次以上同一方法

## 八、元教训：审计无行动 = 未审计（v1.1 新增）

> 用户纠正：上轮技能审计列出了"被忽视的 F8 技能链"和"SOP-01 没读"，但**没有当场修正**。
> 本质：**列出问题 ≠ 解决问题。审计的价值不在记录，在于触发修正行动。**

### 8.1 道——认知机制

「我慢慢发现，用任何办法都不可能消除这个习惯——我就是无法开始做我不感兴趣的事，直到最后期限逼近。」——《深度工作》

**根因**：F1 技能审计被设计为"对话结束前的格式化输出"，变成了**仪式性动作**——输出了审计格式，就觉得自己"做了审计"。但审计的本质是**门禁**，不是**仪式**。门禁不通过 = 不能结束。

「反馈的价值取决于它是否被用来调整行为。」——《刻意练习》

**对照**：刻意练习的核心是"即时反馈 → 立即调整"。如果反馈（审计发现了问题）没有导致调整（当场修正），那反馈就白给了。

### 8.2 法——审计即修正协议（新建）

```
F1 技能审计发现问题？
  ├── "被忽视的"一栏非空
  │   ├── 问题可当场修正（路由表缺失/文件未更新/技能未调用）
  │   │   └── 🔴 立即修正 → 修正后再输出审计报告
  │   ├── 问题需要下轮修正（需要用户确认/需要额外材料）
  │   │   └── 标记为 [下轮必修] → 写入今日记忆作为提醒
  │   └── 问题已无法修正（任务已完成/文件已关闭）
  │       └── 记录到教训文件 → 写入路由表/SOP 防止下次
  │
  └── "被忽视的"一栏为空
      └── 自问：真的没有遗漏吗？是否有"不知道自己不知道"的技能？
          └── 对照技能注册表逐项扫描
```

**铁律**：审计报告中"被忽视的"一栏有内容 → 必须在**同一轮对话**内完成修正或标记 [下轮必修]。不允许"列出来就结束"。

### 8.3 术——审计修正 SOP

**Step 1**：正常执行任务，记录所有技能调用

**Step 2**：任务完成后，执行 F1 审计
```
本次调用了哪些技能/工具？
对照：F8 技能链（按任务类型查 CLAUDE.md）
遗漏了什么？→ 列入"被忽视的"
```

**Step 3**：🔴 审计发现问题 → 立即修正
```
可修正（路由表/SOP/文件缺失）→ Edit/Write 修正 → Read 验证 → 再输出审计
需确认（用户决策）→ 问用户 → 等回复 → 修正
无法修正（任务已结束）→ 写教训文件 + 更新 SOP 防下次
```

**Step 4**：输出修正后的审计报告（不是原样输出有问题的报告）

### 8.4 器——审计自检检查清单

F1 审计输出前，必须过以下检查：

- [ ] "被忽视的"一栏是否有内容？
- [ ] 有内容 → 是否已当场修正？
- [ ] 修正方式：Edit/Write → Read 验证？
- [ ] 无法修正 → 是否已写入教训文件 + [下轮必修] 标记？
- [ ] 审计报告是否反映了修正后的状态（而非修正前的问题）？
