# 飞书 Wiki 权限反复排查浪费 15+ 轮

> **教训 E33** | 2026-06-06 | 来源：本轮对话——wiki 写入 403 死循环

---

## 踩坑过程

1. 第 1 轮：用 tenant_access_token 写 wiki blocks → 403
2. 第 2-5 轮：尝试不同 API 路径（docx blocks / wiki nodes / drive permissions / drive public）→ 全部 403
3. 第 6-8 轮：尝试把自己添加为协作者 → 1063002 Permission denied
4. 第 9-10 轮：尝试在 wiki space 添加 bot 为成员 → 131101 Bot cannot be added via API
5. 第 11-13 轮：建议用户在飞书 UI 搜索「workbuddy」添加 → 搜不到机器人
6. 第 14-15 轮：反复建议开通「wiki:wiki」scope → 用户说「从来没有这个权限」（scope 确实存在但飞书控制台显示名不同）
7. **最终突破**：用户自己用 user_access_token 绕过了权限限制

## 根因

飞书 wiki 的权限模型是**双层架构**：
- **第一层：App scope**（开放平台配置）— 决定 App 能调用哪些 API
- **第二层：wiki space 级权限**（飞书 UI 操作）— 决定 App 能访问哪些 wiki 空间

tenant_access_token（App 身份）即使 scope 齐全，也需要 wiki space 管理员在 UI 里把 App 添加为成员。**但飞书 UI 的「知识库设置 → 成员管理」不支持添加机器人**，形成死锁。

## 为什么看起来行其实不行

- scope 确实有 `wiki:wiki`，但 scope ≠ 实际权限（飞书的权限模型不是简单的 scope → 能力映射）
- 「添加 App 为 wiki 成员」的建议在 UI 中搜不到机器人，因为飞书 wiki 的「添加成员」只支持人不支持 bot
- tenant_access_token 永远无法突破 wiki space 级权限限制——这是飞书的架构设计，不是 bug

## 正确做法

1. **第一时间**确认 wiki 写入是否走 tenant_access_token → 如果是，直接告知用户需要 user_access_token
2. **不要反复尝试不同 API 路径**——所有基于 tenant_access_token 的路径都会被 wiki space 级权限拦截
3. **wiki 写入的唯一可靠方案**：user_access_token（用户 OAuth 授权）或用户手动在 UI 操作

## 退出判据

- wiki 写入返回 403 / 131006 / 1770032 → 立即停止尝试，切换到 user_access_token 方案
- **最多探测 2 轮**（确认是权限问题而非格式问题），超过 2 轮必须切换方案
