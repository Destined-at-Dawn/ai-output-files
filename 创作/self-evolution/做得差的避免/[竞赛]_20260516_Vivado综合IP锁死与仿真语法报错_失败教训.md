# Vivado IP锁死与仿真常见坑教训

## 1. 错误现场
- **Vivado IP 锁死**：手动修改了 `blk_mem_gen_0.xci` 文件（如 `Load_Init_File=true`），但没有清理 Vivado 自动生成的对应的 `.xml` 缓存。Vivado 发现文件与缓存不匹配，抛出 `stale content` 错误，并强制将整个 IP 锁死。之后任何试图 `reset_target` 或 `generate_target` 的操作全部报错。
- **综合找不见模块**：直接运行 `launch_runs synth_1` 时，由于 IP 被锁导致生成失败或尚未综合完，顶层 `lcd_top` 找不见被实例化的 `blk_mem_gen_0`，报错模块未定义。
- **XSIM 语法报错**：在 Testbench 中使用了 `fork/join_any` 或 `disable fork`，这是 SystemVerilog 语法，而 Vivado 2020.1 XSIM 在 `.v` 文件中严格使用 Verilog-2001 解析器，导致大量的语法 `ERROR`。
- **LCD 屏幕方向误判（横竖屏错位）**：代码坐标全按 480x320（横屏）写，但 `MADCTL` 寄存器错配成了 `0x48`（竖屏模式，320x480），结果图像如麻花一般扭曲，错位换行，只显示一半甚至花屏。

## 2. 错误原因分析
1. **IP 缓存机制**：Vivado 对生成的 IP 采用严格的配置与缓存版本校验。手动修改 `.xci` 不会主动触发 XML 缓存的更新，导致内部校验失败而锁死。
2. **顶层与IP的依赖时序**：IP `blk_mem_gen_0` 需要一个独立的综合过程（`blk_mem_gen_0_synth_1`），如果它没有彻底完成，顶层综合就找不到该 IP 的网表。
3. **仿真器解析器差异**：XSIM 默认不能自动推断混用的 SV 语法。在 `.v` 文件中必须保持纯正的 Verilog-2001。
4. **硬件寄存器与软件逻辑脱节**：LCD 的分辨率并非仅由数据量决定，硬件控制器的扫描方向（行列地址指针递增方式）决定了最终的物理映射，`MADCTL` 是此映射的关键。

## 3. 正确做法
1. **修改 IP 配置永远用 TCL**：
   使用 `set_property -dict [list CONFIG... ] [get_ips name]`，绝不手动改 `.xci`。若已锁死，需执行 `set_property is_locked false [get_files xxx.xci]` 并主动删除对应的 `.xml` 文件。
2. **串行执行综合任务**：
   必须确保 IP 自己的 synth run 跑完后再跑系统级 synth。
3. **编写纯正 Verilog-2001 TB**：
   替代语法：使用 `begin : block_name ... disable block_name; end` 替代 `disable fork` 等跳出逻辑。
4. **横竖屏映射对齐**：
   在驱动 LCD 屏幕前，首先将 `MADCTL` 的扫描方向（如是否包含 `MV=1` 行列互换）与预设坐标（480x320 vs 320x480）进行精确对齐。
