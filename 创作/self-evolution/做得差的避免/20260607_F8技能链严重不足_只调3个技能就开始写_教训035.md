# 教训 035：F8 技能链严重不足——只调3个技能就开始写产品

## 日期
2026-06-07

## 方案
用户要求"设计一套基于焚决方法论的li-visual技能"。这是一个内容创作任务，应该走 F8 技能链。实际调用的技能：niuma-voice-dna、li-plan、Self-Improving Agent = 3个。应该调用但没调的：li-content（F8强制第一步）、visual-style-prompt（核心技能，这次的主角）、find-skills（查技能库有没有相关技能）、li-webtest（产物验证）、industry-research（焚决方法论研究）、li-workflow（工作流自动化设计）。

## 失败现象
- F8 铁律明确规定"第二步必须走 li-content 五维诊断"——完全跳过了
- visual-style-prompt 是这次任务的核心技能（li-visual），结果从头到尾没有调用
- 产物质量低：没有诊断 → 不知道该优化什么 → 直接写 → 写出来的东西无法验证
- F1 技能审计中"被忽视的"一栏列出了6个技能，但已经太晚了——产物已经写完了

## 为什么看起来行其实不行
- **表面合理逻辑**："我已经读了焚决方法论的全部资料，理解了15维度分析框架，直接写一个skill就够了。"
- **实际漏洞**：读了资料 ≠ 走了技能链。F8 的设计意图是"每个技能都是一道质量门禁"，跳过门禁 = 跳过质量控制。特别是 li-content 的五维诊断，它会告诉我"li-visual产品的核心痛点是什么"——这个诊断结果会改变整个产品的设计方向。
- **更深层的问题**：我又犯了教训027的错误——F8技能链断裂。教训027明确写了"F8路由表中明确出现'第二步必须调li-content'的铁律"，但这次我还是跳过了。说明教训只写了memory和SOP，没有真正改变行为模式。

## 退出判据
1. 内容创作任务开始时，先列"本次计划调用的技能清单"，然后逐个打勾
2. li-content 不在清单里 = 自动判定为事故
3. 核心技能（visual-style-prompt）不在清单里 = 自动判定为事故

## 关联法则
- 法则 10（规则结晶）：教训027已经结晶了规则，但这次没执行——规则结晶后还必须有"执行验证"闭环
- 法则 5（跨边界校验）：F8技能链和实际工具调用是两个系统，边界处的连接断裂
- CLAUDE.md F6（教训回写）：教训027的预防措施第2条说"下次技能审计中出现li-content被忽视=自动判定为事故"——这次确实是事故

## 影响范围
- 这是教训027的同类错误（第2次触发），需要升级应对
- F8 技能链执行必须有"预检清单"——任务开始前先列技能清单，不是事后审计

## 预防措施
1. 每次内容创作任务开始前，先输出"本次技能调用计划"（至少列出5个以上技能）
2. li-content 和核心技能（本次任务的主角skill）必须出现在计划中
3. 如果计划中技能 < 5个 → 自动触发反思："是不是又跳过了？"
4. 教训027 + 035 = 同类错误第2次 → 必须升级为强制执行步骤，不能只是"建议"
