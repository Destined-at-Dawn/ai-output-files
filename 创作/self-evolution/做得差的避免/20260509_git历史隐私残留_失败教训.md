# 失败教训：git 历史中的隐私残留

**日期**：2026-05-09
**场景**：GitHub 开源仓库隐私审查
**严重程度**：🔴 高（用户质疑后才发现）

## 用户原话

「你确定这会让别人在github上看到我的个人信息吗」

## 我的错误

### 错误 1：用 `git rm --cached` 代替 `git filter-repo`

**做了什么**：
- career-breakthrough 仓库的 CLAUDE.md 包含隐私信息
- 我只做了 `git rm --cached CLAUDE.md` + 更新 `.gitignore` + commit + push
- 报告说"隐私已清除"

**问题**：
- `git rm --cached` 只从当前 HEAD 移除文件
- **旧 commit 中仍然存在完整文件内容**
- 任何人点击 GitHub 上的旧 commit → 浏览文件即可看到

**正确做法**：
- 必须使用 `git filter-repo --invert-paths --path <文件名>` 从所有历史中彻底删除
- 然后 `git push --force` 覆盖远程
- 最后用 `gh api` 验证所有 commit tree 中都找不到该文件

### 错误 2：没有验证"历史是否也干净"

**做了什么**：
- 只验证了 `git ls-tree HEAD` 不含 CLAUDE.md
- 没有检查 `git log --all -- CLAUDE.md` 是否还有旧 commit

**正确做法**：
- 推送后必须检查 **所有 commit**，不只是 HEAD
- 验证命令：`git rev-list --all | xargs -I {} git ls-tree -r {} --name-only | grep <文件名>`

## 根因分析

| 层次 | 原因 |
|------|------|
| **表象** | 认为"从当前版本删除 = 从仓库删除" |
| **错误假设** | git rm 会清除历史（不会！） |
| **知识盲区** | 不知道 `git filter-repo` 的存在和必要性 |
| **流程缺陷** | 没有"隐私文件必须从历史清除"的检查步骤 |

## 正确流程（必须遵守）

```
发现隐私文件在 git 中
  ↓
① git filter-repo --invert-paths --path <文件名>
  ↓
② git push --force origin main
  ↓
③ 验证：git log --all -- <文件名> → 应返回空
  ↓
④ 验证：gh api 遍历所有 commit tree → 确认无命中
  ↓
⑤ 报告：只有全部验证通过才能说"已清除"
```

## 关键教训

> **`git rm --cached` ≠ 从历史中删除**
> **涉及隐私的文件清理，必须用 `git filter-repo`，不能用 `git rm`**
