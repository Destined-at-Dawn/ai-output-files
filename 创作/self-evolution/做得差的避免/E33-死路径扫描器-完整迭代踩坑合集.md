# E33 — 死路径扫描器完整迭代踩坑合集

> 一次工具开发踩了 8 个坑。核心教训：**每个你以为"修好了"的版本，跑一遍才知道还有多少误报。**

## 日期
2026-06-07

## 背景
用户要求"自动化脚本检测所有 SOP 中的死路径"。从 v2.0 到 v3.1 经历了 5 轮迭代，每轮都以为修好了，每轮都还有大量误报。

## 踩坑清单（8个）

### 坑1：扫描卡死（v2.0）
- 现象：脚本跑了2分钟没反应
- 根因：对 `~/.newmax/skills/` 下 7003 个文件逐个做 rglob 搜索
- 修复：skills 目录从默认扫描中排除，只在 `--full` 模式才扫
- 教训：**扫描范围要先评估规模**，不能盲目 rglob

### 坑2：正则转义错误
- 现象：`re.error: bad character range`，脚本启动就崩
- 根因：正则表达式中 `[` 没转义，被当字符集语法
- 修复：`re.escape()` 或手动转义
- 教训：**正则写完先用 Python 测试编译**，不要直接跑脚本

### 坑3：路径分隔符不一致
- 现象：索引中路径用 `\`，查找时用 `/`，匹配全部失败
- 根因：Python `str(Path(...))` 返回 `\`，但路径字符串用 `/`
- 修复：查找时统一 `Path(x).as_posix()` 或统一 `os.path.normpath()`
- 教训：**Windows 路径比较必须统一格式**，这是一个每次都踩的坑

### 坑4：SKILL.md 教学示例被当路径
- 现象：1473 条死路径来自 3 个 SKILL.md 文件
- 根因：skill 文件中有大量示例路径（如 `你的项目/xxx.md`），被正则匹配
- 修复：skills 扫描从默认模式中排除
- 教训：**教学文档中的路径 ≠ 真实引用**，要区分"示例"和"引用"

### 坑5：路由表中文关键词被当路径
- 现象：`公众号/小红书/视频/海报/群公告` 被当路径
- 根因：路由表中用 `/` 分隔多个关键词，正则把它当路径
- 修复：过滤纯中文+标点的路径段
- 教训：**正则只能做格式匹配，不能做语义理解**

### 坑6：模板占位符被当路径
- 现象：`{YYYY-MM-DD}`、`xxx.md`、`your-name` 被当真实路径
- 根因：占位符看起来像路径的一部分
- 修复：全路径段检查（不只检查末尾）
- 教训：**占位符检测要在路径的每个部分做**，不能只看末尾

### 坑7：每轮都说"大幅改进"
- 现象：修完一个bug就报告"误报率大幅降低"
- 根因：修了一个类别没验证总误报率，只看到漏网之鱼减少了
- 教训：**修完必须跑全量，报总数而不是报降幅**。"从 3175 降到 2800"听起来很多，但误报率还是 80%

### 坑8：5轮迭代才到可接受水平
- v2.0：卡死
- v3.0：3175 条（60%+误报）
- v3.1-alpha：正则报错
- v3.1-beta：2800 条（分隔符问题）
- v3.1-gamma：1473 条（SKILL.md噪音）
- v3.1-final：618 条（可接受）
- 教训：**工具开发要预留"验证迭代"的时间**，不要以为一轮能搞定

## 系统性根因

**核心问题**：路径扫描器的本质是"从自然语言文本中识别文件路径"，这是一个 NLP 问题，不是正则能完美解决的。正则只能做格式匹配，不能理解语义。

**正确的架构**：
1. 格式层：正则匹配路径格式（过滤掉明显的非路径）
2. 上下文层：根据源文件位置推断相对路径（不只从工作区根目录找）
3. 语义层：过滤教学示例、路由表、模板变量（这个最薄弱）
4. 验证层：Path.exists() 验证（最终判断）

## 防止再犯的规则

1. 扫描范围先评估规模，不能盲目 rglob
2. 正则写完先 `re.compile()` 测试
3. Windows 路径比较必须统一格式
4. 修完一个 bug 必须跑全量验证
5. 报告总数和误报率，不只报降幅
6. 工具开发预留 3 轮验证迭代的时间
