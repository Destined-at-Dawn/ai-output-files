8|## 现象
9|
10|收到"PDF→MD→深度拆解→生成第63本认知书籍"任务后，从零手写6个Python脚本(v1→v6)：
11|- v1→v5全部失败（格式冲突/变量未定义/太小不达标）
12|- v6才成功（改用程序化批量生成）
13|- 累计约50分钟浪费，5个脚本散落在outputs/未管理
14|
15|## 根因（道法术器四层）
16|
17|### 道：默认行为模式——"先写后查"
18|AI拿到"写脚本"任务后，系统1自动启动"手写代码"模式。li-script skill 存在于可用列表但未被激活——因为路由表没有"写脚本→li-script"的自动映射。这不是懒——是路由缺口。
19|
20|### 法：缺少"动手前强制检索"的门禁
21|li-script 设计了 Phase 1（检索引擎先行），但本次任务没走任何检索步骤就开始写代码。如果写了第一个脚本前先读 script-library/INDEX.md，会发现库虽空但建立了"先搜再写"的肌肉记忆。
22|
23|### 术：脚本管理处于野生状态
24|6个脚本散落在 outputs/，没有入库/归档分类，没有 USAGE 注释，后续任何AI再遇到同类任务=再次从零手写。没有 li-script Phase 3（入库/归档决策门）。
25|
26|### 器：li-script 存在但触发词不匹配
27|用户的原始消息没有触发 li-script 的路由条件。任务实质是"PDF→MD→深度拆解→书籍"，AI通过原生 Python Write 工具直接写了脚本，没有经过 li-script 的路由检测。
28|
