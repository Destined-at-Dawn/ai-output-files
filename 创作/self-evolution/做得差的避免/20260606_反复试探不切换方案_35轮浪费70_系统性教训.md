# 反复试探不切换方案——35 轮浪费 70%

> **教训等级：铁律级** | 2026-06-06 | 来源：本轮对话——飞书 wiki 权限 + API 权限修复
> **本文档目的：建立「方案切换决策树」，消灭"换个路径再试试"的无效循环**

---

## 一、本轮的「反复试探」全景

### 案例 1：飞书 Wiki 权限（15+ 轮）

```
轮次 1-3:  tenant_token 写 wiki blocks → 403
轮次 4-6:  换 API 路径（docx blocks / wiki nodes / drive permissions）→ 全部 403
轮次 7-9:  尝试把自己添加为协作者 → 1063002
轮次 10-12: 尝试在 wiki space 添加 bot → 131101 Bot cannot be added
轮次 13-15: 建议用户搜索机器人添加 → UI 搜不到
最终突破:    用户自己用 user_access_token 绕过
```

**根因**：所有 15 轮的底层障碍是同一个——tenant_access_token 没有 wiki space 级权限。换 API 路径不改变底层权限模型。

### 案例 2：权限修复（10+ 轮）

```
轮次 1-3:  API 修改 permissions/public → 1063002
轮次 4-5:  API 修改 permissions/members → 1063002
轮次 6-7:  API batch_update → 1063002
轮次 8-9:  浏览器打开 → 仍无权限
轮次 10:   最终重建文档
```

**根因**：App 创建的文档，App 自己不是 owner。所有权限管理 API 都需要 owner 身份。

## 二、为什么 AI 会反复试探

| 心理陷阱 | 表现 | 本质 |
|----------|------|------|
| **路径依赖** | "这个 API 不行，换一个类似的试试" | 换路径不换底层假设 |
| **沉没成本** | "已经试了 5 轮了，再试一轮可能就通了" | 不愿意承认方向错误 |
| **乐观偏差** | "这次参数可能对了" | 忽略了底层架构限制 |
| **缺乏退出机制** | 没有预设的「该停手了」判据 | 无限循环直到用户发火 |

## 三、方案切换决策树（铁律）

```
操作失败了？
├── 第 1 次失败
│   → 分析错误码含义
│   → 如果是参数/格式错误 → 修正后重试（最多 2 次）
│   └── 如果是权限/架构限制 → 直接进入方案切换
│
├── 第 2 次失败（同类错误）
│   → 确认是否是同一个底层障碍
│   → 如果是 → 立即切换到完全不同的方案
│   └── 如果不是 → 修正后给最后一次机会
│
└── 第 3 次失败（同类错误）
    → 🛑 停止。这是结构墙，不是平台期
    → 切换到替代方案
    → 记录为负结果
```

### 替代方案优先级（飞书场景）

```
API 失败 → 方案 A: user_access_token（OAuth）
         → 方案 B: 浏览器操作（手动）
         → 方案 C: 创建独立文档 + 用户手动移入
         → 方案 D: 告知用户"当前技术条件下做不到"
```

**禁止**：方案 A 失败后回到方案 A 的变体（换参数、换路径、换端点）。换汤不换药 = 无效循环。

## 四、2 轮规则

**任何操作，同一底层障碍最多探测 2 轮。**

| 底层障碍类型 | 1-2 轮 | 第 3 轮 |
|-------------|--------|--------|
| 权限不足 | 确认是权限问题 | 切换到非 API 方案 |
| 格式错误 | 修正格式 | 检查文档/spec |
| 参数错误 | 修正参数 | 搜索已知 issue |
| 网络超时 | 重试一次 | 检查网络/换时段 |
| 架构限制 | 确认是架构限制 | 停止，记录为结构墙 |

## 五、关联

- 工程法则 9（识别结构墙）：wiki 权限是结构墙，不是平台期
- 工程法则 7（负结果归档）：死路必须归档，不被重复踩
- 工程法则 3（实测优先）：但实测 ≠ 无限试错，实测要有退出判据
- 飞书全流程检查清单（本轮新建）：操作前先读清单，避免踩坑
- 认知科学支撑：「确认偏误」——倾向于寻找支持自己假设的证据，忽略反面证据。每一轮"再试试"都是确认偏误的体现
