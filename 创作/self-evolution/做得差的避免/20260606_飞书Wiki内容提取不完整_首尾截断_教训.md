# 失败教训：飞书 Wiki 内容提取不完整——首尾截断、需要多轮补全

## 日期
2026-06-06

## 场景
从飞书 Wiki 页面提取内容时，第一次提取只拿到了部分内容。原因有两个：① 浏览器方式滚动不到位；② API 方式分页未处理完。

## 踩坑清单

### 坑1：浏览器提取只看到首屏
**表现**：browser_eval 读取 `document.body.innerText` 只拿到前几千字，后面的内容没加载
**根因**：飞书文档是懒加载的，页面滚动到哪才加载到哪。首屏之外的内容在 DOM 里不存在。
**正确做法**：用 browser_scroll 滚到底部 → 等待加载 → 再读取。或者直接用 API 提取。

### 坑2：API 分页未处理
**表现**：飞书 Docx API `GET /docx/v1/documents/{id}/blocks` 默认只返回前几十个 block
**根因**：API 有分页机制（page_token），但第一次调用没处理分页，只拿到了第一页。
**正确做法**：循环调用，直到 `has_more = false`：
```python
all_blocks = []
page_token = ""
while True:
    resp = requests.get(url, params={"page_token": page_token, "page_size": 500})
    data = resp.json()["data"]
    all_blocks.extend(data["items"])
    if not data.get("has_more"):
        break
    page_token = data["page_token"]
```

### 坑3：Markdown 文件写入中文路径失败
**表现**：Write 工具写入成功但后续 Python 读取时文件为空
**根因**：中文路径在不同工具之间传递时编码不一致（Write 工具用 UTF-8，Bash 可能用 GBK 解码路径）。
**正确做法**：所有涉及中文路径的文件操作，统一用 Write 工具写、Read 工具读。不要在 Bash/Python 中用 `open()` 操作中文路径。

## 根因分析

**最深层根因：没有选择最可靠的提取方式。**

内容提取的优先级应该是：
1. **API 提取**（最可靠，有分页机制，不依赖渲染） → 首选
2. **浏览器滚动提取**（依赖懒加载，可能截断） → 备选
3. **复制粘贴**（人工操作，不适合自动化） → 最后手段

## 正确流程

```
1. 获取文档 ID（wiki get_node → obj_token）
2. API 分页读取全部 blocks
3. Python 解析 blocks → 转为 Markdown
4. Write 工具写入本地文件
5. Read 工具验证内容完整性（检查字节数和首尾内容）
```

## 关联工程法则
- 法则3（实测优先）：先验证提取结果是否完整，不要假设拿到了全部内容
- 法则5（跨边界校验）：Wiki API → Blocks → Markdown → 本地文件，每个环节都可能丢失信息
- 法则8（门禁文化）：提取完成后必须检查"内容完整性门禁"
