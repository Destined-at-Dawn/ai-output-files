# E35 — 经验库更新遗漏（只记了表面坑，漏了深层坑）

> 用户说"你总结的还不够"——因为你只记了"发生了什么"，没记"为什么会反复发生"。

## 日期
2026-06-07

## 背景
这次对话踩了 10+ 个坑，但第一轮只记录了 3 个（E30/31/32）。用户指出"你总结的还不够，我要的是再也不会犯错"。

## 遗漏了什么

### 遗漏1：只记了"现象"没记"模式"
- E30 只写了"扫描器迭代踩坑"，没写"每次修完一个bug就宣布改进但误报率依然60%+"
- 这是一个**行为模式**：AI 修完一个问题就报喜，不跑全量验证
- 教训：记录教训时要问"这是不是一种反复出现的行为模式？"

### 遗漏2：漏了工具开发的系统性问题
- 5 轮迭代才到可接受水平
- 每轮都以为修好了，每轮都有新问题
- 这不是"一个坑"，是"工具开发方法论"的问题
- 教训：一个项目踩了多个坑 → 要写一个**综合性教训**，不只是逐个记录

### 遗漏3：没有建立防错机制
- 记录教训 ≠ 防止再犯
- 用户要的是"再也不会犯错"，不是"记录了犯过的错"
- 教训：经验库的终极目标是**防错机制**，不只是**错题本**

## 根因分析

**为什么AI会"只记表面坑"？**

1. **任务驱动思维**：经验库更新被当作"任务完成清单"——做完一项打个勾，不深度复盘
2. **数量代替质量**：写了 3 个文件觉得"已经更新了"，没问"够不够"
3. **缺少用户视角**：AI 从自己的角度看"哪些坑值得记"，没从用户的角度想"怎么确保不再犯"

## 防止再犯的规则

1. 经验库更新前，先全面盘点这次对话的所有操作（不只是明显的错误）
2. 每个教训要回答三个问题：发生了什么 → 为什么 → 怎么确保不再发生
3. 如果一个项目踩了多个坑 → 先写综合教训，再拆分细节
4. 更新完后问自己："用户看到这个，能防止再犯吗？" 不能 → 补充防错机制
5. 用户说"你总结的不够" → 立即停止辩解，全面重写
