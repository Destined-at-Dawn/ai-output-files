## 本轮全部踩坑清单（7个）

### 坑1：跨人物信息污染 — txt中"小黎同辈互助群"误判嘉宾

**表现**：txt文件标题是"AI时代，一个人就是一家公司 — **小黎**同辈互助群分享"
- "小黎"是群名（定语），不是嘉宾名（主语）
- 我错误推断"提到小黎 = 嘉宾是小黎"
- 实际嘉宾是**集阳**（6/1个人IP分享）

**根因**：遇到文本中的人名时不追问"这个人名在此上下文中的角色是主语（谁在说）还是定语（属于谁的）"

**正确做法**：
```
看到"小黎同辈互助群" → 立即自问：
  - "小黎"是群主？→ 是
  - "小黎"是本次分享者？→ 不确定，需要查嘉宾排期表
  - 查排期表 → 6/1是集阳
  - 结论：txt中的"小黎"= 群名，不是嘉宾
```

### 坑2：跳过确认直接生成

**表现**：拿到4个txt文件后直接开始批量生成md，没有先做：
1. 列出所有txt文件清单
2. 匹配每个txt → 嘉宾 → 日期
3. 标注不确定项（如"OPC分享_整理版.txt"文件名里没有嘉宾信息）
4. 把映射表展示给用户确认

**根因**：把"txt→md"当成纯格式转换任务，忽略了它本质上是"内容归属任务"——你要先知道每篇内容是谁说的，才能正确标注。

**正确做法**（强制流程）：
```
Step 0: 用户说"把txt变成md" → 先不写md
Step 1: 列出所有txt文件
Step 2: 用嘉宾排期表 + 文件名关键词 + txt内容首段 → 初步映射
Step 3: 标注不确定项 → 展示映射表给用户确认
Step 4: 用户确认后 → 才动笔生成md
```

### 坑3：没有建立"txt→md"标准工作流

**表现**：社群运营已经做了6天，每天都有嘉宾分享笔记需要整理，但没有形成标准化流程。每次都是"凭现场感觉"转换。

**根因**：没有把"嘉宾笔记整理"设计成一个可复用的工作流。这和教训026（社群分享没有SOP）是同一个根因的不同表现——重复性操作没有标准化。

**缺失的流程步骤**：
- ❌ 没有固定输出目录（直播笔记/md/、直播笔记/txt/ 目录存在但是空的）
- ❌ 没有固定md模板（道法术器结构 vs 其他结构？）
- ❌ 没有固定命名规范（日期-主题-嘉宾.md？）
- ❌ 没有源文件→成品的映射追踪（哪个txt生成了哪个md？）

### 坑4：混淆html模仿和txt转换两个任务

**表现**：用户说"把txt变成md"和"模仿html生成第二天的md"，但我把两件事混在一起做——同时生成了html→md和txt→md的笔记，没有区分边界。

**根因**：听到"模仿"和"变成md"两个词后，没有做任务拆解——直接启动生成模式，把所有文件（html+txt）一起处理。

**正确做法**：
```
用户指令拆解：
  任务A："模仿html"→ 生成第二天格式一致的md笔记
  任务B："把所有txt变成md"→ txt文件批量转换
  → 先做B（txt转md，纯格式转换）→ 确认完成
  → 再问"html的也要转吗？"（不自行扩大范围）
```

### 坑5：动手前没有先读已有教训文件

**表现**：经验库里已经有教训027（跨人物信息污染），如果先读了这个文件，看到"跨人物信息传递时需增加跨人物事实核查环节"，就不会跳进同一个坑。

**根因**：CLAUDE.md要求"每次对话必读 self-evolution/lessons.md 最近5条教训"，但我没有执行。更本质的问题是：**不是"没读到"，是"读到了也不会在做具体任务时回想起来"**——教训和任务执行之间缺少触发连接。

**解决方案**：教训文件除了写"发生了什么"，还需要写一个**触发词列表**——当任务中出现某个关键词时，强制先读对应教训。
```
本教训的触发词：txt转md / 嘉宾笔记 / 整理分享记录 / 把xx变成笔记 / 生成笔记
→ 触发后强制动作：①读教训029 ②做嘉宾映射表 ③确认后再动笔
```

### 坑6：用户第一次纠正后没有立刻写教训草稿

**表现**：用户第一次纠正嘉宾映射时（"这是对应的嘉宾"），我应该立刻：
1. 意识到这是一个教训触发
2. 在本轮对话的当日记忆里写教训草稿
3. 修正所有md文件的嘉宾标注
4. 向用户确认修正完成

实际做法：只修正了文件名，没有写教训，没有系统反思"为什么又判断错了"。

**根因**：用户纠正 = 教训信号，但我把它当成"快速修正"而非"学习机会"——这是教训只记不改的另一种表现：**连记都不及时记**。

### 坑7：用户说"再也不会犯错"——说明之前的教训只写不防

**表现**：用户说"你总结的还不够，我要的是再也不会犯错"——这句话的意思不是"教训写得不详细"，而是"教训写完了，但我下次还是会犯同样的错，因为教训和工作流之间没有强制连接"。

**根因**：当前的教训系统是**被动归档型**——教训写进文件，但下次任务开始时不会自动触发检查。要"再也不会犯错"，需要把教训变成**主动拦截型**——在任务执行的路上设检查点，触发条件匹配时强制停下来。

**解决方案**：每条教训附带一个 `触发检测规则`，在任务开始前自动匹配：
```
触发检测：任务关键词 ∈ {txt转md, 嘉宾笔记, 整理分享, 变成笔记}
→ 强制动作：①读教训029 ②做映射表 ③mcp__ask-user__ask_user确认
→ 未完成强制动作 → 禁止动笔
```

## 根因分析

### 表面根因
AI拿到"把txt变成md"的指令后，跳过了确认步骤直接批量生成。

### 中层根因
没有"txt源文件→嘉宾→日期→格式→成品"的标准工作流，每次都是凭现场判断。

### 深层根因（为什么同类触发3次？）
**教训系统是被动归档型，不是主动拦截型。**

当前的教训系统流程：
```
踩坑 → 写教训文件 → (下次任务) 靠AI自觉想起来读 → 经常想不起来 → 再踩同一个坑
```

需要的教训系统流程：
```
踩坑 → 写教训文件（附带触发词） → (下次任务开始) 关键词匹配 → 强制弹警告 → 必须先读教训再动手
```

三次跨人物信息污染的根因不是"AI记性不好"，是**教训文件的触发条件没有和任务执行流程做硬连接**。教训写进文件但下次任务根本不会主动读——除非碰巧在Step 3（读lessons.md最近5条）时这条教训还在5条以内。

## 跨人物信息污染的通用识别规则（新增）

当在任何文本中看到人名时，必须做以下三步判断：

```
规则：See-Name-Stop（看到人名就停）
1. 这个人是"说的人"还是"被说的人"？（主语 vs 宾语 vs 定语）
2. 这个人是"当前内容的主体"还是"被引用的第三方"？
3. 如果有嘉宾排期表/品牌档案 → 交叉验证（不要靠文本内容推断）

常见误判模式（必须背下来）：
  ✗ "小黎同辈互助群" → 误判：嘉宾是小黎 → 正确：群名含"小黎"，嘉宾另有人
  ✗ "小黎分享了XX观点" → 误判：本段内容是小黎说的 → 正确：可能是转述
  ✗ "XX给吉洋的建议" → 误判：本段是XX的观点 → 正确：要看上下文谁在说
  ✗ 文件路径含人名 → 误判：文件属于这个人 → 正确：路径可能是文件夹名/项目名
```
