## 二、动手前检查清单（每次飞书操作必过，不过不动手）

### 阶段 1：操作前决策（30 秒）

```
□ 确认目标：是创建独立文档？还是写入 Wiki？还是修改已有文档？
□ 确认身份：当前用的是 tenant_access_token（App）还是 user_access_token（用户）？
□ 确认权限：App 是否有对应 scope？Wiki space 是否已添加 App 为成员？
```

**决策树：**

```
要创建文档吗？
├── 是 → 用户能接受独立文档吗？
│   ├── 能 → 用 API 创建（走阶段 2A）
│   └── 不能（需要在知识库里）→ 用浏览器创建 或 Wiki API（走阶段 2B）
└── 否（修改已有文档）→ 文档是独立的还是 Wiki？
    ├── 独立 → API 修改（走阶段 2A）
    └── Wiki → 需要 wiki space 权限（走阶段 2B）
```

### 阶段 2A：API 创建独立文档（6 步，每步必须验证）

```
□ Step 1: 创建文档（POST /docx/v1/documents）
    → 不设任何权限参数（默认 tenant_readable = 最安全默认值）
    → 记录 document_id

□ Step 2: 写入内容（POST /docx/v1/documents/{id}/blocks/batch_create）
    → 首次写入前，先发 1 个测试 block 验证格式
    → block_type 映射：TEXT=2, H1=3, H2=4, H3=5, T13=13, QuoteContainer=34
    → 逐个检查返回值，不要只看汇总统计

□ Step 3: 浏览器验证（browser_open）
    → 用管理员身份打开文档 URL
    → 确认页面可访问、内容完整、格式正确

□ Step 4: 权限确认
    → 不调用 /permissions/{token}/public（禁止）
    → 不调用 /permissions/{token}/members（禁止）
    → 如果需要更高权限 → 告知用户在飞书 UI 手动操作

□ Step 5: 确认文档不是"孤岛"
    → 文档 URL 能打开 ≠ 用户能在飞书 UI 里找到
    → 如果需要在知识库里 → 告知用户手动移动到目标文件夹

□ Step 6: 记录文档 URL 到 memory
    → 防止后续对话找不到已创建的文档
```

### 阶段 2B：Wiki 写入（3 步，阻塞时立即切换方案）

```
□ Step 1: 测试 wiki 写入权限
    → 发 1 个最小化 block 到 wiki 页面
    → 返回 403 / 131006 / 1770032 → 立即停止，切换方案

□ Step 2: 权限不足时的正确切换路径（按顺序）
    → 方案 A：用 user_access_token（需要 OAuth 流程）
    → 方案 B：用浏览器在 Wiki 里手动创建 + 粘贴内容
    → 方案 C：创建独立文档，用户手动移入 Wiki
    → ❌ 禁止：反复尝试不同 API 路径（最多 2 轮确认是权限问题）

□ Step 3: wiki 可用 block 白名单
    → ✅ TEXT(2), H1(3), H2(4), H3(5), Ordered(13), QuoteContainer(34)
    → ❌ Divider(14), Image(27), Table(未确认), Callout(未确认)
```

## 三、反复试探的根因分析（为什么浪费了 30+ 轮）

### 根因 1：没有「死路归档」意识

飞书 wiki 权限问题踩了 15+ 轮。每一轮的逻辑都是："换个 API 路径试试"——但每条路径的底层障碍是同一个（tenant_access_token 没有 wiki space 权限）。

**铁律：同一个底层障碍，最多探测 2 轮。第 3 轮开始必须切换到完全不同的方案。**

### 根因 2：「API 返回成功 ≠ 用户能看到」

API 返回 HTTP 200 只意味着「服务器接受了请求」，不代表「用户能访问」。每次权限修改、文档创建后，必须用浏览器以用户身份验证。

**铁律：API 操作的最后一步永远是浏览器验证，不是看 API 返回码。**

### 根因 3：不了解飞书权限的三层架构

```
第一层：App scope（开放平台配置）→ 决定 API 能不能调
第二层：文档级权限（owner/成员）→ 决定谁能访问这个文档
第三层：Wiki space 级权限（知识库设置）→ 决定 App 能不能操作这个知识库
```

三层互相独立。scope 齐全 ≠ 能操作文档 ≠ 能操作 wiki。每一层的失败信号不同：
- scope 不足 → 400 / 401
- 文档权限不足 → 1063002
- wiki space 权限不足 → 131006 / 1770032

### 根因 4：open_id 的跨 app 陷阱

飞书的 open_id 是**按 app 维度分配**的。同一个用户，不同 app 看到的 open_id 不同。浏览器登录是一个 app，API 用的是另一个 app。

**铁律：任何跨系统的身份标识（open_id/union_id），必须先验证是否一致，不能假设一致。**

## 四、退出判据汇总（每个坑的「该停手了」信号）

| 场景 | 退出信号 | 切换到 |
|------|---------|--------|
| 权限 API 返回 1063002 | 第 1 次就停 | 用户在 UI 操作 |
| Wiki 写入返回 403/131006 | 第 2 次确认后停 | user_access_token 或浏览器 |
| Block 写入返回 invalid param | 连续 3 个同类 block 失败 | 检查 block_type 映射 |
| 内容提取不完整 | 第 2 次补全仍缺失 | 切换提取方式（API↔浏览器） |
| 文档创建后用户打不开 | 第 1 次就停 | 重建文档（不设权限参数） |
