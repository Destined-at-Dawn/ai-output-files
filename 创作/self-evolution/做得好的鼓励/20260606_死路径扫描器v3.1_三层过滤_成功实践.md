# 成功实践：死路径扫描器 v3.1——三层过滤 + 上下文感知

> 日期：2026-06-06
> 来源：dead-path-scanner.py 从 v2→v3.1 的迭代
> 价值：解决了"扫描器自身不可信"的核心问题

---

## 做了什么

将死路径扫描器从 v2（卡死）→ v3（3175 条误报）→ v3.1（618 条真实断链），核心改进：

### 1. 上下文感知路径解析

相对路径不再只从工作区根目录查找，而是按优先级：
```
源文件目录 → 工作区根目录 → .claude/rules/ → projects/xxx/
```

这解决了 CLAUDE.md 引用 `公众号/xxx` 但文件实际在 `projects/.../公众号/xxx` 的问题。

### 2. 三层误报过滤

```
第一层：格式过滤（花括号、通配符、纯数字）
第二层：上下文过滤（CLI命令、代码注释、Markdown代码块、路由表关键词）
第三层：语义过滤（SKILL.md教学示例、占位符变量、标识符类名）
```

### 3. 分层扫描策略

```
--quick   核心文件（SOP/CLAUDE/rules）< 2秒
默认      扩展（含项目层）
--full    全量（含所有 skill）
```

## 效果

| 指标 | v2 | v3 | v3.1 |
|------|-----|-----|------|
| 死路径数 | 1535 | 3175 | **618** |
| 误报率 | 未知 | 60%+ | **低** |
| 耗时 | 卡死 | 3秒 | 2秒 |
| 修复建议 | 无 | 无 | **231条** |

## 可复用的方法论

1. **扫描器本身也要被扫描**——输出结果必须抽样验证，不能盲目信任
2. **分层过滤**——格式→上下文→语义，每层消灭一类误报
3. **先小样本后全量**——用 3-5 个已知文件验证过滤逻辑，再跑全量
4. **路径归一化**——Windows `\` vs `/` 必须统一，用 `pathlib.Path` 或 `.replace('\\', '/')`

## 适用场景

- 任何"从文本中提取结构化信息"的脚本
- 需要区分"引用"和"示例"的场景
- 跨平台路径比较
