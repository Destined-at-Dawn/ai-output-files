## 三、对话生命周期中的技能调用顺序

```
对话开始
  ├─ Step 0:  runtime-snapshot.md
  ├─ Step 0.5: 技能注册表.md（本文件）← 知道所有技能的存在和职责
  ├─ Step 1:  memory/long-term.md
  ├─ Step 2:  memory/{今天日期}.md
  └─ Step 3:  evolution-calendar.md + lessons.md

对话中（按需求匹配）
  ├─ 内容输入 → input-content-guard 闸门（完整性判断+输出矩阵确认）
  ├─ 路由表匹配 → 调用对应技能
  ├─ 内容创作 → F8 技能链（li-analyze → ... → 自审）
  └─ 技能管理 → li-manage / li-bestskill / li-skillfusion

对话结束前
  ├─ Phase 0: li-improve 自审检查清单
  ├─ F1: post-task-audit 技能审计
  ├─ 检查三: post-task-audit 教训兜底（li-improve 未触发时主动触发）
  └─ 有教训 → Phase 1: li-improve 五步闭环
```

## 四、新增/合并/弃用技能时的维护规则

| 操作 | 必须更新的文件 |
|------|---------------|
| **新增技能** | ① 本文件 §一 添加条目 + §二 添加职责边界 ② CLAUDE.md 路由表 ③ projects/_shared/ 三个文件 |
| **合并技能** | ① 被合并技能移入 §一 "已弃用" ② 吸收方更新版本号和职责 ③ 同步更新所有引用方 |
| **弃用技能** | ① 移入 §一 "已弃用" ② 标注替代方案 ③ 保留原文件（不删），标注 DEPRECATED |
| **职责变更** | ① 更新 §二 对应技能的边界表 ② 同步更新 PK-9 组描述 |

**铁律**：改技能时必须先 grep 建引用图，再逐个更新配置文件——不能只改 skill 本身。

**标准流程**：见 `SOPs/18_技能生态治理SOP.md`（7步闭环：引用图→诊断→决策→执行→同步→注册表→验证）。

## 六、能力需求声明标准（v2.0 新增 · SkVM论文驱动）

> **来源**：SkVM论文（上交大，arXiv:2604.03088）——从1.5万高下载Skill中提炼26种原子能力，每种分L1-L3三级。
> **核心原则**：「Skill不是写了就能用的——必须声明它对模型和环境的假设。」

### 6.1 原子能力分级标准（26选6，最常用）

| 原子能力 | L1（基础） | L2（进阶） | L3（专家） |
|---------|-----------|-----------|-----------|
| **指令遵循** | 执行单步简单指令 | 遵循多步有序指令 | 自主判断步骤优先级和依赖 |
| **代码生成** | 生成简单脚本（<50行） | 生成多文件项目 | 架构设计+测试+文档 |
| **格式输出** | 纯文本/Markdown | JSON/CSV结构化输出 | 自定义Schema+嵌套结构 |
| **工具调用** | 单工具调用 | 多工具组合（≤3个） | 工具链编排+错误恢复 |
| **长上下文** | 处理<8K输入 | 处理<128K输入 | 跨文档关联+增量更新 |
| **推理深度** | 直接回答/信息检索 | 多步推理链 | 反事实推演+自反驳 |

### 6.2 能力需求声明模板

每个分享版 Skill 的 SKILL.md 中必须包含：

```markdown
## 🧠 能力需求

| 原子能力 | 最低等级 | 说明 |
|---------|---------|------|
| 指令遵循 | L2 | 需要精确遵循多步指令 |
| 代码生成 | L1 | 仅需简单脚本 |
| ... | ... | ... |

### 推荐环境
- 模型：Claude 4+ / GPT-5+ / Gemini 3+
- Harness：牛马AI（Newmax）/ Claude Code
- 最低上下文：128K tokens

### 降级策略
- **弱模型（L1能力）**：简化指令→跳过高级功能→提供手动操作步骤
- **无工具调用环境**：工具调用步骤替换为"请用户手动执行以下命令"
- **短上下文环境**：分步执行，每次只传入当前步骤所需上下文

### 已知不兼容
- Qwen系列（L1级以下工具调用）：工具调用部分会失败，请使用手动模式
- 短上下文模型（<32K）：请分批输入内容
```

### 6.3 声明等级要求

| 技能类别 | 最低声明要求 |
|---------|------------|
| 🔴 核心引擎（li-improve等） | 全部6项能力需求 + 降级策略 + 已知不兼容 |
| 🟠 内容生产（li-analyze等） | 至少4项能力需求 + 降级策略 |
| 🟡 内容创作与表达 | 至少3项能力需求 + 降级策略 |
| 🟢 全局管理 | 至少2项能力需求 |
| 🟠 专项任务 | 至少3项能力需求 + 环境依赖清单 |

### 6.4 检查时机

| 时机 | 检查什么 |
|------|---------|
| **Skill创建时**（li-skillcreate Phase 1） | 必须填写能力需求段 |
| **Skill分享前**（SOP-16 GATE -1） | 检查能力需求是否合理 + 降级策略是否完整 |
| **技能注册时**（本文件 §一） | 检查是否包含能力需求段 |
| **每月审计**（li-manage Flow I） | 可移植性评分（第六维） |
