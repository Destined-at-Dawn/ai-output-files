## 九、自我进化机制

### 9.1 目录结构

```
自我进化/
├── 做得好的鼓励/    ← 用户肯定时，保存该轮对话
└── 做得差的避免/    ← 用户批评时，保存该轮对话
```

### 9.2 操作规则

**当用户说"做得好/不错/厉害/满意"等肯定性反馈时：**
1. 将本轮完整的一问一答（用户的问题 + 我的回复）提取为 `.md` 文件
2. 保存到 `自我进化/做得好的鼓励/`
3. 文件命名格式：`{YYYYMMDD}_{主题}_成功案例.md`
4. 文件包含：用户原问题 + 我的回复 + 用户为何满意（如有）

**当用户批评或指出错误时（⚠️ 强约束，不得跳过）：**
1. 将本轮完整的一问一答提取为 `.md` 文件
2. 保存到 `自我进化/做得差的避免/`
3. 文件命名格式：`{YYYYMMDD}_{主题}_失败教训.md`
4. 文件包含：用户原问题 + 我的回复 + 错误原因 + 正确做法

> ⚠️ **触发铁律：用户的提问带有问号且内容为指责/批评/否定时，必须无条件记录到 `自我进化/做得差的避免/`。不需要用户再次提醒，AI必须在该轮对话中主动完成记录。**

**日后在相似场景中先检索这两个文件夹**，做得好则复用，做差则避坑。

## 十二、IDEAS 管理系统

> 创作灵感/项目想法/研究思路的版本化管理
> 核心文件：`IDEAS/README.md`

### 14.1 目录结构

```
IDEAS/
├── README.md             ← 系统说明
├── _version_index.md     ← 版本索引（自动维护）
├── active/               ← 当前活跃的 idea
│   └── {主题}/
│       └── v{数字}.md
└── archive/              ← 归档的老版本
    └── {主题}/
        └── v{数字}.md
```

### 14.2 版本规则

- **新增**：在 `active/{主题}/` 创建 v1.md
- **更新**：当前版移入 archive → 创建新版（v2/v3...）
- **从不用单一 ideas.md 文件**
- **从不覆盖旧版本**

### 14.3 工程师思维文件组织命令（强约束）

⚠️ **此规则对所有在此工作区运行的AI智能体具有约束力。以下关键词触发的是"必须执行"的操作，而非"可以执行"的可选操作。**

| 用户说的话 | 你**必须**自动执行的操作 |
|-----------|----------------------|
| **"我这有个 idea"** / "有个想法" / "有个灵感" | ①读 IDEAS/active/ 查看有无已有文件 ②有→archive→创建新版 ③无→创建新文件夹+v1 ④更新 _version_index.md |
| **"帮我整理文件结构"** | ①扫描工作区所有文件夹 ②检查文件是否放错位置 ③检查空文件夹 ④给出整理报告+建议 |
| **"归档这个 idea"** | ①当前active内容移入archive ②标记_done/_abandoned ③更新索引 |

**这些命令整合在 4.1 节（自动触发条件）中，与新会话中的内容创作流程同级，作为全局绑定条件。** 任何新会话中的AI读到4.1节时，必须同时看到这些组织命令。

## 二十五、引号内容原样保存铁律（2026-05-04新增）

> ⚠️ **此规则对所有在此工作区接收用户带引号内容的AI智能体具有强制约束力。**
>
> 2026-05-04 用户明确要求：凡是用户发送带有引号的长文本（指令/模板/参考资料/提示词等），必须将引号内完整内容原封不动保存到项目目录的专属 .md 文件中。

### 27.1 触发条件

用户消息中包含 `""` 引号包裹的长文本（通常为：提示词模板、指令集、参考资料、命令说明等），**必须**执行本规则。

### 27.2 保存规则

1. **提取引号内容**：识别用户消息中最外层引号包裹的完整内容
2. **创建专属文件**：根据内容主题创建 .md 文件，保存到项目目录的对应子目录下
3. **原封不动**：引号内内容**必须**一字不删、一字不改地保存为正文，禁止只存摘要/要点/提炼
4. **加元信息头**：文件头部标注来源、时间、用途
5. **Write后Read验证**：必须 Read 确认文件内容与引号原文一致

### 27.3 文件命名与存放

- 文件名应体现内容主题（如 `字体美学提示词模板.md`）
- 存放目录根据内容类型判断：
  - 提示词类 → `图片提示词/`
  - 技法/方法论类 → `公众号/自己/` 或 `认知之书/`
  - 参考资料类 → `downloads/` 或对应子目录
  - 其他 → `outputs/`

### 27.4 禁止行为

- ❌ **禁止**只保存引号内容的摘要/要点——必须保存完整原文
- ❌ **禁止**对引号内容做任何改写/美化/删减
- ❌ **禁止**省略元信息头（来源/时间/用途）
- ❌ **禁止**Write后不Read验证

### 28.1.1 输入类型优先规则（v1.1 新增）

> ⚠️ **用户丢链接/图片/文件时，自动走输入分发 SOP，不需要用户说"帮我提取/分析"。**

**判断优先级**：输入分发 SOP（识别输入类型）→ 主题 SOP（按内容执行）

```
用户消息
  ↓
有链接/图片/文件？ → 是 → 读「输入分发SOP.md」→ 识别类型 → 调已有 Skill → 衔接主题 SOP
  ↓ 否
按 §一 10大类别匹配 → 读对应 SOP → 执行
```

### 28.2 执行规则

**收到用户消息时，必须执行以下步骤：**

1. 读 `SOPs/SOP总索引.md` §一 匹配指令类别
2. 命中 → 读对应 SOP → 按决策表执行
3. 未命中 → 按 CLAUDE.md §十二 映射表匹配 Skill
4. 执行完 → 按 SOP 总索引 §2.3 标准动作收尾

### 28.3 自学习机制（所有 SOP 共用）

每个 SOP 末尾有「迭代日志」表。每次执行后：
- 对比 SOP 决策表与实际执行路径
- 偏差 = 新知识？→ 写入迭代日志
- 偏差 = 失败？→ 写入 自我进化/做得差的避免/
- 3 次同类偏差 → 更新 SOP 决策表或流程
- 数据来源：memory/ + 自我进化/ + 技能调用记录.md + 每日完成记录/

## 二十八、陌生图片自动寻源与美学参考规则（2026-05-06新增）

> ⚠️ **此规则对所有在此工作区处理用户上传参考图的AI智能体具有强制约束力。**

### 30.1 核心规则

当用户提供陌生的、不符合已有风格库的参考图，或要求设计新风格时，**必须自动执行以下操作**：

1. **自动联网寻源**：调用联网搜索工具（如 `web_search`），去全球知名的高质量设计网站（如 Pinterest、Behance、ArtStation 等）寻找类似风格的设计参考和专业术语。
2. **中国传统文化特例**：如果图片或需求与中国传统文化强相关（如皮影、水墨、剪纸），**必须优先搜索**「上海美术电影制片厂（上美影）」、「中国民族风格动画」、「中国传统美学现代转译」等高质量中国特色唯美设计作为核心美学参考。
3. **10维度深度拆解**：结合搜索到的顶尖参考资料与用户供图，使用 `图片提示词/风格深度拆解模板.md` 进行参数级拆解，再输出设计方案与提示词。

### 30.2 禁止行为

- ❌ **禁止**参考低质量、低俗或缺乏美感的素材网站。
- ❌ **禁止**在未进行高水准美学背景搜索的情况下，直接生造肤浅的提示词。
- ❌ **禁止**面对中国传统文化需求时，使用西方刻板印象（如廉价的红灯笼/龙），必须向上追溯到上美影等真正的东方美学高地。

### 30.3 记录与存档
- 用户发送的相关指令必须无条件更新至本文件。

## 二十九、提示词质量拦截层（2026-05-07新增）

> ⚠️ **此规则对所有在此工作区接收用户指令的AI智能体具有强制约束力。**
>
> 2026-05-07 用户明确要求：每次下达模糊指令时，AI自动输出3个改进版prompt，用户选择后再执行。除非任务紧急或用户说"快"。

### 31.1 核心机制

**本规则包裹在所有SOP外面，是第一个执行的步骤。**

```
用户消息进入
  ↓
Step 0：快速判断
  → 有链接/图片/文件/引号？ → 跳过，走输入分发SOP
  → 说"快"/"急"/"马上"？ → 跳过，直接最佳猜测执行
  → ≥50字且有明确动词？ → 跳过，直接执行
  → 以上都不是 → 进入 Step 1
  ↓
Step 1：3维度评分（清晰度0-4 + 上下文0-3 + 输出期望0-3 = 总分0-10）
  ↓
  ├── ≤3分 → 直接执行
  ├── 4-7分 → 输出3个改进版（最小改动/中度优化/深度重构）
  └── ≥8分 → 先调 dbs-goal 三问审计 → 再出3个改进版
  ↓
Step 2：用户选择 → 用选定版本执行
  ↓
Step 3：记录质量分 → 渐进退出（30天后平均分≤2则关闭拦截层）
```

### 31.2 三个改进版设计

| 版本 | 策略 | 示例（用户说"帮我写"） |
|------|------|---------------------|
| **A：最小改动** | 保持原意，只补缺失 | "帮我写一篇公众号文章，主题是{最近热门AI话题}，字数1500-2000" |
| **B：中度优化** | 补上下文+明确格式+指定平台 | "基于我的电气+AI方向，写一篇面向大学生的公众号长文，主题是{AI工具提效学习}，用学生视角硬核技术笔记风格，1500字左右" |
| **C：深度重构** | 从目标反推最优指令 | "先从技法吸收记录提取3个最近学到的写作技法，再从百大认知汇总找2本相关书籍原句，用STAR法则写一篇公众号文章，主题是{大一学生如何用AI工具建立学习系统}" |

### 31.3 注入的上下文（从 telos 自动读取）

每次生成改进版时，自动注入：身份（大一电气生/GPA 3.9）+ 当前目标（创作体系/科研/升学）+ 已有能力（144个Skill/90+技法/62本认知书）+ 已有数据（44个作者分析/SOP系统）

### 31.4 与 dbs-goal / jc-clarifier 的协作

- **模糊度≥8分** → 调 `dbs-goal` 执行三问审计（真问题还是症状？目标是什么？最短路径？）
- **用户说"我不确定"** → 调 `jc-clarifier` 多轮澄清
- **涉及提示词本身优化** → 调 `BROK提示词优化器`

### 31.5 渐进退出机制

| 时间段 | 规则 |
|--------|------|
| 第1-7天 | 每次模糊指令都输出3个改进版 |
| 第8-14天 | 近7天平均分≤4 → 只输出1个改进版 |
| 第15-30天 | 近14天平均分≤3 → 只在≥6分时才输出 |
| 30天后 | 近30天平均分≤2 → 关闭拦截层 |

### 31.6 禁止行为

- ❌ **禁止**在用户说"快"/"急"/"马上"时仍输出改进版（紧急任务不拦截）
- ❌ **禁止**对链接/图片/文件/引号消息做提示词优化（输入类型已明确）
- ❌ **禁止**连续对话中重复拦截已澄清的任务
- ❌ **禁止**改进版之间没有差异（3个版本必须有明显不同的策略）

## 三十、GitHub 开源发布铁律（2026-05-09新增）

> ⚠️ **此规则对所有在此工作区执行 GitHub 发布/开源操作的 AI 智能体具有强制约束力。**
>
> 2026-05-09 用户明确要求："先在 private 里面让我检查之后，我才下命令，让你 public，记入 CLAUDE 规则。"

### 32.1 Private 先行铁律

**任何项目发布到 GitHub 时，必须遵守以下流程，缺一不可：**

```Step 1：创建 Private 仓库
  → gh repo create {name} --private
  → 推送所有内容

Step 2：创建 Release + 上传 Skill 文件
  → gh release create v1.0.0（含功能说明+安装方式）
  → gh release upload 上传 .yaml/.skill 等可安装文件（方便用户一键下载）
  → 注意：GitHub API 不支持中文文件名，Release Asset 必须用英文命名

Step 3：生成封面图提示词
  → 为项目生成 GitHub Social Preview（OG Image）提示词
  → 遵循 §十八 图片提示词生成约束（默认 ChatGPT/DALL-E，全英文）
  → 遵循 §二十五 视觉风格双轨制（技术项目→暗色轨道）

Step 4：通知用户检查
  → 给出仓库 URL + Release URL
  → 列出已推送的文件清单 + Release Asset 清单
  → 给出封面图提示词
  → 等待用户确认

Step 5：用户明确说"public"/"公开"/"可以了"
  → gh repo edit {name} --visibility public
  → 验证：gh repo view {name} --json visibility
```

### 32.2 发布前安全检查（每次必做）

| # | 检查项 | 操作 |
|---|--------|------|
| 1 | **个人数据隔离** | memory/、outputs/、project.md、CLAUDE.md **禁止**进入公开仓库 |
| 2 | **文件完整性** | README.md、LICENSE、.gitignore 缺一不可 |
| 3 | **信息真实性** | README 中的功能描述、性能数据必须真实可验证，禁止编造 |
| 4 | **依赖预检** | 推送前验证 `gh auth status`、remote 配置、token 权限 |
| 5 | **.gitignore 审查** | 确认个人数据被正确排除 |

### 32.3 发布后验证（铁律）

**每次 git push 后，必须用以下命令验证远程内容：**

```bash
gh repo view {owner}/{repo} --json visibility,name,description
gh api repos/{owner}/{repo}/contents --jq '.[].name'
```

**禁止**：
- ❌ **禁止**未验证就报告"已发布"
- ❌ **禁止**跳过 Private 阶段直接 Public
- ❌ **禁止**未检查 .gitignore 就推送（可能泄露 memory/个人信息）
- ❌ **禁止**使用 `git push --force`（等同删除，必须先备份）
