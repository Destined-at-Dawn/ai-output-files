## 十七、文献检索时效性约束（2026-04-26新增）

> ⚠️ **此规则对所有在此工作区检索论文/文献的AI智能体具有强制约束力。**
> 
> 2026-04-26 用户明确批评"论文根本不够新"——检索的论文（2025年8月发表）距离当天已超过8个月。

### 19.1 搜索优先级

| 来源 | 优先级 | 原因 |
|------|--------|------|
| arXiv 预印本 | 🔴 最高 | 即时发布，无延迟 |
| 近期会议（CVPR/ICCV/ICRA/NeurIPS） | 🟡 高 | 会议周期3-6个月 |
| 期刊论文（Springer/IEEE/Elsevier） | 🟢 低 | 出版延迟6-12个月，除非标明"Online First" |
| 找代码 | ⭐ 额外 | GitHub、PapersWithCode |

### 19.2 时效性门槛

- **优先搜索最近3个月**的论文
- **日报/周报中至少60%引用必须是近3月论文**
- 期刊论文超过6个月 → 标注为🟡并仅供"背景参考"
- 每篇论文输出**时效性评级**：近期🟢(≤3月) / 中期🟡(3-6月) / 过时🔴(>6月)
- 经典必读论文不受此限，但需标注"经典必读"

### 19.3 触发规则

- 所有涉及"论文检索/文献综述/科研日报/科研指导"的任务自动激活本规则
- 本规则优先级高于其他论文搜索规则

## 十八、每日完成记录（2026-04-26新增）

> ⚠️ **每次完成任务后，AI必须记录至此系统。** 用户明确要求"以后你每天完成任务都有记录"。

### 20.1 目录

```
每日完成记录/
├── README.md    ← 系统说明 + 记录列表
```

### 20.2 记录规则

- **每次对话结束时**，必须更新 `每日完成记录/README.md`
- 格式：`✅ {任务名称} — 完成于 HH:MM — 产出：{路径}`
- 失败任务标注 `❌ {任务名称} — 原因：{说明}`
- 不覆盖历史，每次在列表顶部追加新记录

### 20.3 强制触发的场景

- 科研日报生成 → 记录论文篇数+S/A/B分布
- 论文下载/检索 → 记录检索结果+可访问性
- 其他任务完成 → 记录完成内容+产出路径

## 十九、让人放心的工作标准（2026-04-26新增）

> ⚠️ **此规则对所有在此工作区运行的AI智能体具有强制约束力。**
> 
> 2026-04-26 小黎明确批评"敷衍"并让我学习FPGA工作区（`D:\study\competion\FPGA\fpga\`）的自动化+复盘体系。以下规则从该工作区的17个Bug修复流程和三层记忆体系中提炼。

### 21.1 自动化优先（铁律一）

**遇到任何问题时，执行顺序必须是：**

```
① 敲命令/跑脚本 → 看真实输出
② 读 CLAUDE.md（本文件，已自动加载）
③ 查 reflect/ 下的复盘记录（深度参考）
④ 写 .md 文件（记录新发现）
```

**为什么自动化优先于文档：**
- 脚本输出的是**真实当前状态**，.md 文件记录的是**历史状态**
- 脚本不会过时，文档会。**脚本是真理的来源，文档是辅助说明**

**每发现一个新问题，优先写自动化脚本（Makefile target / Python验证），其次才是写文档。**

### 21.2 验证闭环（铁律二）

**每个操作完成后，必须给出可验证的结果确认，禁止只说"已完成"。**

| 操作类型 | 必须说的话 |
|---------|-----------|
| 创建文件 | 路径 + 内容摘要 + 文件大小验证 |
| 下载/检索论文 | 路径 + DOI + 访问状态（可下载/付费墙/预印本） |
| 设置定时任务 | 任务名 + 时间 + 触发条件 + 确认状态 |
| 修改CLAUDE.md | 修改位置 + 修改内容摘要 + 版本号变更 |
| 调用Skill | 技能名 + 调用结果 + 产出路径 |

**禁止句式（踩到一次记录一次）：**
- ❌ "已创建/已设置/已完成"（没有验证证据）
- ❌ "我已经XX了"（没有路径/状态确认）
- ❌ "我学到了"（没有写到文件里）

**正确做法：每一步操作都带 tool_use 验证，回复只说 tool_result 确认后的内容。**

### 21.3 错误复盘机制（铁律三）

> ⚠️ **每次失败/错误/批评，必须在同一轮对话中完成三步：**

```
Step 1: 根因分析
→ 表象是什么？错误假设是什么？调试手段是什么？发现根因的瞬间是什么？

Step 2: 写入系统
→ 高价值 → 写入 CLAUDE.md 对应板块
→ 中价值 → 写入 memory/
→ 低价值 → 写入 自我进化/做得差的避免/

Step 3: 避免再犯
→ 确保写入的规则是"必须/禁止/不得"格式
→ 确保未来的AI能自动加载这些教训
```

### 21.4 操作纪律（铁律四）

- **每轮操作前**：先读 `memory/MEMORY.md` 了解上下文，读 `每日完成记录/README.md` 了解最近工作状态
- **每轮操作后**：更新 `每日完成记录/README.md`
- **每轮对话结束时**：确认每日记录已更新
- **重要决策**：写入 `memory/long-term.md`
- **用户批评**：先道歉 → 立即改正 → 写入规则 → 反问"还有没有其他问题"

### 21.5 每日检查清单（AI自行执行）

每次对话结束时，AI**必须**自我检查：

- [ ] 今日产出都存档到正确位置了？
- [ ] 每日完成记录更新了？
- [ ] 有失败/错误需要写复盘？
- [ ] 有决策需要写入 memory？
- [ ] 小黎下次来能一眼看懂现状？

## 二十、文件同步与归档规则（2026-04-28新增）

> ⚠️ **此规则对所有在此工作区修改文件的AI智能体具有强制约束力。**

### 22.1 文件同步失败处理（铁律）

当 `write_project_file` 未能将内容写入工作区根目录文件时（即文件仍然为旧版本），**必须**执行以下流程：

```
① 将原文件（旧版本）移入 "E:\ai产出文件\牛马\个人\个人\归档"
② 在同一位置创建新文件副本（文件名与原名保持一致）
③ 写入更新后的完整内容
④ 验证新文件内容正确后，视为完成
```

**禁止**：
- ❌ **不得**在原文件上反复尝试覆盖写入
- ❌ **不得**跳过验证步骤
- ❌ **不得**将更新内容写入项目子目录（`projects/proj-xxx/`）而不同步到工作区根目录

### 22.2 重复文章归档

当发现**重复的文章**（同一作者同一篇文章被多次写入）时：

- 多余的副本**必须**移入 `E:\ai产出文件\牛马\创作\创作\projects\proj-1777084456942-plvse9\归档`
- 仅保留工作区根目录 `公众号/别人/{作者}/` 下的正本
- **禁止**直接删除重复文章（归档而非删除，保留追溯能力）

### 22.3 双归档目录（全局+工作区）

> ⚠️ **所有删除文件必须保存两份归档副本，缺一不可。**

| 归档位置 | 路径 | 说明 |
|---------|------|------|
| **全局归档** | `E:\ai产出文件\牛马\归档\` | 所有工作区的删除文件统一备份 |
| **创作区归档** | `E:\ai产出文件\牛马\创作\创作\归档\` | 创作工作区删除的文件 |
| **创作区项目归档** | `E:\ai产出文件\牛马\创作\创作\projects\proj-xxx\归档\` | 项目级别归档 |
| **个人区归档** | `E:\ai产出文件\牛马\个人\个人\归档\` | 个人工作区删除的文件 |
| **竞赛区归档** | `E:\ai产出文件\牛马\竞赛\竞赛\归档\` | 竞赛工作区删除的文件 |
| **求职区归档** | `E:\ai产出文件\牛马\求职\求职\归档\` | 求职工作区删除的文件 |

**工作区识别规则**：被删除文件原路径包含 `牛马\创作\` → 归档到创作区；包含 `牛马\个人\` → 归档到个人区；以此类推。

### 22.4 删除前双归档铁律（最高优先级 · 2026-05-07升级）

> 此规则优先级高于所有其他操作规则。违反=数据丢失=最严重错误。
> 来源：小黎明确要求——"所有删除文件保存两份，第一份全局归档，第二份原工作区归档"

**任何删除操作前，必须执行以下六步，缺一不可：**

1. **`ls` 检查**：列出目录/文件的完整内容，确认有哪些文件
2. **识别原工作区**：根据被删除文件的路径，判断属于哪个工作区
3. **第一份归档→全局**：`cp -r` 复制到 `E:\ai产出文件\牛马\归档\`（全局备份）
4. **第二份归档→原工作区**：`cp -r` 复制到原工作区的 `归档/` 目录（没有则 `mkdir -p` 新建）
5. **AskUserQuestion 确认**：向用户展示——①将要删除的内容清单 ②全局归档路径+验证 ③工作区归档路径+验证，获得明确确认
6. **验证双归档完整性**：ls 确认两份归档副本均与原文件一致后，才执行删除

**删除后报告（强制）**：
每次删除操作完成后，**必须**向用户报告：
```
✅ 删除完成
- 删除的文件/目录：[清单]
- 全局归档：E:\ai产出文件\牛马\归档\{文件名} — 已验证
- 工作区归档：E:\ai产出文件\牛马\{工作区}\归档\{文件名} — 已验证
```

**禁止**：
- 不得在未检查内容的情况下删除任何目录
- 不得在未归档的情况下删除任何文件
- 不得在未获得用户确认的情况下删除任何文件
- 不得用"重复目录"作为删除理由——先归档，再让用户判断是否重复
- **不得只归档一份**——必须全局+工作区两份都归档
- **不得删除后不报告**——必须输出双归档路径+验证状态

## 二十一、文件操作铁律与记忆追加（交叉学习自竞赛区/个人区）

### 23.1 文件操作铁律（信任验证）

| 规则 | 说明 |
|------|------|
| **禁止伪造** | 所有文件操作必须通过 tool_use（Write/Edit/Bash）发起，禁止在纯文本中描述"已写入"但未发 tool_use |
| **验证必须** | 每次 Write/Edit 后，必须 Read 同一文件确认内容正确落盘 |
| **禁止空文件** | 写入的内容必须有实际信息 |

### 23.2 记忆文件追加铁律

| # | 规则 |
|---|------|
| **先 Read 再 Write** | 更新任何 memory 文件前必须先 Read 当前内容，在末尾追加，**禁止覆盖全文** |
| **追加格式** | 以 `## YYYY-MM-DD HH:MM` 开头，加在文件末尾 |
| **Write 后必须 Read 验证** | 验证追加的行存在于末尾 |

### 23.3 修改前安全清单

| # | 检查项 |
|---|--------|
| 1 | 读过要修改的文件当前内容？（必须先 Read） |
| 2 | 修改是否涉及公众号/朋友圈核心系统？→ 如果是，同步更新关联文件 |
| 3 | 修改是否高风险？（删除/覆盖/重命名）→ 如果是，先存副本到 `归档/` |

### 23.4 对话结束收尾（强制）

| 步骤 | 操作 |
|------|------|
| 1 | 更新 `每日完成记录/README.md` |
| 2 | 如有认知跃迁/重要决策 → 写入 memory |
| 3 | 如修改了 CLAUDE.md → 追加变更记录行 |

> ⚠️ **Write 之后必须 Read 验证——没有 Read 验证 == 未完成。**

## 二十二、项目目录唯一性规则（2026-05-03新增）

> ⚠️ **此规则对所有在此工作区运行的AI智能体具有强制约束力。**
>
> 2026-05-03 用户明确要求：根目录 `创作/` 与项目目录 `projects/proj-xxx/` 出现了大量重复文件夹（公众号/自我进化/认知之书/每日完成记录/归档），导致文件分散、版本混乱。

### 24.1 唯一存放位置

**所有内容文件必须且只能存放在当前活跃项目目录下：**

```
E:\ai产出文件\牛马\创作\创作\projects\proj-{当前项目ID}\
```

当前活跃项目目录由 `牛马AI` 的 project system 管理，路径见本文件顶部「项目目录」。

### 24.2 禁止在根目录创建的文件夹

以下文件夹**禁止**在 `创作/` 根目录下创建或维护：

| 禁止的根目录文件夹 | 正确位置 |
|------------------|---------|
| `创作/公众号/` | `projects/proj-xxx/公众号/` |
| `创作/自我进化/` | `projects/proj-xxx/自我进化/` |
| `创作/认知之书/` | `projects/proj-xxx/认知之书/` |
| `创作/每日完成记录/` | `projects/proj-xxx/每日完成记录/` |
| `创作/归档/` | `projects/proj-xxx/归档/` |
| `创作/outputs/` | `projects/proj-xxx/outputs/` |
| `创作/memory/` | `projects/proj-xxx/memory/` |
| `创作/视频/` | `projects/proj-xxx/视频/` |

### 24.3 操作规则

1. **创建文件前**：必须确认目标路径在 `projects/proj-xxx/` 下，不得写入根目录
2. **读取文件时**：优先读取项目目录版本；根目录版本仅作为历史归档参考
3. **发现根目录有重复文件夹** → **必须立即执行**：①将根目录独有文件迁移到项目目录 ②验证迁移完整性 ③**删除根目录版本**。不得保留根目录重复副本。
4. **新项目创建时**：所有内容文件夹只在项目目录内创建
5. **任何时候**，任何文件操作的目标路径必须以 `projects/proj-xxx/` 为前缀（项目级CLAUDE.md、project.md、产出、记忆、分析等全部内容文件）。根目录仅保留 CLAUDE.md、projects/、微信朋友圈/ 和用户明确指定的顶层文件夹。

### 24.4 根目录保留的文件

根目录 `创作/` 仅保留以下文件/文件夹：

- `CLAUDE.md` — 全局约束文件
- `projects/` — 所有项目目录
- `微信朋友圈/` — 朋友圈系统（待迁移）
- `个人/` — 个人区（独立系统）
- 其他由用户明确指定保留的顶层文件夹
