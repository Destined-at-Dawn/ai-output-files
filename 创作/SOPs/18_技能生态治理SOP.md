# 18_技能生态治理SOP

> **工作区：** 创作区（跨工作区通用）
> **版本：** v3.0 | **创建日期：** 2026-06-07 | **最后更新：** 2026-06-15
> **触发条件：** 新增/合并/弃用/重构任何skill时；发现skill功能重叠时；技能注册表需要更新时

## 二、铁律（违反任何一条 = 制造失忆）

| 铁律 | 说明 |
|------|------|
| **改技能前先建引用图** | 用 Grep 搜索所有引用该skill的配置文件，列出完整清单再动手 |
| **改技能后必须同步所有引用方** | 每个引用方的描述/版本号/职责边界必须和skill定义一致 |
| **注册表是单一真相源** | `技能注册表.md` 是所有技能的存在、版本、职责边界的唯一权威 |
| **弃用不删除** | 弃用skill = 标注 DEPRECATED + 保留原文件 + 写迁移指南，永不物理删除 |
| **每次治理必须验证** | 改完后 Grep 确认无残留的旧引用，Read 验证关键文件内容正确 |

## 三、执行流程（7步闭环）

### Phase 1：构建引用图（动手前必做）

**操作**：用 Grep 搜索所有引用该skill名称的文件

```bash
# 搜索skill名称在所有配置文件中的引用
Grep pattern="{skill名}" path="E:\ai产出文件\牛马\创作\创作" glob="*.md"
Grep pattern="{skill名}" path="C:\Users\13975\.newmax\skills" glob="*.md"
```

**输出**：引用方清单（文件路径 + 行号 + 引用内容）

**铁律**：没有引用图就不动手。今天的经验教训——手动grep了30+个文件才发现8个需要更新。

### Phase 2：诊断重叠（如有多个skill做同一件事）

**操作**：逐个读skill定义，填写重叠分析矩阵

| 功能维度 | Skill A | Skill B | Skill C | ... |
|---------|---------|---------|---------|-----|
| 教训记录 | ✅/❌ | ✅/❌ | ✅/❌ | |
| 技能审计 | | | | |
| 双写验证 | | | | |
| ... | | | | |

**决策标准**：
- 两个skill同一功能维度都是 ✅ → 必须合并或明确边界
- 一个skill覆盖了另一个skill的80%功能 → 后者弃用
- 每个功能维度只有一个skill是 ✅ → 分工清晰，无需合并

### Phase 3：整合决策

**四种处置方式**：

| 处置 | 适用场景 | 操作 |
|------|---------|------|
| **合并** | A吸收B的功能 | A升级版本 + B标注DEPRECATED + 迁移指南 |
| **瘦身** | A太大（>500行） | Progressive Disclosure：主文件精简 + references/ 按需加载 |
| **弃用** | 功能已被其他skill完全覆盖 | 标注DEPRECATED + 保留原文件 + 写迁移指南 |
| **新建** | 现有skill都不覆盖 | 按li-skillcreate流程创建 + 注册到技能注册表 |

### Phase 4：执行变更

**操作**：按Phase 3的决策，修改skill定义文件

**铁律**：每次修改skill时，version号必须递增。版本规则见li-skillfusion §四。

### Phase 5：配置同步（🔴 最容易遗漏的一步）

**操作**：按Phase 1的引用图，逐个更新所有引用方

**自动化工具**：`python scripts/skill-version-sync.py {skill名} --fix`
- 从 SKILL.md 读取权威版本号
- 自动更新所有配置文件中的版本引用
- 跳过历史文件（memory/教训/归档）
- 输出修复报告

**手动补充**（脚本不覆盖的部分）：

| 文件类型 | 更新什么 | 脚本覆盖？ |
|---------|---------|-----------|
| CLAUDE.md 路由表 | 版本号 | ✅ 自动 |
| CLAUDE.md 关键文件引用表 | 版本号 | ✅ 自动 |
| 技能注册表.md | 版本号 | ✅ 自动 |
| CLAUDE-子模块.md | 版本号 | ✅ 自动 |
| 技能PK机制.md | 版本号 | ✅ 自动 |
| Skill路由系统.md | 版本号 | ✅ 自动 |
| Skill链式调用路径.md | 版本号 | ✅ 自动 |
| SOP总索引.md | 版本号 | ✅ 自动 |
| 经验库/INDEX.md | 版本号 | ✅ 自动 |
| 职责边界描述 | 新增/修改/删除能力 | ❌ 手动 |
| 替代关系 | 弃用标注+迁移指南 | ❌ 手动 |
| 触发词 | 新增/修改关键词 | ❌ 手动 |

**铁律**：引用图中的每个文件都必须更新，没有"这次不用改"的例外。脚本能覆盖版本号同步，但职责边界和触发词变更必须手动更新。

### Phase 6：注册表更新

**操作**：更新 `技能注册表.md`（根目录）

**必须更新的章节**：
- §一 技能全景：新增/修改/删除条目
- §二 职责边界：更新对应skill的边界表
- §四 维护规则：如有新的维护要求
- §五 版本记录：追加本次变更记录

### Phase 7：验证（🔴 改完必做）

**操作**：
1. Grep 确认无残留的旧skill名引用（如experience-sync在配置文件中不应出现）
2. Read 验证关键文件内容正确（技能注册表、CLAUDE.md路由表）
3. Bash 检查文件行数（skill主文件≤500行，超出则拆references/）

**输出**：验证报告（引用方全部更新完毕 / 无残留引用 / 行数合规）

## 七、认知科学支撑

| 认知机制 | 来源 | 在技能生态治理中的应用 |
|---------|------|----------------------|
| **外部化认知** | 005-深度工作 | 技能注册表=外部化记忆。AI的工作记忆有限（7±2），不可能记住所有skill的存在和职责。注册表把"内部知识"变成"外部文件"，降低认知负荷 |
| **确认偏误** | 010-思考快与慢 | "改了skill就完事"=确认偏误（只确认"我改了"，不验证"改全了"）。Phase 7强制Grep验证=对抗确认偏误 |

| **检查清单效应** | 023-清单革命 | §四的检查清单=防止"改了A忘了B"。飞行员起飞前不靠记忆，靠清单——技能治理同理 |

## 八、skills-refiner 2.0 治理框架参考（v2.0 新增）

> **来源**：skills-refiner 2.0（TecDeTec，GitHub yknothing/skills-refiner）
> **核心原则**：AI judges, scripts collect——脚本收集可验证事实，AI解释，人做决定。

### 8.1 四件套架构（可迁移的设计模式）

| 能力 | 对应现有工具 | 可借鉴的升级点 |
|------|-------------|---------------|
| **设计审计**（skills-refiner） | li-manage Flow I 技能健康诊断 | 增加定位/边界/上下文工程/可移植性维度 |
| **可读解析**（skills-appreciation） | li-bestskill 推荐报告 | 增加"为什么这样设计"的教学型输出 |
| **安装态扫描**（skill-hygiene） | 技能注册表 + dual-write-verify.py | 增加内容哈希、同名冲突检测、失效引用扫描 |
| **canary追踪**（skill-debug） | post-task-audit 检查二 | 增加canary注入机制验证skill是否被执行 |

### 8.2 拓扑感知（防止同名冲突）

**问题**：当前技能注册表只按skill名索引，无法区分：
- 同名但不同来源的skill（已分叉）
- 被修改过但版本号没变的skill（隐式漂移）

**解决**：在 Phase 7 验证中增加内容哈希检查：
```bash
# 计算skill文件哈希，检测隐式修改
python -c "import hashlib; h=hashlib.md5(open('SKILL.md','rb').read()).hexdigest()[:8]; print(h)"
```

### 8.3 过期告警（180天规则）

- 超过180天未更新的skill → 标记为 `[需审查]`（不是"过期"）
- 可配置阈值：核心引擎类90天，专项任务类365天
- **关键**：只报告事实，不自动清理

### 8.4 Canary 追踪机制（验证skill是否被执行）

**原理**：在SKILL.md中注入一条轻量bash，如果AI执行了skill就会触发。
```bash
echo '{"skill":"skill-name","ts":"'$(date -Iseconds)'"}' >> ~/.newmax/activation.jsonl
```

**边界声明**：canary只能证明"bash被执行"，不能证明"skill被发现/加载/遵守"。这是代理证据，不是直接证据。

## 十、Skill 可移植性治理（v3.0 新增 · SkVM 论文驱动）

> **来源**：SkVM论文（上交大，arXiv:2604.03088）——11万Skills分析：15%任务加Skill后性能下降，87%至少在一个模型上无提升。三大错配（模型/框架/环境）是Skill生态的根本瓶颈。
> **核心哲学**：「把Skill当作程序，把大模型当作处理器。」——Skill需要编译才能在不同模型和Harness上可靠运行。

### 10.1 可移植性三原则

| 原则 | 说明 | 检测方式 |
|------|------|---------|
| **显式声明能力假设** | 每个Skill必须声明它对模型能力的假设（不要隐含假设"所有模型都能做到"） | 技能注册表检查：是否有「能力需求」字段？ |
| **自带降级策略** | 每个分享版Skill必须有降级策略——弱模型/缺环境时怎么fallback | SKILL.md检查：是否有「降级策略」段？ |
| **环境依赖前置检查** | Skill的环境依赖必须在安装/分享时先行检查，不等到执行时才发现缺失 | SKILL.md检查：是否有环境依赖清单？ |

### 10.2 可移植性评分（加入技能健康诊断）

在 li-manage Flow I 技能健康度五维诊断中新增**第六维：可移植性**（满分10分）：

| 分数 | 标准 | 说明 |
|------|------|------|
| 0-3 | ❌ 不可移植 | 隐含对特定模型/Harness的假设，无降级策略，无环境依赖声明 |
| 4-6 | ⚠️ 部分可移植 | 有基本的能力需求说明，但降级策略不完整 |
| 7-8 | ✅ 良好可移植 | 有完整能力需求声明+降级策略+环境依赖清单 |
| 9-10 | ⭐ 工业化可移植 | 具备多模型变体（AOT编译产物）+ 自适应降级（JIT）+ 代码固化 |

### 10.3 可移植性审计清单（每次分享Skill前执行）

```
☐ 能力假设：这个Skill默认模型具备什么能力？
  ☐ 已声明最低模型等级（Claude 4+/GPT-5+/Gemini 3+等）
  ☐ 已在技能注册表中标注

☐ 降级策略：如果模型能力不足，Skill怎么fallback？
  ☐ 提供了简化版指令（跳过高级功能）
  ☐ 提供了手动替代方案（不依赖Skill的操作指南）
  ☐ 降级策略写进了SKILL.md

☐ 环境依赖：这个Skill需要什么外部环境？
  ☐ Python包列表（含版本号）
  ☐ CLI工具列表（含安装方式）
  ☐ 系统配置要求
  ☐ 依赖检查脚本（安装时自动运行）

☐ 已知不兼容：
  ☐ 已知不兼容的模型列表（如Qwen系列L1级以下）
  ☐ 已知不兼容的Harness列表
  ☐ 不兼容表现描述（会怎么死）
```

### 10.4 「三大错配」防护规则

| 错配类型 | 根因 | 防护规则 |
|---------|------|---------|
| **模型错配** | Skill为强模型设计，弱模型无法执行复杂指令 | **规则10.4.1**：每个Skill必须标注目标模型等级。如果分享给未知模型的用户，必须附带"弱模型简化版" |
| **框架错配** | 不同Agent Harness的系统提示词深度、工具链不同 | **规则10.4.2**：Skill中涉及工具调用的步骤必须提供"手动替代方案"（不依赖特定Harness的工具） |
| **环境错配** | Skill依赖特定Python包/CLI工具/系统配置 | **规则10.4.3**：Skill必须在SKILL.md头部声明环境依赖。分享时附带依赖检查脚本 |

### 10.5 代码固化——从"LLM推理"到"原生调用"

> SkVM数据：代码固化可将PDF提取从10秒降到200毫秒（50倍加速）。75%的Skills内部嵌有固定结构的代码片段。

**对小黎生态的应用**：

| 高频脚本 | 当前方式 | 固化后 | 预估收益 |
|---------|---------|--------|---------|
| dual-write-verify.py | 每次LLM调用Python | 独立CLI命令 | 零Token+零延迟 |
| 中文标点检查 | LLM自审（不可靠） | post-generation hook | 100%准确+零Token |
| fetch-wechat-article.py | 每次LLM调用 | 独立CLI命令 | 品读流程自动化 |
| privacy-scan.sh | git push前手动 | 固化为git hook | 零遗漏 |

**操作**：将以上脚本从"LLM推理路径"中移除，改为独立CLI命令或Git Hook，在Skill中通过「如果检测到脚本存在→直接调用；否则→LLM生成」的方式调用。
