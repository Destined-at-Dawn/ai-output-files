# SOP-20：工具开发与迭代 SOP

> **编号**：20
> **版本**：v1.0
> **创建日期**：2026-06-07
> **来源**：死路径扫描器 v1→v2→v3 演进（8个坑，3轮重写）+ 文件迁移踩坑 + 经验库更新遗漏
> **核心洞察**：工具/系统开发天然分三阶段，**每个阶段都有"假完成"陷阱**

---

## §〇 核心法则

> 「做了一次就宣称完成，是最危险的完成。」——死路径扫描器的教训：v1 卡死、v2 误报 60%、v3 才能用

**v1→v2→v3 不是失败，是自然规律。** 问题在于：
- v1 就宣称完成 → 用户第一次用就踩坑
- v2 修完一个 bug 就报喜 → 其实还有 5 个同类 bug
- v3 没跑全量验证就宣布成功 → 误报率依然 60%

---

## §一 三阶段演进模型

### 阶段定义

| 阶段 | 目标 | 验收标准 | 典型时长 |
|------|------|---------|---------|
| **v1：最小可用** | 核心逻辑跑通 | 能处理 1 个真实 case | 30% 总工时 |
| **v2：问题暴露** | 在真实数据上跑，暴露所有问题 | 有完整的误报/漏报清单 | 40% 总工时 |
| **v3：精准可用** | 逐类消灭误报，达到可用标准 | 误报率 < 20%，耗时 < 预期 | 30% 总工时 |

### 关键认知

**v1 的陷阱**：能跑 ≠ 能用。
- 死路径扫描器 v1：正则 `\[` 转义错误，脚本根本启动不了
- 防错：v1 完成后必须跑 1 个真实 case，不跑 = v1 未完成

**v2 的陷阱**：修了一个 bug ≠ 问题解决了。
- 扫描器 v2.0→v3.0：修了 Windows 路径分隔符，结果 SKILL.md 教学示例又产生 1473 条误报
- 防错：每修一类 bug，必须跑全量验证，**报总数不报降幅**

**v3 的陷阱**：自己觉得好了 ≠ 真的好了。
- 扫描器 v3.0→v3.1：路由表中文关键词、模板占位符、CLI 命令全被当路径
- 防错：v3 验收必须**人工抽查 10 条**，不能只看数字

## §二 执行流程（6 步）

### Step 1：规模评估（动手前必做）

```
问自己：
1. 输入规模有多大？（文件数、数据量）
2. 有没有"炸弹目录"？（如 7003 个 skill 文件）
3. 最坏情况耗时多少？（扫描器 v1 因 rglob 7003 文件卡死）
4. 需要排除哪些？（第三方库、生成文件、模板文件）
```

**防错信号**：
- [ ] 没评估规模就开始写代码
- [ ] 没考虑"炸弹目录"（大量小文件/嵌套目录）
- [ ] 没有排除策略

### Step 2：v1 — 最小可用版本

**目标**：用最少代码跑通核心逻辑。

**规则**：
- 不加过滤、不做优化、不处理边界
- 但**必须能跑 1 个真实 case**
- 跑完记录：哪些结果对、哪些明显是误报

**v1 验收门禁**：
- [ ] 能成功运行（不报错、不卡死）
- [ ] 处理了 1 个真实 case
- [ ] 输出了结果（不管质量如何）
- [ ] 记录了观察到的误报类型

**禁止**：
- ❌ v1 完成后就宣布"搞定了"
- ❌ v1 没跑真实数据就开始优化
- ❌ v1 写了 500 行但没有 1 个测试 case

### Step 3：v2 — 全量暴露问题

**目标**：在真实规模数据上跑，建立完整的误报/漏报清单。

**规则**：
- 跑全量数据，记录总数
- 人工抽查 10-20 条结果
- 按**误报模式分类**（不是逐条修，是按类修）

**v2 必须产出**：

```markdown
## v2 问题清单

### 误报模式
| 模式 | 数量 | 示例 | 根因 |
|------|------|------|------|
| 模式A | N条 | ... | ... |
| 模式B | M条 | ... | ... |

### 漏报检查
- 有没有明显该报但没报的？
- 边界 case 覆盖了吗？

### 性能问题
- 耗时：X秒（目标：<Y秒）
- 瓶颈：哪个步骤最慢？
```

**防错信号**：
- [ ] 只跑了小样本就开始修 bug
- [ ] 没有按模式分类，逐条修（低效）
- [ ] 没有漏报检查（只关注误报）
- [ ] 修完一个模式就宣布"大幅改进"

### Step 4：v3 — 逐类消灭

**目标**：逐类消灭误报，每修一类跑全量验证。

**迭代节奏**：

```
修模式A → 跑全量 → 报告：总X条（降Y条）→ 人工抽查
修模式B → 跑全量 → 报告：总X条（降Y条）→ 人工抽查
...
直到：误报率 < 20% 或 剩余都是真问题
```

**每轮报告格式**（必须）：

```
修复前：X 条（Y% 误报）
修复后：A 条（B% 误报）
本轮消灭：模式名（Z 条）
剩余误报率：B%（目标 < 20%）
```

**防错信号**：
- [ ] 修完一类就宣布"大幅改进"但没跑全量
- [ ] 只报降幅不报总数（"从 3175 降到 2097"——但 2097 还是不能用）
- [ ] 没有人工抽查就宣布验收
- [ ] 修了 5 轮还没到目标 → 应该重新审视架构，不是继续修

### Step 5：验收（三重门禁）

**门禁 1：功能验收**
- [ ] 全量跑完不报错
- [ ] 误报率 < 20%（人工抽查确认）
- [ ] 没有明显漏报

**门禁 2：性能验收**
- [ ] 耗时在预期范围内
- [ ] 没有"炸弹目录"导致的卡死

**门禁 3：可用性验收**
- [ ] 报告可操作（有修复建议，不只是"这里有死路径"）
- [ ] 用户能直接用报告去修问题
- [ ] 命令行参数直观（`--quick`/`--full`/`--report`）

### Step 6：文档化 + 经验回流

完成后必须：
1. 记录 v1→v2→v3 的演进过程（踩了什么坑、怎么解的）
2. 如果教训有普适性 → 写入经验库 + 防错机制
3. 如果工具会被反复使用 → 写使用文档（一行命令示例）

## §三 跨场景适用指南

本 SOP 的 v1→v2→v3 模式不只适用于"写脚本"，适用于所有**渐进式开发任务**：

| 场景 | v1 | v2 | v3 |
|------|-----|-----|-----|
| **写脚本/工具** | 能跑 1 个 case | 全量暴露误报 | 逐类消灭 |
| **文件迁移** | 搬文件 | 修引用 | 验证完整性 |
| **知识库整理** | 粗分类 | 去重+合并 | 交叉引用+索引 |
| **Skill 创建** | 核心流程能用 | 真实场景测试 | 边界覆盖+文档 |
| **SOP 编写** | 骨架+关键步骤 | 实战验证 | 补充案例+检查清单 |
| **经验库更新** | 记现象 | 挖根因+防错 | 双写验证+系统加固 |

**通用节奏**：

```
最快出能用的 → 在真实场景暴露问题 → 逐类修到验收标准
     30%              40%                    30%
```

## §四 反模式（必须背下来）

| 反模式 | 为什么错 | 正确做法 |
|--------|---------|---------|
| v1 完成就宣布"搞定了" | v1 是骨架，不是成品 | v1 必须跑真实 case + 记录误报 |
| 修了一个 bug 就说"大幅改进" | 还有同类 bug 没修 | 每修一类跑全量，报总数不报降幅 |
| 只跑小样本就优化 | 小样本没有暴露真实问题 | v2 必须全量跑 |
| 误报逐条修 | 效率极低，同类 bug 反复改 | 按模式分类，一类一类消灭 |
| 修了 5 轮还在修 | 架构可能有问题 | ≥3 轮未达标 → 重新审视架构 |
| 只关注误报不检查漏报 | 漏报 = 该报的没报 = 功能缺陷 | v2 阶段必须检查漏报 |
| 没有人工抽查就验收 | 数字可以造假，抽查不能 | 验收门禁：人工抽查 ≥10 条 |
| 工具第一次用就完美 | 不可能。第一次用 = 测试验收标准 | 预留 v2/v3 迭代空间 |

## §五 认知科学支撑

| 机制 | 在本 SOP 中的应用 | 来源 |
|------|-------------------|------|
| **确认偏误** | v1 完成后只找支持性证据（"能跑"），忽略反证（"跑错"） | 010《思考，快与慢》 |
| **完成欲** | 宣布"完成"带来的多巴胺奖励，驱动过早停止迭代 | 001《多巴胺国度》 |
| **锚定效应** | 第一次全量结果成为锚点，后续迭代都围绕它修而不是重新审视 | 010《思考，快与慢》 |
| **过度自信** | "我写的代码没问题" → 跳过验证 → 误报率 60% | 010《思考，快与慢》 |
| **习得性无助** | 修了 5 轮还没好 → 放弃 → 工具半成品永远不能用 | 014《心态》 |
| **生成效应** | 亲手跑一遍全量数据比看代码更能发现问题 | 009《认知天性》 |

## §六 与防错机制的关系

本 SOP 的 Step 4（逐类消灭）直接对应防错机制的：
- **规则 2**：工具开发 SOP（5 步验收流程）
- **规则 6**：验证后才能报结果

本 SOP 是规则 2 的**完整升级版**——从 5 步扩展为 6 步 + 三阶段模型 + 跨场景适用。
