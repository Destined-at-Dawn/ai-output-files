## 二、执行步骤

### Step 1: 需求确认
- **操作**：确认以下信息
  - 目标（创建/修改/查找/删除skill）
  - 范围（本工作区/跨工作区/全局）
  - 优先级

### Step 2: Skill查找
- **操作**：
  - 读取 `E:\ai产出文件\牛马\mutual\mutual\skill-catalog-9cat.md`
  - 按关键词搜索匹配的skill
  - 评估skill是否满足需求

### Step 3: Skill调用/创建
- **操作**：
  - 如找到匹配skill → 直接调用
  - 如需要新skill → 按模板创建
  - 如需要修改skill → 先备份再修改

### Step 4: 更新路由表
- **操作**：
  - 更新 `.claude/skill-rules.json`
  - 更新 `skill-catalog-9cat.md`（如有新skill）
  - 更新 `workspace-skill-mapping.md`（如有新映射）

### Step 5: 测试验证
- **操作**：
  - 测试skill是否正常工作
  - 验证路由是否正确
  - 记录到memory

## 五、GitHub 开源发布流程（⚠️ 2026-05-18 新增 —— 基于遗漏Release教训）

> **触发**：涉及 Git 推送/开源发布/GitHub 操作时，必须执行以下完整流程。

| 步骤 | 操作 | 铁律 |
|------|------|------|
| 1. 隐私扫描 | `Grep` 搜索所有待推送文件中的个人信息 | 扫描报告禁止引用原始隐私值 |
| 2. `.gitignore` | 在首次推送前配置，排除敏感文件 | 含隐私的文件不进版本控制 |
| 3. 仓库设置 | 默认 `private`，确认后再改 public | 默认 private |
| 4. 推送代码 | `git add` → `git commit` → `git push` | Release Asset 用英文命名 |
| 5. 创建 Release | 上传编译产物/附件 | GitHub API 不支持中文文件名 |
| 6. 全量重扫 | 推送后重新扫描，确认无残留 | 修了A处，B处可能还在 |
| 7. Git历史清理 | 已推送的敏感文件用 `git filter-repo` | `git rm --cached` 不清理历史 |
| 8. 验证 | 在浏览器打开仓库确认公开内容无泄露 | 必须人工验证 |

**关联教训**：`自我进化/做得差的避免/20260509_开源GitHub遗漏封面和Release_失败教训.md`

## 七、认知科学支撑（百大认知书籍）

### 系统管理的认知心理学

| 认知机制 | 来源 | 在系统管理中的应用 |
|---------|------|-------------------|
| **习惯回路** | 016-习惯的力量 §概念1 | 系统维护应形成"暗示→惯常行为→奖励"的自动回路。Skill路由表更新的触发信号（暗示）→执行更新（惯常行为）→系统响应变快（奖励）→习惯固化 |
| **确认偏误** | 010-思考快与慢 §概念5 | 审计Skill时，大脑会倾向于"确认Skill正常工作"而非"寻找Skill的问题"。§六审计铁律A7（全量搜索）的底层逻辑就是对抗确认偏误 |
| **沉没成本谬误** | 018-算法之美 §概念2 | "这个Skill我已经写了很久，虽然不好用但不想重写"=沉没成本。Step 3应强制问"如果从零开始，你还会这样设计吗？"→打断沉没成本 |
| **反馈延迟** | 046-反脆弱 §概念2 | Skill修改后效果不立刻显现→修改者不确定改对了没有→倾向于过度修改。Step 5测试验证应设置"24小时冷却"→第二天再评估修改效果 |

### Skill路由的认知架构

| 设计原理 | 来源 | 应用 |
|---------|------|------|
| **选择架构** | 061-助推 §概念1 | Skill路由表的设计=选择架构设计。默认选项（常用Skill自动匹配）应利用"默认效应"——用户不需要主动选择就能到达正确Skill |
| **认知负荷管理** | 025-认知负荷理论 §概念1 | 路由规则≤7条（工作记忆容量）。超过7条规则→AI决策质量下降→错误路由增加。Skill分类应符合"7±2"法则 |
| **冗余编码** | 008-协同智能 §概念2 | 关键Skill应有"冗余路由"——同一个Skill通过多个关键词都能触发。冗余不是浪费，是认知系统的容错设计 |
| **元认知监控** | 023-认知觉醒 §概念3 | 系统管理需要定期"元认知检查"——不仅检查Skill是否工作正常，还要检查"检查机制本身"是否正常。§六审计流程就是元认知监控的工程化 |

### 开源发布的认知风险

| 风险类型 | 来源 | 防御策略 |
|---------|------|---------|
| **注意力隧道** | 043-看不见的猩猩 §概念1 | 推送代码时注意力集中在"功能正确性"→完全忽略"隐私泄露"→非注意盲视。§五隐私扫描的必要性在于打破注意力隧道 |
| **WYSIATI** | 010-思考快与慢 §概念3 | "我扫描了A文件没发现隐私→所有文件都没隐私"=WYSIATI陷阱。§五第6步"全量重扫"的设计就是对抗WYSIATI |
| **乐观偏差** | 046-反脆弱 §概念1 | "应该没问题"的默认乐观→跳过验证步骤→安全事故。§五第8步"必须人工验证"是打破乐观偏差的最后防线 |

## 八、Skill 自动优化方法论（SkillOpt 支撑）

> **来源**：微软 SkillOpt 论文（arXiv:2605.23904），2026-05-27 融入本 SOP
> **核心发现**：手写 Skill 不是最优的——SkillOpt 在 52/52 测试组合中全胜，平均 +23.5 分

### 8.1 训练循环（可手工模仿）

```
Rollout → Reflection → Edit → Validation Gating → 接受/拒绝
```

| 步骤 | 操作 | 手工版做法 |
|------|------|-----------|
| **Rollout** | 让 Agent 带着当前 Skill 做一批任务 | 选 5-10 个典型任务，记录成功率 |
| **Reflection** | 用优化器分析失败原因 | 手动检查失败任务→找出共性错误模式 |
| **Edit** | add/delete/replace，每次 ≤4 条规则 | 针对性补充/删除/修正规则 |
| **Validation Gating** | 改完在验证集上跑，分数不涨就不接受 | 用修改前后的 Skill 各跑 3 个任务对比 |

### 8.2 关键设计原则

| 原则 | 说明 | 数据支撑 |
|------|------|---------|
| **每轮只改 ≤4 条规则** | 一次改太多，好坏混在一起，验证集无法准确判断 | 无限制重写比 L_t=4 低 2-3 分 |
| **rejected-edit buffer** | 被否决的修改存入缓冲区，后续不再重复犯错 | 相当于负结果日志的自动化 |
| **slow/meta update** | 每个 epoch 结束时做跨 epoch 纵向更新，受保护不被 step 编辑覆盖 | 去掉导致暴跌 22.5 分 |
| **紧凑即智能** | 有效 Skill 中位数仅 920 token，有的只需 1 条规则涨 39 分 | 规则精准 > 规则数量 |

### 8.3 对小黎 Skill 生态的直接影响

| 影响维度 | 说明 |
|---------|------|
| **CLAUDE.md 修剪** | 超 1000 行，按 SkillOpt 标准应精简约 50%，保留精确操作级规则 |
| **Skill 迭代** | 从「凭经验改」→「数据驱动改」：每次修改前跑基准测试 |
| **跨模型兼容** | 优化的 Skill 可在 Claude/GPT/国产模型间迁移（+59.7 分跨环境） |
| **成本结构** | 训练一次性，部署零额外成本 → Skill 产品有极高边际优势 |

```bash

cd SkillOpt

```

支持 6 个 benchmark：SearchQA、SpreadsheetBench、OfficeQA、DocVQA、LiveMathematicianBench、ALFWorld

## 2026-05-28 补充：Claude Code Hook 自修复闭环

### 核心判断

不要只靠继续扩写 CLAUDE.md 解决重复错误。更稳的路线是：规则文件负责“方向”，Hook负责“门禁”，记忆负责“跨会话复用”。

### 三类Hook

1. **PostToolUse：写后即时检查**
   - 文件写完后自动格式化、类型检查、lint。
   - 优点：错误在扩散前被捕捉。

2. **Stop：完成前质量门禁**
   - AI声称完成前自动跑测试或验证脚本。
   - 必须检查 stop_hook_active，避免无限循环。

3. **PreToolUse：危险操作前拦截**
   - 拦截 `.env` 写入、危险删除、超大日志直读。
   - 对日志应先提取 ERROR/WARN，而不是全文喂给AI。

### 自动重试上限

任何自动修复循环必须设置最大尝试次数。推荐3次：

- 第1次：按错误信息直接修。
- 第2次：换一个思路。
- 第3次：仍失败则输出已尝试方案、失败原因、下一步人工判断点。

### 对当前系统的提醒

当前规则系统已经很重，下一步不是继续堆CLAUDE.md，而是把高频验证变成自动门禁，把低频知识拆成按需Skill。
