## §〇 核心认知

### 需求翻译的本质

「AI是手段，回应老板的需求才是目的。」——77老师

**你不是在卖AI技术，你是在把模糊需求翻译成可执行方案。** 最值钱的能力不是技术多牛，是让对方心里石头落地——提供确定性、情绪价值、可验证的结果。

### 三层翻译模型

| 层 | 输入 | 输出 | 示例 |
|----|------|------|------|
| L1 语言翻译 | "我想学AI但不知道怎么开始" | 结构化学习路径 | SOP化教程 |
| L2 流程翻译 | "我想做小红书但总是没时间" | 标准化工作流 | SOP-01内容创作 |
| L3 业务翻译 | "我想靠AI赚钱但不知道卖什么" | 商业闭环方案 | 机会筛选+产品+获客+交付 |

**判断**：L1/L2 直接用现有SOP。L3 走本SOP全流程。

## §一 Phase 0：需求闸门（GATE）

> 任何模糊需求进来，先过这个闸门。

### GATE 0：需求分类

| 信号 | 分类 | 走什么 |
|------|------|--------|
| "帮我写XX" | 内容创作 | → F8路由 → 对应SOP |
| "帮我分析XX" | 分析诊断 | → 本SOP §二 |
| "我想做XX但不知道怎么开始" | 机会探索 | → 本SOP §二~§四 |
| "这个值不值得做" | 机会评估 | → 本SOP §三 |
| "帮我想想怎么赚钱/变现" | 商业设计 | → 本SOP §二~§五 |
| "老板/老师说了一句话，我该怎么接" | 项目提取 | → 本SOP §四 |

### GATE 1：信息完整性检查

```
必须确认的信息：
1. 谁的需求？（自己/客户/老板/群成员）
2. 当前状态？（零基础/有方向/有产品/有客户）
3. 目标是什么？（学会/赚钱/解决问题/晋升）
4. 有什么资源？（时间/数据/预算/人脉）
5. 约束是什么？（时间线/风险偏好/领域限制）

信息不足4项 → 用 mcp__ask-user__ask_user 补全
信息充足 → 进入 §二
```

## §二 Phase 1：需求翻译诊断

### Step 1：三层翻译定位

判断需求在 L1/L2/L3 哪一层：
- L1（语言翻译）→ 用现有SOP直接解决
- L2（流程翻译）→ 用SOP+定制化解决
- L3（业务翻译）→ 进入 Step 2

### Step 2：六步拆法（77老师方法论）

从模糊需求中拆出可执行项目：

```
Q1：业务目标 → "你想提升哪块效率/收入/影响力？"
Q2：重复流程 → "最占时间、最重复、最让人烦的是什么？"
Q3：数据输入 → "信息在哪？微信聊天/表格/系统/脑子里？"
Q4：输出定义 → "要产出什么？报告/名单/方案/产品？"
Q5：人工校验 → "哪些必须人看了才能用？出错谁负责？"
Q6：小试点 → "先做1-2周验证，什么算'有效'？"
```

**输出**：一份结构化的需求翻译报告，包含：
- 业务目标（一句话）
- 关键痛点（按频率/痛度排序）
- 数据现状（有什么/缺什么）
- 输出规格（具体到格式）
- 风险点（出错成本+校验需求）
- MVP方案（1-2周试点）

### Step 3：需求翻译报告模板

```markdown
# 需求翻译报告：{项目名}

## 一句话翻译
{把模糊需求翻译成一句话的可执行目标}

## 六步拆法结果
| 步骤 | 回答 |
|------|------|
| 业务目标 | ... |
| 重复流程 | ... |
| 数据输入 | ... |
| 输出定义 | ... |
| 人工校验 | ... |
| 小试点 | ... |

## 真AI化诊断
- 去掉AI后核心流程变没变？{变/不变}
- AI在哪个节点进入？{具体节点}
- 判断：{真AI化/半贴膏药/贴膏药}

## MVP方案
- 第一步：{具体动作}
- 预计耗时：{X天}
- 验证标准：{什么算"有效"}
- 退出判据：{什么情况下放弃}
```

## §三 Phase 2：机会评估（七条标准）

> 任何潜在的产品/服务/项目，过这7条再动手。

### 七条筛选标准

| # | 标准 | 检查问题 | ✅通过 | ❌不通过 |
|---|------|---------|--------|---------|
| 1 | **高频** | 一年发生多次？ | 有复购/续费空间 | 一次性需求→人工交付即可 |
| 2 | **结果可验证** | 能量化效果？ | 省时X%/增收Y元 | 无法验证→无法证明价值 |
| 3 | **付费意愿** | 以前有人为此付费？ | 市场已验证 | 没人付过→先免费试水 |
| 4 | **数据稳定** | AI所需数据能持续获取？ | 数据源稳定 | 零散/权限不稳→做不大 |
| 5 | **出错可控** | 搞砸了后果多严重？ | 后果可承受 | 高风险领域→先别碰 |
| 6 | **可小试点** | 能先做MVP？ | 1-2周可验证 | 需要大投入→降低规模 |
| 7 | **续费空间** | 能变成月费服务？ | 有月费/年费模式 | 纯一次性→考虑转模板 |

### 评估报告格式

```markdown
## 机会评估：{项目名}

| # | 标准 | 结果 | 说明 |
|---|------|------|------|
| 1 | 高频 | ✅/❌ | ... |
| 2 | 可验证 | ✅/❌ | ... |
| ... | ... | ... | ... |

**通过数**：X/7
**结论**：{立即做/先试点/暂不做/需调整}
**下一步**：{具体动作}
```

### 阈值规则

- **≥5条通过** → 立即做MVP
- **3-4条通过** → 先试点验证未通过的条目
- **≤2条通过** → 暂不做，记录原因，定期复查

## §四 Phase 3：方案设计（卖解决方案而非卖时间）

### 产品三层模型

| 层 | 形态 | 定价逻辑 | 交付物 |
|----|------|---------|--------|
| **底层** | 模板/工具/SOP包 | 固定价，边际成本趋零 | 可复用文档/提示词/配置文件 |
| **中层** | 诊断报告 | 按复杂度定价 | 结构化问题分析+方案建议 |
| **上层** | 陪跑服务 | 月费制（结果可验证） | 帮跑通第一个闭环 |

### 设计原则

1. **先卖再做**：有了付费信号再投入开发，不先做再找客户
2. **模板优先**：能做成模板的不做定制，能SOP化的不靠人
3. **结果定价**：按客户获得的结果定价，不按你投入的时间定价
4. **可复用**：每个交付都要问"能不能复用给下一个客户？"

### 真AI化检查

在方案设计完成后，做一次"贴膏药诊断"：

```
把方案中的AI步骤全部去掉：
- 核心流程变没变？
  - 变了 → ✅ 真AI化（AI是流程核心节点）
  - 没变 → ⚠️ 贴膏药（AI只是装饰品）

如果判断为"贴膏药"：
- 回到§二 Step 2，重新设计AI在流程中的角色
- AI应该在"数据采集→分析→输出"的关键节点，不是在"人做完后帮润色"
```

## §五 Phase 4：执行与验证

### MVP执行流程

```
Step 1：锁定最小交付物（1个模板/1份报告/1次诊断）
Step 2：找到第一个客户（朋友圈/社群/咨询转化）
Step 3：交付 + 收集反馈（满意度/改进建议/付费意愿）
Step 4：根据反馈决定：
  - 反馈好 → 扩大规模（模板化/SOP化/产品化）
  - 反馈差 → 分析原因（产品问题/定位问题/渠道问题）
  - 无反馈 → 换渠道/换人群/换产品
```

### 验证标准

| 维度 | 指标 | 达标线 |
|------|------|--------|
| 付费验证 | 有人愿意付费 | ≥1人 |
| 满意度 | 交付后满意度 | ≥4.0/5.0 |
| 复购意愿 | "愿意继续/推荐给别人" | ≥50% |
| 可复制 | 能模板化/SOP化交付 | 是 |

### 退出判据

以下任一条件满足 → 停止投入，记录负结果：
- 3次MVP尝试均无付费信号
- 满意度 < 3.0/5.0 且无法通过迭代改善
- 数据源不稳定导致交付质量波动 > 30%
- 出错成本超出预期（客户投诉/退款/信任损失）
